# 개발자를 위한 레디스 9장 학습 정리

레디스의 자체 고가용성 기능인 센티널(Sentinel)을 다룬 챕터다. 책 249–275쪽이 범위다.

레디스는 인메모리 데이터베이스라 인스턴스가 재시작되면 데이터가 통째로 사라질 수 있다. 8장의 복제를 붙여도 마스터에 장애가 나면 사람이 직접 복제본을 마스터로 승격시키고 애플리케이션의 엔드포인트를 바꿔 배포해야 한다. 센티널은 이 과정을 자동으로 처리해 다운타임을 최소화한다. 마스터를 감시하다가 죽으면 복제본 하나를 새 마스터로 자동 승격(페일오버)시키고, 클라이언트에게 바뀐 마스터 주소를 알려주는 별도의 프로그램이다.

## 1. 고가용성 기능의 필요성

복제 구조만 있고 별도 고가용성 장치가 없으면, 마스터 장애 시 복구가 전부 수동이다.

1. 복제본 노드에 직접 접속해 `REPLICAOF NO ONE`으로 읽기 전용 상태를 해제한다.
2. 애플리케이션 코드에서 레디스 엔드포인트를 복제본 IP로 바꾼다.
3. 배포한다.

운영 중에 이 과정을 사람이 거치는 동안 장애 처리가 지연되고, 그대로 서비스 장애로 이어진다. 특히 레디스를 look aside 캐시로 쓰는 경우가 위험하다. look aside 구성에서는 캐시에 접근하지 못하면 애플리케이션이 MySQL 같은 원본 소스에서 직접 데이터를 읽는다. 캐시를 보던 세션이 한꺼번에 원본 DB로 몰리면 커넥션이 급증하고 서버 부하가 치솟아 서비스 전체에 영향을 준다.

## 2. 센티널이란

센티널은 데이터를 저장하는 레디스 인스턴스와는 다른 역할을 하는 별도 프로그램이다. 데이터를 직접 들고 있지 않고, 다음 세 가지 기능을 제공한다.

- **모니터링**: 마스터와 복제본 인스턴스의 상태를 실시간으로 확인한다.
- **자동 페일오버**: 마스터의 비정상 상태를 감지하면 정상 복제본 중 하나를 마스터로 승격시킨다. 기존 복제본들은 새 마스터에 다시 연결된다.
- **구성 정보 안내**: 클라이언트에게 현재 마스터 정보를 알려준다. 페일오버가 일어나면 바뀐 마스터 정보를 다시 전달하므로, 클라이언트는 엔드포인트 정보를 직접 바꿀 필요가 없다.

세 번째 기능 덕분에 수동 복구의 2단계(엔드포인트 변경)가 사라진다. 클라이언트는 항상 센티널에 먼저 물어보고 마스터에 연결하기 때문이다.

## 3. 분산 시스템과 쿼럼

고가용성을 도입하는 이유는 레디스가 SPOF(Single Point Of Failure)가 되는 걸 막기 위해서다. 그런데 정작 고가용성을 책임지는 센티널이 SPOF가 돼 버리면 도입할 이유가 없다. 그래서 센티널은 그 자체로 SPOF가 되지 않도록 **최소 3대 이상**일 때 정상 동작하게 설계됐고, 하나가 죽어도 나머지가 역할을 이어간다.

오탐을 줄이기 위해 **쿼럼(quorum)** 개념을 쓴다. 쿼럼은 "마스터가 비정상이다"라는 데 동의해야 하는 센티널의 수다. 동의한 센티널이 쿼럼 이상이어야 페일오버 절차를 시작한다. 보통 센티널 3대에 쿼럼 2로 설정한다. 최소 2대가 마스터 비정상에 동의하면 페일오버가 트리거된다.

쿼럼을 이용한 과반수 선출을 쓰기 때문에 센티널은 **3대 이상의 홀수**로 구성하는 게 좋다. 보통 3대, 더 견고하게 가려면 5대로 구성한다.

## 4. 센티널 인스턴스 배치 방법

센티널은 마스터의 장애를 독립적으로 감지해야 하므로, 서로 영향받지 않는 물리 서버, 보통 서로 다른 가용 영역에 배치한다.

**복제본 2개 구성** (그림 9-3~9-5). 서버 A, B, C 세 대에 각각 레디스와 센티널 프로세스를 함께 띄운다. A가 마스터, B와 C가 복제본이며 quorum은 2다. 가장 일반적인 배치다.

- 서버 A에 장애가 나 마스터와 센티널에 접근할 수 없게 되면, B와 C의 센티널이 "마스터에 접근 불가"에 동의한 뒤 페일오버를 진행한다.
- 새 마스터로 선출된(예: B) 복제본이 마스터로 승격되고, C는 B를 바라보도록 복제 연결을 바꾼다. 기존 마스터를 보던 클라이언트는 모두 B로 연결되고, 신규 커넥션도 B의 주소를 받는다.
- 나중에 A가 복구되면 센티널들이 A를 **새 마스터 B의 복제본**으로 자동 편입시킨다. 운영자 개입이 필요 없다.

**복제본 1개 구성** (그림 9-6~9-7). 서버가 부족하면 2대(레디스+센티널)만 데이터를 들고, 나머지 1대(서버 C)에는 **센티널 프로세스만** 띄운다. 이 서버는 데이터를 저장하지도, 클라이언트 요청을 받지도 않으므로 최저 사양으로 구성해도 된다. 비용을 아끼면서도 마스터 장애를 감지해 자동 페일오버할 수 있는 구조다.

## 5. 센티널 인스턴스 실행하기

그림 9-8의 예제 구성을 기준으로 보자. `192.168.0.11`, `192.168.0.22` 두 서버에는 레디스와 센티널을 모두 띄우고, `192.168.0.33`에는 센티널만 띄운다. quorum은 2, 레디스는 6379 포트, 센티널은 26379 포트를 쓴다.

먼저 마스터와 복제본 간 복제 연결을 만들어 둔다. 복제본 노드에서:

```
REPLICAOF 192.168.0.11 6379
```

센티널을 띄우려면 `sentinel.conf`라는 별도 구성 파일이 필요하다.

```
port 26379
sentinel monitor master-test 192.168.0.11 6379 2
```

- `port`: 센티널 프로세스가 실행될 포트(여기서는 26379).
- `sentinel monitor <이름> <ip> <port> <쿼럼>`: 모니터링할 마스터에 이름을 붙이고 주소와 쿼럼을 지정한다. 마스터 이름에는 특수문자나 공백을 못 쓰고 알파벳, 숫자, `.`, `-`, `_`만 허용된다.

복제본 정보는 구성 파일에 적지 않아도 된다. 센티널이 시작하면 마스터에 붙은 복제본을 **자동으로 찾아낸다**.

센티널은 두 가지 방법으로 실행하며, 둘은 동일하게 동작한다.

```
# redis-sentinel 이용
redis-sentinel /path/to/sentinel.conf

# redis-server 이용
redis-server /path/to/sentinel.conf --sentinel
```

지정한 위치에 `sentinel.conf`가 없거나 그 경로에 데이터를 쓸 수 없으면 인스턴스는 시작되지 않는다. 실행 후에는 레디스처럼 `redis-cli`로 센티널에 직접 접속할 수 있다.

```
$ redis-cli -p 26379
```

## 6. 센티널 상태 조회 커맨드

센티널에 접속하면 모니터링 중인 마스터와 복제본, 다른 센티널의 정보를 볼 수 있다(레디스가 가진 실제 데이터는 못 본다).

`SENTINEL master <name>`은 마스터의 IP와 포트, 연결된 복제본 수 등을 반환한다. 구성이 정상인지 확인할 때 다음 플래그가 유용하다.

- `num-other-sentinels`: 같은 마스터를 모니터링하는 **다른** 센티널 수. 3대 구성이면 2가 정상.
- `flags`: 마스터 상태. 정상이면 `master`, 비정상으로 판단되면 `s_down`, `o_down` 등으로 바뀐다.
- `num-slaves`: 마스터에 연결된 복제본 수.

이 밖에 상세 조회 커맨드가 있다.

```
SENTINEL replicas  <name>   # 마스터에 연결된 복제본 상세
SENTINEL sentinels <name>   # 마스터를 함께 보는 다른 센티널 상세
SENTINEL ckquorum  <name>   # 정상 센티널 수가 쿼럼 이상인지 확인
```

`SENTINEL ckquorum`은 페일오버가 가능한 상태인지 알려준다. 센티널 3대, 쿼럼 2 기준으로:

```
# 3대 모두 정상
OK 3 usable Sentinels. Quorum and failover authorization can be reached

# 1대 장애, 2대 정상 (쿼럼 2 충족)
OK 2 usable Sentinels. Quorum and failover authorization can be reached

# 2대 장애, 1대만 정상 (쿼럼 미달)
(error) NOQUORUM 1 usable Sentinels. Not enough available Sentinels to reach
the specified quorum for this master. Not enough available Sentinels to
reach the majority and authorize a failover
```

정상 센티널 수가 쿼럼보다 작아지면, 마스터에 장애가 나도 투표를 못 해 자동 페일오버를 실행할 수 없다.

## 7. 페일오버 테스트

운영에 투입하기 전, 마스터에 실제 이슈가 생겼을 때 페일오버가 잘 도는지 확인하는 게 좋다. 두 가지 방법이 있다.

**수동 페일오버.** 센티널에 접속해 다음 커맨드를 쓰면 다른 센티널의 동의 없이 즉시 페일오버가 발생한다. 마스터 상태가 정상이어도 마스터↔복제본 간 롤 체인지가 일어난다. 네트워크 단절로 페일오버가 실패하진 않는지, 애플리케이션 커넥션이 새 마스터로 잘 옮겨가는지 확인할 때 쓴다.

```
SENTINEL FAILOVER <master name>
```

**자동 페일오버.** 마스터 노드를 직접 셧다운시켜 센티널이 비정상 상태를 감지하는지 본다.

```
$ redis-cli -h <master-host> -p <master-port> shutdown
```

센티널은 주기적으로 마스터에 PING을 보내 상태를 파악한다. `sentinel.conf`의 `down-after-milliseconds`로 지정한 시간 동안 응답이 없으면 비정상으로 판단해 페일오버를 트리거한다. 이 옵션의 기본값은 30,000밀리초(30초)이므로, 마스터를 다운시킨 뒤 약 30초가 지나 페일오버가 발생하는지 확인하면 된다.

## 8. 센티널 운영하기

**패스워드 인증.** 마스터와 복제본에 `requirepass`/`masterauth`로 패스워드를 걸었다면 센티널 설정에도 패스워드를 지정해야 한다. 센티널 구성에서는 모든 노드가 마스터가 될 가능성이 있으므로, 하나의 복제 그룹 안에서 `requirepass`와 `masterauth` 값은 **모든 노드에서 동일**하게 맞춰야 한다.

```
sentinel auth-pass <master-name> <password>
```

**복제본 우선순위.** 모든 레디스 인스턴스는 `replica-priority` 파라미터를 가진다. 페일오버 시 센티널은 이 값이 **가장 작은** 복제본을 마스터로 선출한다. 기본값은 100이고, **0이면 그 복제본은 절대 마스터로 선출되지 않는다**.

**운영 중 구성 변경.** 실행 도중 모니터링 대상을 추가, 제거, 변경할 수 있다. 단 마스터를 보는 센티널이 여러 대라면 **각 센티널에 모두 적용**해야 하며, 한 곳에서 바꿔도 다른 센티널로 전파되지 않는다.

```
SENTINEL MONITOR <name> <ip> <port> <quorum>   # 새 마스터 모니터링 시작
SENTINEL REMOVE  <name>                         # 모니터링 중단, 내부 상태에서 제거
SENTINEL SET     <name> <option> <value> ...    # 특정 마스터의 파라미터 변경
```

```
sentinel> SENTINEL SET mymaster down-after-milliseconds 1000
OK
sentinel> SENTINEL SET mymaster quorum 1
OK
```

레디스 6.2 이상부터는 특정 마스터에 종속되지 않는 센티널 고유 설정도 런타임에 바꿀 수 있다(레디스의 `CONFIG GET`/`SET`과 유사).

```
SENTINEL CONFIG GET <configuration name>
SENTINEL CONFIG SET <configuration name> <value>
```

**센티널 초기화.** 센티널은 비정상이라 판단한 노드(페일오버 후 접근 불가가 된 옛 마스터 등)도 임의로 모니터링을 멈추지 않고 계속 PING한다. 더 이상 쓰지 않는 노드의 모니터링을 끊으려면 `SENTINEL RESET`으로 상태 정보를 초기화한다. 마스터 이름을 직접 지정하거나, `*`로 전체를 초기화할 수 있다.

```
SENTINEL RESET <master name>
```

**노드 추가/제거.** 새 센티널을 추가하는 건 간단하다. 마스터를 모니터링하도록 설정한 센티널을 실행시키면, 자동 검색 메커니즘에 의해 기존 센티널들의 목록(known-list)에 자동 등록된다. 단 한 번에 여러 대를 추가할 때는 10초 이상 간격을 두고 1대씩 천천히 넣어야 오류 가능성이 줄어든다. 제거할 때는 센티널 노드끼리 오래 응답이 없어도 목록에서 지우지 않으므로, 프로세스를 먼저 종료한 뒤 `SENTINEL RESET *`로 모니터링 정보를 리셋한다. 이때도 최소 30초 간격으로 차례대로 한다. 추가나 제거가 끝나면 `SENTINEL MASTER`의 `num-other-sentinels` 값으로 정상 반영 여부를 확인한다.

## 9. 자동 페일오버 과정

여러 센티널이 마스터를 함께 감시하며 오탐을 줄이는 페일오버는 다음 단계로 진행된다. 각 단계마다 센티널 로그에 특정 문구가 남는다.

**① 마스터 장애 감지.** 센티널은 `down-after-milliseconds` 이상 동안 마스터로 보낸 PING에 유효한 응답을 받지 못하면 다운으로 판단한다. 유효한 응답은 `+PONG`, `-LOADING`, `-MASTERDOWN`이고, 그 외 응답이나 무응답은 모두 유효하지 않은 것으로 본다.

**② sdown → odown.** 한 센티널이 마스터 응답을 못 받으면 우선 마스터를 `sdown`(subjectively down, 주관적 다운)으로 플래깅한다. 이후 다른 센티널들에게 `SENTINEL is-master-down-by-addr ...` 커맨드로 장애 사실을 전파하고, 자신을 포함해 **쿼럼 이상**의 센티널이 장애에 동의하면 `odown`(objectively down, 객관적 다운)으로 바꾼다. 단 `odown`과 장애 전파는 **마스터에 대해서만** 일어난다. 복제본은 장애가 나도 `sdown`까지만 표시되고 odown으로 승격되지 않는다(다만 sdown 상태의 복제본은 페일오버 때 새 마스터 후보에서 제외된다).

**③ 에포크 증가.** 처음으로 마스터를 odown으로 인지한 센티널이 페일오버를 시작하며, 그 전에 `epoch` 값을 1 증가시킨다. 에포크는 페일오버의 버전 번호로, 같은 에포크 값을 공유함으로써 모든 센티널이 같은 페일오버 작업을 진행 중임을 보장한다.

**④ 센티널 리더 선출.** 에포크를 올린 센티널이 다른 센티널들에게 리더 선출 투표를 요청한다. 받은 에포크가 자신의 것보다 크면 자신의 에포크를 올리고 투표한다. 하나의 에포크에서 센티널은 하나에게만 투표할 수 있고 변경할 수 없다.

> **NOTE: 쿼럼과 과반수는 다르다**
> 마스터를 `sdown → odown`으로 바꾸는 데는 **쿼럼** 이상의 동의면 된다. 하지만 실제 페일오버를 위해 센티널 리더를 뽑을 때는 쿼럼이 아니라 **전체 센티널 중 과반수**의 동의가 필요하다. 예를 들어 센티널 5대, 쿼럼 2일 때, 2대만 동의하면 odown으로 바뀌어 페일오버가 트리거될 수는 있어도, 리더 선출에 필요한 과반수(3대)를 못 채워 실제 페일오버는 일어나지 않는다.

**⑤ 복제본 선정 후 승격.** 과반수가 동의하면 리더 센티널이 새 마스터가 될 복제본을 고른다. 마스터로부터 오래 끊겼던 복제본은 자격이 없고, 자격 있는 복제본은 다음 순서로 선출된다.

1. `replica-priority`가 더 낮은 복제본
2. 마스터로부터 더 많은 데이터를 받은 복제본(`master_repl_offset`이 큰 쪽)
3. 그래도 같으면 `runID`가 사전순으로 작은 복제본 (특별한 의미 없이 하나를 임의 선택하는 용도)

선정된 복제본에 `slaveof no one`을 실행해 기존 마스터로부터의 복제를 끊는다.

**⑥ 복제 연결 변경.** 나머지 복제본들이 새로 승격된 마스터를 바라보도록 각각 `replicaof <new-ip> <new-port>`를 수행한다. 복제 그룹의 모든 센티널도 구성 정보를 갱신한다.

**⑦ 장애 조치 완료.** 모든 과정이 끝나면 센티널은 새 마스터를 모니터링하기 시작한다.

단계별로 남는 센티널 로그를 순서대로 보면 다음과 같다.

```
+sdown
+odown
+new-epoch
+try-failover
+vote-for-leader
+elected-leader
+failover-state-select-slave
+selected-slave
+failover-state-send-slaveof-noone
+failover-state-wait-promotion
+promoted-slave
+failover-state-reconf-slaves
+slave-reconf-sent
+slave-reconf-inprog
+slave-reconf-done
+config-update-from sentinel
+failover-end
+switch-master
```

## 10. 스플릿 브레인 현상

스플릿 브레인(split brain)은 네트워크 파티션으로 분산 저장소가 둘로 끊기고, 끊긴 양쪽이 각각 자신을 정상 서비스라고 인식하는 현상이다. 센티널 구성에서도 네트워크 단절이 생기면 발생할 수 있다(그림 9-14~9-16).

마스터와 센티널 A가 한쪽, 복제본과 센티널 B, C가 다른 쪽으로 끊겼다고 하자.

- 단절이 길어지면 B와 C는 마스터 접근이 안 된다고 판단하고, 자기 쪽 복제본을 새 마스터로 승격시킨다. **과반수 센티널(B와 C)이 같은 파티션에 있으므로** 승격이 가능하다.
- 그런데 실제로는 마스터에 장애가 난 게 아니라 단지 네트워크만 끊긴 상황이라면, 이제 마스터가 **2개**가 되는 스플릿 브레인이 된다.
- 클라이언트는 센티널에서 마스터 주소를 받아 마스터에 직접 연결한다. 단절에 영향받지 않은 기존 클라이언트는 계속 옛 마스터에 쓰고, 새로 붙는 클라이언트는 센티널이 알려준 새 마스터에 쓴다. 같은 데이터가 두 마스터로 갈라진다.
- 네트워크가 복구되면 옛 마스터는 센티널 B와 C에 의해 **새 마스터의 복제본으로 강등**된다. 복제본이 되는 과정에서 그 노드의 데이터는 모두 삭제되므로, 단절 동안 옛 마스터에 쓴 데이터는 전부 유실된다. 클라이언트 입장에서는 분명히 입력했는데 사라진 데이터가 생기는 셈이다.

## 헷갈리기 쉬운 지점

- **복제 ≠ 고가용성.** 복제는 데이터 사본을 만들 뿐, 마스터가 죽었을 때 자동으로 승격하고 전환해 주는 건 센티널의 몫이다. 복제만으로는 장애 시 수동 복구를 피할 수 없다.
- **쿼럼 vs 과반수.** odown 전환에 필요한 건 쿼럼, 리더 선출에 필요한 건 전체 센티널의 과반수다. 둘이 다르기 때문에 "odown은 됐는데 페일오버는 안 되는" 상황(5대, 쿼럼 2에서 2대만 동의)이 생긴다.
- **sdown vs odown.** sdown은 한 센티널의 주관적 판단, odown은 쿼럼 이상이 동의한 객관적 판단이다. odown과 장애 전파는 **마스터에만** 적용되고, 복제본은 sdown까지만 간다.
- **replica-priority는 작을수록 우선.** 값이 클수록이 아니라 작을수록 마스터로 선출되며, 0은 영구 제외다.
- **센티널 구성 변경은 전파되지 않는다.** `SENTINEL SET`/`MONITOR` 등은 실행한 센티널에만 적용된다. 모든 센티널에 일일이 적용해야 한다.
- **스플릿 브레인의 데이터 유실 지점.** 데이터는 페일오버 순간이 아니라, **네트워크가 복구돼 옛 마스터가 복제본으로 강등될 때** 삭제된다. 단절 동안 옛 마스터에 쌓인 쓰기가 그때 통째로 사라진다.

> 본문의 IP(`192.168.0.x`)와 포트(6379, 26379), 쿼럼 값은 책의 예제 구성값이다. `down-after-milliseconds` 같은 기본값은 레디스 버전에 따라 달라질 수 있으니 참고치로 본다.
