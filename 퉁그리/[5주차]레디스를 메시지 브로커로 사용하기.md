# 개발자를 위한 레디스 6장 학습 정리

레디스를 메시지 브로커로 쓰는 법을 다룬 챕터다. 책 155–200쪽이 범위고 스터디는 5주차에 진행했다. 책 본문에 스터디원 PR 세 건(#27, #28, #30)을 같이 본 뒤 다시 정리했다.

레디스가 메시지 브로커로 쓸 수 있는 도구는 세 가지다. Pub/Sub, List, Stream. 셋 다 "어디로 메시지를 보내고 누군가 받는다"는 점은 같지만 영속성과 보장 수준이 꽤 다르다. 가장 가벼운 게 Pub/Sub이고 가장 묵직한 게 Stream이다.

## 1. 왜 메시지 브로커가 필요한가

마이크로서비스에서 서비스 간 호출을 전부 동기로 묶으면, B의 장애나 지연이 A로 그대로 전염된다. 비동기 통신을 끼우면 이걸 풀 수 있다.

B가 다운돼 있어도 A는 메시지를 브로커에 던지고 다음 일을 계속한다. B가 복구되면 쌓인 메시지를 차례로 처리한다. A의 응답 시간이 B의 응답 시간에 영향받지 않는다.

요약하면 "당장 처리하지 못해도 보낸 메시지를 어딘가 쌓아두고 나중에 처리할 수 있게 한다"가 메시지 브로커의 핵심이다.

## 2. 메시징 큐와 이벤트 스트림

용어는 다른데 본질은 비슷하다. 양쪽 다 넣는 쪽과 빼는 쪽이 있다. 메시징 큐 쪽에서는 생산자(Producer)와 소비자(Consumer)라고 부르고, 이벤트 스트림 쪽에서는 발행자(Publisher)와 구독자(Subscriber)라고 부른다.

차이는 두 가지다.

**push냐 pull이냐.** 메시징 큐에서는 생산자가 소비자 큐로 직접 push한다. 두 소비자에게 같은 메시지를 보내려면 큐도 두 개, push도 두 번이다. 생산자가 소비자 수만큼 책임을 진다. 반대로 이벤트 스트림은 생산자가 하나의 저장소에 넣고 구독자들이 거기서 pull해 간다. 메시지를 복제할 필요가 없다.

**영속성.** 큐는 소비자가 읽으면 그 자리에서 삭제된다. 새 소비자가 도중에 붙어도 이전 메시지는 볼 수 없다. 스트림은 구독자가 읽어도 데이터가 남고, 보존 정책에 따라 일정 기간 유지된다. 도중에 붙은 구독자도 이전 히스토리를 조회할 수 있다.

그래서 큐는 1:1 작업 지시에 어울리고 스트림은 n:n 데이터 분배에 어울린다. 레디스 도구 셋을 모델에 매칭하면, Pub/Sub은 이벤트 스트림 모델이지만 영속성이 없고, List는 메시징 큐, Stream은 Kafka 영향을 받은 이벤트 스트림이다.

## 3. Pub/Sub

레디스가 제공하는 가장 가벼운 메시지 전달 방식이다. 발행자가 채널에 메시지를 보내면 그 채널을 듣고 있는 모든 구독자가 즉시 받고, 메시지는 전달 직후 사라진다.

```
> PUBLISH hello world
(integer) 1            # 메시지를 받은 구독자 수
```

```
> SUBSCRIBE event1 event2
Reading messages... (press Ctrl-C to quit)
1) "subscribe"
2) "event1"
3) (integer) 1
```

### 한계

발행자는 어떤 구독자가 메시지를 받았는지, 누가 못 받았는지 알 수 없다. 구독자도 메시지가 언제 누구에 의해 만들어졌는지 알 수 없다. 게다가 구독자가 끊겨 있던 동안 발행된 메시지는 영영 못 받는다.

그래서 정합성이 중요한 데이터에는 부적합하다. 가벼운 알림이나 채팅 broadcast처럼 fire-and-forget 패턴(작업 결과를 안 기다리는 비동기 방식)에 잘 맞는다.

### PSUBSCRIBE: 패턴 구독

prefix가 정해진 채널을 한 번에 구독할 수 있다.

```
> PSUBSCRIBE mail-*
```

`mail-track`, `mail-album` 같은 채널이 한꺼번에 잡힌다. 받는 메시지 타입은 `pmessage`로, 일반 SUBSCRIBE의 `message`와 구분된다. 같은 채널을 `SUBSCRIBE mail-1`과 `PSUBSCRIBE mail-*`로 동시에 구독하고 있으면 `mail-1`에 메시지가 1개 발행될 때 구독자는 메시지를 2개 받게 된다.

### 구독자가 쓸 수 있는 커맨드

구독 모드에 들어간 클라이언트는 다음만 실행 가능하다.

`SUBSCRIBE`, `SSUBSCRIBE`, `UNSUBSCRIBE`, `PSUBSCRIBE`, `PUNSUBSCRIBE`, `PING`, `RESET`, `QUIT`

### 클러스터에서의 비효율과 Sharded Pub/Sub

기본 Pub/Sub은 클러스터의 어느 노드에 발행해도 메시지가 모든 노드에 자동 전파된다. 구독자가 어디 붙어 있어도 받을 수 있게 한 설계다. 단순한 대신 클러스터의 본래 목적인 데이터 분산과 어긋난다. 네트워크 부하와 CPU 사용량이 늘어난다.

7.0에서 도입된 Sharded Pub/Sub은 이걸 해결한다. 채널을 키처럼 슬롯에 매핑하고, 같은 슬롯을 담당하는 노드들 사이에서만 전파한다.

```
10.0.0.1:6379> SPUBLISH apple a
-> Redirected to slot [7092] located at 10.0.0.2:6379
(integer) 1
```

`SPUBLISH`/`SSUBSCRIBE`는 채널 키의 슬롯을 담당하는 노드에서만 동작하고, 다른 노드에 연결돼 있으면 리다이렉트된다.

## 4. List를 메시징 큐로

List는 양 끝에서 push/pop이 가능해서(`LPUSH/LPOP/RPUSH/RPOP`) 큐를 직접 구현하기 좋다. 애플리케이션 입맛대로 큐 동작을 짤 수 있다는 점이 장점이다.

### 기본 FIFO 큐

```
# Producer
> RPUSH tasks "send-email-1"
> RPUSH tasks "send-email-2"

# Consumer (FIFO)
> LPOP tasks
"send-email-1"
```

### RPUSHX와 트위터 타임라인 캐시

트위터(현 X)는 유저별 타임라인 캐시를 List로 관리한다. 유저 A가 트윗을 쓰면 A를 팔로우하는 모든 유저의 타임라인 캐시에 그 트윗을 추가해야 한다. 다만 거의 안 들어오는 유저(D)의 캐시까지 굳이 만들 필요는 없다.

`RPUSHX`는 키가 이미 있을 때만 추가하는 변종이다.

```
> RPUSHX Timelinecache:userB data3   # B 캐시 있음 → 추가됨
(integer) 26
> RPUSHX Timelinecache:userC data3   # C 캐시 있음 → 추가됨
(integer) 5
> RPUSHX Timelinecache:userD data3   # D 캐시 없음 → 무시
(integer) 0
```

"캐시가 있는지 확인" + "있으면 push"를 한 커맨드로 합쳤기 때문에 애플리케이션 라운드트립이 줄어든다.

### 블로킹: BLPOP / BRPOP

이벤트 큐를 폴링으로 처리하면 두 가지 문제가 있다. 큐에 데이터가 없어도 주기적으로 체크하니까 CPU/네트워크가 낭비되고, 폴링 인터벌만큼 처리가 지연된다.

`BLPOP`/`BRPOP`은 데이터가 있으면 즉시 반환하고 없으면 들어올 때까지 대기한다.

```
> BRPOP queue:a 5
1) "queue:a"
2) "data"
```

`5`는 타임아웃(초)이다. `0`이면 무한 대기. 타임아웃이 끝나면 `nil`을 반환한다. 여러 클라이언트가 같은 리스트에 동시에 블로킹돼 있으면, 데이터가 들어왔을 때 먼저 요청한 클라이언트가 가져간다.

### BRPOP이 2개 값을 반환하는 이유

```
> BRPOP queue:a queue:b queue:c 1000
1) "queue:b"     # 어느 큐에서 왔는지
2) "DATA"        # 실제 값
```

여러 큐를 동시에 대기할 수 있도록 설계됐기 때문이다. 어느 큐에서 들어왔는지 모르면 분기 처리가 불가능하므로 큐 이름과 값을 함께 반환한다. `BRPOP high mid low`처럼 우선순위 큐를 구현할 때도 유용하다.

### 원형 큐: RPOPLPUSH

같은 아이템을 반복 순회하거나, 병렬로 같은 아이템을 처리해야 하는 클라이언트에 어울리는 패턴이다. 큐에서 빼낸 즉시 반대쪽으로 다시 push한다.

```
> LPUSH clist A
> LPUSH clist B
> LPUSH clist C
> LRANGE clist 0 -1
1) "C"
2) "B"
3) "A"

> RPOPLPUSH clist clist
"A"                          # tail의 A를 빼서 head로

> LRANGE clist 0 -1
1) "A"
2) "C"
3) "B"
```

## 5. Stream

레디스 5.0에서 도입된 자료구조다. 대용량 메시지 데이터를 빠르게 처리할 수 있도록 설계됐고, Kafka에서 영향을 받았다. append-only 구조에 소비자 그룹 개념을 얹은 형태다.

활용은 두 갈래로 본다. 백엔드에서는 대량 데이터를 빠르게 처리하는 메시징 플랫폼으로, 데이터 엔지니어링에서는 다수 생산자가 만든 데이터를 다수 소비자가 처리하는 중간 큐잉 시스템으로 쓴다.

### Kafka와의 비교

| 항목 | Kafka | Redis Stream |
|---|---|---|
| 데이터 단위 | 토픽 | stream key |
| 메시지 식별 | partition 내 sequence number, 토픽 전체에서는 비유니크 | `<ms>-<seq>`, stream 전체에서 유니크 |
| 생성 | 토픽을 먼저 만들어야 함 | XADD가 stream을 자동 생성 |
| 순서 보장 | partition 내에서만 | stream 전체에서 |
| 부하 분산 단위 | partition | 소비자 그룹 |

### 메시지 ID

```
<millisecondsTime>-<sequenceNumber>
```

ms 파트는 저장 시점의 노드 로컬 시간이다. seq 파트는 같은 ms 안에 여러 메시지가 들어올 때의 순서로, 64bit 정수라 사실상 무제한이다. ID 자체에 시간이 들어있어서 시간 범위 검색이 가능하다.

### XADD

```
> XADD Email * subject "first" body "hello?"
"1659114481311-0"
```

`*`는 ID 자동 생성을 뜻한다. 필드-값 쌍 구조라 hash와 비슷하게 저장된다. 키가 없으면 stream을 자동 생성한다. 직접 ID를 지정할 수도 있지만 직전 ID보다 작은 값은 못 쓴다.

```
> XADD mystream 0-1 "hello" "world"
0-1
> XADD mystream 0-2 "hi" "redis"
0-2
```

### XREAD: 실시간 리스닝

```
> XREAD BLOCK 0 STREAMS Email 0
```

`BLOCK 0`은 데이터가 없어도 끊지 않고 무한 대기한다. `BLOCK 1000`이면 1초 대기다. `STREAMS Email 0`은 Email stream에서 ID > 0인 것 모두 받겠다는 뜻이라 결국 처음부터 전부 가져온다. ID 자리에 `$`를 넣으면 커맨드 실행 이후의 새 메시지부터 받는다. 특정 ID를 직접 넣어서 그 시점 이후만 받는 것도 된다.

### XRANGE: 특정 범위 조회

```
> XRANGE Email - +                          # 처음부터 끝까지
> XRANGE Email 1659114481310 1659114481319  # ms 범위
> XRANGE Email (1659114481311-0 +           # 해당 ID 다음부터 (exclusive)
```

`-`는 stream 최소 ID, `+`는 최대 ID, `(`는 exclusive를 뜻한다. `LIMIT`로 개수 제한도 가능하다. 역순은 `XREVRANGE`.

XREAD와의 차이는 명확하다. XRANGE는 현 시점 데이터를 모두 반환한 뒤 종료하지만, XREAD는 반환 후에도 새 메시지를 계속 기다린다.

### fan-out과 소비자 그룹

처음 읽을 때 이 둘이 가장 헷갈렸다. 둘 다 "여러 소비자가 stream을 읽는다"는 점은 같은데 의미가 다르다.

XREAD를 여러 소비자가 각자 부르는 fan-out 패턴에서는 같은 데이터를 모든 소비자가 다 받는다. BI 분석, 알림, 이메일 시스템이 동시에 같은 이벤트를 봐야 하는 경우에 쓴다.

XREADGROUP으로 소비자 그룹을 만들면 그룹 내 소비자들이 데이터를 나눠서 받는다. 작업을 분산해 처리량을 높이려고 쓰는 패턴이다.

여러 그룹이 같은 stream을 구독하면 그룹 단위 fan-out에 그룹 내부 분산이 겹친 모양이 된다. BIGroup이랑 EmailGroup이 같은 Email stream을 보는데, 각 그룹 안에서는 소비자들이 나눠 처리하는 식이다.

### 순서 보장에서 Kafka와 다른 점

Kafka는 partition 내에서만 순서가 보장된다. 메시지가 해시 함수로 partition에 분배되고 소비자는 partition 존재를 모른 채 토픽에서 데이터를 읽기 때문에, partition 간 순서는 보장되지 않는다. 순서를 지키고 싶으면 소비자 그룹과 partition을 일대일로 묶어야 한다.

Redis Stream은 stream 전체에서 ID 순서가 자연히 보장된다. 소비자 그룹 안의 소비자는 다른 소비자가 아직 안 읽은 메시지만 가져간다. partition 분할 자체가 없으니 순서가 깨질 일이 없다.

### XGROUP CREATE

```
> XGROUP CREATE Email EmailServiceGroup $
```

마지막 인자 `$`는 현재 시점 이후 메시지부터 리스닝하겠다는 뜻이다. `0`이면 stream 처음부터.

### XREADGROUP

```
> XREADGROUP GROUP EmailServiceGroup emailService1 COUNT 1 STREAMS Email >
```

`GROUP <그룹> <소비자>`는 어느 그룹의 어느 소비자로 읽을지를 지정한다. `COUNT 1`은 1개만 가져오겠다는 뜻. `>`는 다른 소비자가 아직 안 받은 새 메시지만 받겠다는 뜻이다.

소비자 이름은 처음 등장할 때 자동 생성된다. 명시적으로 만들 필요는 없다. XREADGROUP은 소비자 그룹의 상태를 변경하므로 일종의 쓰기 커맨드라, 마스터 노드에서만 호출 가능하다.

마지막 인자 자리에 ID 숫자(예: `0`)를 넣으면 새 메시지가 아니라 해당 소비자의 보류 리스트에서 ID > 입력값인 메시지를 반환한다. 장애 복구 때 유용한 옵션이다.

### 한 그룹으로 여러 stream 리스닝

```
> XGROUP CREATE Email BIGroup 0
> XGROUP CREATE Push BIGroup 0
> XREADGROUP GROUP BIGroup BI1 COUNT 2 STREAMS Email Push > >
```

각 stream에 동일한 이름의 그룹을 미리 만들어 두면 한 번의 XREADGROUP으로 둘 다 읽을 수 있다.

### ACK와 보류 리스트(PEL)

장애를 고려해서 Redis Stream은 소비자별로 보류 리스트(pending list)를 유지한다. XREADGROUP으로 메시지를 가져가면 PEL에 들어가고, 처리가 끝나면 XACK로 명시적으로 확인 응답을 보내야 PEL에서 제거된다. last_delivered_id는 그룹에 마지막으로 전달된 ID로, 중복 전달을 방지하는 데 쓰인다.

```
> XPENDING Email EmailServiceGroup
1) (integer) 9              # 그룹의 보류 메시지 총 개수
2) "1659114481311-0"        # 보류 중 최소 ID
3) "1659170735630-0"        # 보류 중 최대 ID
4) 1) 1) "es1"              # 소비자별 보류 개수
      2) "1"
   2) 1) "es2"
      2) "1"
   3) 1) "es3"
      2) "7"

> XACK Email EmailServiceGroup 1659114481311-0
(integer) 1
```

소비자가 죽었을 때 PEL을 보고 처리 못 한 메시지를 다시 가져갈 수 있다. 그래서 애플리케이션은 XREADGROUP으로 데이터를 처리한 뒤 주기적으로 XACK를 호출하는 패턴으로 짜야 한다.

### 메시지 보증 전략

세 가지가 있다.

- **at most once**: 받자마자 ACK 후 처리한다. 빠르지만 처리 도중 죽으면 메시지가 손실된다.
- **at least once**: 처리 후 ACK를 보낸다. 안전한 대신 ACK 직전에 죽으면 같은 메시지가 재처리된다.
- **exactly once**: 추가 메커니즘이 필요하다. Redis Stream만으로는 보장이 안 되고, 처리 여부를 set 같은 곳에 따로 기록해 둬야 한다.

at least once는 처리가 멱등(idempotent)함을 전제로 한다. 결제처럼 멱등하지 않은 작업이라면 중복 처리가 곧 버그라서 별도 방어 로직이 필요하다.

### XCLAIM: 메시지 재할당

소비자 A에 장애가 나서 A의 보류 메시지를 B로 옮기고 싶을 때 쓴다.

```
> XCLAIM Email EmailServiceGroup EmailService1 3600000 1626569498055-0
```

`3600000`은 min-idle-time(ms)이다. 메시지가 그 시간 이상 보류 상태였을 때만 옮긴다. 두 소비자가 동시에 XCLAIM을 걸어도 한쪽만 성공한다. 첫 호출이 idle을 0으로 리셋하기 때문에 두 번째 호출은 min-idle-time 조건을 못 채우게 된다.

### XAUTOCLAIM: 자동 재할당

XPENDING으로 ID를 찾아 일일이 XCLAIM 거는 게 번거로워서 도입됐다.

```
> XAUTOCLAIM Email EmailServiceGroup es1 360000 0-0 COUNT 1
```

지정 그룹에서 min-idle을 만족하는 보류 메시지를 지정 소비자로 옮기고, 다음 후보 ID도 같이 반환한다. 그래서 반복 호출이 가능하고, 다음 후보가 없으면 `0-0`이 돌아온다.

### Dead letter

메시지가 처리되지 못해 여러 소비자에게 반복 재할당되면 그 메시지의 counter가 1씩 증가한다. counter가 일정 값을 넘으면 정상 처리가 불가능한 메시지일 가능성이 높으니, 별도 stream(dead letter)으로 옮겨 사람이 들여다보게 한다.

### Stream 상태 확인: XINFO

```
> XINFO STREAM email          # stream 자체 정보 (length, 인코딩, 첫/마지막 메시지 ID)
> XINFO GROUPS email          # stream의 소비자 그룹 목록
> XINFO CONSUMERS email <그룹> # 그룹 내 소비자 상태 (pending, idle 시간)
```


## 6. 세 도구 선택 기준

상황별로 정리하면 이렇다.

| 상황 | 선택 |
|---|---|
| 단발성 알림, 메시지 손실 허용 | Pub/Sub |
| 한 소비자가 처리하는 작업 큐 | List + BLPOP/BRPOP |
| 여러 소비자가 같은 데이터를 받고 영속성 필요 | Stream + XREAD |
| 여러 소비자가 작업을 분산해 처리, 실패 시 재처리 필요 | Stream + 소비자 그룹 + ACK |
| Kafka 도입은 부담, 비슷한 기능 필요 | Stream |

## 7. 정리하며 헷갈렸던 것들

fan-out과 소비자 그룹의 차이가 처음엔 가장 안 잡혔다. 같은 stream을 여러 소비자가 XREAD로 읽으면 모두 동일 데이터를 받는다(fan-out). 반면 같은 그룹의 여러 소비자가 XREADGROUP으로 읽으면 데이터가 그룹 안에서 분산된다. 여러 그룹이 같은 stream에 붙는 경우는 그룹 단위 fan-out에 그룹 내부 분산이 겹친 모양이 된다고 이해하고 나니 그제야 정리됐다.

XREAD/XREADGROUP의 마지막 인자 자리에 들어가는 기호 셋(`0`, `$`, `>`)도 의미가 비슷해 보여서 매번 헷갈렸다. `0`은 stream 처음부터, `$`는 커맨드 실행 이후의 새 메시지부터, XREADGROUP의 `>`는 그룹이 아직 가져가지 않은 새 메시지를 뜻한다. 다 "어디부터 읽을지"인데 기준이 시간이냐 그룹 상태냐가 다르다.

at least once의 함정도 한 번 정리해 둬야겠다. 메시지가 처리됐는데 ACK를 보내기 직전에 소비자가 죽으면, 복구 시 보류 리스트에 그 메시지가 남아 있어서 두 번 처리된다. 멱등하지 않은 작업이라면 중복 방어 로직이 따로 필요하다. XCLAIM의 min-idle-time도 비슷한 맥락의 안전장치다. 두 소비자가 동시에 같은 메시지를 가로채려고 해도, 첫 XCLAIM이 idle을 0으로 리셋해서 두 번째 XCLAIM은 조건을 못 채워 자연히 무시된다.

자잘한 제약이 두 가지 있다. Stream에 ID를 직접 지정할 때는 이전 ID보다 작거나 같은 값을 쓸 수 없으니, 자동 생성을 안 쓸 거면 단조 증가는 호출 측 책임이 된다. 그리고 Pub/Sub은 클러스터 환경에서 기본 동작이 전체 broadcast라 노드가 많고 트래픽이 큰 경우 부담스럽다. 7.0 이상이면 Sharded Pub/Sub으로 슬롯 단위 격리를 검토하는 게 맞다.
