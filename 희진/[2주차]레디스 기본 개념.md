# 레디스 기본 개념

## 자료 구조

### String

String은 레디스에서 데이터를 저장할 수 있는 가장 간단한 자료 구조다. 최대 512MB의 문자열 데이터를 저장할 수 있으며, binary-safe하게 처리되기 때문에 JPEG 이미지 같은 바이너리 값이나 HTTP 응답값 등 다양한 형태의 데이터도 저장할 수 있다.

String은 키와 실제 저장되는 값이 일대일로 연결되는 유일한 자료 구조다. 다른 자료 구조에서는 하나의 키에 여러 개의 아이템이 저장된다.

**주요 커맨드**

```
> SET hello world
OK

> GET hello
"world"
```

기존 키에 값이 이미 있으면 새 값으로 덮어쓴다. 기존 값이 다른 자료 구조였어도 동일하게 동작한다.

**NX / XX 옵션**

- `NX` — 키가 없을 때만 새로 저장
- `XX` — 키가 이미 있을 때만 덮어쓰기

```
> SET hello newval NX
(nil)     // 이미 hello가 있으므로 저장되지 않음

> SET hello newval XX
OK        // 키가 있으므로 덮어쓰기 성공
```

**숫자 연산 — INCR / DECR**

String에는 숫자 형태의 데이터도 저장할 수 있고, 원자적으로 증감 연산을 할 수 있다.

```
> SET counter 100
OK

> INCR counter
(integer) 101

> INCRBY counter 50
(integer) 151
```

원자적이라는 것은 여러 클라이언트가 동시에 같은 키에 접근해도 레이스 컨디션이 발생하지 않는다는 의미다. 클라이언트 A와 B가 동시에 INCR 커맨드를 수행해도, 데이터를 읽고-증가하고-저장하는 일련의 과정이 하나의 원자적인 단위로 처리된다. `DECR`과 `DECRBY`는 감소 커맨드다.

**여러 키 한 번에 조작 — MSET / MGET**

```
> MSET a 10 b 20 c 30
OK

> MGET a b c
1) "10"
2) "20"
3) "30"
```

대규모 시스템에서 밀리초 단위의 네트워크 통신 시간도 서비스 전체 성능에 영향을 줄 수 있다. MSET, MGET 같은 커맨드를 활용하면 왕복 횟수를 줄여 응답 속도를 개선할 수 있다.

### List

List는 순서를 가지는 문자열의 목록이다. 하나의 List에는 최대 42억여 개의 아이템을 저장할 수 있다. 인덱스를 이용해 데이터에 직접 접근할 수 있으며, 일반적으로 스택이나 큐로 사용된다.

**주요 커맨드**

- `LPUSH` — 왼쪽(head)에 데이터 추가
- `RPUSH` — 오른쪽(tail)에 데이터 추가
- `LPOP` — 왼쪽에서 아이템을 꺼내면서 삭제
- `LRANGE` — 범위 조회

```
> RPUSH mylist A B C
(integer) 3

> LRANGE mylist 0 -1
1) "A"
2) "B"
3) "C"
```

인덱스는 음수를 사용할 수 있다. tail의 인덱스는 `-1`, 그 앞은 `-2`다. `LRANGE mylist 0 -1`은 전체 데이터를 조회한다는 의미다.

`LPOP`에 숫자를 함께 전달하면 지정한 개수만큼 반복해서 반환한다.

```
> LPOP mylist 2
1) "A"
2) "B"
```

**LTRIM — 범위 외 아이템 삭제**

`LTRIM`은 지정한 범위만 남기고 나머지는 모두 삭제한다. `LPOP`과 달리 삭제되는 아이템을 반환하지 않는다.

```
> LRANGE mylist 0 -1
1) "A"
2) "D"
3) "E"
4) "B"

> LTRIM mylist 0 1
OK

> LRANGE mylist 0 -1
1) "A"
2) "D"
```

**LPUSH + LTRIM 조합 — 고정 크기 큐**

로그처럼 계속 쌓이는 데이터를 최근 N개만 유지하고 싶을 때 유용하다.

```
LPUSH logdata <data>
LTRIM logdata 0 999
```

1,001번째 데이터가 들어오는 순간부터 가장 오래된 데이터(tail)가 삭제된다. List의 tail에서 데이터를 삭제하는 작업은 `O(1)`로 동작하기 때문에, 배치 처리로 주기적으로 삭제하는 것보다 훨씬 효율적이다.

**성능 특성**

- List의 양 끝에 데이터를 넣고 빼는 `LPUSH`, `RPUSH`, `LPOP`, `RPOP`은 `O(1)`로 처리된다.
- 인덱스나 데이터를 이용해 List 중간에 접근할 때는 `O(N)`으로 처리되므로, 데이터가 많을수록 성능이 저하된다.

**그 외 커맨드**

- `LINSERT` — 특정 데이터의 앞(`BEFORE`)이나 뒤(`AFTER`)에 데이터 추가
- `LSET` — 지정한 인덱스의 데이터를 덮어쓰기
- `LINDEX` — 원하는 인덱스의 데이터 조회

### Hash

Hash는 필드-값 쌍을 가진 아이템의 집합이다. 레디스에서 키-값으로 데이터를 저장하는 것처럼, 하나의 Hash 자료 구조 안에서 아이템은 필드-값 쌍으로 저장된다. 필드는 하나의 Hash 내에서 유일하다.

Hash는 객체를 표현하기 적절한 자료 구조이며, 관계형 데이터베이스의 테이블 데이터로 변환하기에도 간편하다. 다만 칼럼이 고정된 RDBMS 테이블과 달리, Hash에서는 각 아이템마다 다른 필드를 가질 수 있고 동적으로 필드를 추가할 수 있다는 차이가 있다.

**주요 커맨드**

- `HSET` — 아이템 저장 (한 번에 여러 필드-값 쌍 저장 가능)
- `HGET` — 특정 필드의 값 조회
- `HMGET` — 여러 필드의 값을 한 번에 조회
- `HGETALL` — Hash 내의 모든 필드-값 쌍을 반환

```
> HSET user:1 name Alice age 30
(integer) 2

> HGET user:1 name
"Alice"

> HGETALL user:1
1) "name"
2) "Alice"
3) "age"
4) "30"
```

### Set

Set은 정렬되지 않은 문자열의 모음이다. 하나의 Set 안에서 아이템은 중복해서 저장되지 않으며, 집합 연산(교집합, 합집합, 차집합) 관련 커맨드를 제공한다. 객체 간의 관계를 계산하거나 유일한 원소를 구해야 할 때 사용된다.

**주요 커맨드**

- `SADD` — 아이템 저장 (한 번에 여러 개 가능)
- `SMEMBERS` — 전체 아이템 조회 (순서는 랜덤)
- `SREM` — 아이템 삭제
- `SPOP` — 랜덤한 아이템 하나를 반환하면서 삭제

```
> SADD myset A B C
(integer) 3

> SMEMBERS myset
1) "B"
2) "A"
3) "C"
```

**집합 연산**

- `SUNION` — 합집합
- `SINTER` — 교집합
- `SDIFF` — 차집합

### Sorted Set

Sorted Set은 스코어 값에 따라 정렬되는 고유한 문자열의 집합이다. 모든 아이템은 스코어-값 쌍을 가지며, 저장될 때부터 스코어 값으로 정렬된다. 같은 스코어를 가진 아이템은 사전 순으로 정렬된다.

Sorted Set은 다른 자료 구조들의 특성을 함께 가진다.

- 중복 없이 유일하게 저장되므로 Set과 유사하다
- 각 아이템이 스코어라는 데이터에 연결되어 있어 Hash와 유사하다
- 스코어 순으로 정렬되어 있어 List처럼 인덱스로 접근할 수 있다

> **참고:** List와 Sorted Set 모두 인덱스로 아이템에 접근할 수 있지만, 인덱스를 자주 사용해야 한다면 Sorted Set이 더 효율적이다. List에서 인덱스 접근은 `O(N)`이지만, Sorted Set에서는 `O(log N)`으로 처리되기 때문이다.

**주요 커맨드**

- `ZADD` — 스코어-값 쌍으로 아이템 저장
- `ZRANGE` — 저장된 데이터 조회 (시작과 끝 범위 지정 필수)

```
> ZADD score:220817 100 user:B
(integer) 1

> ZADD score:220817 150 user:A 150 user:C 200 user:F 300 user:E
(integer) 4
```

저장하려는 데이터가 이미 있으면 스코어만 업데이트되며, 업데이트된 스코어로 재정렬된다.

**ZADD 옵션**

- `XX` — 아이템이 이미 존재할 때만 스코어 업데이트
- `NX` — 아이템이 없을 때만 신규 삽입
- `GT` — 새 스코어가 기존보다 클 때만 업데이트 (없으면 삽입)
- `LT` — 새 스코어가 기존보다 작을 때만 업데이트 (없으면 삽입)

**ZRANGE 조회 방식**

ZRANGE는 여러 옵션으로 다양한 조건의 조회가 가능하다.

_인덱스로 조회 (기본)_

```
> ZRANGE score:220817 0 -1 WITHSCORES
```

- `WITHSCORES` — 데이터와 함께 스코어도 반환
- `REV` — 역순 조회
- List처럼 음수 인덱스(`-1`)로 마지막 아이템 접근 가능

_스코어로 조회 — BYSCORE_

```
> ZRANGE score:220817 100 150 BYSCORE
```

스코어가 100 이상 150 이하인 값을 조회한다. 스코어 앞에 `(` 문자를 붙이면 해당 값을 포함하지 않는다. `-inf`, `+inf`로 최솟값/최댓값을 표현할 수 있다.

_사전 순으로 조회 — BYLEX_

스코어가 같은 아이템들을 사전식 순서로 조회하는 옵션이다. `(` 또는 `[` 문자로 포함/미포함을 지정하며, `-`와 `+`로 처음/끝을 나타낸다. 한글 문자열도 아스키 바이트 순서에 따라 사전식으로 정렬된다.

### 비트맵 (Bitmap)

비트맵은 독자적인 자료 구조가 아니라, String 자료 구조에 bit 연산을 수행할 수 있도록 확장한 형태다. String이 binary-safe하고 최대 512MB까지 저장할 수 있기 때문에, 2³² 비트의 비트맵으로 사용할 수 있다.

비트맵의 가장 큰 장점은 저장 공간을 획기적으로 줄일 수 있다는 점이다. 예를 들어 정수 ID로 구분되는 40억 명이 넘는 유저에 대한 y/n 데이터를 512MB 안에 저장할 수 있다.

**주요 커맨드**

- `SETBIT` — 비트 저장
- `GETBIT` — 비트 조회
- `BITFIELD` — 여러 비트를 한 번에 SET
- `BITCOUNT` — 1로 설정된 비트 개수 카운팅

```
> SETBIT mybitmap 7 1
(integer) 0

> BITCOUNT mybitmap
(integer) 1
```

### HyperLogLog

HyperLogLog는 집합의 원소 개수(카디널리티)를 추정할 수 있는 자료 구조다. 대량 데이터에서 중복되지 않는 고유한 값을 집계할 때 유용하다.

Set 같은 자료 구조는 중복을 피하기 위해 모든 데이터를 기억해야 하므로 데이터가 많아질수록 메모리 사용량도 늘어난다. 반면 HyperLogLog는 입력된 데이터 자체를 저장하지 않고 자체적인 방식으로 변환해 처리하기 때문에, 저장되는 데이터 개수와 관계없이 일정한 메모리를 유지한다.

**특징**

- 하나의 HyperLogLog는 최대 12KB의 크기를 가진다
- 카디널리티 추정의 오차는 0.81%다
- 최대 2⁶⁴개의 아이템을 저장할 수 있다

**주요 커맨드**

- `PFADD` — 아이템 저장
- `PFCOUNT` — 카디널리티(고유 원소 개수) 추정

```
> PFADD visitors user1 user2 user3
(integer) 1

> PFCOUNT visitors
(integer) 3
```

### Geospatial

Geospatial은 경도, 위도 데이터 쌍의 집합으로 지리 데이터를 저장하는 자료 구조다. 내부적으로는 Sorted Set으로 저장되며, 하나의 자료 구조 안에 키는 중복 저장되지 않는다.

**주요 커맨드**

- `GEOADD` — `GEOADD <key> 경도 위도 멤버` 순서로 저장 (`XX`, `NX` 옵션 지원)
- `GEOPOS` — 저장된 위치 데이터 조회
- `GEODIST` — 두 아이템 사이의 거리 반환
- `GEOSEARCH` — 특정 위치를 기준으로 거리 내 아이템 검색 (`BYRADIUS`는 반경, `BYBOX`는 직사각형 기준)

```
> GEOADD locations 127.027 37.498 "GangnamStation"
(integer) 1

> GEOPOS locations "GangnamStation"
1) 1) "127.02699..."
   2) "37.49800..."
```

### Stream

Stream은 레디스를 메시지 브로커로 사용할 수 있게 하는 자료 구조다. Kafka에서 영향을 받아 만들어졌으며, 컨슈머 그룹 개념을 통한 분산 처리를 지원한다.

데이터는 계속해서 추가되는 append-only 방식으로 저장되므로, 실시간 이벤트나 로그성 데이터 저장에 적합하다. Stream의 자세한 사용 방법은 7장에서 다룬다.

## 레디스에서 키를 관리하는 법

### 키의 자동 생성과 삭제

Stream, Set, Sorted Set, Hash처럼 하나의 키가 여러 아이템을 가지는 자료 구조에서는 명시적으로 키를 생성하거나 삭제하지 않아도 자동으로 관리된다. 세 가지 규칙을 따른다.

**1. 키가 없을 때 아이템을 넣으면 빈 자료 구조를 먼저 생성한다**

```
> LPUSH mylist A
(integer) 1       // mylist 키가 자동 생성됨
```

단, 저장하려는 키에 이미 다른 자료 구조가 생성되어 있으면 에러를 반환한다.

**2. 모든 아이템을 삭제하면 키도 자동으로 삭제된다 (Stream은 예외)**

```
> LPOP mylist    // 마지막 아이템까지 삭제하면 mylist 키 자체가 없어짐
```

**3. 키가 없는 상태에서 읽기 전용 커맨드는 에러 대신 "비어 있음"처럼 동작한다**

```
> DEL mylist
> LLEN mylist     // 에러가 아니라 0 반환
(integer) 0
```

### 키의 조회

**EXISTS**

```
EXISTS key [key ...]
```

키가 존재하면 1, 없으면 0을 반환한다.

**KEYS**

```
KEYS pattern
```

매칭되는 패턴의 모든 키 리스트를 반환한다. 글롭 패턴으로 동작한다.

- `h?llo` — `hello`, `hallo` 등 한 글자 매칭
- `h*llo` — 임의 길이 매칭
- `h[ae]llo` — 대괄호 안의 문자 중 하나
- `h[^e]llo` — 대괄호 안의 문자 제외
- `h[a-b]llo` — 범위 지정

> **주의:** `KEYS`는 위험한 커맨드다. 레디스에 100만 개의 키가 저장되어 있다면 모든 키를 반환하는 동안 다른 클라이언트의 커맨드는 전부 대기한다. 레디스는 싱글 스레드로 동작하기 때문에 오래 걸리는 커맨드 하나가 전체 서비스를 마비시킬 수 있다. 마스터가 응답하지 못하는 사이 모니터링 도구의 헬스 체크도 실패해서 의도하지 않은 페일오버가 발생할 수도 있다. 운영 환경에서는 `KEYS` 대신 `SCAN`을 사용해야 한다.

**SCAN**

```
SCAN cursor [MATCH pattern] [COUNT count] [TYPE type]
```

커서를 기반으로 특정 범위의 키만 조회하는 커맨드다. `KEYS`의 대안이다.

```
> SCAN 0
1) "17"               // 다음 커서
2) 1) "key:1"
   2) "key:2"
   ...

> SCAN 17
1) "0"                // 0이면 검색 완료
2) ...
```

첫 호출 시 커서에 `0`을 입력하고, 반환된 커서를 다음 호출에 사용한다. 반환 커서가 `0`이 되면 모든 키를 순회한 것이다.

- `COUNT` — 한 번에 반환할 키 개수 조정 (기본 10개). 단, 정확히 지정한 개수만큼 반환되지는 않는다.
- `MATCH` — 패턴에 맞는 키만 조회. 단, 스캔 후 필터링하는 방식이라 지정한 개수보다 적은 결과나 빈 결과가 반환될 수 있다.
- `TYPE` — 지정한 타입의 키만 조회. `MATCH`와 마찬가지로 반환 직전에 필터링한다.

`SCAN`과 비슷하게 `SSCAN`, `HSCAN`, `ZSCAN`이 있다. 각각 Set, Hash, Sorted Set에서 `SMEMBERS`, `HGETALL`, `ZRANGE` 대신 사용할 수 있어 서버에 부담을 덜 주며 반복 호출이 가능하다.

**SORT**

List, Set, Sorted Set에서만 사용할 수 있는 커맨드로, 키 내부의 아이템을 정렬해 반환한다. `LIMIT`으로 일부 데이터만, `ASC`/`DESC`로 정렬 순서 변경, `ALPHA`로 문자열 사전 순 정렬을 할 수 있다. `BY`와 `GET` 옵션으로 정렬 결과를 이용해 다른 키에 접근하는 것도 가능하다.

### 키의 변경

**RENAME / RENAMENX**

```
RENAME key newkey
RENAMENX key newkey
```

키 이름을 변경한다. `RENAMENX`는 변경할 키(newkey)가 존재하지 않을 때만 동작한다.

**COPY**

```
COPY source destination [DB destination-db] [REPLACE]
```

source 키를 destination에 복사한다. destination이 이미 있으면 에러가 반환되지만, `REPLACE` 옵션을 사용하면 덮어쓴다.

**TYPE**

지정한 키의 자료 구조 타입을 반환한다.

**OBJECT**

키에 대한 상세 정보를 반환한다. `ENCODING`, `IDLETIME` 등의 서브커맨드로 해당 키가 내부적으로 어떻게 저장됐는지, 얼마나 오래 호출되지 않았는지 등을 확인할 수 있다.

### 키의 삭제

**FLUSHALL**

```
FLUSHALL [ASYNC | SYNC]
```

레디스에 저장된 모든 키를 삭제한다. 기본적으로 동기 방식으로 동작하기 때문에, 모든 데이터가 삭제될 때까지 다른 응답을 처리할 수 없다. `ASYNC` 옵션을 사용하면 백그라운드로 실행된다.

`lazyfree-lazy-user-flush` 옵션이 `yes`면 `ASYNC` 없이도 백그라운드로 동작한다. 버전 7 기준 기본값은 `no`다.

**DEL**

키와 키에 저장된 모든 아이템을 삭제한다. 기본적으로 동기 방식으로 동작한다.

**UNLINK**

`DEL`과 비슷하게 키와 데이터를 삭제하지만, 백그라운드에서 다른 스레드에 의해 처리된다. 우선 키와 데이터의 연결을 끊고 실제 삭제는 이후에 진행한다.

> **실무 포인트:** Set이나 Sorted Set처럼 하나의 키에 많은 아이템이 저장된 경우, `DEL`로 삭제하면 레디스가 그동안 다른 요청을 처리하지 못한다. 100만 개 아이템을 가진 키를 `DEL`로 삭제하는 것은 전체 키가 100만 개인 레디스에서 동기 방식으로 `FLUSHALL`을 수행하는 것과 같다. 아이템이 많은 키를 삭제할 때는 `UNLINK`를 사용하는 것이 좋다.

`lazyfree-lazy-user-del` 옵션이 `yes`면 모든 `DEL`이 `UNLINK`처럼 동작한다. 버전 7 기준 기본값은 `no`다.

### 키의 만료 시간

**EXPIRE**

```
EXPIRE key seconds [NX | XX | GT | LT]
```

키의 만료 시간을 초 단위로 정의한다.

- `NX` — 만료 시간이 정의되지 않았을 때만 적용
- `XX` — 만료 시간이 이미 정의되어 있을 때만 적용
- `GT` — 기존보다 더 긴 만료 시간일 때만 적용
- `LT` — 기존보다 더 짧은 만료 시간일 때만 적용

**EXPIREAT**

유닉스 타임스탬프로 만료 시점을 직접 지정한다. 사용할 수 있는 옵션은 `EXPIRE`와 동일하다.

**EXPIRETIME**

키가 삭제되는 유닉스 타임스탬프를 반환한다. 만료 시간이 설정되지 않은 키는 `-1`, 존재하지 않는 키는 `-2`를 반환한다.

**TTL**

키가 몇 초 뒤에 만료되는지 반환한다. 만료 시간이 없으면 `-1`, 키가 없으면 `-2`를 반환한다.

`PEXPIRE`, `PEXPIREAT`, `PEXPIRETIME`, `PTTL`은 위와 동일하게 동작하되 밀리초 단위로 계산한다.
