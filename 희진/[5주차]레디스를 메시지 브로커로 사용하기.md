# 레디스를 메시지 브로커로 사용하기

## 메시징 큐와 이벤트 스트림

메시지 브로커는 크게 메시징 큐와 이벤트 스트림 두 가지 형태로 나눌 수 있다. 용어부터 정리하면 다음과 같다.

| 형태          | 보내는 쪽         | 받는 쪽            |
| ------------- | ----------------- | ------------------ |
| 메시징 큐     | 생산자(Producer)  | 소비자(Consumer)   |
| 이벤트 스트림 | 발행자(Publisher) | 구독자(Subscriber) |

두 방식의 차이는 세 가지로 정리할 수 있다.

| 구분                  | 메시징 큐                           | 이벤트 스트림                              |
| --------------------- | ----------------------------------- | ------------------------------------------ |
| **방향성**            | 생산자가 소비자의 큐에 직접 푸시    | 발행자가 저장소에 보내면 구독자가 풀(Pull) |
| **데이터 영속성**     | 소비자가 읽어가면 큐에서 삭제       | 읽어가도 삭제되지 않음 (설정에 따라 보존)  |
| **새 소비자 추가 시** | 추가된 시점 이후 메시지만 확인 가능 | 이전 히스토리도 조회 가능                  |

메시징 큐는 1대1 상황에서 한 서비스가 다른 서비스에 동작을 지시할 때 적합하고, 이벤트 스트림은 N대N 상황에서 유리하다. 같은 메시지를 두 서비스에 전달할 때 메시징 큐는 두 개의 큐에 각각 푸시해야 하지만, 이벤트 스트림은 하나의 저장소에 보내면 구독자들이 알아서 가져간다.

## Redis가 제공하는 세 가지 메시지 브로커 기능

Redis는 메시지 브로커 용도로 사용할 수 있는 세 가지 기능을 제공한다. 각 기능은 성격이 다르므로 목적에 맞게 선택해야 한다.

| 기능        | 성격                 | 메시지 보존   | 전달 보장        | 적합한 상황                  |
| ----------- | -------------------- | ------------- | ---------------- | ---------------------------- |
| **Pub/Sub** | 이벤트 스트림과 유사 | 보존되지 않음 | 보장 없음        | 알림 서비스, Fire-and-forget |
| **List**    | 메시징 큐            | 소비 시 삭제  | 단순한 큐로 사용 | 작업 큐, 이벤트 큐           |
| **Stream**  | 이벤트 스트림        | 보존됨        | ACK 기반 보장    | 대규모 메시징, 분산 처리     |

- **Pub/Sub** — 빠르고 간단하다. 발행자가 채널에 데이터를 전송하면 채널을 듣고 있는 모든 소비자가 즉시 데이터를 받는다. 한 번 전파된 후 삭제되는 일회성 방식이라 메시지 전달이 보장되지 않는다. Fire-and-forget 패턴이 필요한 알림 서비스에 유용하다.
- **List** — 메시징 큐로 사용하기 알맞다. push/pop이 빠르고, 새 데이터가 들어올 때까지 대기하는 블로킹 기능을 제공한다.
- **Stream** — 완벽한 스트림 플랫폼으로 사용할 수 있다. Apache Kafka에서 영감을 받아 만들어졌으며, 메시지가 소비되어도 보존되고 Consumer Group을 통해 분산 처리도 가능하다.

## Pub/Sub

Pub/Sub은 Redis 노드에 접근 가능한 모든 클라이언트가 발행자와 구독자가 될 수 있는 가벼운 메시지 전달 방식이다. 발행자는 특정 채널에 메시지를 전송하고, 구독자는 특정 채널을 리스닝한다.

매우 가볍기 때문에 최소한의 메시지 전달 기능만 제공한다.

- 발행자는 어떤 구독자가 메시지를 읽어갔는지, 정상적으로 전달됐는지 알 수 없다
- 구독자는 메시지가 언제 어떤 발행자에 의해 생성됐는지 메타데이터를 알 수 없다
- 한 번 전파된 데이터는 Redis에 저장되지 않으며, 특정 구독자에 장애가 생겨도 알 수 없다

> Pub/Sub은 정합성이 중요한 데이터에는 적합하지 않다. 필요하면 애플리케이션 레벨에서 송수신 로직을 추가해야 한다.

### 기본 커맨드

**PUBLISH** — 채널에 메시지를 전송한다. 수신한 구독자 수를 반환한다.

```
> PUBLISH hello world
(integer) 1
```

**SUBSCRIBE** — 특정 채널을 구독한다. 여러 채널을 동시에 구독할 수 있다. 구독자 상태에서는 새 채널 구독은 가능하지만, Pub/Sub과 관련 없는 다른 커맨드는 수행할 수 없다.

**PSUBSCRIBE** — 패턴 매칭으로 한 번에 여러 채널을 구독한다. glob-style 패턴을 지원한다.

```
> PSUBSCRIBE mail-*
```

위 커맨드는 `mail-track`, `mail-album` 등 `mail-`로 시작하는 모든 채널의 메시지를 수신한다. 이때 메시지는 일반 `message` 타입이 아닌 `pmessage` 타입으로 전달된다. 구독자가 `SUBSCRIBE mail-1`과 `PSUBSCRIBE mail-*`을 동시에 구독한 상태에서 `mail-1`에 메시지가 발행되면, 구독자는 두 개의 메시지(message, pmessage)를 받는다.

### 클러스터 환경에서의 비효율

클러스터 구조에서 메시지를 발행하면 클러스터에 속한 모든 노드에 자동으로 전달된다. 아무 노드에 연결해 `SUBSCRIBE`해도 데이터를 수신할 수 있지만, 이는 클러스터의 주요 목적인 대규모 데이터 분산 처리에 부합하지 않는 비효율적인 구조다. 모든 노드에 메시지가 복제되면서 불필요한 리소스 사용과 네트워크 부하가 발생한다.

### Sharded Pub/Sub

Redis 7.0에서 도입된 기능이다. 각 채널이 슬롯에 매핑되고, 같은 슬롯을 가진 노드 간에만 메시지가 전파된다. 키를 슬롯에 할당하는 것과 동일한 방식이다.

| 구분             | 일반 Pub/Sub           | Sharded Pub/Sub                    |
| ---------------- | ---------------------- | ---------------------------------- |
| 메시지 전파 범위 | 클러스터 내 모든 노드  | 같은 슬롯을 담당하는 노드와 복제본 |
| 리소스 사용      | 모든 노드에 복제       | 필요한 노드에만 전달               |
| 도입 버전        | 전 버전                | 7.0 이상                           |
| 주요 커맨드      | `PUBLISH`, `SUBSCRIBE` | `SPUBLISH`, `SSUBSCRIBE`           |

**SPUBLISH** — 해당 채널의 슬롯을 담당하는 노드와 그 복제본에만 메시지를 전달한다.

```
10.0.0.1:6379> SPUBLISH apple a
-> Redirected to slot [7092] located at 10.0.0.2:6379
(integer) 1
```

redis-cli로 데이터를 전파할 때 연결된 노드에서 처리할 수 없으면 해당 슬롯을 담당하는 노드로 리다이렉트된다. `SSUBSCRIBE`도 마찬가지로 특정 노드에서만 수행할 수 있다.

## List를 메시징 큐로 사용하기

Redis의 List는 큐로 사용하기 적절한 자료 구조다. `LPUSH`, `RPUSH`, `LPOP`, `RPOP`으로 tail이나 head에서 데이터를 넣고 뺄 수 있어, 애플리케이션 특성에 맞는 메시징 큐를 직접 구현할 수 있다.

### RPUSHX — 조건부 추가

**RPUSHX**는 List가 이미 존재할 때만 아이템을 추가하는 커맨드다. 트위터의 타임라인 캐시 관리가 대표적인 활용 사례다.

트위터는 각 유저의 타임라인 캐시 데이터를 List 자료구조로 관리한다. 유저 A가 트윗을 작성하면 A를 팔로우하는 유저들의 타임라인 캐시 List에 새 아이템이 추가되어야 한다. 이때 `RPUSHX`를 사용하면 이미 캐시된 타임라인에만 데이터를 추가한다.

```
> RPUSHX TimelineCache:userB data3
(integer) 26

> RPUSHX TimelineCache:userD data3
(integer) 0
```

자주 접속하지 않는 유저 D는 타임라인 캐시가 없으므로 0이 반환되고 데이터가 추가되지 않는다. 캐시 존재 여부를 애플리케이션에서 확인하지 않고 Redis에서 직접 제어할 수 있어, 불필요한 확인 과정을 줄여 성능이 향상된다.

### BRPOP / BLPOP — 블로킹 기능

이벤트 큐를 폴링(Polling)으로 처리하면 두 가지 단점이 있다.

- 폴링 인터벌 동안 큐를 반복 확인하면서 애플리케이션과 큐의 리소스를 불필요하게 소모한다
- 이벤트가 들어와도 폴링 인터벌이 지나야만 처리할 수 있어 즉시 처리가 어렵다

**BRPOP**과 **BLPOP**은 `RPOP`/`LPOP`에 블로킹 기능을 추가한 커맨드다. 데이터가 있으면 즉시 반환하고, 없으면 데이터가 들어올 때까지 대기하다가 들어온 값을 반환한다. 타임아웃을 설정할 수 있고, 타임아웃 시 nil을 반환한다.

```
> BRPOP queue:a 5
1) "queue:a"
2) "data"
```

`queue:a`에 데이터가 들어올 때까지 최대 5초 대기한다. 타임아웃을 0으로 설정하면 무한 대기한다. 여러 클라이언트가 동시에 블로킹할 경우 데이터가 들어오면 가장 먼저 요청한 클라이언트가 가져간다.

`BRPOP`은 `RPOP`과 달리 두 값을 반환한다. 첫 번째는 팝된 리스트의 키, 두 번째는 값이다. 여러 리스트에서 동시에 대기할 수 있도록 설계됐기 때문이다.

```
> BRPOP queue:a queue:b queue:c timeout 1000
1) "queue:b"
2) "DATA"
```

여러 리스트 중 어느 하나에 데이터가 들어오면 가장 먼저 들어온 데이터를 가져온다.

### RPOPLPUSH — 원형 큐 구현

특정 아이템을 반복 접근하거나 여러 클라이언트가 병렬로 같은 아이템에 접근해야 할 때 **RPOPLPUSH**로 원형 큐를 구현할 수 있다. tail에서 pop한 데이터를 head에 push하는 순환 구조다.

```
> LPUSH clist A
> LPUSH clist B
> LPUSH clist C

> LRANGE clist 0 -1
1) "C"
2) "B"
3) "A"

> RPOPLPUSH clist clist
"A"

> LRANGE clist 0 -1
1) "A"
2) "C"
3) "B"
```

tail에 있던 `A`가 head로 이동했다.

## Stream — Kafka에서 영감을 받은 자료구조

Stream은 Redis 5.0에서 추가된 자료구조로, 대용량 메시징 데이터를 빠르게 처리하도록 설계됐다. Append-only 방식, 즉 로그 파일처럼 데이터가 쌓이기만 하는 구조다.

활용 관점은 두 가지다.

- **백엔드 개발자 관점** — 대량의 데이터를 효율적으로 처리하는 플랫폼
- **데이터 엔지니어 관점** — 여러 생산자와 소비자가 처리할 수 있는 데이터 저장소이자 중간 큐잉 시스템

대규모 메시징 데이터 처리에서 가장 먼저 고려되는 시스템은 Apache Kafka다. LinkedIn에서 처음 개발되어 2011년 오픈소스로 공개됐고, 현재는 Confluent에서 관리한다. Redis Stream은 Kafka에서 영향을 많이 받아 유사한 기능을 가지며, 일부 영역에서는 Kafka보다 뛰어난 점도 있다.

스트림 자체의 개념은 연속적인 데이터의 흐름, 일정한 데이터 조각의 연속을 의미한다. 10GB의 텍스트 파일을 단어나 줄 단위로 잘게 쪼개 처리하는 바이트 스트림, 채팅 프로그램의 메시지를 처리하는 JSON 스트림이 대표적이다. 외부 입력뿐 아니라 애플리케이션 내부 서버 간 데이터 이동도 스트림이다.

직접 스트리밍 플랫폼을 구축하기는 어렵다. Kafka나 Redis Stream을 활용하면 더 쉽고 정확하게 처리할 수 있고, 스트리밍 플랫폼을 시스템 중심에 두면 모든 데이터 처리를 통합할 수 있다.

## Stream의 데이터 저장과 식별 — Kafka와의 비교 (1)

Kafka와 Redis Stream은 데이터를 저장하고 식별하는 방식에서 차이가 있다.

| 구분               | Kafka                          | Redis Stream                |
| ------------------ | ------------------------------ | --------------------------- |
| **데이터 단위**    | 토픽(Topic)                    | Stream 자료구조 (하나의 키) |
| **메시지 식별**    | 시퀀스 넘버 (0부터 증가)       | 메시지 ID (시간 기반)       |
| **유니크 범위**    | 토픽 내 파티션 안에서만 유니크 | Stream 전체에서 유니크      |
| **시간 기반 검색** | 불가                           | 가능                        |
| **사전 생성**      | 토픽을 미리 생성 필요          | XADD로 자동 생성            |

### Redis 메시지 ID 구조

```
<millisecondsTime>-<sequenceNumber>
```

- **밀리세컨드 파트** — stream에 아이템이 저장될 시점의 Redis 노드 로컬 시간
- **시퀀스 파트** — 동일 밀리세컨드에 여러 아이템이 저장될 수 있어 순서를 의미. 64bit

모든 데이터가 유니크 ID를 가지고 ID 값이 시간을 의미하므로, 시간으로 특정 데이터를 검색할 수 있다는 장점이 있다.

### 스트림 생성 방식 차이

Kafka는 토픽을 먼저 생성한 뒤 프로듀서로 메시지를 전송한다.

```
$ kafka-topics --zookeeper 127.0.0.1:6000 --topic Email --create partitions 1 --replication-factor 1
$ kafka-console-consumer --brokers-list 127.0.0.1:7000 --topic Email
> "I am first email"
> "I am second email"
```

Redis는 별도의 stream 생성 과정이 필요 없다. `XADD` 커맨드로 새 이름의 stream에 데이터를 저장하면 자동 생성된다.

```
> XADD Email * subject "first" body "hello?"
"1659114481311-0"
```

## Stream의 데이터 입력과 조회

### XADD — 데이터 입력

`XADD`에서 `*` 자리에 들어가는 값이 메시지 ID다. `*`로 입력하면 Redis가 자동으로 타임스탬프 기반 ID를 생성한다. 메시지는 키-값 쌍으로 저장되어 Hash처럼 메시지마다 유동적인 데이터를 저장할 수 있다.

자동 생성 대신 ID를 직접 지정할 수도 있다.

```
> XADD mystream 0-1 "hello" "world"
0-1

> XADD mystream 0-2 "hi" "redis"
0-2
```

> 최소 ID 값은 `0-1`이며, 이후 저장되는 ID는 이전에 저장된 ID보다 작은 값으로 지정할 수 없다.

### XREAD — 실시간 리스닝

데이터를 읽는 주체가 동작하는 방식은 Kafka와 Redis 사이에 명확한 차이가 있다.

- **Kafka 소비자** — 토픽을 실시간으로 리스닝하면서 메시지를 받는다. 기본은 리스닝 시작 시점 이후 메시지만 반환하고, `--from-beginning` 옵션으로 처음부터 읽을 수 있다.
- **Redis Stream 소비자** — 두 가지 방식으로 데이터를 읽을 수 있다. Kafka처럼 실시간 리스닝하거나, ID로 필요한 데이터를 검색한다.

```
XREAD [COUNT count] [BLOCK milliseconds] STREAMS key [key ...] ID [ID ...]
```

Email stream을 처음부터 읽고 새 메시지가 들어올 때까지 계속 리스닝하는 예제다.

```
> XREAD BLOCK 0 STREAMS Email 0
```

- `BLOCK 0` — stream에 더 이상 가져올 데이터가 없어도 연결을 끊지 않고 계속 리스닝
- `BLOCK 1000` — 데이터가 없으면 1초 동안 대기
- `STREAMS Email 0` — Email stream에서 ID가 0보다 큰 값을 읽음 (처음부터 모든 데이터)
- `STREAMS Email $` — 커맨드 실행 이후 저장되는 메시지만 가져옴 (`$`는 최대 ID 의미)

### XRANGE — 특정 데이터 조회

`XRANGE`는 ID로 원하는 시간대의 데이터를 조회한다.

```
XRANGE key start end [COUNT count]
XREVRANGE key end start [COUNT count]
```

- `-` — 가장 작은 ID
- `+` — 가장 큰 ID
- `XREVRANGE` — `XRANGE`의 역순

타임스탬프를 이용한 조회도 가능하다.

```
> XRANGE Email 1659114481310 1659114481319
```

기본은 입력한 타임스탬프를 포함하며, 특정 ID를 제외하고 다음부터 조회하려면 `(` 문자를 사용한다.

| 커맨드   | 동작 방식                                          |
| -------- | -------------------------------------------------- |
| `XREAD`  | 기존 데이터 반환 후 신규 데이터 계속 반환 (리스닝) |
| `XRANGE` | 커맨드 실행 시점에 저장된 모든 데이터 반환 후 종료 |

## Consumer Group — Kafka와의 비교 (2)

여러 소비자가 같은 stream을 읽는 방식에는 두 가지가 있다.

**Fan-out**은 같은 데이터를 여러 소비자에게 전달하는 개념이다. Kafka는 같은 토픽을 여러 소비자가 읽도록 하면 간단하게 fan-out이 된다. Redis Stream에서도 여러 소비자가 `XREAD`를 수행하면 fan-out이 가능하다.

같은 데이터를 여러 소비자가 나눠서 가져가야 할 때도 있다. 메시지를 병렬로 처리해 성능을 향상시키는 경우다. 이때 순서 보장이 필요한지에 따라 처리 방식이 달라진다.

- **순서 보장 필요** — 티켓 결제(카드 유효성 검사 → 잔고 확인 → 결제)
- **순서 보장 불필요** — 회원 가입

### 순서 보장 방식의 차이

| 구분               | Redis Stream | Kafka                            |
| ------------------ | ------------ | -------------------------------- |
| **유니크 키 범위** | Stream 전체  | 파티션 내부                      |
| **순서 보장**      | 항상 보장    | 파티션 내부에서만 보장           |
| **fan-out 시**     | 순서 유지됨  | 여러 파티션에서 읽으면 순서 깨짐 |

Redis Stream은 데이터를 저장할 때마다 고유 ID를 부여하기 때문에 순서대로 저장되며, 소비자에게 전달될 때도 순서가 항상 보장된다.

Kafka는 유니크 키가 파티션 내에서만 보장된다. 메시지가 토픽에 저장될 때 해시 함수로 여러 파티션에 랜덤하게 분배되는데, 소비자는 파티션의 존재를 모르고 전체 파티션에서 읽기 때문에 여러 파티션 간 정렬이 보장되지 않는다.

> Kafka에서 메시지 순서를 보장하려면 Consumer Group을 사용해야 한다.

### Consumer Group 동작 방식의 차이

| 구분                        | Kafka Consumer Group         | Redis Consumer Group                       |
| --------------------------- | ---------------------------- | ------------------------------------------ |
| **부하 분산 방식**          | 소비자-파티션 1:1 매핑       | last_delivered_id로 상태 관리              |
| **소비자별 할당**           | 각 소비자가 특정 파티션 담당 | 다른 소비자가 읽지 않은 데이터를 순차 분배 |
| **소비할 메시지 개수 지정** | 불가                         | COUNT 옵션으로 가능                        |

- **Kafka** — 여러 소비자가 토픽 내 파티션과 1대1로 연결된다. 파티션 내부에서는 순서가 보장되므로 각 소비자는 순서가 보장된 메시지를 읽는다.
- **Redis Stream** — 소비자 그룹 내 한 소비자는 다른 소비자가 아직 읽지 않은 데이터만 읽는다. 소비자 그룹은 stream의 상태를 나타내는 개념으로, `last_delivered_id`를 관리하면서 동일한 메시지가 중복 전달되는 것을 방지한다.

### XGROUP CREATE — 소비자 그룹 생성

```
> XGROUP CREATE Email EmailServiceGroup $
```

- `$` — 현재 시점 이후의 데이터부터 리스닝
- `0` — stream의 처음부터 리스닝

### XREADGROUP — 소비자 그룹으로 데이터 읽기

`XREADGROUP`은 `XREAD`와 형태가 비슷하지만 지정한 소비자 그룹을 통해 메시지를 읽는다.

```
> XREADGROUP GROUP EmailServiceGroup emailService1 COUNT 1 STREAMS Email >
```

매번 소비자가 소비자 그룹으로 작업할 때마다 그룹 내에서 소비자를 고유하게 식별할 이름을 지정해야 한다. 처음 언급될 때 자동으로 생성되므로 명시적으로 생성할 필요는 없다.

`>` 의미는 다음과 같다.

- `>` — 다른 소비자에게 전달되지 않은 새 메시지 전달
- `0`이나 다른 숫자 ID — 입력한 ID보다 큰 ID 중 보류 리스트(Pending List)에 속하던 메시지 반환

> `XREADGROUP`은 읽어오는 동작이 소비자 그룹의 상태에 영향을 미치므로 일종의 쓰기 커맨드로 간주되며, 마스터 노드에서만 호출할 수 있다.

### 하나의 소비자 그룹에서 여러 stream 리스닝

하나의 stream에 여러 소비자 그룹이 존재할 수 있고, 각 그룹은 독립적으로 동작한다. 소비자 그룹 1의 소비자가 a 메시지를 읽으면 같은 그룹에서는 다시 읽을 수 없지만, 소비자 그룹 2나 다른 일반 소비자는 해당 메시지를 읽을 수 있다. 반대로 하나의 소비자 그룹에서 여러 stream을 리스닝하는 것도 가능하다.

```
> XGROUP CREATE Email BIGroup 0
> XGROUP CREATE Push BIGroup 0

> XREADGROUP GROUP BIGroup BI1 COUNT 2 STREAMS Email Push > >
```

`BIGroup`이 Email과 Push 두 stream을 함께 리스닝한다.

## 메시지 처리 보장과 보류 리스트 (PEL)

여러 서비스가 메시지 브로커를 통해 데이터를 처리할 때, 예상치 못한 장애로 시스템이 종료되면 이를 인지하고 재처리할 수 있어야 한다. 메시지 브로커는 각 소비자에게 어떤 메시지를 전달했고, 그 메시지가 정상적으로 처리됐는지 인지하고 있어야 한다.

Redis Stream은 이를 위해 **보류 리스트(Pending Entries List, PEL)**를 관리한다. 소비자 그룹에 속한 소비자가 메시지를 읽으면 각 소비자별로 읽어간 메시지 리스트가 새로 생성되고, 마지막으로 읽어간 데이터 ID로 `last_delivered_id`가 업데이트된다.

이메일 서비스가 stream에 ACK를 보내면, stream은 해당 소비자의 보류 리스트에서 ACK를 받은 메시지를 삭제한다. 보류 리스트를 이용하면 소비자가 처리한 데이터를 명확히 파악할 수 있다.

> 이메일 서비스에 장애가 발생해 재부팅해야 하는 상황에서, stream의 보류 리스트에 남아 있는 데이터가 있다면 해당 데이터를 먼저 불러와 처리하는 작업을 선행할 수 있다. 이렇게 하면 예상치 못한 서비스 중단 상황에서도 모든 메시지를 놓치지 않고 처리할 수 있다. 데이터 처리가 완료된 후에는 애플리케이션에서 `XACK`를 주기적으로 전송하는 작업이 필요하다.

### XPENDING — 보류 메시지 확인

```
XPENDING <key> <groupname> [<start-id> <end-id> <count> [<consumer-name>]]
```

```
> XPENDING Email EmailServiceGroup
1) (integer) 9
2) "1659114481311-0"
3) "1659170735630-0"
4) 1) 1) "es1"
      2) "1"
   2) 1) "es2"
      2) "1"
   3) 1) "es3"
      2) "7"
```

반환되는 값의 의미는 다음과 같다.

| 인덱스 | 의미                                  |
| ------ | ------------------------------------- |
| 1)     | ACK를 받지 못해 보류 중인 메시지 개수 |
| 2)     | 보류 중인 메시지 ID의 최솟값          |
| 3)     | 보류 중인 메시지 ID의 최댓값          |
| 4)     | 각 소비자별 보류 리스트 개수          |

### XACK — 처리 완료 처리

```
> XACK Email EmailServiceGroup 1659114481311-0
(integer) 1
```

Email stream의 `EmailServiceGroup` 그룹에 속한 소비자가 `1659114481311-0` ID를 가진 메시지를 처리했다는 의미다. `XACK` 이후 `XPENDING`으로 확인하면 보류 메시지 개수가 줄어든 것을 확인할 수 있다.

Kafka도 Redis Stream과 비슷하게 파티션별 오프셋을 관리한다. Kafka는 내부적으로 `__consumer_offsets` 토픽에 데이터를 기록하는데, 소비자가 특정 파티션의 메시지를 읽으면 소비자 그룹, 토픽, 파티션 정보가 통합되어 저장된다. 다만 Kafka에서 오프셋은 소비자가 마지막으로 읽은 위치가 아니라, **다음으로 읽어야 할 위치**를 기록한다는 점이 특징이다.

## 메시지 보증 전략 — Kafka와의 비교 (3)

메시징 시스템에는 세 가지 메시지 보증 전략이 있다. 애플리케이션 특성에 따라 적절히 선택해야 한다.

| 전략              | ACK 시점            | 메시지 손실 가능성 | 중복 처리 가능성 | 적합한 상황                         |
| ----------------- | ------------------- | ------------------ | ---------------- | ----------------------------------- |
| **at most once**  | 처리 전             | 있음               | 없음             | 빠른 응답이 필요하고 일부 손실 허용 |
| **at least once** | 처리 후             | 없음               | 있음             | 멱등한 서비스                       |
| **exactly once**  | 처리 후 + 중복 체크 | 없음               | 없음             | 정확히 한 번만 처리되어야 하는 경우 |

### at most once

메시지를 최대 한 번 보내는 것을 의미한다. 소비자는 메시지를 받자마자 실제 처리하기 전에 먼저 ACK를 보낸다. 속도는 향상되지만, ACK를 보낸 뒤 소비자에 문제가 생겨 서비스가 다운되면 이미 ACK를 보냈지만 실제로 처리하지 못한 데이터는 잃을 수 있다. 메시지가 일부 손실되더라도 빠른 응답이 필요한 경우 선택하는 전략이다.

### at least once

소비자는 받은 메시지를 모두 처리한 뒤 ACK를 보낸다.

예를 들어 이메일 서버 1에서 `129-0`과 `130-1`을 받은 뒤 `129-0`은 처리했지만 `130-1`은 처리하지 못해 ACK를 보내지 못한 상태에서 서비스가 재시작된 경우, 보류 리스트에 남아 있는 메시지를 다시 전부 읽어가기 때문에 `129-0`은 두 번 처리된다.

> ACK 전송이 지연되어 실제로 메시지가 처리됐지만 ACK 전송 전에 소비자가 종료되면, 보류 리스트에 처리된 메시지도 남아 있어 이미 처리한 메시지를 한 번 더 처리하게 된다. 멱등함이 보장되는 서비스라면 상관없지만, 그렇지 않다면 문제가 될 소지가 있다.

### exactly once

모든 메시지가 무조건 한 번씩 전송되는 것을 보장한다. 이런 방식으로 메시지를 전송하려면 추가적인 기능이 필요할 수 있다. Redis Stream에서 exactly once를 구현하려면 Set 같은 추가 자료 구조를 이용해 이미 처리된 메시지인지를 확인하는 과정이 필요하다.

## 메시지 재할당

Redis는 소비자에게 장애가 날 경우를 대비해 소비자별 보류 리스트를 유지한다. 소비자 서버에 장애가 발생해 복구되지 않는다면, 해당 소비자가 처리하던 보류 중인 메시지들은 다른 소비자가 대신 처리해야 한다.

### XCLAIM — 수동 재할당

`XCLAIM`은 메시지의 소유권을 다른 소비자에게 할당하는 커맨드다.

```
XCLAIM <key> <group> <consumer> <min-idle-time> <ID-1> <ID-2> ... <ID-N>
```

`min-idle-time`(최소 대기 시간)을 지정해야 한다. 메시지가 보류 상태로 머무른 시간이 최소 대기 시간을 초과한 경우에만 소유권을 변경할 수 있도록 해서, 같은 메시지가 두 소비자에게 중복 할당되는 것을 방지한다.

예를 들어 `EmailService3`에 문제가 생겨 이 소비자가 처리하던 메시지를 `EmailService1`과 `EmailService2`가 가져가려고 하는 경우, 두 소비자가 모두 `XCLAIM`을 실행하더라도 충돌이 방지된다.

```
EmailService 1: XCLAIM Email EmailServiceGroup EmailService3 3600000 1626569498055-0
EmailService 2: XCLAIM Email EmailServiceGroup EmailService3 3600000 1626569498055-0
```

`EmailService1`의 커맨드가 먼저 실행되면 메시지의 보류 시간이 즉시 0으로 재설정된다. `EmailService2`의 `XCLAIM`은 최소 대기 시간보다 메시지의 보류 시간이 짧으므로 무시되어 중복 할당이 방지된다.

### XAUTOCLAIM — 자동 재할당

보류 메시지를 직접 확인하고 재할당하는 작업이 자주 발생하면 번거롭다. `XAUTOCLAIM`은 자동으로 메시지를 가져와 처리하며, 할당 대기 중인 다음 메시지 ID를 반환하므로 반복 호출할 수 있다.

```
XAUTOCLAIM <key> <group> <consumer> <min-idle-time> <start> [COUNT count] [JUSTID]
```

`XCLAIM`과 달리 재할당할 메시지를 직접 입력할 필요가 없다.

| 반환값  | 의미                                       |
| ------- | ------------------------------------------ |
| 첫 번째 | 다음 보류 메시지 ID (더 이상 없으면 `0-0`) |
| 두 번째 | 소유권이 이전된 메시지 정보                |

### Dead letter

Stream 내 각 메시지는 counter 값을 가진다. `XREADGROUP`으로 할당하거나 `XCLAIM`으로 재할당할 때마다 counter가 1씩 증가한다.

문제가 있는 메시지는 여러 소비자에게 할당이 반복되며 counter가 계속 증가한다. 특정 값에 도달하면 다른 stream으로 보내 관리자가 추후 처리하도록 하는 것이 현명한데, 이런 메시지를 보통 **dead letter**라고 부른다.

## Stream 상태 확인

`XINFO` 커맨드로 stream의 상태를 확인할 수 있다.

| 커맨드                          | 용도                                                   |
| ------------------------------- | ------------------------------------------------------ |
| `XINFO CONSUMERS <key> <group>` | 특정 소비자 그룹에 속한 소비자 정보                    |
| `XINFO GROUPS <key>`            | stream에 속한 전체 소비자 그룹 리스트                  |
| `XINFO STREAM <key>`            | stream 자체 정보 (내부 인코딩, 첫·마지막 메시지 ID 등) |

```
> XINFO CONSUMERS Email EmailServiceGroup
1) "name": "es1", "pending": 1, "idle": 650129
2) "name": "es2", "pending": 0, "idle": 437738623
3) "name": "es3", "pending": 7, "idle": 858725
```
