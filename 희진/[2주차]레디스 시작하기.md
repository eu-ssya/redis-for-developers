# 레디스 시작하기

## 설치 방식 — 패키지 vs 소스 빌드

리눅스에 레디스를 설치하는 방식은 두 가지다.

- **패키지 설치** — yum, apt 등 패키지 매니저를 사용한다. 빠르고 간편하지만, 레디스 관련 파일이 여러 디렉터리에 흩어져 저장되므로 관리가 어려울 수 있다. 또한 인터넷이 연결되지 않은 서버에서는 사용할 수 없다.
- **소스 빌드** — 소스 코드를 다운로드해 직접 빌드한다. 원하는 버전을 선택할 수 있고, 파일이 한 디렉터리에 모여 있어 관리가 편하다. 하나의 서버에 여러 개의 레디스 인스턴스를 띄울 때에도 소스 빌드 방식이 유리하다.

운영 환경에서 여러 인스턴스를 관리하거나 특정 버전을 고정해야 하는 경우에는 소스 빌드 방식이 더 적합하다.

## 레디스 실행 전 서버 환경 설정

레디스를 설치한 뒤 바로 실행하는 것보다, 운영에 적합한 환경을 먼저 구성하는 것이 좋다. 레디스의 특성상 몇 가지 서버 설정을 조정하지 않으면 성능 저하나 장애로 이어질 수 있다.

### Open files — 파일 디스크립터 수 확보

파일 디스크립터(File Descriptor)는 OS가 프로그램이 사용하는 파일이나 네트워크 연결에 붙이는 번호다. 프로그램이 파일을 열거나 네트워크 연결을 맺을 때마다 하나씩 소비된다. OS는 한 프로세스가 동시에 열 수 있는 파일 디스크립터 수에 제한을 둔다.

레디스는 클라이언트 하나가 접속할 때마다 파일 디스크립터를 하나씩 사용한다. 따라서 동시에 많은 클라이언트를 받으려면 그만큼 많은 파일 디스크립터가 필요하다.

레디스의 기본 `maxclients` 값은 10000이다. 레디스 프로세스가 받아들일 수 있는 최대 클라이언트 개수를 의미한다.

다만 이 값은 서버의 파일 디스크립터 수에 영향을 받는다. 레디스가 내부적으로 예약하는 파일 디스크립터 32개를 포함해, 서버의 최대 파일 디스크립터가 `maxclients + 32` 이상이어야 한다. 기본값인 10000을 유지하려면 서버의 파일 디스크립터 수를 최소 10032 이상으로 설정해야 한다.

이 값이 부족하면 레디스가 실행될 때 자동으로 maxclients를 낮추기 때문에, 예상보다 적은 클라이언트만 수용할 수 있게 된다.

### THP (Transparent Huge Pages) 비활성화

리눅스는 메모리를 페이지(Page) 단위로 관리한다. CPU가 메모리에 접근할 때는 가상 주소를 실제 물리 메모리 주소로 변환해야 하는데, 이 매핑 정보를 빠르게 조회하기 위해 캐시 역할을 하는 TLB(Translation Lookaside Buffer)라는 테이블을 사용한다.

기본 페이지 크기는 4KB로 고정되어 있는데, 서버의 메모리가 커질수록 관리해야 할 페이지 수가 많아지면서 TLB 크기도 함께 커지고, 오버헤드가 발생한다. 이를 줄이기 위해 페이지를 더 크게 만들어(보통 2MB) 관리하는 기능이 THP(Transparent Huge Pages)다. 페이지가 커지면 같은 메모리 영역을 더 적은 수의 페이지로 관리할 수 있어 TLB 부담이 줄어든다.

다만 레디스와 같은 데이터베이스 애플리케이션에서는 THP를 사용하면 오히려 성능이 떨어지고 응답 지연 시간(레이턴시, Latency)이 증가하는 현상이 발생한다. 레디스가 `fork()`로 백그라운드 프로세스를 만들 때, 큰 페이지를 복사하는 비용이 더 크기 때문이다. 레디스를 사용할 때는 THP를 비활성화하는 것이 권장된다.

### vm.overcommit_memory = 1

레디스는 디스크에 데이터를 저장할 때 `fork()`라는 시스템 콜을 사용한다. `fork()`는 현재 실행 중인 프로세스를 복제해 자식 프로세스를 만드는 기능으로, 레디스는 이 자식 프로세스에게 디스크 저장 작업을 맡기고, 자신은 계속 클라이언트 요청을 처리한다.

`fork()`로 자식 프로세스가 만들어지면 부모 프로세스의 메모리를 그대로 복사하는 대신, 처음에는 동일한 메모리 페이지를 공유한다. 이후 어느 한 쪽이 데이터를 변경할 때만 해당 페이지를 복사해서 각자 보유하게 되는데, 이를 COW(Copy-On-Write) 메커니즘이라고 한다.

레디스가 저장 작업 중에 클라이언트 요청으로 데이터가 많이 변경되면, 그때마다 페이지가 복사되면서 메모리 사용량이 순간적으로 빠르게 증가할 수 있다.

리눅스는 기본적으로 시스템이 감당할 수 있는 물리 메모리를 초과해 메모리 할당을 요청하면 이를 거부한다(`vm.overcommit_memory=0`). 실제로는 COW 특성상 모든 페이지가 동시에 복사되는 경우는 드물어, 실제 물리 메모리보다 더 많은 할당이 요청되더라도 대부분 문제없이 동작한다. 하지만 리눅스의 기본 설정은 이런 가능성을 계산에 넣지 않고 보수적으로 거부하기 때문에, 레디스의 `fork()`가 실패해 백그라운드 저장이 안 되는 상황이 발생할 수 있다.

`vm.overcommit_memory=1`로 설정하면 리눅스가 메모리 초과 할당 요청을 허용한다. 이렇게 해야 레디스의 `fork()`가 안정적으로 수행될 수 있다.

### somaxconn과 syn_backlog

클라이언트가 서버에 TCP로 접속할 때는 3-way handshake라는 과정을 거친다. 클라이언트가 연결 요청(SYN)을 보내고, 서버가 이에 응답(SYN+ACK)하고, 클라이언트가 최종 확인(ACK)을 보내는 세 단계의 패킷 교환을 통해 연결이 성립되는 방식이다.

이 과정에서 연결은 여러 단계를 거친다. 아직 완성되지 않은 상태(SYN만 받은 상태), 이미 완성되어 애플리케이션이 받아가기를 기다리는 상태 등으로 구분된다. OS는 이 연결들을 두 개의 큐에 담아서 관리하는데, 이를 TCP backlog 큐라고 부른다.

- `syn_backlog` — 아직 연결 성립이 진행 중인 요청을 담는 큐 (SYN 패킷을 받은 상태)
- `somaxconn` — 연결 성립이 완료되어, 애플리케이션이 accept() 하기를 기다리는 요청을 담는 큐

두 큐가 가득 차면 새로 들어오는 연결 요청이 거부된다. 트래픽이 몰리는 시점에 일부 클라이언트가 연결에 실패하는 원인이 될 수 있다.

레디스 설정 파일의 `tcp-backlog` 파라미터는 레디스가 사용할 backlog 큐의 크기를 지정하며, 기본값은 511이다. 다만 이 값은 서버의 `somaxconn`과 `syn_backlog` 값을 넘을 수 없다. 서버 설정이 더 작으면 레디스의 `tcp-backlog` 값은 서버 설정값으로 제한된다.

대량의 연결 요청이 몰릴 때 연결이 거부되는 상황을 피하려면, 서버의 `somaxconn`과 `syn_backlog` 값을 `tcp-backlog` 이상으로 설정해야 한다.

## redis.conf 주요 설정값

레디스는 `redis.conf` 파일로 설정을 관리한다. 운영 관점에서 반드시 알아야 할 설정값은 다음과 같다.

### port

- **기본값** — 6379
- 레디스 서버가 클라이언트 연결을 받는 포트를 지정한다.

### bind

- **기본값** — `127.0.0.1 -::1`
- 레디스 인스턴스가 어떤 네트워크 인터페이스(IP)로 들어오는 연결을 허용할지를 결정한다.

네트워크 인터페이스는 서버가 네트워크와 통신하기 위한 창구를 뜻한다. 서버는 여러 개의 네트워크 인터페이스를 가질 수 있고, 각각이 서로 다른 IP를 가진다. 예를 들어 하나의 서버가 내부망 통신용 IP, 외부망 통신용 IP, 루프백 IP를 각각 가질 수 있다. 루프백 IP(`127.0.0.1`)는 외부로 나가지 않고 서버 자기 자신으로만 연결되는 특별한 IP다.

기본값인 `127.0.0.1`은 루프백 IP이므로, 서버 내부에서 요청하는 연결만 허용한다는 의미다. 외부 서버에서 레디스에 접근하려면 해당 네트워크 인터페이스의 IP를 추가해야 한다.

`0.0.0.0`으로 설정하면 모든 IP로 들어오는 연결을 허용한다. 다만 레디스가 인터넷에 노출된 상황에서 이 값을 사용하면 보안상 위험할 수 있으므로, 운영 환경에서는 항상 특정한 값을 지정하는 것이 좋다.

### protected-mode

- **기본값** — `yes`
- 이 설정이 `yes`일 경우 패스워드를 설정해야만 레디스에 접근할 수 있다. 패스워드가 설정되어 있지 않다면 서버는 로컬에서 들어오는 연결만 수신한다. 외부에서 레디스에 접근하려는데 패스워드 설정 없이 연결이 되지 않는다면 이 설정을 확인해야 한다.

### requirepass / masterauth

- **기본값** — 없음
- `requirepass` — 레디스 서버에 접속하기 위한 패스워드를 지정한다.
- `masterauth` — 복제(Replication) 구조를 사용할 때, 리플리카가 마스터에 연결하기 위해 사용하는 패스워드다. 복제 연결을 사용할 예정이라면 `requirepass`와 `masterauth`를 같은 값으로 설정하는 것이 좋다.

### daemonize

- **기본값** — `no`
- 레디스 프로세스를 데몬(Daemon)으로 실행할지를 결정한다.

데몬은 사용자와 직접 상호작용하지 않고 백그라운드에서 계속 실행되는 프로세스를 말한다. 보통 서버 프로그램(웹 서버, 데이터베이스 등)은 데몬 형태로 실행된다. `daemonize`를 `yes`로 설정하면 레디스가 터미널에 묶여 있지 않고 백그라운드에서 계속 동작하기 때문에, 터미널을 종료해도 레디스는 계속 실행된다.

데몬으로 실행되면 PID 파일이 함께 생성된다. PID 파일은 현재 실행 중인 프로세스의 ID를 기록해두는 파일로, 나중에 이 프로세스를 종료하거나 재시작할 때 참조할 수 있게 해준다. PID 파일 경로는 `pidfile` 파라미터로 제어되며, 기본값은 `/var/run/redis_6379.pid`다.

### dir

- **기본값** — `./`
- 레디스의 워킹 디렉터리를 지정한다. 로그 파일, RDB/AOF 백업 파일 등 인스턴스 실행 중에 생성되는 파일이 이 디렉터리에 저장된다. 기본값은 현재 디렉터리이므로, 운영 환경에서는 명시적인 경로를 지정해두는 것이 좋다.

## 레디스 접속 방식

레디스 설치 시 함께 제공되는 `redis-cli`를 이용해 레디스에 접속할 수 있다. 접속 방식은 두 가지다.

- **대화형 모드** — `redis-cli` 명령으로 접속한다. 사용자가 연결을 끊을 때까지 세션이 유지되며, 커맨드를 입력하면 레디스가 처리 후 응답한다.
- **커맨드라인 모드** — `redis-cli <command>` 형태로 실행한다. 특정 커맨드를 수행한 뒤 응답을 반환하고 바로 종료된다.

`requirepass`에 패스워드가 설정되어 있다면, 접속 시 `-a` 옵션으로 패스워드를 함께 입력하거나, 접속 후 `AUTH` 커맨드로 인증해야 정상적으로 사용할 수 있다.

애플리케이션에서 레디스에 연결하는 것도 간단하다. Java, Python, JavaScript, Go 등 다수의 언어에서 레디스 클라이언트를 제공하기 때문에, 각 언어에 맞는 클라이언트 라이브러리를 사용하면 된다.
