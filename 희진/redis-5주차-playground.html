Chapter 06 · Section 1

Pub/Sub

발행자가 채널에 메시지를 던지면, 그 채널을 구독 중인 모든 구독자에게 즉시 전달됩니다.

기본 동작 — SUBSCRIBE와 PUBLISH06.1.1

Pub/Sub은 메시지 전달 모델 중 가장 단순한 형태입니다. 발행자는 누가 듣고 있는지 신경 쓰지 않고, 구독자는 누가 보냈는지 신경 쓰지 않습니다. 둘 사이를 연결하는 것은 채널 이름 하나뿐입니다.

구독

구독자는 하나 이상의 채널을 구독합니다. 같은 채널에 여러 구독자가 동시에 있을 수 있습니다.

> SUBSCRIBE news.tech
1) "subscribe"
2) "news.tech"
3) (integer) 1

응답의 세 번째 값은 현재 클라이언트가 구독 중인 채널의 총 개수입니다.

발행

발행자는 채널 이름과 메시지를 함께 보냅니다. 반환값은 그 메시지를 받은 구독자의 수입니다. 0이 반환되면 메시지를 받은 구독자가 한 명도 없었다는 뜻이며, 그 메시지는 그대로 사라집니다.

> PUBLISH news.tech "Redis 7.4 released"
(integer) 1
Diagram
Pub/Sub fan-out
발행자는 채널에만 메시지를 보내고, 구독자는 채널을 통해 동시에 수신합니다. 발행자는 누가 받는지 알 필요가 없습니다.
Try it · Pub/Sub 기본 동작
플레이그라운드 동작 실제 redis-cli에서는 SUBSCRIBE 이후 입력이 막히지만, 이 페이지는 같은 세션에서 PUBLISH도 가능하도록 가상 구독으로 처리합니다. 메시지가 도착하면 분홍색 [push] 라인으로 표시됩니다.
패턴 구독 — PSUBSCRIBE06.1.2

채널 이름을 하나씩 지정하는 대신 글롭 패턴으로 여러 채널을 한 번에 구독할 수 있습니다. 채널이 동적으로 생기는 환경에서 유용합니다.

> PSUBSCRIBE news.*
1) "psubscribe"
2) "news.*"
3) (integer) 1

> PUBLISH news.tech "v1"
(integer) 1

> PUBLISH news.sports "v2"
(integer) 1

패턴 구독에서는 메시지가 도착할 때 어떤 패턴이 매칭됐는지도 함께 전달됩니다. 일반 메시지의 message 대신 pmessage 타입으로 구분됩니다.

Try it · 패턴 구독
비용 패턴 매칭은 PUBLISH가 발생할 때마다 모든 활성 패턴과 비교해야 합니다. 패턴 구독자가 많을수록 PUBLISH의 비용이 O(N)으로 증가합니다.
한계와 메시지 손실06.1.3

Pub/Sub은 빠르고 단순한 만큼, 운영 환경에서 주의해야 할 한계가 있습니다.

구독자가 없으면 메시지가 사라짐

구독자가 한 명도 없을 때 발행된 메시지는 그대로 사라집니다. 구독자가 잠시 끊겨 있어도 그 사이에 발행된 메시지는 받을 수 없습니다.

처리 보장 없음

구독자가 메시지를 받았는지, 정상적으로 처리했는지 발행자는 알 수 없습니다. 처리 보장이 필요한 시나리오라면 Stream을 사용해야 합니다.

클러스터 환경에서의 비효율

일반 Pub/Sub은 클러스터의 모든 노드에 메시지가 브로드캐스트됩니다. 노드가 많을수록 네트워크 비용이 커집니다.

Redis 7.0부터는 같은 슬롯에 속한 노드끼리만 메시지를 주고받는 Sharded Pub/Sub 방식이 SSUBSCRIBE, SPUBLISH 명령어와 함께 추가됐습니다.

Redis 클러스터에서의 메시지 전파 — 일반 Pub/Sub vs Sharded Pub/Sub

Redis 클러스터는 16,384개 슬롯을 여러 노드에 분산 저장합니다. 채널 이름은 해시 함수로 슬롯에 매핑되는데, 두 방식은 이 슬롯 정보를 활용하는 방식이 다릅니다.

일반 Pub/Sub은 채널이 어느 슬롯에 매핑됐는지 무시하고 클러스터의 모든 노드에 메시지를 보냅니다. 노드 수만큼 네트워크 비용이 증가하는 구조입니다. Sharded Pub/Sub은 채널이 매핑된 슬롯의 책임 노드(와 그 복제본)에만 메시지를 전달해 비용을 절감합니다.
언제 쓰면 좋은가06.1.4

Pub/Sub은 메시지가 사라져도 영향이 적은 시나리오에 적합합니다. 대표적인 사용처는 다음과 같습니다.

  • 실시간 알림 — 새로운 댓글, 좋아요 같은 이벤트를 즉시 전달
  • 캐시 무효화 — 여러 인스턴스에 캐시 갱신 신호를 동시에 보내기
  • 채팅의 타이핑 표시 — 상대방이 입력 중이라는 일시적 정보 전달
  • 대시보드 업데이트 — 실시간 지표를 모든 뷰어에게 전송

반대로 결제, 주문, 외부 시스템 연동처럼 한 건이라도 누락되면 안 되는 이벤트는 Stream이나 외부 메시지 브로커를 사용해야 합니다.

Chapter 06 · Section 2

List를 메시징 큐로 사용하기

List는 LPUSH, RPUSH, LPOP, RPOP으로 양 끝에서 데이터를 넣고 뺄 수 있어, 애플리케이션 특성에 맞는 메시징 큐를 직접 구현하기 적절한 자료 구조입니다.

List 큐의 기본 동작06.2.1

한쪽 끝으로 넣고 반대쪽 끝으로 꺼내면 FIFO 큐가 됩니다. 가장 흔한 조합은 LPUSH(왼쪽 삽입)와 RPOP(오른쪽 추출)입니다. 둘 다 O(1)이라 큐의 길이와 상관없이 일정한 속도로 동작합니다.

> LPUSH tasks "task-1"
(integer) 1
> LPUSH tasks "task-2"
(integer) 2
> LPUSH tasks "task-3"
(integer) 3

# 큐 상태:  [task-3] [task-2] [task-1]
#            ↑ head(왼쪽)         tail(오른쪽) ↑

> RPOP tasks
"task-1"
> RPOP tasks
"task-2"

먼저 들어간 task-1이 먼저 나옵니다. 반대로 LPUSH + LPOP 조합은 LIFO 스택이 됩니다.

Animation
List 큐 — LPUSH + RPOP
왼쪽 head에 새 아이템이 들어오고 오른쪽 tail에서 먼저 들어간 아이템이 빠져나갑니다. 두 동작 모두 O(1)이라 큐 길이와 상관없이 일정한 속도로 동작합니다.
Try it · 기본 큐 동작
조건부 추가 — RPUSHX06.2.2

RPUSHX는 List가 이미 존재할 때만 아이템을 추가하는 커맨드입니다. 키가 없으면 아무 일도 일어나지 않고 0을 반환합니다.

트위터 타임라인 캐시

트위터는 각 유저의 타임라인 캐시 데이터를 List 자료구조로 관리합니다. 유저 A가 트윗을 작성하면 A를 팔로우하는 모든 유저의 타임라인 캐시 List에 새 아이템이 추가되어야 합니다. 다만 자주 접속하지 않는 유저까지 캐시를 유지할 필요는 없습니다.

이때 RPUSHX를 사용하면 이미 캐시된 타임라인에만 데이터를 추가할 수 있습니다.

# userB는 활성 유저 — 타임라인 캐시가 이미 존재함
> RPUSHX TimelineCache:userB "new-tweet-from-A"
(integer) 26

# userD는 비활성 유저 — 타임라인 캐시가 없음
> RPUSHX TimelineCache:userD "new-tweet-from-A"
(integer) 0

비활성 유저 D의 캐시는 만들지 않고 그냥 넘어갑니다. 캐시 존재 여부를 애플리케이션에서 미리 확인하는 과정이 사라지므로, 불필요한 왕복이 줄어듭니다.

Try it · RPUSHX 동작 비교

같은 동작을 하는 왼쪽 버전 LPUSHX도 있습니다.

블로킹 기능 — BRPOP / BLPOP06.2.3

이벤트 큐를 폴링 방식으로 처리하면 두 가지 단점이 있습니다.

  • 폴링 인터벌 동안 큐를 반복 확인하면서 애플리케이션과 큐의 리소스를 불필요하게 소모합니다
  • 이벤트가 들어와도 폴링 인터벌이 지나야만 처리할 수 있어 즉시 처리가 어렵습니다

BRPOP과 BLPOP은 RPOP과 LPOP에 블로킹 기능을 추가한 커맨드입니다. 데이터가 있으면 즉시 반환하고, 없으면 데이터가 들어올 때까지 대기하다가 들어온 값을 반환합니다. 타임아웃을 설정할 수 있으며, 타임아웃 시에는 nil을 반환합니다.

> BRPOP queue:a 5
# queue:a에 데이터가 들어올 때까지 최대 5초 대기
# 0을 주면 무한 대기

1) "queue:a"
2) "data"

BRPOP은 RPOP과 달리 두 개의 값을 반환합니다. 첫 번째는 팝된 리스트의 키, 두 번째는 값입니다. 여러 리스트에서 동시에 대기할 수 있도록 설계됐기 때문입니다.

여러 리스트 동시 대기 — 우선순위 큐

BRPOP은 여러 키를 한 번에 받을 수 있습니다. 왼쪽에 적힌 키부터 확인해서, 데이터가 있는 첫 큐의 값을 반환합니다. 우선순위 큐를 이렇게 구현할 수 있습니다.

> BRPOP queue.high queue.normal queue.low 0
# high → normal → low 순서로 확인
# 셋 다 비어 있으면 데이터가 들어올 때까지 무한 대기

1) "queue.normal"
2) "task-A"
Try it · 블로킹과 폴링
커넥션 점유 BRPOP이 실행 중인 커넥션은 블로킹 상태로 점유됩니다. 같은 커넥션에서 다른 명령은 수행되지 않으므로, 블로킹 커맨드는 별도 커넥션에서 호출해야 합니다.
원형 큐 — RPOPLPUSH06.2.4

특정 아이템을 반복적으로 접근하거나, 여러 클라이언트가 같은 아이템 집합을 병렬로 처리해야 할 때 원형 큐가 필요합니다. RPOPLPUSH는 tail에서 꺼낸 데이터를 head에 다시 넣는 순환 구조를 만듭니다.

> LPUSH clist A
> LPUSH clist B
> LPUSH clist C

> LRANGE clist 0 -1
1) "C"
2) "B"
3) "A"

> RPOPLPUSH clist clist
"A"

> LRANGE clist 0 -1
1) "A"
2) "C"
3) "B"

tail에 있던 A가 head로 이동했습니다. 같은 키로 두 번 지정하면 자기 자신을 순환합니다. 다른 키로 지정하면 한 리스트에서 다른 리스트로 데이터가 옮겨갑니다.

Try it · 원형 큐

Redis 6.2부터는 RPOPLPUSH 대신 방향을 자유롭게 지정할 수 있는 LMOVE가 권장됩니다. LMOVE src dst RIGHT LEFT가 RPOPLPUSH src dst와 같은 동작입니다.

List vs Stream06.2.5

List 큐는 단순하고 빠르지만 처리 보장 측면에서 한계가 있습니다. 컨슈머가 RPOP 또는 BRPOP으로 메시지를 꺼내는 순간 그 메시지는 큐에서 사라집니다. 처리 도중 컨슈머가 죽으면 그 메시지는 복구할 방법이 없습니다.

책 다음 절에서 다루는 Stream은 이 한계를 Consumer Group과 보류 리스트(PEL)로 해결합니다. 두 자료구조를 정리하면 다음과 같습니다.

구분ListStream
메시지 보존소비 시 삭제보존됨 (ACK와 무관)
처리 보장없음ACK 기반
장애 복구불가PEL로 재할당 가능
다수 컨슈머경쟁 소비 (한 명만 받음)Consumer Group으로 분산
적합한 상황단순한 작업 큐, 이벤트 큐대규모 메시징, 분산 처리

처리 보장이나 장애 복구가 필요하면 Stream을 선택하고, 단순한 작업 분배라면 List로 충분합니다.

Chapter 06 · Section 3

Stream의 도입과 데이터 식별

Stream은 Redis 5.0에서 추가된 자료구조로, Append-only 방식으로 데이터가 쌓이는 로그 형태의 구조입니다. 이 섹션에서는 Stream이 등장한 배경과, Kafka와 비교했을 때 데이터를 어떻게 식별하는지를 정리합니다.

Stream의 도입 배경06.3.1

Section 1에서 다룬 메시징 큐와 이벤트 스트림의 구분에서, Stream은 이벤트 스트림 쪽에 해당합니다. 메시지를 소비해도 사라지지 않고 보존되며, 여러 소비자가 같은 데이터를 다른 시점에 읽을 수 있습니다.

등장 배경

대규모 메시징 데이터 처리에서 가장 먼저 고려되는 시스템은 Apache Kafka입니다. LinkedIn에서 처음 개발되어 2011년 오픈소스로 공개됐고, 현재는 Confluent에서 관리하고 있습니다. 정확하게는 Apache Software Foundation 산하 프로젝트로, Confluent는 Kafka의 창립 개발자들이 세운 회사로 주요 컨트리뷰터이자 상업 지원을 맡고 있습니다. Redis Stream은 Kafka에서 영향을 받아 만들어졌으며 유사한 기능을 제공합니다.

Stream을 활용하는 관점은 크게 두 가지로 나눌 수 있습니다.

  • 백엔드 개발자 관점 — 대량의 메시지를 효율적으로 처리하는 플랫폼으로 사용
  • 데이터 엔지니어 관점 — 여러 생산자와 소비자가 데이터를 주고받는 저장소이자 중간 큐잉 시스템으로 사용

스트림이라는 개념

스트림은 연속적인 데이터의 흐름, 일정한 데이터 조각의 연속을 의미합니다. 10GB짜리 텍스트 파일을 단어나 줄 단위로 잘게 쪼개 처리하는 바이트 스트림, 채팅 프로그램의 메시지를 처리하는 JSON 스트림이 대표적입니다. 외부 입력뿐 아니라 애플리케이션 내부 서버 간 데이터 이동도 스트림에 해당합니다.

스트리밍 플랫폼을 직접 구축하기는 쉽지 않습니다. Kafka나 Redis Stream을 활용하면 더 안정적으로 처리할 수 있고, 스트리밍 플랫폼을 시스템 중심에 두면 모든 데이터 처리를 한 곳에서 관리할 수 있습니다.

Kafka는 토픽 안 파티션마다 0부터 시퀀스가 시작되어 유니크 범위가 파티션 내부로 한정됩니다. Redis Stream의 ID는 밀리세컨드-시퀀스 형식으로 stream 전체에서 유니크하며, 값 자체가 시간을 의미합니다.

  • 시퀀스 파트 — 동일한 밀리세컨드에 여러 아이템이 저장될 수 있어 순서를 의미. 64bit 정수

스트림 생성 방식 차이

Kafka는 토픽을 먼저 생성한 뒤 프로듀서로 메시지를 전송하지만, Redis는 별도의 Stream 생성 과정이 필요하지 않습니다. XADD 명령어로 새 이름의 Stream에 데이터를 저장하면 자동 생성됩니다. 다음 절에서 실제 명령어로 비교해 봅니다.

Chapter 06 · Section 4

Stream 데이터 입력과 조회

XADD로 데이터를 입력하고, XREAD로 실시간 리스닝, XRANGE로 시간대 조회를 합니다. Kafka CLI와 직접 비교하면서 차이를 정리합니다.

데이터 입력 — XADD06.4.1

Kafka는 토픽을 먼저 만들고 프로듀서로 메시지를 보내는 두 단계를 거칩니다.

$ kafka-topics --create --topic Email --partitions 1 --replication-factor 1
$ kafka-console-producer --bootstrap-server localhost:9092 --topic Email
> "I am first email"
> "I am second email"

Redis Stream은 토픽 생성 과정이 없습니다. XADD로 곧바로 데이터를 넣으면 Stream이 자동 생성됩니다.

> XADD Email * subject "first" body "hello?"
"1659114481311-0"

커맨드 구조

XADD의 첫 번째 인자는 Stream 키, 두 번째 인자는 메시지 ID입니다. ID 자리에 *를 넣으면 Redis가 타임스탬프 기반 ID를 자동 생성합니다. 메시지 본문은 키-값 쌍의 연속으로, Hash처럼 메시지마다 유동적인 필드를 저장할 수 있습니다.

ID 직접 지정

ID를 자동 생성하는 대신 직접 지정할 수도 있습니다.

> XADD mystream 0-1 hello world
"0-1"

> XADD mystream 0-2 hi redis
"0-2"

최소 ID 값은 0-1이며, 이후 저장되는 ID는 이전에 저장된 ID보다 작은 값으로 지정할 수 없습니다. 단조 증가 제약을 어기면 에러가 발생합니다.

> XADD mystream 0-1 should fail
(error) ERR The ID specified in XADD is equal or smaller than the target stream top item

실시간 리스닝 — XREAD06.4.2

XREAD는 하나 또는 여러 Stream에서 데이터를 읽는 명령어입니다. Kafka 소비자가 토픽을 실시간 리스닝하는 것과 비슷한 역할을 합니다.

XREAD [COUNT count] [BLOCK milliseconds] STREAMS key [key ...] ID [ID ...]

ID 인자의 의미

Stream 키 뒤의 ID는 어디서부터 읽을지를 지정합니다.

  • 0 — 처음부터 모든 데이터를 읽음 (지정한 ID보다 큰 값)
  • $ — 커맨드 실행 이후 저장되는 메시지만 읽음 (현재 시점의 최대 ID)
  • 특정 ID — 해당 ID보다 큰 메시지부터 읽음
# Email Stream을 처음부터 읽고, 최대 2건만
> XREAD COUNT 2 STREAMS Email 0

# 커맨드 실행 이후 새로 들어오는 메시지만 대기
> XREAD BLOCK 0 STREAMS Email $

BLOCK 옵션

BLOCK은 새 메시지가 들어올 때까지 대기하도록 합니다. BLOCK 0은 무한 대기, BLOCK 1000은 1초 대기를 의미합니다.

Kafka와의 비교

Kafka 소비자는 기본적으로 리스닝 시작 시점 이후의 메시지만 받으며, --from-beginning 옵션을 줘야 처음부터 읽습니다. Redis Stream은 이 차이를 ID 값으로 제어합니다. 0이면 처음부터, $면 현재 시점부터입니다.

Try it · XREAD 동작
플레이그라운드 동작 실제 Redis에서는 BLOCK 옵션이 새 데이터가 들어올 때까지 커넥션을 점유하며 대기하지만, 이 페이지의 Mock 엔진에서는 BLOCK 인자가 무시되고 즉시 결과를 반환합니다. 옵션 형식과 의미만 확인하는 용도로 사용합니다.
시간대 조회 — XRANGE06.4.3

XRANGE는 ID 범위를 지정해서 Stream의 데이터를 조회합니다. XREAD가 리스닝에 가깝다면 XRANGE는 검색에 가깝습니다.

XRANGE key start end [COUNT count]
XREVRANGE key end start [COUNT count]

특수 ID 표기

  • - — 가장 작은 ID (Stream의 시작)
  • + — 가장 큰 ID (Stream의 끝)
  • ( — 해당 ID를 제외하고 다음부터 조회 (열린 구간)
# 전체 데이터 조회
> XRANGE Email - +

# 처음 2건만
> XRANGE Email - + COUNT 2

# 특정 타임스탬프 범위
> XRANGE Email 1659114481310 1659114481319

# 특정 ID를 제외하고 그 다음부터
> XRANGE Email (1659114481311-0 +

XREVRANGE

XREVRANGE는 XRANGE의 역순입니다. 가장 최근 데이터부터 조회할 때 사용합니다. 인자 순서가 end start로 바뀐다는 점에 주의합니다.

> XREVRANGE Email + - COUNT 1
# 가장 최근 메시지 1건

XREAD vs XRANGE06.4.4

두 명령어는 모두 Stream 데이터를 읽지만 동작 방식이 다릅니다.

구분XREADXRANGE
동작 방식리스닝 (대기 가능)조회 (즉시 반환)
종료 조건BLOCK 시간 종료 또는 결과 수신지정 범위 끝까지 읽으면 종료
역순 지원없음XREVRANGE로 지원

실시간으로 새 메시지를 받아 처리해야 하면 XREAD, 특정 시간대의 데이터를 찾아야 하면 XRANGE를 사용합니다.

Consumer Group

여러 소비자가 같은 Stream을 나눠서 처리해야 할 때 사용하는 기능입니다. Kafka의 Consumer Group과 이름은 같지만 동작 방식에는 차이가 있습니다.

Consumer Group의 도입 배경06.5.1

여러 소비자가 같은 Stream을 읽는 방식은 크게 두 가지입니다.

Fan-out — 같은 데이터를 여러 소비자에게 전달

같은 메시지를 여러 소비자가 각자 받아 처리하는 방식입니다. Kafka에서는 같은 토픽을 여러 소비자가 읽도록 설정하면 자연스럽게 fan-out이 됩니다. Redis Stream에서도 여러 소비자가 각자 XREAD를 실행하면 같은 데이터를 모두 받습니다.

분산 처리 — 같은 데이터를 나눠서 처리

같은 데이터를 여러 소비자가 나눠서 가져가야 할 때도 있습니다. 메시지를 병렬로 처리해 성능을 높이려는 경우입니다. 이때 순서 보장이 필요한지 여부에 따라 처리 방식이 달라집니다.

  • 순서 보장 필요 — 티켓 결제 (카드 유효성 검사 → 잔고 확인 → 결제)
  • 순서 보장 불필요 — 회원 가입 이메일 발송 (가입 순서와 무관)

fan-out 시 메시지 순서 — Kafka vs Redis Stream
Kafka는 메시지가 해시로 여러 파티션에 분배되어, 컨슈머가 파티션을 순회하면서 읽을 때 도착 순서가 원본과 달라집니다. Redis Stream은 단일 stream에 시간 순으로 저장되므로 도착 순서가 그대로 유지됩니다.

Redis Stream은 데이터를 저장할 때마다 고유 ID를 부여하기 때문에 순서대로 저장되며, 소비자에게 전달될 때도 순서가 항상 보장됩니다. Kafka는 메시지가 토픽에 저장될 때 해시 함수로 여러 파티션에 분배되는데, 파티션 사이의 순서는 보장되지 않습니다. Kafka에서 메시지 순서를 보장하려면 Consumer Group을 사용해야 합니다.

Consumer Group 분배 방식 — Kafka vs Redis Stream
Kafka는 소비자와 파티션이 고정적으로 매핑되어 각 소비자가 자신의 파티션만 읽습니다. Redis Stream은 어떤 소비자가 먼저 XREADGROUP을 호출하든 last_delivered_id가 갱신되어 다음 메시지가 순차 분배됩니다.

Kafka 방식

여러 소비자가 토픽 내 파티션과 1:1로 연결됩니다. 파티션 내부에서는 순서가 보장되므로 각 소비자는 자신이 담당하는 파티션의 메시지를 순서대로 읽습니다. 메시지 처리 위치는 Kafka가 자체적으로 관리합니다.

Redis Stream 방식

Consumer Group은 Stream의 상태를 나타내는 개념입니다. 그룹 내 한 소비자가 메시지를 읽으면 다른 소비자는 그 메시지를 읽지 않으며, last_delivered_id가 갱신되어 동일한 메시지가 중복 전달되는 것을 방지합니다. 소비자는 한 번에 읽을 메시지 개수를 COUNT 옵션으로 직접 지정할 수 있습니다.

XGROUP CREATE — 소비자 그룹 생성06.5.3

Consumer Group을 사용하려면 먼저 XGROUP CREATE로 그룹을 생성해야 합니다.

XGROUP CREATE <key> <groupname> <id> [MKSTREAM]

마지막 인자는 그룹이 어느 시점부터 메시지를 받을지를 지정합니다.

  • $ — 현재 시점 이후 새로 들어오는 메시지부터 리스닝
  • 0 — Stream의 처음부터 리스닝 (저장된 모든 데이터 포함)
  • 특정 ID — 해당 ID 이후의 메시지부터 리스닝
# Email Stream의 처음부터 리스닝하는 그룹 생성
> XGROUP CREATE Email EmailServiceGroup 0
OK

# 같은 이름으로 다시 생성하면 에러
> XGROUP CREATE Email EmailServiceGroup 0
(error) BUSYGROUP Consumer Group name already exists

Stream이 없는 상태에서 그룹을 만들려면 MKSTREAM 옵션을 함께 줍니다. 빈 Stream이 자동 생성됩니다.

XREADGROUP — 소비자 그룹으로 데이터 읽기06.5.4

XREADGROUP은 XREAD와 형태가 비슷하지만 지정한 소비자 그룹을 통해 메시지를 읽습니다.

XREADGROUP GROUP <group> <consumer> [COUNT n] [BLOCK ms] [NOACK]
           STREAMS <key> [<key> ...] <id> [<id> ...]

소비자 이름

그룹 안에서 소비자를 고유하게 식별할 이름을 지정합니다. 처음 호출될 때 자동 생성되므로 별도로 만들 필요는 없습니다.

ID 인자의 의미

  • > — 다른 소비자에게 전달되지 않은 새 메시지만 받음 (가장 자주 쓰는 형태)
  • 0 또는 다른 ID — 해당 ID보다 큰 ID 중 이 소비자의 보류 리스트(Pending List)에 있는 메시지 반환
# EmailServiceGroup의 emailService1 소비자가 새 메시지 1개를 가져옴
> XREADGROUP GROUP EmailServiceGroup emailService1 COUNT 1 STREAMS Email >

# 같은 소비자가 자신의 보류 리스트에 있는 메시지를 다시 확인
> XREADGROUP GROUP EmailServiceGroup emailService1 STREAMS Email 0

하나의 그룹으로 여러 Stream 리스닝

여러 Stream을 한 그룹에서 함께 리스닝할 수도 있습니다. BI 서비스가 Email과 Push 두 Stream을 동시에 처리하는 경우가 예시입니다.

> XGROUP CREATE Email BIGroup 0
> XGROUP CREATE Push BIGroup 0 MKSTREAM

> XREADGROUP GROUP BIGroup BI1 COUNT 2 STREAMS Email Push > >

Chapter 06 · Section 6

보류 리스트와 메시지 처리 보장

소비자가 메시지를 받았다고 처리가 끝난 것은 아닙니다. 처리 도중 장애가 나면 그 메시지는 어떻게 될까요. Redis Stream은 보류 리스트(Pending Entries List, PEL)로 이 문제를 다룹니다.

보류 리스트 (PEL)06.6.1

여러 서비스가 메시지 브로커를 통해 데이터를 처리할 때, 예상치 못한 장애로 시스템이 종료되면 이를 인지하고 재처리할 수 있어야 합니다. 메시지 브로커는 각 소비자에게 어떤 메시지를 전달했고, 그 메시지가 정상적으로 처리됐는지 인지하고 있어야 합니다.

Redis Stream은 이를 위해 보류 리스트(Pending Entries List, PEL)를 관리합니다. 소비자 그룹에 속한 소비자가 메시지를 읽으면, 각 소비자별로 읽어간 메시지 리스트가 새로 생성되고, 마지막으로 읽어간 데이터의 ID로 last_delivered_id가 갱신됩니다.

ACK 처리

소비자가 메시지 처리를 완료하면 Stream에 ACK를 보냅니다. Stream은 해당 소비자의 보류 리스트에서 ACK를 받은 메시지를 삭제합니다. 보류 리스트에 남아 있는 메시지는 전달은 됐지만 아직 처리 완료 신호를 받지 못한 메시지입니다.

장애 복구 시나리오

이메일 서비스에 장애가 발생해 재부팅해야 하는 상황에서, Stream의 보류 리스트에 남아 있는 데이터를 먼저 불러와 처리하면 메시지를 놓치지 않고 처리할 수 있습니다. 데이터 처리가 완료된 후 애플리케이션에서 XACK를 주기적으로 전송하는 작업이 필요합니다.

XPENDING / XACK06.6.2

XPENDING은 소비자 그룹의 보류 리스트 상태를 확인하고, XACK는 메시지 처리 완료를 알립니다.

이어서 따라가기 이 섹션의 Try-it은 6.5.4의 XREADGROUP 실행 결과를 그대로 사용합니다. emailService1과 emailService2가 메시지를 하나씩 받은 상태에서 시작합니다. Reset을 눌렀거나 처음부터 시작한다면 6.4.1과 6.5.3, 6.5.4 순서로 다시 실행해 주세요.

Chapter 06 · Section 7

메시지 보장 전략

메시징 시스템은 메시지 전달을 어떻게 보장할 것인가에 대해 세 가지 전략을 가집니다. 애플리케이션의 특성에 따라 적절한 전략을 선택해야 합니다.

세 가지 보장 전략 개요06.7.1
전략ACK 시점메시지 손실중복 처리적합한 상황
at most once처리 전가능없음일부 손실 허용, 빠른 응답 우선
at least once처리 후없음가능멱등한 서비스, Redis 기본 동작
exactly once처리 후 + 중복 체크없음없음정확히 한 번만 처리되어야 하는 경우
Interactive
메시지 보장 전략 — ACK 시점 비교
ACK가 처리 전에 전송됩니다. 빠르지만, ACK 후 소비자가 종료되면 처리되지 않은 메시지를 잃을 수 있습니다.

Redis Stream은 기본적으로 at least once에 해당합니다. ACK 시점과 추가 로직에 따라 다른 전략으로 변형할 수 있습니다.

at most once / at least once06.7.2

at most once — 최대 한 번 전송

소비자가 메시지를 받자마자 실제 처리 전에 먼저 ACK를 보내는 방식입니다. 응답 속도는 빠르지만, ACK를 보낸 뒤 소비자에 문제가 생겨 서비스가 다운되면 실제로 처리하지 못한 데이터는 잃을 수 있습니다. 메시지 일부가 손실되어도 빠른 응답이 더 중요한 경우에 선택합니다.

at least once — 최소 한 번 전송

소비자가 받은 메시지를 모두 처리한 뒤 ACK를 보냅니다. Redis Stream의 기본 동작입니다.

예를 들어 이메일 서버 1이 메시지 129-0과 130-1을 받은 뒤 129-0은 처리했지만 130-1은 처리하지 못한 상태에서 서비스가 재시작됐다고 가정합니다. 보류 리스트에 남아 있는 메시지를 다시 전부 읽어가기 때문에, 이미 처리한 129-0도 다시 처리하게 됩니다.

같은 메시지를 두 번 이상 처리해도 결과가 동일한 멱등(idempotent)한 서비스라면 문제가 되지 않지만, 그렇지 않은 경우 중복 처리가 문제를 일으킬 수 있습니다.

중복 처리 시나리오

ACK 전송이 지연되어 실제로 메시지가 처리됐지만 ACK 전송 전에 소비자가 종료되는 경우에도, 보류 리스트에 처리된 메시지가 남아 있어 이미 처리한 메시지를 한 번 더 처리하게 됩니다. 멱등성이 보장되지 않으면 결제 두 번 처리, 이메일 두 번 발송 같은 문제가 될 수 있습니다.

exactly once06.7.3

모든 메시지가 무조건 한 번씩 전송되는 것을 보장하는 방식입니다. Redis Stream만으로는 직접 지원되지 않으며, 추가 자료구조를 활용해 구현해야 합니다.

Set으로 중복 체크

처리한 메시지의 ID를 Set 자료구조에 저장해 두고, 메시지를 처리하기 전에 이미 처리된 메시지인지 확인하는 패턴입니다.

# 메시지 처리 전 — 이미 처리한 ID인지 확인
> SISMEMBER processed:Email 1659114481311-0

# 처음 보는 ID라면 처리 후 Set에 기록 + XACK
> SADD processed:Email 1659114481311-0
> XACK Email EmailServiceGroup 1659114481311-0

Set 확인과 메시지 처리, ACK 전송을 트랜잭션으로 묶거나 Lua 스크립트로 처리하면 정확성을 높일 수 있습니다. 다만 Set이 무한히 커지는 것을 막기 위해 TTL을 설정하거나 주기적으로 정리하는 작업이 추가로 필요합니다.

비용

exactly once는 가장 안전하지만 추가 자료구조와 처리 단계 때문에 비용이 듭니다. 멱등성을 애플리케이션 레벨에서 보장할 수 있다면 at least once만으로 충분합니다. 결제처럼 한 번만 처리되어야 하는 작업에서 멱등성을 보장하기 어려운 경우에 exactly once를 고려합니다.

Chapter 06 · Section 8

메시지 재할당

소비자에 장애가 발생해 메시지 처리가 멈추면, 그 소비자의 보류 메시지를 다른 소비자가 대신 처리해야 합니다. Redis Stream은 이를 위한 두 가지 명령어와 dead letter 패턴을 제공합니다.

메시지 재할당의 필요성06.8.1

Redis는 소비자에 장애가 날 경우를 대비해 소비자별 보류 리스트를 유지합니다. 하지만 소비자 서버에 장애가 발생해 복구되지 않는다면, 해당 소비자가 처리하던 보류 중인 메시지들은 다른 소비자가 대신 처리해야 합니다.

이때 단순히 다른 소비자가 같은 메시지를 읽어가는 것만으로는 부족합니다. 같은 메시지가 여러 소비자에게 동시에 할당되면 중복 처리가 발생할 수 있어, 소유권을 명시적으로 이전하는 명령어가 필요합니다.

XCLAIM — 수동 재할당06.8.2

XCLAIM은 메시지의 소유권을 다른 소비자에게 할당하는 명령어입니다.

XCLAIM <key> <group> <consumer> <min-idle-time> <ID-1> <ID-2> ... <ID-N>

min-idle-time

min-idle-time(최소 대기 시간)을 지정해야 합니다. 메시지가 보류 상태로 머무른 시간이 최소 대기 시간을 초과한 경우에만 소유권을 변경할 수 있어, 같은 메시지가 두 소비자에게 중복 할당되는 것을 방지합니다.

예를 들어 EmailService3에 문제가 생겨 이 소비자가 처리하던 메시지를 EmailService1과 EmailService2가 가져가려고 하는 경우, 두 소비자가 모두 XCLAIM을 실행하더라도 충돌이 방지됩니다.

EmailService1: XCLAIM Email EmailServiceGroup EmailService3 3600000 1626569498055-0
EmailService2: XCLAIM Email EmailServiceGroup EmailService3 3600000 1626569498055-0

EmailService1의 커맨드가 먼저 실행되면 메시지의 보류 시간이 즉시 0으로 재설정됩니다. EmailService2의 XCLAIM은 최소 대기 시간(3600000ms = 1시간)보다 메시지의 보류 시간이 짧으므로 무시되어 중복 할당이 방지됩니다.

XAUTOCLAIM — 자동 재할당06.8.3

보류 메시지를 일일이 확인하고 재할당하는 작업이 자주 발생하면 번거롭습니다. XAUTOCLAIM은 조건을 만족하는 보류 메시지를 한 번에 가져옵니다.

XAUTOCLAIM <key> <group> <consumer> <min-idle-time> <start> [COUNT count] [JUSTID]

반환값

인덱스의미
1)다음 호출 시 사용할 start ID (남은 보류 메시지가 없으면 0-0)
2)소유권이 이전된 메시지 목록

반환된 다음 ID로 XAUTOCLAIM을 다시 호출하면 이어서 처리할 수 있습니다. 0-0이 반환되면 보류 메시지를 모두 가져왔다는 의미입니다.

> XAUTOCLAIM Email EmailServiceGroup emailService1 0 0
1) "0-0"
2) 1) 1) "1659114481311-0"
      2) 1) "subject"
         2) "first"
         3) "body"
         4) "hello?"

JUSTID 옵션을 주면 메시지 본문 없이 ID만 반환합니다. 본문이 필요 없는 경우 네트워크 부하를 줄일 수 있습니다.

Dead Letter06.8.4

Stream 내 각 메시지는 counter 값을 가집니다. XREADGROUP으로 할당되거나 XCLAIM으로 재할당될 때마다 counter가 1씩 증가합니다.

특정 메시지가 처리에 계속 실패해 여러 소비자에게 반복 할당되면 counter가 계속 증가합니다. 데이터 자체에 문제가 있어 처리되지 않는 메시지일 가능성이 높습니다.

Dead Letter 패턴

counter가 일정 값을 초과한 메시지를 다른 Stream으로 분리해 관리자가 추후 확인하도록 하는 패턴이 일반적입니다. 이렇게 분리된 메시지를 보통 dead letter라고 부릅니다.

Redis가 dead letter 처리를 자체적으로 제공하지는 않습니다. 애플리케이션에서 다음과 같은 흐름으로 직접 구현합니다.

  • 주기적으로 XPENDING으로 보류 메시지의 counter 확인
  • 임계값을 초과한 메시지를 다른 Stream에 XADD
  • 원본 Stream에서 XACK로 보류 리스트에서 제거
  • 관리자가 별도 Stream을 분석해 문제 원인 파악
