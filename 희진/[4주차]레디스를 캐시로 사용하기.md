# 레디스를 캐시로 사용하기

## 캐시란

캐시는 데이터의 원본보다 더 빠르고 효율적으로 액세스할 수 있는 임시 데이터 저장소다. 사용자가 같은 정보를 반복적으로 액세스할 때, 원본이 아닌 캐시에서 데이터를 가져옴으로써 리소스를 줄일 수 있다.

### 캐시 도입 조건

다음 조건을 만족하는 애플리케이션이라면 캐시 도입으로 성능을 효과적으로 개선할 수 있다.

- 원본 데이터 저장소에서 데이터를 찾는 시간이 오래 걸리거나, 매번 계산이 필요하다
- 캐시에서 데이터를 가져오는 것이 원본보다 빠르다
- 캐시에 저장된 데이터는 잘 변하지 않는다
- 캐시에 저장된 데이터는 자주 검색된다

### 캐시 도입 효과

- **응답 속도 개선** — 원본 데이터 저장소에서 데이터를 가져오는 시간을 단축
- **원본 부하 감소** — 원본 DB로의 커넥션 수를 줄여 확장성 확보
- **리소스 절약** — 같은 계산을 반복하지 않아 CPU/메모리 사용량 감소
- **장애 대응** — 원본 DB에 장애가 발생해도 캐시에서 데이터를 가져올 수 있음

## 캐시로서의 레디스

### 왜 레디스인가

관계형 데이터베이스로 캐시를 구성할 수도 있지만, 캐시의 핵심 요건인 **빠른 응답 속도**를 만족시키기는 쉽지 않다. RDBMS는 디스크 기반이기 때문에 데이터를 검색할 때 디스크 접근이 필요하다.

레디스는 다음과 같은 이유로 캐시 저장소로 적합하다.

- **사용이 간단** — 키-값 형태로 저장하므로 데이터 저장과 반환이 직관적
- **다양한 자료 구조 지원** — 애플리케이션에서 사용하던 List, Hash 등을 변환 없이 그대로 저장
- **빠른 응답 속도** — 모든 데이터가 메모리에 있어 평균 읽기/쓰기 1ms 미만, 초당 수백만 건 처리 가능
- **고가용성** — Sentinel과 Cluster를 통해 마스터 장애 시 자동 페일오버
- **수평 확장** — Cluster를 통한 자체 샤딩으로 스케일아웃이 간단

캐시 저장소에 접근할 수 없게 되면 곧바로 서비스 장애로 이어질 수 있기 때문에, 캐시도 일반 데이터 저장소처럼 안정적인 운영 조건을 갖추는 것이 중요하다. 레디스는 이 조건을 자체적으로 충족한다.

## 캐싱 전략

레디스를 캐시로 사용할 때, 어떻게 배치하느냐에 따라 서비스 성능에 큰 영향을 끼친다. 캐싱 전략은 데이터 유형과 액세스 패턴에 따라 달라지므로, 서비스에 맞는 전략 선택이 중요하다.

### 읽기 전략 — Look aside

가장 일반적인 배치 방식이다. 애플리케이션은 찾고자 하는 데이터가 캐시에 있는지 먼저 확인하고, 없으면 DB에서 가져와 캐시에 저장한다.

| 상황           | 동작                                           |
| -------------- | ---------------------------------------------- |
| **Cache hit**  | 캐시에서 데이터 반환                           |
| **Cache miss** | DB에서 데이터 조회 → 캐시에 저장 → 데이터 반환 |

**장점** — 레디스에 장애가 발생해도 DB에서 데이터를 가져올 수 있어 즉시 서비스 장애로 이어지지 않는다.

**단점** — 기존에 레디스를 통한 연결이 많았다면, 장애 시 모든 커넥션이 한꺼번에 DB로 몰려 부하를 발생시킬 수 있다.

**Lazy loading과 캐시 워밍**

Look aside 방식은 캐시에 데이터가 없을 때만 저장되기 때문에 lazy loading이라고도 부른다. 신규 서비스에 처음 레디스를 도입하거나 새 데이터가 DB에만 저장되는 경우, 매번 캐시 미스가 발생해 지연이 누적된다.

이를 방지하기 위해 미리 DB에서 캐시로 데이터를 밀어넣는 작업을 **캐시 워밍**이라고 한다. 예를 들어 공연 예매 서비스에서 신규 공연 정보를 오픈 전에 미리 캐시에 적재해두면, 오픈 직후 캐시 미스로 인한 지연을 피할 수 있다.

### 쓰기 전략과 캐시 일관성

캐시는 DB의 데이터를 복사한 값이므로 원본과 동일한 값을 유지하는 것이 필수다. 데이터가 변경될 때 캐시에 반영되지 않으면 **캐시 불일치**가 발생한다.

쓰기 전략은 크게 세 가지가 있다.

**1. Write through**

DB에 업데이트할 때마다 캐시에도 함께 업데이트하는 방식이다.

- **장점** — 캐시가 항상 최신 데이터를 보유
- **단점** — 매번 두 저장소에 써야 하므로 쓰기 시간 증가. 사용되지 않을 데이터도 캐시에 저장되어 리소스 낭비
- **권장** — 만료 시간을 함께 설정해 사용

**2. Cache invalidation**

DB에 값을 업데이트할 때 캐시에서는 데이터를 삭제하는 전략이다.

- **장점** — 삭제는 신규 저장보다 리소스를 적게 사용. Write through의 단점 보완
- 데이터는 다음에 조회될 때 캐시 미스를 통해 다시 적재됨

**3. Write behind (Write back)**

데이터를 캐시에 먼저 업데이트하고, 일정 건수나 시간 간격으로 비동기 DB 반영하는 방식이다.

- **장점** — 쓰기가 빈번한 서비스에서 DB 디스크 I/O 부하 감소
- **단점** — 캐시 장애 시 DB에 미반영된 데이터 손실 가능성
- **적합한 상황** — 실시간 정확도가 필수가 아닌 데이터 (예: 유튜브 좋아요 수)

## 캐시에서의 데이터 흐름

캐시는 DB의 서브셋이며, 메모리 용량 제약이 있기 때문에 가득 차지 않게 일정량을 유지하면서 새 데이터가 들어오고 기존 데이터가 삭제되도록 관리되어야 한다. 따라서 데이터 저장 시 적절한 TTL을 함께 설정하는 것이 좋다.

### 만료 시간 (TTL)

레디스에서 TTL은 데이터가 얼마나 오래 저장될 것인지를 나타내는 시간 설정이다. 일반적으로 초 단위로 표현되며, 만료 시간이 지난 키는 자동으로 삭제된다.

- `EXPIRE` — 키에 만료 시간 설정 (초)
- `SET ... EX` — 데이터 저장과 동시에 만료 시간 설정
- `TTL` — 남은 만료 시간 확인 (만료 없으면 -1, 키 없으면 -2)
- `PEXPIRE`, `PTTL` — 밀리초 단위 버전

```
> SET a 100
"OK"

> EXPIRE a 60
(integer) 1

> TTL a
(integer) 58
```

`INCR`로 데이터를 조작하거나 `RENAME`으로 키 이름을 바꿔도 만료 시간은 유지된다. 다만 `SET`으로 키를 덮어쓰면 만료 시간이 사라진다.

```
> SET b 100
"OK"

> EXPIRE b 60
(integer) 1

> SET b banana
"OK"

> TTL b
(integer) -1
```

### 키 만료의 두 가지 방식

레디스에서 키가 만료됐다고 즉시 메모리에서 삭제되는 것은 아니다. 두 가지 방식으로 처리된다.

- **Passive 방식** — 클라이언트가 키에 접근할 때 만료 여부를 확인하고 삭제. 단, 접근하지 않는 키는 계속 메모리를 차지한다.
- **Active 방식** — TTL이 있는 키 중 20개를 랜덤 추출 후, 25% 이상이 만료됐다면 다시 20개를 추출하는 과정을 반복한다. 1초에 10번씩 수행된다.

이렇게 즉시 삭제하지 않는 이유는 리소스 절약이지만, 그만큼 메모리를 더 사용할 가능성이 있다. 최악의 경우 전체 메모리의 1/4은 이미 만료된 키일 수 있다.

## 메모리 관리와 maxmemory-policy

만료 시간을 설정하더라도 너무 많은 키가 저장되면 메모리가 가득 찰 수 있다. 이때 어떤 키를 삭제할지 결정하는 것이 `maxmemory-policy` 설정이다.

`maxmemory`는 데이터의 최대 저장 용량을 설정하고, `maxmemory-policy`는 이 용량 초과 시 처리 방식을 결정한다.

### Eviction 정책 정리

| 정책              | 동작                                                | 적용 대상         |
| ----------------- | --------------------------------------------------- | ----------------- |
| `noeviction`      | 메모리 가득 차도 삭제 안 함, 새 데이터 저장 시 에러 | 전체              |
| `volatile-lru`    | LRU 방식으로 삭제                                   | 만료 시간 있는 키 |
| `allkeys-lru`     | LRU 방식으로 삭제                                   | 모든 키           |
| `volatile-lfu`    | LFU 방식으로 삭제                                   | 만료 시간 있는 키 |
| `allkeys-lfu`     | LFU 방식으로 삭제                                   | 모든 키           |
| `volatile-random` | 랜덤 삭제                                           | 만료 시간 있는 키 |
| `allkeys-random`  | 랜덤 삭제                                           | 모든 키           |
| `volatile-ttl`    | 만료 시간이 가장 작은 키부터 삭제                   | 만료 시간 있는 키 |

### 각 정책의 특징

**Noeviction (기본값)**

- 메모리가 가득 차면 새 데이터 저장 시 에러를 반환
- 캐시 용도로는 권장하지 않음 (장애로 이어질 수 있음)
- 데이터 관리를 캐시에 맡기지 않고 애플리케이션이 직접 관리할 때 사용

**LRU (Least Recently Used)**

- 가장 최근에 사용되지 않은 데이터부터 삭제
- 최근에 액세스되지 않은 데이터는 나중에도 액세스될 가능성이 낮다는 가정
- 공식 문서는 캐시 용도로 `allkeys-lru`를 권장
- `volatile-lru`는 만료 시간이 없는 키만 있는 경우 noeviction과 동일하게 동작 (위험)

**LFU (Least Frequently Used)**

- 가장 자주 사용되지 않은 데이터부터 삭제
- 액세스 패턴에 따라 우선순위가 동적으로 변경
- 키가 오랫동안 사용되지 않았더라도 과거에 자주 액세스됐다면 우선순위가 높게 유지

> **참고:** LRU와 LFU 모두 근사 알고리즘으로 구현되어 있다. 정확한 키를 찾는 것이 CPU/메모리 리소스를 불필요하게 소비할 수 있기 때문이다. 일반적으로 noeviction을 사용하지 않는 한, 저장된 데이터는 특정 상황에서 삭제될 수 있다는 가정하에 동작한다.

**Random eviction**

- 임의로 키를 골라 삭제. 알고리즘 사용하지 않아 부하는 적음
- 나중에 사용될 데이터를 삭제할 가능성이 높아 권장하지 않음

**Volatile-ttl**

- 만료 시간이 가장 작은 키, 즉 곧 만료될 키부터 삭제
- 근사 알고리즘 사용

## 캐시 스탬피드

레디스를 캐시로 활용할 때 모든 키에 만료 시간을 설정하는 것이 권장되지만, 대규모 트래픽 환경에서 만료 시간 설정에 따라 **캐시 스탬피드(cache stampede)** 같은 문제가 발생할 수 있다.

### 문제 상황

여러 애플리케이션이 Look aside 방식으로 레디스를 사용 중이라고 가정하자. 특정 키가 만료되는 시점에 여러 서버가 동시에 만료를 인지하면, 모두가 한꺼번에 DB에 접근해 데이터를 읽어온다. 이를 **중복 읽기(duplicate read)**라고 한다.

이후 각 애플리케이션에서 읽어온 데이터를 다시 레디스에 쓰는 작업도 반복된다. 이를 **중복 쓰기(duplicate write)**라고 한다.

정렬이나 카운팅처럼 DB에서 오랜 시간이 걸리는 쿼리를 캐시해둔 경우, 캐시에 데이터가 없으면 여러 애플리케이션이 동시에 무거운 쿼리를 실행하게 된다. 이는 DB 부하로 이어지며, 한 번 발생한 캐시 스탬피드는 **계단식 실패(cascading failure)**를 유발할 수 있다.

### 해결 방법 1 — 적절한 만료 시간 설정

가장 간단한 방법은 만료 시간을 너무 짧지 않게 설정하는 것이다. 반복적으로 사용될 데이터는 저장 시점부터 만료 시간을 충분히 길게 설정한다.

### 해결 방법 2 — 선계산

키가 실제로 만료되기 전에 미리 값을 갱신해두는 방식이다.

```python
def fetch(key, expiry_gap):
    ttl = redis.ttl(key)
    if ttl - (random() * expiry_gap) > 0:
        return redis.get(key)
    else:
        value = db.fetch(key)
        redis.set(value, KEY_TTL)
        return value
```

`hello` 키의 만료 시간이 10초일 때, `random()`이 6을 반환하면 `10 - (6 * 2) < 0`이 되어 만료 전에 미리 DB에서 데이터를 가져와 갱신한다. 단순 GET보다는 리소스를 더 사용하지만, 캐시 스탬피드를 줄여 전체 성능을 향상시킬 수 있다. `expiry_gap` 값 설정이 중요하며, 잘못 설정하면 오히려 불필요한 작업이 늘어 성능을 저하시킬 수 있다.

### 해결 방법 3 — PER 알고리즘

2015년에 연구된 **PER(Probabilistic Early Recomputation)** 알고리즘은 캐시 값이 만료되기 전에 언제 DB에 접근해 값을 읽어올지 최적으로 계산하는 방식이다. 핵심 수식은 다음과 같다.

```
currentTime - (timeToCompute * beta * log(rand())) > expiry
```

- `currentTime` — 현재 남은 만료 시간
- `timeToCompute` — 캐시된 값을 다시 계산하는 데 걸리는 시간
- `beta` — 기본적으로 1.0보다 큰 값으로 설정 가능
- `rand()` — 0~1 사이의 랜덤 값
- `expiry` — 키를 재설정할 때 새로 넣을 만료 시간

만료 시간이 가까워질수록 조건이 참이 될 확률이 높아진다. true를 반환하는 애플리케이션은 DB로 이동해 데이터를 다시 계산한다. 데이터를 가져오는 과정에서 GET 대신 이 함수를 사용하면 캐시 스탬피드를 줄이고 성능을 최적화할 수 있다.

## 세션 스토어로서의 레디스

### 세션이란

세션은 서비스를 사용하는 클라이언트의 상태 정보를 의미한다. 누가 로그인되어 있는지, 어떤 활동을 하고 있는지 저장하며, 유저가 서비스를 떠나면 세션 스토어에서 정보를 삭제한다.

예를 들어 쇼핑몰에서 장바구니에 담긴 물건이나 최근 본 아이템 정보를 세션에 저장하면, 사용자가 로그인한 동안 해당 정보가 유지된다. 사용자 행동 분석 데이터로도 활용할 수 있다.

세션 스토어로 레디스가 적합한 이유는 다음과 같다.

- 빠른 응답 속도 (계속 읽고 쓰기 때문에 필수)
- 키-값 형식으로 사용이 간단
- String, Set, Hash 등 다양한 자료 구조 제공

### 세션 스토어의 발전 흐름

세션 스토어가 왜 필요한지, 그리고 왜 레디스가 적합한지를 단계별로 살펴보면 다음과 같다.

**1단계 — 단일 서버 (세션 스토어 불필요)**

서비스 초창기나 프로토타입에서는 별도의 세션 스토어가 필요 없다. 각 웹 서버가 자체적으로 세션을 관리할 수 있다.

**2단계 — 다중 서버 + Sticky Session**

서비스가 확장되어 웹 서버가 여러 대로 늘어나면, 트래픽을 분산할 수 있다. 다만 각 웹 서버가 세션 스토어를 따로 관리하면 유저는 자신의 세션 정보가 있는 특정 웹 서버에 종속되어야 한다. 이를 **sticky session**이라고 한다.

문제점:

- 특정 웹 서버에 트래픽이 집중되어도 다른 서버로 분산할 수 없다
- 트래픽 분산이라는 본래 목적을 달성하지 못한다

**3단계 — All-to-all (세션 복제)**

유저의 세션 정보를 모든 웹 서버에 복제해서 저장하는 방식이다.

문제점:

- 한 유저는 한 번에 하나의 서버만 사용하므로 다른 서버에 저장된 세션은 무의미
- 불필요한 저장 공간 차지
- 데이터 복제 과정에서 네트워크 트래픽 발생

**4단계 — DB를 세션 스토어로**

DB를 세션 스토어로 사용하는 방법도 고려할 수 있다. 다만 세션은 활성화된 동안 매우 활발하게 액세스되기 때문에, DB의 응답 속도가 느려지면 곧바로 클라이언트 응답 저하로 이어진다. 유저가 많아질수록 DB가 전체 응답 속도를 저하시키는 요인이 된다.

**5단계 — 레디스를 세션 스토어로**

레디스를 세션 스토어로 사용하면 모든 이슈가 해결된다.

- 유저는 어떤 웹 서버에 연결되어도 동일한 세션 데이터 조회 가능
- 트래픽을 효율적으로 분산
- 데이터 일관성 고려 불필요
- RDBMS보다 훨씬 빠르고 접근이 간편

### 세션 스토어로 Hash 활용

레디스의 Hash 자료 구조는 세션 데이터를 저장하기에 알맞다. `usersession:1`, `usersession:2` 같은 키에 Name, ID, Hits, Lastpage 등의 필드-값을 저장한다.

```
> HMSET usersession:1 Name Garimoo IP 10:20:104:30 Hits 1
OK

> HINCRBY usersession:1 Hits 1
(integer) 2
```

레디스는 키-값 저장소이고 세션 또한 ID로 데이터를 저장하기 때문에, 변환 없이 데이터를 그대로 저장할 수 있다.

## 캐시와 세션의 차이

캐시와 세션은 비슷해 보이지만 데이터를 읽고 쓰는 패턴에서 차이가 있다.

| 구분                 | 캐시                          | 세션                                         |
| -------------------- | ----------------------------- | -------------------------------------------- |
| **데이터 위치**      | DB의 서브셋 (DB에도 존재)     | 세션 스토어에만 존재 (유일할 수 있음)        |
| **데이터 유실 영향** | DB에서 복구 가능              | 데이터 손실 가능성                           |
| **공유 범위**        | 여러 애플리케이션이 함께 사용 | 특정 사용자 ID에 한해 유효, 사용자 간 공유 X |
| **운영 신중도**      | 상대적으로 자유               | 더 신중한 운영 필요                          |

### 캐시의 데이터 흐름

캐시에 저장된 데이터는 모두 DB에 저장되어 있다. 즉, 캐시는 DB의 완벽한 서브셋으로 동작한다. 캐시 내부의 데이터가 유실되더라도 해당 데이터는 DB에서 찾을 수 있다.

캐시에 저장된 데이터는 여러 애플리케이션이 함께 사용할수록 효율적이다.

### 세션의 데이터 흐름

일반적인 세션 스토어에서는 유저가 로그인하면 세션 데이터가 세션 스토어에 저장된다. 유저가 로그인해 있는 동안, 즉 세션이 활성화되어 있는 동안에는 애플리케이션이 유저의 데이터를 DB가 아닌 세션 스토어에만 저장한다.

예를 들어 최근 봤던 아이템이나 장바구니에 담긴 아이템은 세션 스토어에만 담겨 있다. 유저가 로그아웃할 때 세션이 종료되며, 이때 데이터의 종류에 따라 DB에 영구적으로 보관할지 삭제할지가 결정된다.

- 최근 봤던 상품 리스트 — 휘발 가능
- 장바구니에 담아 놨던 상품 — DB에 저장해 다음 로그인 시 확인 가능

세션 스토어가 가진 데이터는 유일할 수 있기 때문에, 장애 발생 시 내부 데이터가 손실될 가능성이 있다. 따라서 레디스를 세션 스토어로 활용할 때는 캐시로 사용할 때보다 더 신중한 운영이 필요하다.
