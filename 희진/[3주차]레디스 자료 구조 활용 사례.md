# 레디스 자료 구조 활용 사례

## Sorted Set을 이용한 실시간 리더보드

### 상황

게임이나 학습 애플리케이션 등에서 사용자의 점수를 기반으로 실시간 순위표를 제공해야 한다. 리더보드는 두 가지 유형이 있다.

- **절대적 리더보드** — 전체 사용자를 정렬해 상위권 목록을 보여준다.
- **상대적 리더보드** — 특정 사용자 기준으로 인접한 사용자들의 순위를 보여주거나, 사용자가 속한 그룹 내의 순위를 보여준다.

리더보드의 요구사항은 까다롭다. 사용자가 늘어날수록 가공할 데이터가 배로 늘어나고, 점수 변경은 실시간으로 반영되어야 하며, 그룹별·기간별로 다양한 통계가 필요하다.

### 왜 Sorted Set인가

관계형 데이터베이스로 이 기능을 구현하면 점수 정렬 자체가 큰 부하가 될 수 있다. 사용자가 늘어날수록 처리 시간도 길어진다. 반면 Sorted Set은 데이터가 저장될 때부터 스코어 순으로 정렬되어 들어가기 때문에, 매번 정렬을 다시 할 필요가 없다.

### 핵심 커맨드

**데이터 저장 — ZADD**

```
> ZADD daily-score:220817 28 player:286
(integer) 1

> ZADD daily-score:220817 400 player:234 357 player:24 199 player:143
(integer) 3
```

**조회 — ZRANGE / ZREVRANGE**

`ZRANGE`는 스코어 오름차순으로, `ZREVRANGE`는 내림차순으로 반환한다. 상위 3명만 보고 싶다면 다음과 같이 사용할 수 있다.

```
> ZREVRANGE daily-score:220817 0 2 WITHSCORES
1) "player:234"
2) "400"
3) "player:24"
4) "357"
5) "player:143"
6) "199"
```

**스코어 업데이트 — ZADD / ZINCRBY**

같은 아이템을 다른 스코어로 저장하면 스코어만 업데이트되고 자동 재정렬된다. 직접 값을 지정하지 않고 `ZINCRBY`로 증감 연산도 가능하다.

```
> ZINCRBY daily-score:220817 100 player:24
"457"
```

**랭킹 합산 — ZUNIONSTORE**

주간 리더보드처럼 여러 일자의 점수를 합산하려면 `ZUNIONSTORE`를 사용한다.

```
> ZUNIONSTORE weekly-score:2208-3 3 daily-score:220815 daily-score:220816 daily-score:220817
(integer) 4
```

`WEIGHTS` 옵션으로 가중치를 줄 수도 있다. 8월 16일에 점수 두 배 이벤트가 있었다면 다음과 같이 처리할 수 있다.

```
> ZUNIONSTORE weekly-score:2208-3 3 daily-score:220815 daily-score:220816 daily-score:220817 WEIGHTS 1 2 1
(integer) 4
```

각 일자별 스코어에 1, 2, 1을 곱한 값으로 합산된다.

## Sorted Set을 이용한 최근 검색 기록

### 상황

쇼핑몰에서 사용자별로 최근 검색한 키워드를 5개까지 보여주는 기능을 추가하려고 한다. 요구사항은 다음과 같다.

- 사용자별로 다른 키워드를 노출
- 검색 내역은 중복 제거
- 가장 최근 검색한 5개의 키워드만 노출

### 왜 Sorted Set인가

관계형 데이터베이스로 구현하면 같은 키워드를 검색했는지 매번 확인하고 업데이트해야 하며, 오래된 데이터를 삭제하는 배치 작업도 필요하다. 데이터를 가져올 때마다 정렬해야 하므로 사용자와 검색 기록이 늘어날수록 부담이 커진다.

Sorted Set은 중복을 자동으로 제거하고, 시간을 스코어로 사용하면 검색 시점 순으로 정렬할 수 있다.

### 핵심 커맨드

**데이터 저장 — ZADD**

검색 시간을 스코어로 사용해 저장한다.

```
> ZADD search-keyword:123 20221106143501 코듀로이
(integer) 1
```

**최근 5개 조회 — ZREVRANGE**

```
> ZREVRANGE search-keyword:123 0 4 WITHSCORES
```

**중복 처리**

같은 키워드를 다시 검색해도 별도의 중복 체크가 필요 없다. 동일 아이템에 대해 `ZADD`를 다시 호출하면 스코어만 업데이트되어 자동 재정렬된다.

**5개 초과 데이터 삭제 — ZREMRANGEBYRANK**

음수 인덱스를 활용하면 매번 아이템 개수를 확인하지 않고도 5개 이상 저장되지 않도록 강제할 수 있다. 음수 인덱스는 마지막 값이 -1이고 역순으로 증가하는 값인데, 데이터가 6개일 때 가장 오래된 데이터의 음수 인덱스는 -6이다.

```
> ZADD search-keyword:123 20221106152734 버킷햇
(integer) 1

> ZREMRANGEBYRANK search-keyword:123 -6 -6
(integer) 1
```

`ZADD` 직후 항상 -6번째 인덱스를 삭제하는 로직을 추가하면, 6개 이상이 되는 순간 가장 오래된 데이터가 자동으로 제거된다. 아이템이 5개 이하일 때는 -6 인덱스가 존재하지 않으므로 영향을 주지 않는다.

## Set을 이용한 태그 기능

### 상황

블로그 게시물에 태그를 추가하고, 특정 태그를 포함한 게시물 목록을 조회하는 기능이 필요하다.

### 왜 Set인가

관계형 데이터베이스로 태그 기능을 구현하려면 보통 태그 테이블, 태그-게시물 관계 테이블 두 개가 필요하다. "특정 태그들을 모두 포함한 게시물 조회"는 `GROUP BY` + `HAVING` 절로 처리해야 하는데, 데이터가 많아지면 데이터베이스에 부하가 생긴다.

Set은 중복 없이 저장되며, 집합 연산(교집합, 합집합)을 빠르게 수행할 수 있다.

### 핵심 커맨드

**양방향 인덱싱**

게시물 → 태그 방향과 태그 → 게시물 방향을 모두 Set으로 저장한다.

```
> SADD post:47:tags IT REDIS DataStore
(integer) 3

> SADD post:53:tags DataStore IT MySQL
(integer) 3

> SADD tag:DataStore:posts 53
(integer) 1

> SADD tag:IT:posts 53
(integer) 1
```

**특정 태그를 가진 게시물 조회 — SMEMBERS**

```
> SMEMBERS tag:IT:posts
1) "22"
2) "47"
3) "53"
```

**여러 태그를 모두 포함한 게시물 조회 — SINTER**

```
> SINTER tag:IT:posts tag:DataStore:posts
1) "47"
2) "53"
```

IT와 DataStore 태그를 모두 가진 게시물만 반환된다.

## 랜덤 데이터 추출

### 상황

게임에서 유저를 랜덤 매핑하거나, 이벤트 응모자 중 당첨자를 추첨하거나, 가챠에서 아이템을 뽑는 기능을 구현해야 한다.

### 왜 레디스인가

관계형 데이터베이스에서는 보통 `ORDER BY RAND()`를 사용하는데, 조건에 맞는 모든 행을 읽어 임시 테이블에 정렬한 뒤 추출한다. 데이터가 1만 건 이상이면 부하가 크다.

레디스는 `O(1)`의 시간 복잡도로 랜덤 데이터를 추출할 수 있다.

### 핵심 커맨드

자료 구조별로 랜덤 추출 커맨드가 제공된다.

- `RANDOMKEY` — 전체 키 중 무작위로 하나 반환
- `HRANDFIELD` — Hash에서 랜덤 필드 추출
- `SRANDMEMBER` — Set에서 랜덤 멤버 추출
- `ZRANDMEMBER` — Sorted Set에서 랜덤 멤버 추출

```
> HRANDFIELD user:hash
"id:4615"

> HRANDFIELD user:hash 1 WITHVALUES
1) "id:4615"
2) "Jinnji"
```

**COUNT 옵션의 양수/음수 차이**

- 양수 — 중복되지 않는 랜덤 데이터 반환
- 음수 — 중복될 수 있는 랜덤 데이터 반환

```
> HRANDFIELD user:hash 2
1) "id:4615"
2) "id:134"

> HRANDFIELD user:hash -2
1) "id:134"
2) "id:134"
```

`SRANDMEMBER`, `ZRANDMEMBER`도 동일한 방식으로 동작한다.

## 레디스의 다양한 카운팅 방법

### 좋아요 처리하기

**상황** — 뉴스 댓글에 좋아요를 누르는 기능을 구현한다. 트래픽이 많은 사이트라면 댓글 하나에 좋아요가 초당 수만 건 발생할 수 있다. 한 명의 유저는 같은 댓글에 한 번만 좋아요를 누를 수 있어야 하므로, 단순 개수가 아니라 누가 좋아요를 눌렀는지도 추적해야 한다.

**해결** — Set을 활용하면 중복 없이 유저 ID를 저장할 수 있다.

```
> SADD comment-like:12554 967
(integer) 1

> SCARD comment-like:12554
(integer) 3
```

`SCARD`로 좋아요 개수를 빠르게 확인할 수 있다.

### 읽지 않은 메시지 수 카운팅

**상황** — 채팅 애플리케이션에서 사용자가 속한 채널별로 읽지 않은 메시지 수를 관리한다. 메시지가 도착할 때마다 관계형 데이터베이스를 업데이트하면 부하가 크기 때문에, 레디스에 일시 저장한 뒤 필요 시점에 한꺼번에 동기화하는 방식을 사용한다.

**해결** — Hash 자료 구조로 사용자별 채널별 카운트를 관리한다. `HINCRBY`로 증감 연산을 수행한다.

```
> HINCRBY user:234 channel:4234 1
(integer) 1
```

메시지가 삭제되었다면 음수값을 입력해 감소시킬 수 있다.

```
> HINCRBY user:234 channel:3435 -1
(integer) 26
```

Hash 구조는 사용자를 식별하고 해당 채널에 접근해 데이터를 업데이트하는 작업이 직관적이고, 객체 단위 카운트를 효율적으로 관리할 수 있다.

### DAU 구하기

**상황** — 하루 동안 서비스를 방문한 고유 사용자 수(DAU, Daily Active User)를 실시간으로 측정한다. 하루 1,000만 명이 방문하는 서비스라면, 유저 ID를 Set에 저장하는 방식은 메모리도 많이 쓰고 키 하나에 너무 많은 아이템이 쌓여 성능이 저하될 수 있다. 일반적으로 키 하나당 200~300만 개 이하의 아이템을 권장한다.

**해결** — 비트맵을 활용하면 메모리 사용량을 획기적으로 줄일 수 있다. 사용자 ID를 비트의 오프셋으로 사용하면, 1,000만 명을 약 1.2MB로 표현할 수 있다.

```
> SETBIT uv:20221106 14 1
(integer) 0

> BITCOUNT uv:20221106
(integer) 3
```

`SETBIT`으로 접속한 유저의 ID 위치를 1로 설정하고, `BITCOUNT`로 1로 설정된 비트 개수, 즉 방문 유저 수를 확인할 수 있다.

**비트 연산 — BITOP**

`BITOP`을 사용하면 AND, OR, XOR, NOT 연산을 레디스 서버에서 직접 수행할 수 있다. 출석 이벤트로 11월 1일~3일 동안 매일 방문한 유저를 구하려면 다음과 같이 사용한다.

```
> BITOP AND event:202211 uv:20221101 uv:20221102 uv:20221103
(integer) 2
```

3일 동안 모두 출석한 유저들이 새 비트맵 `event:202211`에 저장된다.

## HyperLogLog를 이용한 애플리케이션 미터링

### 상황

클라우드 서비스에서 사용자별 API 호출 횟수를 측정해야 한다. 1초에 100개씩 로그가 쌓이는 서버가 여러 대 있다면, 한 달이면 수억 건의 로그가 쌓일 수 있다. 이 데이터를 모두 저장하면서 고유 호출 수를 카운팅하는 것은 큰 부하가 된다.

### 왜 HyperLogLog인가

다음 조건을 만족한다면 HyperLogLog가 적합하다.

- 집합 내 유일한 데이터의 개수를 카운팅해야 한다
- 1% 미만의 오차는 허용 가능하다
- 카운팅에 사용한 정확한 데이터를 다시 확인하지 않아도 된다

Set은 중복을 피하기 위해 모든 데이터를 기억해야 하므로 메모리 사용량이 데이터 개수에 비례한다. HyperLogLog는 입력된 데이터를 변환해 처리하기 때문에, 저장 공간이 12KB로 고정된다.

### 핵심 커맨드

**저장 및 카운팅 — PFADD / PFCOUNT**

```
> PFADD 202211:user:245 49483
(integer) 1

> PFADD 202211:user:245 32714
(integer) 1

> PFADD 202211:user:245 49483
(integer) 1

> PFCOUNT 202211:user:245
(integer) 2
```

같은 데이터를 두 번 저장해도 고유 개수만 카운팅된다.

**여러 HyperLogLog 합산 — PFMERGE**

분기별, 연도별 합산 데이터를 간편하게 계산할 수 있다.

```
> PFMERGE 2022:user:245 202211:user:245 202212:user:245
"OK"

> PFCOUNT 2022:user:245
(integer) 7
```

## Geospatial Index를 이용한 위치 기반 애플리케이션

### 상황

지도 애플리케이션이나 맞춤 광고 서비스에서 사용자의 위치 데이터를 처리해야 한다. 위치 데이터는 경도와 위도 좌표 쌍으로 표현되며, 다음과 같은 요구사항이 있다.

- 사용자의 현재 위치 파악
- 사용자 이동에 따른 실시간 위치 업데이트
- 사용자 위치 기준 근처 장소 검색

사용자가 늘어날수록 데이터는 몇십 배로 증가하고, 위치 데이터끼리의 연산은 처리가 까다롭다.

### 왜 레디스인가

관계형 데이터베이스는 위치 데이터를 단순히 저장할 뿐, 데이터 가공은 외부에서 처리해야 한다. 레디스는 데이터 저장과 실시간 위치 연산을 함께 수행할 수 있어, 네트워크 트래픽과 애플리케이션 코드의 복잡성을 줄일 수 있다.

또한 다른 기능과 조합하기 쉽다. 예를 들어 Geo Set과 Pub/Sub을 함께 사용하면, 특정 위치에서 이벤트가 발생할 때 그 지역 근처 사용자에게 실시간 알림을 보낼 수 있다.

### 핵심 커맨드

**저장 — GEOADD**

```
> GEOADD restaurant 50.192837912837 14.912837912837 ukalendu
(integer) 1
```

`GEOADD <key> 경도 위도 멤버` 순서로 저장한다. 같은 멤버에 새 좌표를 입력하면 위치가 업데이트된다.

**조회 — GEOPOS**

```
> GEOPOS restaurant ukalendu
1) 1) "50.192837912837"
   2) "14.912837912837"
```

**근처 검색 — GEOSEARCH**

특정 위치 기준으로 일정 거리 내의 데이터를 검색할 수 있다.

```
> GEOSEARCH restaurant FROMLONLAT 50.192837912837 14.912837912837 BYRADIUS 1 KM
1) "ukalendu"
```

옵션 정리:

- `FROMLONLAT` — 직접 경도/위도를 지정해 검색
- `FROMMEMBER` — 같은 데이터셋 내의 다른 멤버를 기준으로 검색 (위경도 직접 입력 불필요)
- `BYRADIUS` — 반지름 기준 원형 영역 검색
- `BYBOX` — width와 height를 지정한 직사각형 영역 검색

**BYRADIUS와 BYBOX 차이점**

`BYBOX`를 사용할 때 width 4km, height 2km로 설정하면, 기준점을 중심으로 좌우 2km, 상하 1km 범위의 직사각형 영역을 검색한다. `BYRADIUS`처럼 입력값만큼 떨어진 거리가 아니라 절반씩 적용된다는 점에 주의해야 한다.
