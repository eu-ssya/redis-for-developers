# 마이크로서비스 아키텍처와 레디스

## 배경

모놀리틱 아키텍처는 전체 애플리케이션을 하나의 패키지로 개발·배포하는 구조다. 한 모듈을 수정하면 전체를 재배포해야 하고, 한 곳의 장애가 전체 장애로 이어지며, 특정 서비스만 따로 확장하기 어렵다.

마이크로서비스 아키텍처는 기능별로 분리된 서비스가 독립적으로 동작하는 구조다. 서비스별로 배포, 확장, 기술 스택 선택이 독립적이다. 마이크로서비스 아키텍처에서는 각 서비스가 독립된 데이터 저장소를 갖는 방향을 지향하며, 비즈니스 특성에 따라 RDBMS 또는 NoSQL을 선택할 수 있다.

NoSQL은 Not Only SQL의 약자로, 관계형 데이터베이스와 다른 방식으로 데이터를 저장하는 저장소를 가리킨다. 저장 방식에 따라 그래프, 칼럼, 문서, 키-값 유형으로 나뉜다. Redis는 이 중 키-값 유형에 해당한다.

---

## Redis가 왜 빠른가 — 인메모리 구조

데이터베이스는 데이터를 어디에 두고 처리하느냐에 따라 두 가지로 나뉜다.

- **온디스크(Disk-Based) DB** — 데이터를 최종적으로 디스크에 두고, 필요할 때 메모리로 읽어서 처리한다.
- **인메모리(In-Memory) DB** — 데이터를 메모리에 두고 바로 처리한다.

대부분의 데이터베이스는 온디스크 형태다. 자주 사용되는 데이터는 캐싱되어 메모리에 올라와 있기도 하지만, 그렇지 않은 데이터를 찾을 때에는 디스크에서 직접 읽어야 한다. 디스크의 데이터를 페이지 단위로 메모리에 올린 뒤 검색하고, 원하는 데이터가 없으면 다시 다른 페이지를 가져오는 과정을 반복한다. 디스크 접근 빈도가 증가할수록 성능은 저하된다.

| 저장 매체    | 접근 속도 |
| ------------ | --------- |
| RAM (메모리) | ~100ns    |
| SSD          | ~100μs    |
| HDD          | ~10ms     |

RAM은 SSD보다 약 1,000배, HDD보다 약 100,000배 빠르다.

Redis는 모든 데이터가 메모리에서 관리되기 때문에 일반적인 읽기/쓰기 과정에서 디스크 접근이 필요 없다. 이것이 Redis가 빠른 가장 핵심적인 이유다.

다만 인메모리이기 때문에 서버가 종료되면 데이터가 사라질 수 있다. 이를 보완하기 위해 AOF와 RDB 방식을 제공한다.

## 싱글스레드인데 왜 성능이 나오는가

Redis는 클라이언트의 커맨드를 처리하는 부분이 싱글스레드로 동작한다.

싱글스레드라면 성능이 떨어질 것 같지만, Redis에서는 오히려 이 구조가 유리하게 작용한다.

Redis는 이벤트 루프 기반의 I/O 멀티플렉싱(Multiplexing)을 사용한다. 하나의 스레드가 여러 클라이언트의 연결을 동시에 감시하다가, 요청이 들어온 순서대로 커맨드를 처리한다. 대부분의 Redis 커맨드는 메모리에서 단순한 자료구조 연산을 수행하기 때문에 처리 시간이 매우 짧다. 짧은 커맨드들을 하나의 스레드에서 빠르게 순차 처리하므로 높은 처리량을 낼 수 있다.

싱글스레드이므로 멀티스레드 환경에서 필요한 Lock이나 Context Switching 비용이 발생하지 않는다. 동기화 오버헤드가 없어 단순하고 예측 가능한 성능을 제공한다. 최소 하나의 코어만 있어도 좋은 성능을 낼 수 있다.

## 오래 걸리는 커맨드와 운영상 주의점

싱글스레드 구조이기 때문에 주의할 점도 있다. 한 사용자가 오래 걸리는 커맨드를 수행하면, 다른 사용자는 그 커맨드가 완료될 때까지 대기할 수밖에 없다.

대부분의 커맨드는 메모리에서 동작하므로 빠르지만, O(N) 시간복잡도를 가진 커맨드는 주의가 필요하다.

| 커맨드        | 설명                     | 위험 요인                    |
| ------------- | ------------------------ | ---------------------------- |
| `KEYS *`      | 모든 키 검색             | 키가 많을수록 전체 응답 지연 |
| `SMEMBERS`    | Set의 모든 멤버 반환     | 멤버 수가 많으면 지연        |
| `HGETALL`     | Hash의 모든 필드·값 반환 | 필드 수가 많으면 지연        |
| `LRANGE 0 -1` | List의 전체 요소 반환    | List가 길면 지연             |

대안으로 `SCAN`, `SSCAN`, `HSCAN`과 같은 커서(Cursor) 기반 커맨드를 사용할 수 있다. 한 번에 전체를 가져오는 대신, 현재 위치를 기억해두고 조금씩 나누어 가져오는 방식이기 때문에 전체 서비스에 미치는 영향을 최소화할 수 있다.

이런 커맨드의 특성을 모르고 운영 환경에서 사용하면 전체 서비스 지연으로 이어질 수 있기 때문에, 시간복잡도가 높은 커맨드는 미리 파악해두는 것이 중요하다.

## 영속성 — AOF와 RDB

Redis는 인메모리 기반이지만, AOF와 RDB로 디스크에 데이터를 저장할 수 있다.

- **RDB** — 특정 시점의 메모리 상태를 스냅샷으로 저장한다. 파일이 compact하고 복구가 빠르다. 다만 마지막 스냅샷 이후 데이터는 유실 가능성이 있다.
- **AOF(Append Only File)** — 모든 쓰기 명령을 로그로 순서대로 기록한다. 더 높은 내구성을 기대할 수 있지만, 파일이 커지고 rewrite 관리가 필요하다.

실무에서는 두 방식의 trade-off를 고려하여 함께 사용하는 경우가 많다.

## Redis의 활용 범위

### 다양한 자료 구조

Redis는 키-값 저장소이지만, 값에 단순한 String만 저장할 수 있는 것은 아니다. Hash, Set, Sorted Set, List, Stream 등 다양한 자료 구조를 지원한다. 프로그래밍 언어의 기본 자료 구조와 밀접하기 때문에 추가적인 데이터 가공 없이 애플리케이션에서 바로 사용할 수 있다.

기존 RDBMS에서는 테이블의 행과 열 구조와 프로그래밍 언어의 객체/자료구조 간에 형태 차이가 있어, 데이터를 변환하는 과정이 필요했다. 이를 Impedance Mismatch(임피던스 불일치)라고 한다. Redis는 내장된 자료 구조가 프로그래밍 언어의 자료 구조와 유사하기 때문에 이 불일치가 적다.

> 참고) [Redis 기본 자료구조 (String, List, Set, Hash, Sorted Set)](https://hoojjang.tistory.com/49)

### 고가용성과 확장성

- **복제(Replication)** — 마스터 노드의 데이터를 리플리카 노드에 복사하는 구조다. 읽기 분산과 장애 대응의 기반이 된다.
- **Sentinel** — Redis 인스턴스를 모니터링하는 별도 프로세스다. 마스터에 장애가 발생하면 이를 탐지해 리플리카를 마스터로 자동 승격(Failover)시킨다.
- **Cluster** — 여러 Redis 노드를 하나의 클러스터로 묶어 운영하는 구조다. 16384개의 해시 슬롯(Hash Slot)을 기반으로 데이터가 자동 샤딩되어 수평 확장이 가능하다.

### 메시지 브로커

Redis는 데이터 저장소뿐 아니라 메시지 브로커로도 활용할 수 있다.

- **Pub/Sub** — 발행자(Publisher)가 메시지를 보내면 구독자(Subscriber)가 실시간으로 받는 방식이다. 메시지는 전달된 뒤 저장되지 않고 삭제되는 Fire and Forget(보내고 잊기) 구조다.
- **List** — Redis의 List 자료구조를 메시징 큐처럼 활용할 수 있다. push/pop이 빠르고, 데이터가 들어올 때까지 대기하는 블로킹 명령을 지원한다.
- **Stream** — Apache Kafka에서 영감을 받은 자료구조다. Pub/Sub과 달리 메시지가 소비되어도 보존되며, 여러 소비자가 메시지를 나누어 처리할 수 있는 Consumer Group을 지원한다.
