# 개발자를 위한 레디스 — 5장 정리 · 캐시와 세션 스토어

## 도입 — 캐시로서의 레디스

레디스가 캐시 저장소로 많이 활용되는 데는 이유가 있다.

- **키-값 형태로 저장**하므로 데이터를 저장하고 반환하는 것이 간단
- **인메모리 데이터 저장소**이므로 검색·반환이 빠름 (디스크 I/O 없음)
- **자체 고가용성 기능**을 가짐 - 센티널/클러스터로 마스터 장애 시 자동 failover
- **스케일 아웃이 쉬움** - 클러스터로 수평 확장이 간단함

캐시 저장소를 새로 도입할 때 레디스가 자주 사용되는 이유는 **캐시, 세션 등** 다양한 역할을 하나의 인스턴스로 처리할 수 있기 때문이다.

### 캐시는 데이터 스토어의 서브셋

캐시는 데이터 스토어가 갖고 있는 데이터 중 사용자가 자주 사용할 만한 데이터를 임시로 저장하는 저장소다. 즉, **DB의 서브셋(subset)** 이다.

레디스는 메모리에 모든 데이터를 저장하므로, 메모리 용량은 서버 스토리지보다 훨씬 적은 양만 보관할 수 있다는 한계가 있다. 따라서 캐시는 **가득 차지 않게 일정량의 여유 공간을 유지**해야 하며, 새로운 데이터가 저장되고 기존 데이터는 삭제될 수 있도록 관리되어야 한다.

이를 위해 캐시로 레디스를 사용할 때는 **데이터를 저장함과 동시에 TTL을 적절하게 지정**하는 것이 좋다.

> 캐시 용량을 무한정 늘리면 인메모리의 장점이 사라진다. 캐시는 항상 DB보다 작아야 한다.

## 읽기 전략 — Look-Aside

가장 보편적인 읽기 캐시 전략이다. 애플리케이션은 찾는 데이터가 먼저 캐시에 있는지 확인한 뒤, 있으면 캐시에서 반환하고(**캐시 히트**), 없으면 DB에서 가져와 캐시에 저장한 후 반환한다(**캐시 미스**).

### 캐시 히트와 미스 흐름

**캐시 히트** : 흐름이 단순하다. DB 접근 없이 즉시 반환한다.

```
App  -> Redis: GET user:42
Redis ->  App: "{name: ...}" 반환
```

**캐시 미스** : DB로 fallback 후 캐시에 적재한다.

```
App  ->  Redis: GET user:42
Redis ->  App:   (nil) — 없음
App  ->  DB:    SELECT * FROM users WHERE id=42
DB   ->  App:   User 데이터
App  ->  Redis: SET user:42 "..." EX 300
Redis ->  App:   OK
```

### RedisTemplate 직접 호출 vs @Cacheable 추상화

같은 Look-Aside 패턴을 두 가지 방식으로 구현할 수 있다.

**방식 1 — RedisTemplate 직접 호출**: 흐름이 코드에 그대로 드러나서 동작 이해가 쉽다. 디버깅 및 예외 처리도 명시적으로 할 수 있다.

```java
public User fetch(String key) {
    // 1. 캐시 조회
    User value = redisTemplate.opsForValue().get(key);

    // 2. 캐시 미스 -> DB 조회 후 캐시에 저장
    if (value == null) {
        value = userRepository.findById(key)
            .orElseThrow(() -> new NoSuchElementException("miss: " + key));
        redisTemplate.opsForValue().set(key, value, CACHE_TTL);
    }
    return value;
}
```

**방식 2 — Spring Cache 추상화 (@Cacheable)**: 어노테이션 한 줄로 방식1과 같은 결과를 얻을 수 있다. 캐시 히트 시 메서드 본문은 실행되지 않는다 - Spring 프록시가 가로채서 DB 호출 자체를 막는다.

```java
@Cacheable(value = "users", key = "#key")
public User fetch(String key) {
    // 이 메서드가 실행됐다 = 캐시 미스다
    // 캐시 히트 시 Spring이 프록시에서 가로채 DB 호출 자체를 차단
    return userRepository.findById(key)
        .orElseThrow(() -> new NoSuchElementException("Not Found: " + key));
}
```

**선택 기준**
- 일반적인 단순 캐싱 -> `@Cacheable`로 간결하게 구현
- 조건부 캐시 미스 시 별도 로직이나 복잡한 키 생성이 필요하면 `RedisTemplate`으로 직접 호출하는 게 좋다.

### Look-Aside의 장점과 함정

**장점 - 캐시 장애에도 서비스는 살아있음**

레디스에 문제가 생겨 접근할 수 없더라도 바로 서비스 장애로 이어지지 않고 **DB에서 데이터를 가져오면 된다**. 캐시가 보조 역할이라는 점이 두드러지는 부분이다.

**함정 - 캐시 장애 시 DB로 부하가 몰림**

하지만 기존에 애플리케이션이 레디스를 통해 데이터를 가져오는 연결이 매우 많았다면, 모든 커넥션이 한꺼번에 원본 DB로 몰려 큰 부하가 발생하게 될 것이다. (캐시 스템피드)

1. 원본 DB의 응답이 느려짐
2. 리소스(커넥션, CPU, I/O)를 과도하게 차지
3. 결국 애플리케이션 전체 성능에 영향

>  **캐시 장애 = DB 장애 '가능성'**
- 평소 캐시에 의존하던 트래픽이 한꺼번에 DB로 쏟아지면 DB도 같이 죽을 수 있다. 이를 방지하려면 **Circuit Breaker** 패턴(예: Resilience4j)을 함께 적용해 일정 비율 이상의 캐시 미스 발생 시 일부 요청을 차단하거나 fallback 응답을 주는 것이 좋다.

### Lazy Loading의 본질과 캐시 워밍

찾고자 하는 데이터가 레디스에 없을 때만 레디스에 데이터가 저장되므로, Look-Aside는 **lazy loading**이라고도 불린다.

#### Cold start 문제

다음 상황을 살펴보자.

- 서비스에 처음으로 레디스를 투입한 직후
- DB에만 새로운 데이터를 저장한 직후
- 레디스 인스턴스를 재시작해 메모리가 비어있는 상태

이 상황에서 애플리케이션은 데이터를 찾기 위해 매번 레디스에 먼저 접근하지만, 그때마다 **캐시 미스**가 일어나 DB에 다시 접근한다. 결국 매 요청마다 두 번의 네트워크 호출이 발생하면서 지연이 누적된다.

#### 해결 — 캐시 워밍 (Cache Warming)

이럴 때는 **미리 DB에서 캐시로 데이터를 밀어넣어주는 작업**인 캐시 워밍을 해주면 된다.

```
Batch -> DB:    SELECT 자주 조회되는 키 N개
DB    -> Batch: 결과 반환
Batch -> Redis: SET key1 ... · SET key2 ... · ...
Redis ->        초기 캐시 적재 완료. 이후 요청은 모두 캐시 히트가 됨.
```

> **실무 팁**
> Spring Batch + `RedisTemplate.executePipelined()`를 사용하면 수만 건의 캐시 워밍을 수 초 안에 완료할 수 있다. 새벽 배포 직전에 배치를 돌려 cold start를 방지하는 것이 흔한 패턴이라고 한다.

## 쓰기 전략과 캐시 일관성

### 일관성 문제 — 모든 쓰기 전략의 출발점

다음 상황을 살펴보자.

- 최초에 `a = 3`이라는 값이 DB와 캐시 모두에 저장됨
- 어떤 사용자가 `a = 28`로 DB만 갱신
- 캐시에는 여전히 `a = 3`이 남아있음

이 상태에서 다른 애플리케이션이 a를 조회하면 캐시에서 먼저 찾으므로 **잘못된 값(3)** 을 반환받게 된다. 즉, 데이터 정합성 문제가 발생한다.

```
초기:    DB: a=3 · Cache: a=3
↓ User -> UPDATE table SET a=28
현재:    DB: a=28 · Cache: a=3 -> 불일치
↓ App-2 -> GET a (캐시에서 조회)
Cache -> a=3 (잘못된 값 반환)
```


### Write-Through

**DB와 캐시를 동시에 갱신해 항상 최신 상태를 유지**하는 전략

#### 동작 방식

DB에 데이터를 갱신할 때마다 매번 캐시에도 데이터를 함께 갱신한다.

```
App  -> DB:    UPDATE a=28
DB   ->        a=28 저장 OK
App  -> Redis: SET a 28 EX 300
Redis ->       a=28 캐시 OK
결과: DB와 캐시가 항상 일치
```

**장점** : 캐시는 항상 최신 데이터를 가지고 있을 수 있음

**단점** : 불필요한 데이터를 캐싱하고 TTL 필요성이 증가함

1. 매번 두 저장소에 쓰기 : 시간 비용 증가
   - 데이터는 매번 2개의 저장소에 저장되어야 하기 때문에 **데이터를 쓸 때마다 시간이 많이 소요**될 수 있다. 쓰기가 잦은 시스템에서는 평균 응답 시간이 늘어난다.
2. 다시 사용되지 않을 데이터까지 캐시에 들어감
   - 캐시에는 **다시 사용될 만한 데이터**가 저장되는 것이 좋다. 그런데 Write-Through 방식은 다시 사용되지 않을 데이터도 캐시에 저장될 수 있어 **리소스 낭비**가 발생할 수 있다.
   - 예를 들어 1년에 한 번 갱신되는 통계 데이터를 매번 캐시에도 저장하는 식이다. 메모리만 차지하고 정작 활용은 안 되게 된다.

이를 해결할 방법은 바로 만료 시간(TTL)을 넣는 것이다.
데이터를 저장할 때, **자주 안 쓰는 데이터는 TTL이 지나면 자동으로 빠지도록 만료 시간을 사용하는 게 좋다**.

```bash
> SET a 28 EX 300   # 5분 후 자동 만료
OK
```

#### 적합한 사용 케이스
- 읽기와 쓰기가 비슷한 빈도로 발생하는 데이터
- 갱신 직후 곧바로 다시 조회될 가능성이 높은 데이터
  - 예: 사용자 프로필 편집 후 즉시 재조회

### Cache Invalidation

Write-Through의 단점을 보완하여, **DB만 갱신하고 캐시에서는 해당 키를 삭제**하는 전략이다.

#### 동작 방식

DB에 값을 업데이트할 때마다 캐시에서는 데이터를 삭제한다. 이후 조회 요청이 오면 캐시 미스가 나면서 자연스럽게 새 값이 캐시로 적재된다.

```
App  -> DB:    UPDATE a=28
DB   ->        a=28 저장 OK
App  -> Redis: DEL a
Redis ->       키 삭제 (1)
─────────
App-2 -> Redis: GET a (다음 조회)
Redis ->        (nil) — 미스
App-2 -> DB:    조회 → 28 → 캐시에 SET
```

#### Write-Through 대비 리소스 효율

> 삭제가 저장보다 가볍다

저장소에서 **특정 데이터를 삭제하는 것이 새로운 데이터를 저장하는 것보다 훨씬 리소스를 적게 사용**하기 때문이다.

- 저장: 직렬화 + 메모리 할당 + (Redis Cluster의 경우) 슬롯 라우팅 + 복제
- 삭제: 키 lookup → 1개 명령어

**다시 사용되지 않을 데이터는 캐시에 안 들어옴**

Write-Through와 달리 **재조회가 일어날 때만** 캐시에 다시 적재된다. 따라서 자주 안 쓰는 데이터는 자연스럽게 캐시에 안 남는다.

| 비교 항목 | Write-Through | Cache Invalidation |
|-----------|---------------|---------------------|
| 쓰기 시 동작 | DB + 캐시 둘 다 SET | DB SET + 캐시 DEL |
| 쓰기 비용 | 높음 (직렬화 + 저장 ×2) | 낮음 (저장 + 삭제) |
| 다시 안 쓸 데이터 | 캐시에 들어감 (낭비) | 캐시에 안 들어감 |
| 일관성 | 강함 (동시 갱신) | 강함 (재조회 시 DB 최신값) |
| 다음 조회 성능 | 즉시 히트 | 한 번은 미스 발생 (성능 비용) |

#### Spring 구현 - @CacheEvict

일반적으로 캐시 무효화 전략이 더 자주 쓰인다고 한다. `@CacheEvict` 어노테이션을 쓰면 쉽게 구현할 수 있다.

```java
@CacheEvict(value = "users", key = "#user.id")
public void update(User user) {
    userRepository.save(user);  // DB 갱신 후 자동으로 캐시 키 삭제
}
```

**Cache Invalidation의 일관성에 관한 미묘한 점**

> Cache Invalidation은 "강한 일관성"으로 분류되지만, 분산 환경에서는 **DB 갱신과 캐시 DEL 사이의 짧은 윈도우**에서 다른 reader가 stale 캐시를 볼 수 있는 race condition이 존재한다. 단일 인스턴스 환경에서는 무시 가능한 수준이고, 분산 환경에서도 대부분의 경우 허용 가능하다. 절대적 일관성이 필요한 도메인(결제·재고 등)은 분산 락이나 더 엄격한 패턴을 함께 써야 한다.

### Write-Behind (Write-Back)

**캐시에만 먼저 쓰고 나중에 비동기로 DB에 반영**하는 고성능 전략

#### 왜 필요한가

DB에 대량의 쓰기 작업이 발생하면, 이는 많은 디스크 I/O를 유발해 성능 저하가 발생할 수 있다. 따라서 먼저 데이터를 빠르게 접근할 수 있는 **캐시(레디스)에 업데이트한 뒤, 이후 건수나 특정 시간 간격을 봐서 비동기적으로 DB에 갱신**하는 것이다.

```
App   -> Redis:  INCR likes:product:42
Redis ->         즉시 응답 (10μs)
App   ->         사용자에게 응답 완료
─────────
Batch:  5분마다 (cron)
Batch -> Redis:  GET likes:product:* 일괄 수집
Batch -> DB:     UPDATE products SET likes=... (bulk)
```

#### 데이터 유실 위험이라는 트레이드오프

**캐시 장애 시 데이터가 유실될 수 있다**

Write-Behind의 가장 큰 단점이다. **캐시에 문제가 생겨 데이터가 날아갈 경우, 최대 지정한 시간(예: 5분) 동안의 데이터가 유실될 위험**을 감수해야 한다.

예를 들어 5분 간격으로 DB에 동기화하는 시스템에서 캐시가 4분 59초 시점에 죽으면, 그 4분 59초 동안 발생한 모든 쓰기가 사라진다.

**적합한 사용 케이스**

저장되는 데이터가 **실시간으로 정확한 데이터가 아니어도 되는 경우**에는 이 방법이 유용할 수 있다.

- 쇼핑몰의 **상품 좋아요 수** : 매번 실시간 집계 불필요
- **조회수 카운터** : 1~2분 지연돼도 사용자 경험에 영향 없음
- **분석/통계용 이벤트 로그** : 약간의 유실이 나도 비즈니스 영향 없음

### 세 전략 한눈에 비교

| 비교 축 | Write-Through | Cache Invalidation | Write-Behind |
|---------|---------------|---------------------|---------------|
| 쓰기 응답 속도 | 느림 (2저장소) | 중간 (DB+삭제) | **매우 빠름** |
| 일관성 | **강함** | **강함** (다음 조회 시 DB 최신) | 약함 (eventual, 지연 후 일치) |
| 유실 위험 | 없음 | 없음 | **있음** (지연 시간만큼) |
| 적합한 데이터 | 읽기·쓰기 균형 | 일반적 | 고빈도 쓰기, 실시간 정확도 불필요 |
| Spring 어노테이션 | `@CachePut` | `@CacheEvict` | 직접 구현 |

## 만료 시간 관리 (TTL)

캐시는 가득 차면 안 되므로 모든 키는 자동 만료되어야 한다. TTL 관련 명령과 보존 규칙을 정확히 이해해야 한다.

### 기본 명령어 — EXPIRE · TTL · SET ex

레디스에서 만료 시간(TTL)은 데이터가 얼마나 오래 저장될 것인지를 나타내는 시간 설정이다. 키에 만료 시간을 설정하면 데이터의 수명을 관리하고 메모리 공간을 효율적으로 사용할 수 있다.

#### 주요 명령어

- `EXPIRE key seconds` : 이미 저장된 키에 만료 시간 부여
- `SET key value EX seconds` : 저장과 동시에 만료 시간 지정
- `TTL key` : 남은 만료 시간 조회
- `PERSIST key` : 만료 시간 제거 (영구 키로 전환)

#### TTL 반환값의 의미

- **양수 (예: 58)** — 남은 만료 시간 (초)
- **−1** — 키는 있지만 만료 시간이 설정되지 않음 (영구)
- **−2** — 키 자체가 존재하지 않음

```bash
> SET a 100
"OK"

> EXPIRE a 60
(integer) 1

> TTL a
(integer) 58     # 2초 흐름
```

---

### TTL 보존 규칙 — 운영의 핵심 함정

**여기는 실무에서 가장 많이 실수할 수 있는 부분이다.** TTL이 어떤 명령어에서는 보존되고, 어떤 명령어에서는 사라지는지 정확히 알아야 한다.

#### 보존되는 경우 — INCR / DECR / RENAME / HSET 등

`INCR`로 데이터를 조작하거나 `RENAME`으로 키 이름을 바꾸더라도 **설정된 만료 시간은 변경되지 않는다**. 키를 "수정"하는 명령은 TTL을 보존한다.

```bash
> SET a 100 EX 60
"OK"

> INCR a
(integer) 101

> TTL a
(integer) 51     # TTL 그대로 유지 (단지 9초 흘렀을 뿐)

> RENAME a apple
"OK"

> TTL apple
(integer) 41     # RENAME 후에도 TTL 보존
```

#### 소멸되는 경우 — SET으로 덮어쓰기

그러나 **기존 키에 새로운 값을 저장해 키를 덮어쓸 때는 이전에 설정한 만료 시간이 없어진다**. SET은 "새로 쓰기"이므로 새 키와 동일하게 취급된다.

```bash
> SET b 100 EX 60
"OK"

> TTL b
(integer) 57

> SET b banana
"OK"             # EX 옵션 없이 덮어씀

> TTL b
(integer) -1    # TTL 사라짐 (영구 키가 됨)
```

#### 안전한 패턴 - KEEPTTL 옵션 (Redis 6.0+)

SET 명령에 `KEEPTTL` 옵션을 주면 기존 TTL을 보존한 채로 값만 갱신할 수 있다.

```bash
> SET k 100 EX 60
"OK"

> SET k 200 KEEPTTL
"OK"

> TTL k
(integer) 47    # 기존 TTL 유지
```

#### 실습 - Refresh Token Rotation 시나리오

이 개념들을 정리한 시나리오의 예로 **Refresh Token Rotation**을 들어보자. 4장에서 다룬 RefreshToken을 다시 생각해 보자.

**Hash로 저장한 경우 (안전)**

```bash
HSET RefreshTokenEntity:1 memberId 1 refreshToken token_v1
EXPIRE RefreshTokenEntity:1 1209600

# Rotation - 토큰 필드만 갱신
HSET RefreshTokenEntity:1 refreshToken token_v2
TTL RefreshTokenEntity:1   # 1209598 (TTL 보존됨)
```

**String으로 저장한 경우 (위험)**

```bash
SET refresh:1 token_v1 EX 1209600

# Rotation — EX 옵션 빠뜨림
SET refresh:1 token_v2
TTL refresh:1   # -1 (영구 저장 & 보안 사고)

# 안전한 방법 — KEEPTTL 또는 명시적 EX
SET refresh:1 token_v2 KEEPTTL
# 또는
SET refresh:1 token_v2 EX 1209600
```

### 만료 키 삭제 방식

레디스에서 **키가 만료되었다고 해도 바로 메모리에서 삭제되는 것은 아니다**. passive와 active, 두 가지 삭제 방식을 섞어서 사용하는 게 실무적으로 좋다.

#### Passive 방식

클라이언트가 키에 접근하고자 할 때 키가 만료됐다면 **그제야 메모리에서 수동적으로 삭제**한다. 사용자가 접근할 때만 수동적으로 삭제되기 때문에 passive 방식의 만료라고 한다.

```
12:00 - SET key1 EX 60
12:01 - TTL 만료 시점
12:01 ~ 13:00 — 메모리에 여전히 존재 (아무도 접근 안 함)
13:00 - App: GET key1
        Redis: 만료 확인 → 삭제 → (nil) 반환
```

**한계**: 이 방식만으로는 키 관리가 어렵다. **사용자가 다시 접근하지 않는 만료된 키**도 있기 때문에, 그런 키는 영원히 메모리에 남게 된다.

> 메모리가 점진적으로 차오르는데도 알아채기 어렵다. `maxmemory` 한계에 도달하면 그제야 큰 문제가 발생한다. 그래서 active 방식이 추가로 필요하다.

#### Active 방식 — 1초 10회 샘플링

passive의 한계를 보완하기 위해, 레디스는 백그라운드에서 능동적으로 만료 키를 정리해야 한다.

**알고리즘**

1. TTL 값이 있는 키 중 **20개를 랜덤하게 뽑음**
2. 20개 중 **만료된 키를 모두 메모리에서 삭제**
3. 만약 **25% 이상의 키가 삭제됐다면** 다시 1번으로 돌아가 20개를 더 뽑음 (만료 키가 많다는 신호)
4. 아니라면 그 20개 집합을 다시 한번 확인

이 과정을 **1초에 10번씩 수행**한다 (Redis의 `hz` 설정값 기본 10).

**왜 랜덤 샘플링인가?**

> 모든 키를 순회하면 메인 스레드를 너무 오래 점유한다. 통계적 샘플링으로 확률을 구현해 메인 스레드 차단을 최소화한 것이다. passive와 active 두 방식을 조합해 메모리 관리 부담을 분산시킨다.

## 메모리 관리 — maxmemory-policy

메모리 한계 도달 시 어떤 키를 삭제할지 결정하는 8가지 정책이다.

### 1. Noeviction

레디스의 키 삭제 기본 정책은 **noeviction (축출하지 않음)** 이다.

#### 동작

레디스에 데이터가 가득 차더라도 **임의로 데이터를 삭제하지 않는다.** 그래서 더 이상 데이터를 저장할 수 없을 때는 에러가 반환될 것이다.

```bash
> SET bigkey verybigvalue
(error) OOM command not allowed when used memory > 'maxmemory'
```

#### 캐시로는 부적합

- 캐시에 데이터를 저장하지 못해 에러가 발생할 경우, 로직에 따라 **장애 상황**으로 이어질 수 있음
- 이런 상황에서는 **관리자가 레디스의 데이터를 직접 지워야 하는** 위험성을 안고 있음
- 새벽 시간 트래픽 폭증 시 자동 회복이 안 됨 -> 운영적으로 부담됨

 **Critical**
- 레디스를 **캐시 용도로 쓴다면** noeviction은 캐시로써 사용한다면 이 정책은 사용하지 말자. 단, 레디스를 **주 데이터 저장소**로 쓴다면(예: 세션 스토어, 메시지 큐) 되려 임의 삭제가 위험하므로 noeviction이 적합하다.

---

### LRU eviction

**LRU = Least Recently Used**. 가장 최근에 사용되지 않은 데이터부터 삭제하는 정책이다.

#### 가정

**최근에 액세스되지 않은 데이터는 나중에도 액세스될 가능성이 낮다**는 가정을 전제로 한다. 캐시는 **나중에 사용될 가능성이 있는 데이터를 유지하는 것을 목표**로 하기 때문에 효율적인 메모리 관리 방법 중 하나다.

#### 두 가지 변형

**volatile-lru** - 만료 시간이 설정된 키에 한해서만 LRU 적용

- 만료 시간이 설정된 키는 언젠가 삭제될 키이므로, 그중 가장 오래 사용되지 않은 것을 삭제
- **임의로 삭제되면 안 되는 영구 키**가 있을 때 적합
- **허점**: 모든 키에 만료 시간이 없다면, 즉 지정 안 해놨다면 noeviction과 동일하게 동작하여 나중에 캐시가 꽉 찰 수 있다.

**allkeys-lru** - 모든 키에 LRU 적용

- 적어도 메모리가 꽉 차서 장애가 발생할 상황은 방지할 수 있음

- 순수 캐시 용도라면 `allkeys-lru`가 가장 안전한 기본값이다. 일부 키를 영구 보존해야 한다면 (예: 설정값) 그것들에만 TTL을 안 걸고 나머지에는 TTL을 걸어 `volatile-lru`를 쓰는 패턴도 활용해볼 수 있겠다.

---

### LFU 계열

**LFU = Least Frequently Used**. 가장 사용이 안 된 데이터부터 삭제하는 정책이다.

#### 가정

**자주 사용되지 않은 데이터는 나중에도 액세스될 가능성이 낮다**는 가정을 전제한다. LRU와 비슷하지만 **접근 빈도**를 보는 점이 다르다.

#### LRU vs LFU

LFU는 LRU와 유사하지만, 키에 접근하는 패턴에 따라 우선순위가 **유동적으로 바뀐다는 점**에서 특정 케이스에서는 LFU가 더 효율적일 수 있다.

예를 들어, 키가 **오랫동안 사용되지 않았더라도 과거에 자주 액세스했던 키라면 나중에도 자주 사용될 수 있다**는 가정 하에 우선순위가 높아지게 됩니다.

| 시나리오 | LRU 판단 | LFU 판단 |
|----------|---------|---------|
| 1시간 전 1번 접근한 keyA<br>방금 1번 접근한 keyB | keyA 삭제 (오래됨) | 둘 다 빈도 1, 거의 동일 |
| 1시간 전 100번 접근한 keyA<br>방금 1번 접근한 keyB | keyA 삭제 (오래됨) | **keyB 삭제** (빈도 낮음) |
| 매일 1번씩만 조회되는 keyA | 최근 조회면 보존 | 상대적으로 빈도 낮아 우선 삭제 후보 |

#### 두 가지 변형

- **volatile-lfu** — 만료 시간이 설정된 키에 한해서만 LFU 적용 (volatile-lru와 동일한 허점 존재)
- **allkeys-lfu** — 모든 키에 LFU 적용

**언제 LFU를 선택하면 좋을까**
> 뉴스 기사 사이트의 **인기 글 캐시**처럼, 일부 키가 매우 자주 조회되고 나머지는 골고루 한두 번 조회되는 long-tail 패턴에서 LFU가 LRU보다 캐시 히트율이 높다. 반면 일반적인 사용 패턴에서는 LRU가 충분하다.

---

### Random 계열

레디스에 저장된 키 중 **하나를 임의로 골라내 삭제**한다.

#### 장점

삭제될 키 값을 계산하지 않아도 된다는 점에서 레디스의 부하를 줄여줄 수 있습니다.

#### 단점

그러나 **랜덤**으로 데이터를 삭제하는 거라, **나중에 사용될 수도 있는 데이터를 삭제할 가능성**이 있다. 나중에 사용될 데이터가 삭제되면, 데이터 저장소에서 다시 데이터를 가져와 캐시에 넣어주는 불필요한 작업이 발생한다.

**왜 권장되지 않는가**
> 레디스는 **근사 알고리즘**을 사용하므로 LFU/LRU의 데이터를 찾는 데 큰 자원을 소모하지 않는다. 부하를 줄인다고 random eviction을 쓰는 것은 보통 잘못된 최적화일 가능성이 높다.

#### 두 가지 변형

- **volatile-random** — 만료 시간이 설정된 키에 한해 랜덤 삭제
- **allkeys-random** — 모든 키에 대해 랜덤 삭제

**근사 알고리즘이란?**
> 완벽하지는 않지만 최적에 가까운 해(근사해)를 빠르게 찾아내는 알고리즘. 레디스는 LRU/LFU 판정을 위해 모든 키를 스캔하는 대신, 작은 샘플(`maxmemory-samples` 기본 5개)을 뽑아 그 안에서 가장 오래된/덜 쓰인 키를 고르는 방식으로 부하를 낮춥니다.

---

### volatile-ttl

**만료 시간이 가장 작은 키를 삭제**한다. 즉, 삭제 예정 시간이 얼마 남지 않은 키를 추출해 해당 키를 미리 삭제하는 옵션이다.

#### 알고리즘 특성

이 알고리즘은 **근사 알고리즘**을 사용하므로, 저장된 모든 키를 스캔하여 만료 시간을 비교하지 않아 간단하게 키를 찾아낼 수 있습니다.

#### 언제 적합한가

- 모든 키에 TTL이 설정되어 있고
- "어차피 곧 만료될 키"를 미리 비워 메모리 압박을 완화하려 할 때

---

### 정책 선택 가이드

| 정책 | 대상 | 삭제 기준 | 주요 사용처 |
|------|------|----------|--------------|
| `noeviction` | - | 삭제 안 함 (에러) | 주 데이터 저장소 |
| `allkeys-lru` | 전체 키 | 가장 오래 미사용 | **일반적인 캐시 (권장)** |
| `volatile-lru` | TTL 있는 키 | 가장 오래 미사용 | 일부 영구 키 보존 |
| `allkeys-lfu` | 전체 키 | 접근 빈도 낮음 | 인기/비인기 편차 큰 데이터 |
| `volatile-lfu` | TTL 있는 키 | 접근 빈도 낮음 | 위 + 영구 키 보존 |
| `allkeys-random` | 전체 키 | 무작위 | 거의 비권장 |
| `volatile-random` | TTL 있는 키 | 무작위 | 거의 비권장 |
| `volatile-ttl` | TTL 있는 키 | 만료 임박 | 모든 키에 TTL이 있을 때 |


## 캐시 스탬피드와 PER 알고리즘

### 캐시 스탬피드란

> **캐시 스탬피드(Cache Stampede)** — 인기 있는 데이터의 캐시가 만료(TTL)될 때, 수많은 동시 요청이 일시에 DB 등 원본 저장소로 몰려 과부하를 일으키는 현상

#### 발생 시나리오

Look-Aside 방식에서 애플리케이션은 레디스에 먼저 데이터가 있는지 질의한 후, 없을 때 DB에서 데이터를 읽어오는 과정을 반복한다.

이때 레디스에서 **특정 키가 만료되는 시점**을 생각해보자. 여러 개의 애플리케이션 서버에서 바라보던 키가 레디스에서 만료되어 삭제된다면, 이 서버들은 한꺼번에 데이터베이스에 접근해 데이터를 읽어오는 과정을 거친다.

```
12:00:00 - popular_key 캐시 (5분 TTL) -> 정상
12:05:00 - popular_key 만료 -> 캐시에서 삭제
─────────
App-1 -> Redis: GET popular_key → nil
App-2 -> Redis: GET popular_key → nil
...
App-N -> Redis: GET popular_key → nil  (전부 미스!)
─────────
App-1..N -> DB: 동일 쿼리 N번 (중복 읽기)
App-1..N -> Redis: SET popular_key ... N번 (중복 쓰기)
```

#### 두 가지 비용

- **중복 읽기** - 여러 서버가 동시에 DB에 동일 쿼리. 무거운 쿼리(JOIN, 집계)일수록 치명적
- **중복 쓰기** - 같은 데이터를 여러 서버가 캐시에 SET (불필요)

**계단식 실패 (Cascading Failure)**
> 일반적으로 DB에서 쿼리하는 데 오랜 시간이 걸리는 작업을 미리 계산해서 캐시에 저장해두기 때문에, 캐시에 데이터가 없는 경우 이 데이터를 다시 DB에 접근해서 계산하는 작업이 필요하다. 더 많은 데이터가 이 현상의 영향을 받게 돼 더 큰 문제로 이어질 수 있어 **계단식 실패** 라고도 불린다.

---

### 완화의 첫 단계

캐시 스탬피드를 줄이려면 **되도록 만료 시간을 너무 짧지 않게, 적절한 만료 시간을 설정**해야 한다.

#### 장수명 키가 적합한 경우

다음 조건에 해당하는 데이터라면 저장 시점부터 만료 시간을 충분히 길게 설정해주는 것이 좋다.

- **여러 애플리케이션에서 한꺼번에 접근해야 하는 데이터** : 인기 콘텐츠, 메인 페이지 데이터 등
- **반복적으로 사용되어야 하는 데이터** : 공통 코드, 설정값, 자주 조회되는 마스터 데이터

#### 너무 짧은 TTL의 함정

1분 TTL을 걸면 매분 스탬피드가 발생할 수 있다. 반대로 12시간 TTL을 걸면 12시간마다 1번만 스탬피드 발생 가능하여 빈도가 약 700배 줄어든다.

**한계점**
> 긴 TTL만으로는 **스탬피드를 막지 못한다**. 만료 빈도만 줄일 뿐이다. 결국 만료 순간이 오면 똑같은 문제가 발생하므로, 이때는 PER 알고리즘을 활용하는 게 좋다.

---

### PER 알고리즘

**PER (Probabilistic Early Recomputation)** 이라 불리는 이 알고리즘을 이용하면 캐시 값이 만료되기 전에 언제 DB에 접근해서 값을 읽어오면 되는지 최적으로 계산할 수 있다.

#### 해결하려는 문제

이 알고리즘이 해결하고자 하는 문제는 **캐시 스탬피드**이고, 이를 막기 위해 **만료 직전부터 확률적으로 DB를 미리 조회해 캐시를 갱신**하는 알고리즘이다. 만료에 가까워질수록 DB에서 다시 가져와 캐시를 갱신할 확률이 높아진다.

#### 핵심 수식

```python
# expiryGap: 만료 임박 구간(초)
if ttl - (random() * expiryGap) > 0:
    return cache.get(key)         # 아직 캐시 사용
else:
    return db.fetch(key)          # 선제적으로 DB 조회 후 갱신
```

random()이 [0.0, 1.0) 범위의 균일 분포를 반환할 때:

- **TTL이 충분히 클 때** (예: 100초 남음, gap=2): `100 - r×2`는 거의 항상 양수 -> 모든 서버가 캐시 사용 (스탬피드 가능성 낮음)
- **TTL이 거의 없을 때** (예: 1초 남음, gap=2): `1 - r×2`가 음수일 확률 50% -> 각 서버가 독립적으로 50% 확률로 갱신 시도. 통계적으로 일부 서버만 미리 갱신
- 중요: **여러 서버가 한꺼번에 DB로 가지 않고, 만료 시점 직전부터 점차 분산되어 가게 됨**

#### Java 구현 예

```java
public <T> T fetch(String key, long expiryGapSec, Function<String, T> dbFetcher) {
    // 1. 남은 TTL 조회
    Long ttl = redisTemplate.getExpire(key, TimeUnit.SECONDS);

    // 2. 확률적 조기 재계산 판단
    if (ttl != null && ttl > 0) {
        double random = ThreadLocalRandom.current().nextDouble();
        if (ttl - (random * expiryGapSec) > 0) {
            T cached = (T) redisTemplate.opsForValue().get(key);
            if (cached != null) return cached;
        }
    }

    // 3. DB 조회 후 캐시 갱신 (스탬피드 회피)
    T value = dbFetcher.apply(key);
    redisTemplate.opsForValue().set(key, value, KEY_TTL);
    return value;
}
```

> **근데 모든 캐시에 적용할 필요는 없다**
> 다음 조건 중 하나라도 해당하면 PER 적용을 고려하자:
> - **트래픽이 많은 키** - 전사 조직도, 공통 코드, 메인 화면 집계 데이터
> - **동시 만료 위험이 있는 캐시** - 배치로 일괄 적재되어 TTL이 정확히 동일하게 끝나는 경우
> - **DB 조회 비용이 비싼 캐시** - Oracle 풀 스캔이나 복잡한 JOIN 쿼리를 감싸는 경우
>
> 실무에서는 평범한 get 메서드 대신 `fetch(key, gap, dbFetcher)` 형태로 PER 판정식을 호출하는 게 좋다.

## 세션 스토어로서의 레디스

왜 많은 서비스가 레디스를 세션 스토어로 쓰는가? 발전 과정을 단계별로 따라가며 알아보자.

### 발전 단계 — 5가지 시도와 한계

#### Stage 0 · 단일 웹 서버 — 별도 세션 스토어 불필요

서비스 초창기나 프로토타입용 서비스에서는 별도 세션 스토어가 굳이 필요하지 않습니다.

```
[User] → [Web Server (with session memory)]
                ↓
              [DB]
```

- 간단함. 웹 서버 자체에 세션 저장
- 서버를 늘리면 즉시 무너짐 → 다음 단계 필요

각 웹 서버에 세션 스토어를 두고 자체적으로 세션을 관리할 수 있습니다. 하지만 **서비스가 확장되어 웹 서버가 여러 대로 늘어나는 상황**이 곧 옵니다.

#### Stage 1 - Sticky Session

각 웹 서버별로 세션 스토어를 따로 관리한다면 **유저는 자신의 세션 정보를 갖고 있는 웹 서버에 종속**되어야 한다. 그렇지 않다면 유저 데이터 정합성에 문제가 생긴다.

예를 들어 쇼핑몰 장바구니에 아이템을 저장했는데, 서버에 접속할 때마다 아이템이 사라졌다 생겼다 한다면 정상적인 서비스 이용이 힘들 것이다.

**해결책**: Sticky Session을 활용한다. 즉, 로드밸런서가 같은 사용자를 항상 같은 서버로 라우팅한다.

```
[User-A] -> [LB] -> [WAS1: User-A 세션]
[User-B] -> [LB] -> [WAS2: User-B 세션]
[User-A] -> [LB] -> [WAS1] (강제 라우팅)
```

- 웹 서버 자체 세션 관리 그대로 유지 가능
- 특정 서버에 트래픽 몰리면 분산 불가능 -> 결국 **트래픽 분산이라는 본래 목적**을 잃어버림
- 그 서버가 죽으면 그 서버에 종속된 모든 사용자가 로그아웃됨

#### Stage 2 - All-to-All 복제

다음 알아볼 단계는 **모든 웹 서버에 모든 사용자의 세션을 복제**하는 방식이다. 이러면 어느 서버로 라우팅돼도 같은 세션을 볼 수 있다.

```
[WAS1: A,B,C 세션 모두 보유]
[WAS2: A,B,C 세션 모두 보유]
[WAS3: A,B,C 세션 모두 보유]

복제 트래픽: O(N²) — 서버 수가 많아질수록 폭증
```

- 유저는 여러 웹 서버에 분산 가능
- 유저의 세션 데이터는 여러 서버로 복사 후 저장되기 때문에 **불필요한 저장 공간**을 차지
- 하나의 유저는 한 번에 하나의 웹 서버에만 접속하기 때문에, 다른 웹 서버에 저장된 유저의 세션 정보는 무의미함
- 데이터를 복제하는 과정에서 **불필요한 네트워크 트래픽**이 다수 발생

10대 서버에 10만 명 사용자라면, 같은 세션 데이터를 10번 저장하는 셈입니다.

#### Stage 3 - DB 활용

세션을 RDBMS에 저장하면 어떨까? 이론상 모든 서버가 같은 DB를 보므로 일관성 문제는 없다.

```
[WAS1] ─┐
[WAS2] ─┼→ [Single DB: 모든 세션 저장]
[WAS3] ─┘
```

- 일관성 문제 없음. 모든 웹 서버가 동일 DB 참조
- 웹 서버 추가/제거가 자유로움
- 데이터베이스를 활용한다면 세션 스토어의 응답 속도가 느려질 때 **곧바로 클라이언트의 응답 속도도 저하**
- 서비스가 커져 유저가 많아질수록 운영하기 힘든 구조

매 요청마다 세션을 읽고 갱신하는데, RDBMS는 디스크 I/O와 트랜잭션 오버헤드가 있어 ms 단위 응답을 하고 있다. 인메모리 캐시는 μs 단위 이므로 **1000배 차이**나는 성능을 보여줄 수 있다.

#### Stage 4 - 레디스

레디스를 세션 스토어로 사용해 **서버, DB와 분리**한 뒤 여러 서버에서 세션 스토어를 바라보도록 구성한다면 앞선 사례들의 단점을 모두 커버할 수 있다.

```
[WAS1] ─┐
[WAS2] ─┼→ [Redis: 모든 세션 (인메모리)]   [DB: 영구 데이터]
[WAS3] ─┘                                ↑
          ↑                              │
   μs 응답 └─ 정기 동기화 (선택)      ────────┘
```

- 유저는 어떤 웹 서버에 연결되더라도 **동일한 세션 데이터를 조회**할 수 있음
- 트래픽을 효율적으로 분산할 수 있고 **데이터의 일관성도 고려할 필요가 없음**
- **RDBMS보다 훨씬 빠르고 접근하기도 간편**하므로 데이터를 가볍게 저장할 수 있음
- 레디스의 **hash 자료구조**가 세션 데이터를 저장하기에 알맞음 (필드 단위 갱신 가능)

#### Hash로 세션을 표현하는 예

```bash
> HSET session:abc123 userId 42 cart "item-1,item-2" theme dark
(integer) 3

> EXPIRE session:abc123 3600       # 1시간 세션
(integer) 1

> HGETALL session:abc123
1) "userId"    2) "42"
3) "cart"      4) "item-1,item-2"
5) "theme"     6) "dark"

> HSET session:abc123 theme light    # 일부 필드만 갱신 — TTL 보존됨
```

### 캐시 vs 세션 스토어

캐시와 세션은 비슷해 보이지만 데이터를 읽고 쓰는 패턴에서 약간의 차이를 보인다. 레디스를 각각으로 활용할 때를 비교해 보자.

#### 캐시로 사용할 때

레디스(캐시)는 **완벽한 서브셋으로 동작**한다.

- 캐시가 갖고 있는 데이터는 모두 DB에 저장되어 있음
- 캐시 내부의 데이터가 유실되더라도 해당 데이터는 DB에서 찾을 수 있음
- 캐시에 저장된 데이터는 **여러 애플리케이션에서 함께 사용**할 수 있음
  - 함께 사용할수록 더 효율적임

#### 세션 스토어로 사용할 때

이와 다르게 세션 스토어로 활용하면, 저장된 데이터는 **여러 사용자 간 공유되지 않으며 특정 사용자(ID)에 한해 유효**하다.

- 유저가 로그인하면 세션 데이터는 세션 스토어에 저장됨
- 로그인해 있는 동안에는 애플리케이션은 유저의 데이터를 **세션 스토어에만** 저장
- 유저가 로그아웃할 때 세션이 종료되며, 데이터의 종류에 따라 DB에 영구 저장하거나 삭제
- 예: 최근에 봤던 상품 리스트는 날아가도 괜찮지만, 장바구니는 DB로 옮겨 보존함

#### 핵심 차이 확인

| 비교 축 | 캐시 | 세션 스토어 |
|---------|------|--------------|
| 원본 데이터 | DB에 존재 | **레디스에 저장** |
| 유실 시 영향 | DB에서 다시 가져옴 (자가복구) | **데이터 손실** |
| 공유 범위 | 여러 사용자가 공유 | 특정 사용자만 |
| 운영 방식 | 비교적 자유 | 더 신중한 운영 필요 |
| 적합한 자료구조 | String, Hash 등 다양 | Hash (필드 단위 갱신) |
| TTL 의미 | 오래된 사본 정리 | 로그인 유지 기간 |
| 고가용성 중요도 | 중간쯤. 죽어도 DB가 있으니까. | **매우 높음**. 죽으면 전원 로그아웃 |
| 권장 maxmemory-policy | `allkeys-lru` | `noeviction` |

**운영 관점**
> 세션 스토어가 가지고 있는 데이터는 유일할 수 있다. 이때 세션 스토어에 장애가 발생하면 내부 데이터가 손실될 가능성이 있으므로, **레디스를 세션 스토어로 활용할 때는 캐시로 사용할 때보다 더 신중한 운영이 필요**하다. RDB/AOF 영속화, 센티널/클러스터 구성, 모니터링 등을 필수로 해야 한다.
