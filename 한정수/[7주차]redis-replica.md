# 8장. 복제 (Replication)


## 1. 가용성과 복제 도입 배경

### 1.1 가용성(Availability)
```
Availability = Available for Use Time / Total Time
```
- 시스템이 정상 동작한 시간의 비율이며, 1에 가까울수록 고가용성 시스템이라고 볼 수 있다.

### 1.2 복제본을 추가하는 이유
1. **하드웨어 장애 대비**: 마스터 다운 시 대신 사용할 여분의 노드가 필요하다.
2. **트래픽 분산**: 대규모 서비스에서 일부 트래픽을 복제본이 처리 -> 마스터 노드에 가는 부하를 감소시킨다.
3. **백업 부하 분리**: 운영 중 마스터에서 매번 백업을 받는 것은 부담됨 -> 복제본에서 백업을 수행한다.

### 1.3 레디스 복제 구조의 특징
- MySQL/PostgreSQL과 달리 **멀티 마스터 미지원**
- 마스터는 다른 노드의 복제본이 될 수 없다.
- v2.6+ 부터 복제본은 **기본 읽기 전용** -> 모든 쓰기는 마스터에서 발생한다.
- 복제본은 마스터에서 변경된 데이터를 그대로 받아옴

## 2. 복제 구조 구성하기

### 2.1 기본 명령어
```
REPLICAOF <master-ip> <master-port>
```

예시: 노드 B(192.168.0.2)를 노드 A(192.168.0.1:6379)의 복제본으로 설정한다.
```
B> REPLICAOF 192.168.0.1 6379
```

- 마스터 A 장애 시 애플리케이션의 연결 설정을 B로 변경하면 서비스를 계속 운용할 수 있다.

### 2.2 복제 그룹의 트리 구조

```
            MASTER
           /      \
       REPLICA   REPLICA ── REPLICA  (복제본 아래에 또 복제본 가능)
                              |
                          REPLICA
```

- 한 복제 그룹에는 **항상 하나의 마스터만 존재**
- 가장 상위 마스터만 데이터 업데이트 커맨드 수행 가능
- 하위 복제본은 모두 읽기 전용

### 2.3 패스워드 설정 (v6.0 ACL 이전)

| 노드 | 옵션 | 값 |
|------|------|-----|
| 마스터 | `requirepass` | `"mypassword"` |
| 복제본 | `masterauth` | `"mypassword"` |

런타임에 변경하려면:
```
> CONFIG SET masterauth mypassword
OK
> CONFIG REWRITE
OK
```

## 3. 복제 메커니즘: 두 가지 동기화 방식

### 3.1 방식 1: 디스크 사용 (v7 이전 기본)

```conf
repl-diskless-sync no
```

**디스크를 사용하는 복제 연결 메커니즘**
```
A(master) ←(1) REPLICAOF─── B(replica)
   │                            ↑
   │(2) fork()                  │
   ↓                  Replication buffer
RDB 스냅숏 ─(3) 데이터셋 변경 커맨드 저장─┘
   │                  │
   │(4) RDB 파일 전달   │(6) 버퍼링된 커맨드 전달
   └──────────────→ RDB 스냅숏 (B 디스크)
                        │
                        │(5) RDB 파일 read
                        ↓
                      B 메모리
```

**디스크 사용 복제 연결 과정**:
1. 복제본이 `REPLICAOF` 커맨드로 복제 연결 시도
2. 마스터가 `fork`로 자식 프로세스 만든 뒤 디스크에 **RDB 스냅숏 생성**
3. 2번이 진행되는 동안 마스터에서 수행된 모든 데이터셋 변경 작업은 **RESP 형태로 Replication buffer에 저장**
4. RDB 파일 생성 완료 -> 복제본 노드로 복사
5. 복제본은 **기존 데이터 모두 삭제 후 RDB 파일로 데이터 로딩**
6. 복제 과정 동안 버퍼링됐던 복제 버퍼의 데이터를 복제본으로 전달해 수행

**특징**: 복제 속도가 **디스크 I/O 처리량**에 직접적으로 영향을 받고, NAS 같은 원격 디스크 사용 시 더 느리다.

### 3.2 방식 2: 디스크리스 (v7 이상 기본)

```conf
repl-diskless-sync yes
```

**디스크를 사용하지 않는 복제 연결 메커니즘**
```
A(master) ←(1) REPLICAOF─── B(replica)
   │                             │
   │(2) 소켓 통신으로 RDB 직접 스트리밍 |  
   │─────────────────────────→   │
   │(3) 변경 커맨드 저장             │(4) 소켓에서 받은 RDB 데이터를 디스크에 저장
   ↓                             ↓
Replication buffer          RDB 스냅숏 (B 디스크)
   │                             │
   │(6) 버퍼링된 커맨드 전달          │(5) 메모리 flush 후 RDB 로딩
   └────────────────────────────→ B 메모리
```

**디스크리스 복제 연결 과정**:
1. `REPLICAOF` 커맨드로 복제 연결 시도
2. 마스터가 **소켓 통신**으로 복제본에 바로 연결, RDB 파일은 **생성과 동시에 점진적으로 복제본 소켓에 전송**
3. 2번 진행되는 동안 변경 작업은 RESP 형태로 마스터 복제 버퍼에 저장
4. 복제본은 소켓에서 읽어온 RDB 파일을 **자신의 디스크에 저장**
5. 복제본 기존 데이터 삭제 후 RDB 파일 내용을 메모리에 로딩
6. 복제 버퍼 데이터를 복제본으로 전달해 수행

**중요한 설계 결정**: `repl-diskless-load`가 기본 `disabled`라서 복제본은 소켓에서 받은 RDB를 **바로 메모리에 로드하지 않고 일단 디스크에 저장**한다.
- 이유: 자기 데이터를 삭제하기 전에 소켓으로 받아온 RDB가 정상인지 미리 확인할 수 없기 때문이다. 만약 받다가 끊겼는데 이미 자기 데이터를 삭제했다면 회복이 불가하다.
   - 따라서 디스크 저장 단계를 선행함으로써 **데이터 안정성**을 확보해야 한다.

### 3.3 디스크리스 방식의 동시 연결 제약

- **디스크 방식**: RDB 파일 생성 중 들어온 다른 노드의 복제 요청은 큐에 저장함 -> 기존 저장 완료 시 여러 복제본이 한 번에 복제 연결을 할 수 있다.
- **디스크리스 방식**: 이미 하나의 복제본으로 복제 연결이 시작되면 다른 복제본은 큐에서 대기함 -> 복제 과정 끝나기 전까지 다른 복제본은 연결이 안 된다.

해결책: `repl-diskless-sync-delay 5` (기본값 5초)
```conf
repl-diskless-sync-delay 5
```
- 새 복제 연결 들어오면 기본 5초 대기 후 복제 시작
   - 이 5초 사이 다른 복제 연결이 들어오면 마스터가 **여러 복제본에 동시에 RDB로 전송**한다.
- 네트워크 유실 후 재동기화 요청은 한 번에 여러 복제본에서 들어오는 게 일반적이므로 활성화를 권장한다.

### 3.4 두 방식 선택 기준
| 조건 | 권장 방식 |
|------|----------|
| 디스크 I/O가 느림 (NAS, HDD) | 디스크리스 |
| 네트워크가 빠름 | 디스크리스 |
| 디스크 I/O가 빠르고 네트워크 대역폭이 좁음 | 디스크 사용 |

## 4. 비동기로 동작하는 복제 연결

### 4.1 동작 메커니즘
```
client ──(1) SET a b──→ MASTER
client ←──(2) OK──────  MASTER
                          │ (3) SET a b (별도)
                          ↓
                        REPLICA
                          │ (4) OK
                          ↑
                        MASTER
```

- 마스터에서 커맨드 처리 -> **즉시 클라이언트에 OK 응답**을 줌
- 복제본 전달은 별도의 비동기 작업으로 진행
- 클라이언트는 복제본에 데이터가 정확히 전달됐는지 확인하지 않음 -> **짧은 지연 시간과 높은 성능**을 보여줌

### 4.2 비동기 복제의 trade-off
- **장점**: 복제 구조를 써도 마스터의 응답 속도가 거의 영향받지 않음
- **위험**: (2)번 과정 직후 마스터가 비정상 종료되면 복제본에 전달되지 못한 데이터가 유실될 수 있음
- **현실**: 실제 복제본 전달 속도는 매우 빠르기 때문에 유실이 빈번하게 발생하지는 않음

## 5. 복제 ID와 오프셋

### 5.1 기본 개념
- **모든 레디스 인스턴스는 랜덤 문자열의 replication ID 보유**
- 복제 기능을 사용하지 않는 인스턴스라도 ID는 존재한다.
- **복제 ID는 오프셋과 쌍**으로 존재: 데이터 수정 커맨드 수행할 때마다 오프셋 1씩 증가한다.
- 복제 ID + 오프셋이 같으면 **두 노드는 정확히 일치한 상태**가 된다.

### 5.2 INFO REPLICATION 확인 예시

**마스터 단독 상태**:
```
> INFO Replication
# Replication
role:master
connected_slaves:0
master_failover_state:no-failover
master_replid:e3b06d3eba522894a240a8a9ce3e808dd5ccfd7a
master_replid2:0000000000000000000000000000000000000000
master_repl_offset:709
second_repl_offset:-1
repl_backlog_active:1
repl_backlog_size:67108864
repl_backlog_first_byte_offset:1
repl_backlog_histlen:709
```

**복제본 1개 연결 후**:
```
role:master
connected_slaves:1
slave0:ip=127.0.0.1,port=6002,state=online,offset=709,lag=0
...
master_repl_offset:807
```

### 5.3 복제 ID 일치

```
   6001 (master)
   replid: e3b06...
   offset: 807
   ┌────┴────┐
   ↓         ↓
 6002       6003
 (replica)  (replica)
 replid:    replid:
 e3b06...   e3b06... ← 마스터와 정확히 일치
 offset:    offset:
 801        915
```

복제 ID는 같고 오프셋이 다르면, 차이 나는 만큼 복제본이 마스터를 따라잡아야 한다는 의미다.

## 6. 부분 재동기화 (Partial Resynchronization)

### 6.1 문제 의식
- 복제 연결이 끊길 때마다 마스터에서 RDB 파일을 새로 만들어 전체 전송 -> 네트워크 불안정 환경에서 성능 폭락
- 해결책: **부분 재동기화(partial resynchronization)** 기능을 활용

### 6.2 백로그 버퍼(Backlog Buffer)
- 마스터는 **커넥션 유실에 대비해 백로그 버퍼**라는 메모리 공간에 복제본에 전달한 커맨드 데이터를 저장
- 복제 연결이 잠시 끊겼다 재연결되면 복제본은 `PSYNC` 커맨드로 자신의 replication ID와 오프셋을 마스터에 전달

### 6.3 부분 재동기화 동작
```
   ┌─ 백로그 (master)   ─┐
   │  offset 900~915   │  ← master_repl_offset: 915
   │  ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒ │
   └───────────────────┘
                    ↑
   ┌───────────────PSYNC (replication id, offset 900)
   │
 replica
 master_repl_offset: 900
```
- 복제본이 offset 900에서 끊겼다가 재연결됨
- 마스터 백로그에 offset 901~915가 남아있으면 -> **RDB 새로 저장 없이** 백로그 내용만 전달

### 6.4 전체 재동기화로 fallback되는 경우
- 마스터 백로그 버퍼에 원하는 데이터가 남아있지 않거나
- 복제본이 보낸 replication ID가 현재 마스터와 일치하지 않으면 **전체 재동기화(full resync)** 수행

### 6.5 백로그 관련 옵션

```conf
repl-backlog-size 1mb    # 기본값 1MB
repl-backlog-ttl 3600    # 복제본 모두 끊어진 후 백로그 유지 시간
```

- **백로그 크기가 클수록** 복제본이 부분 재동기화가 가능한 시간이 길어짐
- 백로그는 1개 이상의 복제본이 연결된 경우에만 할당됨
- 마지막 복제본이 끊어진 후 `repl-backlog-ttl`만큼 시간이 경과하면 메모리에서 백로그 공간이 삭제됨
- 단, **복제본은 언제든 마스터로 승격될 가능성이 있으므로** 복제본에 직접 연결된 복제 연결이 따로 없더라도 백로그 버퍼를 해제하지 않음. 추후 다른 복제본과 부분 재동기화 시 사용한다.

## 7. Secondary 복제 ID (페일오버 대응)

### 7.1 기본 원칙: 같은 그룹은 같은 복제 ID

**정상 상태**
```
    A (master)
    replid: aaaaa
    ┌──┴──┐
    ↓     ↓
    B     C
    replid: aaaaa
    (마스터와 동일)
```

- 한 복제 그룹 내 모든 노드는 동일한 복제 ID를 보유함

### 7.2 페일오버 발생 시
```
    A (장애)              
    replid: aaaaa         
                          
    B ────────→ C         
    (새 master)           
    replid: bbbbb         
    replid2: aaaaa  ← Secondary로 기존 ID 보관
```

- A에 장애가 발생하면 B가 새 마스터로 승격됨
- **B는 새 replication ID `bbbbb`를 갖게 되고, 기존 ID `aaaaa`는 `master_replid2`에 저장됨**
- 기존 A의 복제본이던 C는 B에 연결되며, B의 복제 ID `bbbbb`를 갖게 됨. C도 기존에 A에 연결돼 있었기 때문에 `master_replid2`에는 `aaaaa`를 보유함

### 7.3 왜 2개의 복제 ID가 필요한가?

**핵심**: 페일오버 후 복제본이 새 마스터에 빠르게 재연결하기 위해서이다.

- C가 B에 복제 연결될 때, 두 노드의 `master_replid2`가 같음 → C는 B에 **부분 재동기화 시도**
- B와 C 모두 기존 A 노드의 복제본으로 **동일한 데이터셋을 가지고 있었기 때문에**, 노드 B의 백로그가 재동기화에 필요한 데이터를 가지고 있다면 부분 재동기화만으로 연결 가능함
   - **RDB 파일을 새로 전송하는 전체 재동기화를 피할 수 있음** -> 불필요한 작업을 제거하고, 빠른 복제 상태를 구축할 수 있음

**만약 복제 ID가 하나만 있었다면?**
- B가 마스터로 승격된 후에도 기존 ID `aaaaa`를 유지하다 A와 다시 연결되면, 같은 복제 ID인데 두 노드가 다른 데이터를 갖는 모순이 발생함 -> "같은 복제 ID = 같은 데이터셋" 원칙을 위반함
   - 그래서 승격 시 새 ID를 받고 기존 ID는 보조 자리(`master_replid2`)로 옮겨둔다.

## 8. 읽기 전용 모드와 그 함정

### 8.1 기본 동작
- 2.6 이상 버전 복제본은 기본 `replica-read-only yes` -> 이러면 새 데이터 저장을 못함
- 클라이언트가 복제본에 SET 등 데이터 조작 커맨드를 보내면 거부됨

### 8.2 해제 시 동작
```
    A (master)
    │
    ↓
    B (replica, replica_read_only = NO) <- set key value
    │
    ↓
    C (sub-replica)
```

- B의 `replica-read-only`를 해제하면 클라이언트가 B에 직접 쓸 수 있음
- 그러나 **B에 쓴 데이터는 오직 B 로컬에만 유지**하고, 하위 복제본 C로는 **전파되지 않음**
- 서브 복제본 C는 항상 **최상위 마스터 A가 중간 복제본 B로 보낸 것과 동일한 복제 프로토콜**만 전달받음

### 8.3 함정
- 복제본에 직접 쓴 데이터는 **복제본을 재시작하거나 커넥션을 유실했을 때 전체 재동기화가 발생했다가 사라진다**.
- 마스터에서 수행하기 부담스러운 오래 걸리는 연산 결과를 테스트하는 용도 정도로만 제한적으로 사용을 권장한다.

## 9. 유효하지 않은 복제본 데이터 (Stale Data)

### 9.1 Stale 데이터의 정의
- 복제 구조에서 **복제본의 데이터와 마스터의 데이터가 일치하지 않는 경우**의 데이터
- 발생 시점:
  - 복제본이 마스터와 연결이 끊어진 상태
  - 복제 연결이 시작된 뒤 아직 완료되지 않은 상태

### 9.2 replica-serve-stale-data 옵션
```conf
replica-serve-stale-data yes  # 기본값
```

| 설정값 | 동작 |
|--------|------|
| `yes` (기본) | 데이터가 유효하지 않다고 판단될 때에도 **클라이언트의 모든 읽기 요청에 데이터를 반환** |
| `no` | INFO, CONFIG, PING 등 일부 기본 커맨드 외 모든 커맨드에 대해 `SYNC with master in progress` 오류 반환 |

**선택 기준**:
- 정확성보다 가용성이 중요한 캐시 용도라면 `yes`를 유지
- 데이터 정확성이 중요한 서비스라면 `no`로 설정 (오래된 데이터를 반환하느니 차라리 오류를 내는 게 낫다)

## 10. 백업과 복제의 관계: 함정 시나리오

### 10.1 백업 없는 복제 사용 시 위험 시나리오

```
1. MASTER (data) ──복제──→ REPLICA (data)
                              │
2. MASTER 장애 → 자동 재시작
   ↓
   MASTER (memory cleared) ← 메모리 초기화
                              │
3. REPLICA (data 있음) → MASTER로 복제 연결 시도
                              │
4. MASTER ──빈 RDB 스냅숏──→ REPLICA (data 사라짐!)
```

**시나리오 4단계 정리**:
1. 백업 기능을 사용 안 하는 마스터-복제본 노드가 존재
2. 마스터 노드가 장애로 종료됨 -> 레디스 프로세스가 자동 재시작되는 시스템에 의해 노드가 재부팅됨 -> **메모리 내용 초기화**됨
3. 복제본 노드에는 데이터가 존재하지만, 마스터 노드로의 복제 연결을 시도함
4. 마스터에서 복제본으로 **빈 데이터셋을 전달**하면 -> 복제본 데이터까지 모두 손실남

### 10.2 백업을 함께 사용했다면?
- 2번 상황에서 레디스가 재부팅될 때 **백업 파일을 자동으로 읽어옴** -> 데이터 복원 -> 복원된 내용이 복제본으로 정상 전달됨

### 10.3 운영 권장사항
| 백업 사용 여부 | 권장 설정 |
|----------------|-----------|
| 사용 | 자동 재시작 활성화해도 됨 |
| 사용 안 함 | **마스터의 자동 재시작 비활성화 필수** |

- 자동 재시작을 끄면, 마스터 장애 시 애플리케이션 연결을 복제본으로 수동 변경해서 서비스를 계속 운영할 수 있음
- 또는 복제본에서 데이터를 새로 백업받아 마스터에 전달한 뒤 마스터를 재시작하면 됨

> **결론**: 데이터의 안정성을 위해 복제 기능을 사용할 경우 **백업 기능도 함께 사용**하는 것이 좋으며, 그렇지 않을 경우 마스터에서는 **인스턴스의 자동 재시작을 활성화하지 않는 것을 권장**한다.
