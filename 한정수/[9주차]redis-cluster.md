# 10장. 클러스터 (Redis Cluster)

## 1. 레디스 클러스터와 확장성

### 1-1. 확장성(Scalability)이란

확장성은 운영 중인 시스템에서 **증가하는 트래픽에 유연하게 대응**할 수 있는 능력이다. 사용자나 데이터가 늘어 처리량을 키워야 할 때, 리소스를 투입하는 방식에 따라 스케일 업과 스케일 아웃으로 나뉜다.

### 1-2. 스케일 업 vs 스케일 아웃

| 구분 | 스케일 업 (Scale-up / 수직 확장) | 스케일 아웃 (Scale-out / 수평 확장) |
|------|-------------------------------|----------------------------------|
| 방식 | 서버 하드웨어를 고사양으로 업그레이드 (CPU·메모리·디스크 증설) | 비슷한 사양의 장비를 추가로 연결 |
| 예시 | 2vCPU/4GB → 4vCPU/16GB → 8vCPU/32GB | 2vCPU/4GB 서버를 여러 대 나란히 |
| 장점 | 간단하고 초기 비용이 적음 | 용량 + 처리량을 함께 확장, 선형 확장 가능 |
| 단점 | 하드웨어 허용 범위 내에서만 가능(상한 존재) | 데이터 분산 처리 로직을 추가 개발해야 함 |

### 1-3. 레디스에서의 확장성

- **eviction이 자주 발생**한다면 : 메모리가 부족하다는 신호이므로 **스케일 업**을 고려 (서버 메모리를 늘리고 `maxmemory` 값 증가). eviction은 인스턴스의 `maxmemory`만큼 데이터가 찬 상태에서 또 저장을 시도할 때 발생한다.
- **처리량(throughput)을 늘려야** 한다면 : 스케일 업만으로는 한계가 있다. 레디스는 **단일 스레드로 동작**하므로 CPU 코어를 추가해도 여러 코어를 동시에 활용할 수 없기 때문이다. 따라서 데이터를 여러 서버로 분할(샤딩)해 **요청을 병렬 처리**함으로써 처리량을 선형적으로 확장한다.

## 2. 레디스 클러스터의 기능

클러스터 모드를 쓰면 **애플리케이션 아키텍처 변경 없이** 여러 인스턴스 간 수평 확장, 데이터 분산 처리·복제·자동 페일오버를 모두 사용할 수 있다.

### 2-1. 데이터 샤딩(Data Sharding)

데이터 저장소를 수평 확장하며 여러 서버 간에 데이터를 분할하는 DB 아키텍처 패턴을 **샤딩**이라 한다.

- 클러스터에서 데이터는 **키를 이용해 샤딩**되며, 하나의 키는 항상 하나의 마스터 노드에 매핑된다.
- 모든 노드는 "어떤 키가 어느 노드에 저장돼야 하는지" 알고 있어, 클라이언트가 엉뚱한 노드에 읽기/쓰기를 시도하면 **올바른 마스터로 리디렉션**한다. (이 처리는 노드와 클라이언트가 담당)
- 애플리케이션 소스 코드 로직 변경 없이 분할 저장이 되므로 샤딩의 번거로움이 줄어든다.
- 클라이언트는 "특정 키 → 특정 마스터" 매핑 정보를 **캐싱(cluster map)** 해 키를 찾아가는 시간을 단축한다. 노드 추가/변경이 없는 한 키는 특정 마스터에 고정 매핑된다.

#### 동작 예시
```
client -> (GET key:1234) -> 잘못된 노드 도착
client <- (ASK / MOVED Master-5) <- 리디렉션 정보 반환
client -> (GET key:1234) -> 올바른 마스터(Master-5) 재요청
```
각 마스터(Master-1 ~ Master-5)는 자신의 복제본(Replica)으로 데이터를 복제한다.

### 2-2. 고가용성 (High Availability)

- 클러스터는 보통 **최소 3대의 마스터 + 각 마스터당 복제본**으로 구성하는 것이 일반적이다.
- 하나의 클러스터에 속한 각 노드는 **서로를 모니터링**한다.
- 마스터에 장애가 발생하면, 이를 인지한 다른 노드들이 해당 마스터의 복제본을 **마스터로 자동 페일오버**시킨다 → 사용자 개입 없이 가용성 확보.
- 마스터에 연결된 복제본 개수를 파악해, **잉여 복제본을 복제본이 없는 마스터에 연결**하는 **복제본 마이그레이션(replica migration)** 도 수행한다.
- 토폴로지: **데이터 복제(실선)** + **클러스터 버스(점선)** 로 모든 노드가 상호 연결된 형태.

> 레디스 클러스터는 마스터를 **최대 1,000개까지 확장** 가능하다.

### 2-3. 클러스터 버스 (Cluster Bus)

노드 간 통신은 **클러스터 버스**라는 독립적인 통신 채널을 사용한다.

- 클라이언트 커맨드용 TCP 포트와 **별개로 동작**하는 추가 TCP 포트를 사용.
- 포트는 `cluster_bus_port`를 지정하지 않으면 **일반 포트 + 10000** 으로 자동 설정.
  - 예: 레디스가 `6379` 포트면 클러스터 버스는 `16379` 포트.
- 클러스터는 **풀 메쉬(full-mesh) 토폴로지**: N개 노드면 각 노드는 N-1개의 다른 노드와 송수신 TCP 연결을 상시 유지한다.
- 어떤 노드에 PING을 보냈을 때 PONG 응답이 늦으면 해당 노드로의 연결을 새로 시도한다.

> **오버헤드 걱정은 불필요**: 풀 메쉬지만 **가십 프로토콜 + 구성 업데이트 메커니즘** 덕분에 정상 상태에서는 노드 간 메시지가 적다. 노드가 증가해도 메시지가 기하급수적으로 늘지 않도록 설계됨.

## 3. 레디스 클러스터 동작 방법

### 3-1. 해시슬롯을 이용한 데이터 샤딩

클러스터 구조에서 모든 데이터는 **해시슬롯**에 저장된다. 레디스는 총 **16,384개**의 해시슬롯을 가지며, 마스터 노드들이 이를 나눠 갖는다.

#### 3대 마스터 구성 시 분배
```
Master-1 : 0      ~ 5460
Master-2 : 5461   ~ 10922
Master-3 : 10923  ~ 16383
```

#### 키 -> 해시슬롯 매핑 함수
```
HASH_SLOT = CRC16(key) mod 16384
```
키를 CRC16으로 한 번 해싱한 뒤 16384로 나눈 나머지로 해시슬롯이 결정된다. **데이터 저장 때 뿐만 아니라 읽을 때도** 이 함수로 적절한 마스터를 찾아간다.

예: `ID:0817`이라는 키 -> 알고리즘에 의해 해시슬롯 `5459`에 저장 -> 클라이언트는 5459를 가진 Master-1에서 데이터를 가져온다.
예: `ID:87345` -> 해시슬롯 `5462` -> Master-2에 저장.

#### 해시슬롯의 이동성
- 해시슬롯은 마스터 노드 내에서 **자유롭게 옮겨질 수 있고**, 옮겨지는 중에도 데이터는 정상 접근 가능.
- 이 특성 덕분에 클러스터 내 **마스터 노드 추가/삭제가 매우 간단**하다.

#### 마스터 노드 추가
신규 마스터를 추가한 뒤, 기존 노드들의 해시슬롯 일부를 신규 마스터로 이동시킨다.
```
Master-1, 2, 3에서 일부 슬롯 -> Master-4로 이동
Master-4 : 0~665, 5461~6127, 10923~11588 (각 마스터에서 조금씩)
```
마스터 삭제도 동일하게 간단: 삭제할 노드의 해시슬롯을 전부 다른 마스터로 이동시킨 뒤 클러스터에서 제외.

### 3-2. 해시태그 (Hash Tag)

#### 문제: 클러스터에서는 다중 키 커맨드 제약
`MGET`처럼 한 번에 여러 키에 접근하는 다중 키 커맨드는 **서로 다른 해시슬롯에 속한 키에 대해서는 사용할 수 없다**.

```
MGET user1:name user2:name
```
`user1:name`과 `user2:name`이 서로 다른 해시슬롯(예: 6001, 6003)에 저장돼 있으면, 클러스터는 키마다 다른 마스터로 리디렉션해야 하므로 **한 번에 처리할 수 없어 ERROR** 반환.

#### 해결: 해시태그
키에 **대괄호 `{}`** 를 사용하면, 전체 키가 아닌 **대괄호 사이의 값만으로 해싱**한다. 이를 해시태그라 한다.

```
user:{123}:profile
user:{123}:account
```
위 두 키는 대괄호 사이가 모두 `123`이므로 **같은 해시슬롯 = 같은 마스터**에 저장됨이 보장된다.

```
MGET {user}1:name {user}2:name   # 둘 다 'user'로 해싱 -> 다중 키 커맨드로 가능
```

#### 해시태그 파싱 규칙
- 대괄호 사이에 **아무 문자열이 없으면** 전체 키로 해싱한다.
- `{}`가 여러 개면 **가장 처음의 `{` 부터 가장 처음의 `}` 사이** 값으로 해싱한다.

| 키 | 해싱되는 값 |
|----|-----------|
| `{user1000}.followers` | `user1000` |
| `user{}id` | `user{}id` (전체 — 사이가 비어 있음) |
| `user{{name}}id` | `{name` (첫 `{`부터 첫 `}`까지) |
| `user{name}{id}` | `name` |

> **주의**: 너무 많은 키가 같은 해시태그를 가지면 **하나의 해시슬롯에 데이터가 몰리는 현상**이 생길 수 있으므로 키 분배 모니터링이 필요하다.

### 3-3. 자동 재구성 (Reconfiguration)

센티널과 마찬가지로 클러스터도 복제 + 자동 페일오버로 고가용성을 확보한다. 차이점은 **별도 센티널 인스턴스 없이 데이터 노드들이 서로 감시**하고, 인스턴스에 문제가 생기면 클러스터 버스를 통해 자동으로 구조를 재구성한다는 점이다.

클러스터의 재구성은 **두 가지**다.
1. **자동 페일오버** — 마스터 장애 시 복제본을 마스터로 승격
2. **복제본 마이그레이션** — 잉여 복제본을 다른 마스터에 재연결

#### (1) 자동 페일오버
```
초기: 6001 -> 6005, 6002 -> 6006, 6003 -> 6004  (마스터 -> 복제본)
6001 마스터 장애 발생
  -> 6005 복제본이 다른 마스터들에게 "페일오버해도 될지" 투표 요청
  -> 6001이 정상 아니라고 판단한 마스터들이 6005에 투표
  -> 과반수 이상 득표 시 6005가 마스터로 승격
결과: 6005(M), 6002(M), 6003(M)
```

**`cluster-require-full-coverage`** (기본값 `yes`)
- `yes`면 일부 해시슬롯을 사용 못 하게 되는 순간(일부 노드만 다운돼도) **데이터 정합성을 위해 클러스터 전체 상태가 `fail`** 이 되어, 문제 슬롯을 포함한 **전체 해시슬롯에 대한 조작이 실패**한다.
- 즉, 위 예시에서 승격된 6005에 또 장애가 나면 마스터 하나가 비정상이 되어 전체 클러스터를 못 쓰게 된다.
- 가용성이 중요하면 → **복제본 마이그레이션이 가능하도록 아무 마스터에 복제본을 하나 더 추가**하는 것이 좋다.

#### (2) 복제본 마이그레이션
```
구성: 6001 -> 6005, 6002 -> 6006, 6003 -> 6004,6007  (6003만 복제본 2개)
6001 장애 → 6005 승격 (6005는 이제 복제본 없는 마스터)
  → 클러스터가 복제본 불균형 감지
  → 6003의 복제본 2개 중 하나(6007)를 6005의 복제본으로 이동
```
- 가장 많은 복제본이 연결된 마스터의 복제본 중 하나가 옮겨진다.
- **FAIL 상태가 아닌 복제본 중 노드 ID가 가장 작은 복제본**이 이동 대상으로 선택된다.
- 복제본 마이그레이션은 **모든 마스터가 최소 1개 이상의 복제본**을 갖도록 보장 -> 클러스터 전체 안정성 향상.

**관련 설정**
```
cluster-allow-replica-migration yes   # 기본값 yes일 때 동작
cluster-migration-barrier 1           # 복제본을 떠나보낸 뒤에도 원래 마스터에 "남아 있어야" 할 최소 정상 복제본 수
```
- `cluster-migration-barrier`의 정확한 정의: **복제본 하나가 다른(고아) 마스터로 마이그레이션을 떠나려면, 떠난 뒤에도 자기 원래 마스터에 이 값만큼의 정상 복제본이 남아 있어야 한다.**
- 예: barrier=2이고 6003이 복제본 2개(6004, 6007)를 가진 경우 -> 하나를 보내면 1개만 남아 "2개가 남아야 한다"는 조건에 미달하므로 **마이그레이션이 발생하지 않는다.**
- 기본값 1이면, 복제본 2개짜리 마스터는 하나를 보내도 1개가 남아 조건(1개 이상)을 만족 -> 마이그레이션 발생.

## 4. 레디스 클러스터 실행하기

### 4-1. 사전 준비

- 클러스터 모드는 **최소 3개 마스터 노드** 필요
- 실제 운영은 **3 마스터 + 각 복제본 = 총 6개 노드**가 일반적
- 예제: `192.168.0.11` ~ `192.168.0.66` 6대 서버 사용

### 4-2. 클러스터 초기화

#### redis.conf 설정
```
cluster-enabled yes
```
이 설정을 `yes`로 바꿔 클러스터 모드로 변경한 뒤, 6대 서버에 레디스를 각각 실행

#### 클러스터 생성 커맨드
```
redis-cli --cluster create [host:port] --cluster-replicas 1
```
- `--cluster create` : 새 클러스터 생성, 뒤에 추가할 노드 `ip:port` 나열
- `--cluster-replicas 1` : 각 마스터마다 복제본 1개씩 추가

#### 실행 예시
```bash
$ src/redis-cli --cluster create 192.168.0.11:6379 192.168.0.22:6379 \
  192.168.0.33:6379 192.168.0.44:6379 192.168.0.55:6379 192.168.0.66:6379 \
  --cluster-replicas 1 -a nhncloud

>>> Performing hash slots allocation on 6 nodes...
Master[0] -> Slots 0 - 5460
Master[1] -> Slots 5461 - 10922
Master[2] -> Slots 10923 - 16383
Adding replica 192.168.0.55:6379 to 192.168.0.11:6379
Adding replica 192.168.0.66:6379 to 192.168.0.22:6379
Adding replica 192.168.0.44:6379 to 192.168.0.33:6379
...
Can I set the above configuration? (type 'yes' to accept):
```
- 입력한 순서대로 **앞 3개는 마스터, 나머지는 복제본**으로 구성
- 각 마스터의 해시슬롯 할당, 복제본 연결 정보를 보여준 뒤 `yes` 입력 시 진행

```
>>> Nodes configuration updated
>>> Assign a different config epoch to each node
>>> Sending CLUSTER MEET messages to join the cluster
Waiting for the cluster to join
>>> Performing Cluster Check (using node 192.168.0.11:6379)
...
[OK] All nodes agree about slots configuration.
>>> Check for open slots...
>>> Check slots coverage...
[OK] All 16384 slots covered.
```
`[OK] All 16384 slots covered.`가 뜨면 정상 초기화 완료

#### 마스터 vs 복제본 슬롯 표기
```
slots:[0-5460] (5461 slots) master   # 마스터 노드 (슬롯 할당됨)
slots: (0 slots) slave               # 복제본 노드 (슬롯 할당 안 됨)
```
- **해시슬롯은 마스터에만 할당**되고 복제본에는 할당되지 않는다.
- 복제본은 마스터와 동일한 데이터를 저장(슬롯 내부 데이터는 동일)하지만, 슬롯 자체를 할당받진 않는다.

### 4-3. 클러스터 상태 확인하기

#### CLUSTER NODES
```bash
$ redis-cli cluster nodes
73abfbb...229cdf1c3a3c0f2d05 192.168.0.22:6379@16379 master - 0 1670429890051 2 connected 5461-10922
f6c15801...d6cd3 192.168.0.66:6379@16379 slave 73abfbb...2d05 0 1670429890553 2 connected
...
```
랜덤 순서로 노드를 출력하며, 각 라인은 다음 필드를 갖는다.
```
<id> <ip:port@cport> <flags> <master> <ping-sent> <pong-recv> <config-epoch> <link-state> <slot> <slot> ... <slot>
```

| 필드 | 설명 |
|------|------|
| **id** | 노드 생성 시 자동 생성되는 랜덤 클러스터 ID. 한 번 할당되면 바뀌지 않음 |
| **ip:port@cport** | 노드의 ip, port, 클러스터 버스 포트. cport = port + 10000 자동 설정 |
| **flags** | 노드 상태 (아래 표 참조, 복수 표시 가능) |
| **master** | 복제본이면 마스터 노드의 ID, 마스터면 `-` |
| **ping-sent** | 보류 중인 PING 없으면 0, 있으면 마지막 PING 전송 유닉스 타임 |
| **pong-recv** | 마지막 PONG 수신 유닉스 타임 |
| **config-epoch** | 현재 노드의 구성 에포크. 페일오버 때마다 증가, 구성 충돌 시 에포크가 높은 노드 구성으로 변경 |
| **link-state** | 클러스터 버스 링크 상태 (connected / disconnected) |
| **slot** | 노드가 가진 해시슬롯의 범위 |

**flags 값**
- `myself` : redis-cli로 접근한 노드
- `master` : 마스터 노드
- `slave` : 복제본 노드
- `fail?` : PFAIL 상태 (정상 접근 불가를 확인해 다른 노드에 확인 시작)
- `fail` : FAIL 상태 (과반수 이상 노드가 접근 불가 인지 후 PFAIL → FAIL로 변경)
- `handshake` : 새 노드를 인지하고 핸드셰이킹하는 단계
- `nofailover` : 복제본이 페일오버를 시도하지 않음
- `noaddr` : 해당 노드의 주소를 모름
- `noflags` : 플래그 없음

#### CLUSTER NODES 라인 해석 예시
```
첫 라인 id: 73abfbb...2d05, ip:port = 192.168.0.22:6379
클러스터 버스 포트 = 16379 (6379 + 10000)
flag = master → 마스터 인스턴스
PING 활성 없음(0), 마지막 PONG 수신 = 1670429890051
config-epoch = 2, link-state = connected
보유 해시슬롯 = 5461 ~ 10922
```

### 4-4. redis-cli로 클러스터 접근 & 리디렉션

클러스터에 접속하려면 **클러스터 모드를 지원하는 클라이언트**가 필요하다. 이런 클라이언트만 리디렉션 기능을 제공한다.

#### 일반 모드 — MOVED 에러
```bash
$ redis-cli
127.0.0.1:6379> set user:1 true
(error) MOVED 10778 192.168.0.22:6379
```
- `192.168.0.11`에 연결해 `user:1`을 저장하려 했으나, 이 키의 해시슬롯은 `10778`이고 해당 슬롯은 `192.168.0.22` 마스터 소유라고 응답
- 일반 클라이언트는 데이터가 저장될 노드가 정해져 있고 **해당 노드에서만 커맨드 수행 가능**

#### 클러스터 모드 (`-c`) — 자동 리디렉션
```bash
$ redis-cli -c
127.0.0.1:6379> set user:1 true
-> Redirected to slot [10778] located at 192.168.0.22:6379
OK
192.168.0.22:6379>
```
- `-c` 옵션으로 클러스터 모드 접속 → 리디렉션 기능 제공
- `Redirected` 메시지 후 정상 저장(`OK`), 연결도 `192.168.0.22`로 자동 변경
- 단, **`redis-cli -c` 자체는 슬롯 맵을 캐싱하지 않는다.** 매 요청마다 리디렉션을 그때그때 따라갈 뿐이다. 맵 캐싱은 **라이브러리 클라이언트(Jedis, Redisson, lettuce 등)의 기능**이다.
- 이런 라이브러리 클라이언트는 리디렉션 정보를 **클라이언트 측 메모리에 캐싱해 슬롯 맵(cluster map)을 생성**한다. 다음에 같은 키 커맨드 시 캐싱된 노드로 바로 보낸다. 이에 따라 성능 향상을 노려볼 수 있다.
- 이 맵은 **클러스터(서버)가 아니라 클라이언트가 보관**한다. 마스터 추가/삭제, 페일오버 등으로 클러스터 구조가 바뀌면, 클라이언트는 MOVED 응답을 받거나 `CLUSTER SHARDS`/`CLUSTER SLOTS`로 **맵을 리프레시**한다.

### 4-5. 페일오버 테스트

클러스터 구성 후 **페일오버가 정상 동작하는지** 확인하는 것이 좋다. 노드 간 통신, 네트워크 단절 여부 등을 점검할 수 있다. 두 가지 방법으로 페일오버를 발생시킬 수 있다.

#### (1) 커맨드를 이용한 페일오버 — 수동 페일오버
- 페일오버시킬 마스터에 **1개 이상의 복제본**이 연결돼 있어야 함
- 페일오버를 발생시킬 **복제본 노드**에서 `cluster failover` 실행

```bash
192.168.0.55:6379> INFO REPLICATION
# Replication
role:slave
master_host:192.168.0.11
master_port:6379
...
192.168.0.55:6379> CLUSTER FAILOVER
OK
```
실행 후 다시 확인하면 역할이 바뀜:
```bash
192.168.0.55:6379> INFO REPLICATION
# Replication
role:master
connected_slaves:1
slave0:ip=192.168.0.11,port=6379,state=online,offset=613998,lag=0
```

**수동 페일오버 동작 과정**
1. 기존 마스터에 연결된 클라이언트가 잠시 블락된다.
2. 페일오버 시작 전 복제 딜레이를 기다려, 마스터의 복제 오프셋을 복제본이 따라잡는다.
3. 따라잡기 완료되면 페일오버 시작 -> 클러스터 정보 변경
4. 모든 작업 완료 후 클라이언트는 새 마스터로 리디렉션

> 수동 페일오버는 데이터 유실 없이(복제 오프셋 동기화 후) 안전하게 역할을 교체한다.

#### (2) 마스터 동작을 중지시켜 페일오버 — 자동 페일오버
실제로 마스터에 장애를 발생시켜, 마스터 비정상 시 다른 노드가 이를 인지하는지 확인

```bash
$ redis-cli -h <master-host> -p <master-port> shutdown
```

**`cluster-node-timeout`** (기본값 15,000ms = 15초)
- 복제본은 이 시간 동안 마스터에서 응답이 없으면 마스터가 비정상이라 판단해 페일오버를 트리거한다.

```bash
$ redis-cli -h 192.168.0.33 -p 6379 shutdown
$ redis-cli cluster nodes
...
ab1b4edfa...dea66 192.168.0.33:6379@16379 master,fail - 1670858509870 1670858507353 7 disconnected
52e66ec38...cdf9dd 192.168.0.55:6379@16379 master - 0 1670858803348 9 connected 0-148 5461-5511 10923-16383
...
```
15초 후 `cluster nodes`로 확인하면 다운된 노드는 `master,fail`로, 복제본이 승격되어 새 마스터로 동작하는 것을 볼 수 있다.

## 5. 레디스 클러스터 운영하기

### 5-1. 클러스터 리샤딩 (Resharding)

마스터 노드가 가진 해시슬롯 중 일부를 다른 마스터로 이동하는 것을 **리샤딩**이라 한다. `redis-cli`의 `cluster reshard` 옵션으로 수행한다.

#### 대화형 리샤딩
```bash
$ redis-cli --cluster reshard 192.168.0.66 6379
>>> Performing Cluster Check (using node 192.168.0.66:6379)
...
[OK] All 16384 slots covered.
How many slots do you want to move (from 1 to 16384)? 100
What is the receiving node ID? ab1b4edfa9085b104fc3fd9f3f9d5a740f7dea66
Please enter all the source node IDs.
  Type 'all' to use all the nodes as source nodes for the hash slots.
  Type 'done' once you entered all the source nodes IDs.
Source node #1: all

Ready to move 100 slots.
...
Resharding plan:
  Moving slot 5461 from 73abfbb...2d05
  Moving slot 5462 from 73abfbb...2d05
  ...
Do you want to proceed with the proposed reshard plan (yes/no)? yes
```
- 클러스터에 속한 **아무 노드(마스터든 복제본이든)** 를 지정해도 클러스터 구조를 파악해 동일하게 수행됨
- 이동시킬 슬롯 개수 입력 (예: 100)
- 받을 노드(receiving node) ID 입력 — 리샤딩 시작 시 파악한 구성에서 확인 가능
- source node에 **`all`** 입력 → 모든 마스터에서 조금씩 이동 / 특정 노드만 지정하려면 하나씩 입력 후 `done`
- 리샤딩 플랜 확인 후 `yes` → **중단 없이 진행**

리샤딩 후 결과:
```
192.168.0.33 노드의 해시슬롯이 0-148, 5461-5511, 10923-16383 (총 5,561개)로 변경됨
```

#### CLUSTER CHECK로 상세 확인
```bash
$ redis-cli --cluster check 192.168.0.22 6379
192.168.0.22:6379 (73abfbb3...) -> 1 keys | 5411 slots | 1 slaves.
192.168.0.11:6379 (5a15c78c...) -> 0 keys | 5412 slots | 1 slaves.
192.168.0.33:6379 (ab1b4edf...) -> 0 keys | 5561 slots | 1 slaves.
[OK] 1 keys in 3 masters.
...
[OK] All 16384 slots covered.
```
`cluster check`는 `cluster nodes`보다 조금 더 자세한 구성을 보여준다.

#### 클러스터 리샤딩 — 간단 버전 (비대화형)
운영상 슬롯 이동이 잦거나 자동화하려면, 사용자 인터랙션 없이 바로 이동시키는 방법이 있다. (단, 커맨드 실행 즉시 데이터가 옮겨져 중간 취소·확인이 어렵다.)

```bash
redis-cli --cluster reshard <host>:<port> --cluster-from <node-id> \
  --cluster-to <node-id> --cluster-slots <number of slots> --cluster-yes
```

예: `192.168.0.11`에서 `192.168.0.33`으로 100개 슬롯 이동
```bash
$ redis-cli --cluster reshard 192.168.0.11:6379 \
  --cluster-from 5a15c78c...a1445 \
  --cluster-to ab1b4edfa...dea66 \
  --cluster-slots 100 --cluster-yes
```
- `--cluster-yes` : 모든 프롬프트에 자동으로 `yes` 입력. 커맨드 시작 즉시 자동 리샤딩 진행.

### 5-2. 클러스터 확장 — 신규 노드 추가

운영 중 클러스터에 새 노드를 추가할 수 있다. **마스터로** 추가하거나(확장), **복제본으로** 추가(가용성)할 수 있다. 두 경우 모두 **추가하려는 레디스에는 데이터가 없어야** 한다.

#### 비어 있지 않은 노드 추가 시 에러
```
[ERR] Node 192.168.0.77:6379 is not empty. Either the node already knows
other nodes (check with CLUSTER NODES) or contains some key in database 0.
```
또한 추가하려는 노드도 설정 파일에 `cluster-enabled yes`로 클러스터 노드로 실행된 상태여야 한다.

#### 마스터로 추가하기
```bash
redis-cli --cluster add-node <추가할 노드 IP:PORT> <기존 노드 IP:PORT>
```
- 첫 번째 인수: 새로 추가할 노드
- 두 번째 인수: 기존 클러스터에 속한 노드 중 하나

```bash
$ redis-cli --cluster add-node 192.168.0.77:6379 192.168.0.11:6379
>>> Adding node 192.168.0.77:6379 to cluster 192.168.0.11:6379
>>> Performing Cluster Check (using node 192.168.0.11:6379)
...
>>> Send CLUSTER MEET to node 192.168.0.77:6379 to make it join the cluster.
[OK] New node added correctly.
```
- `192.168.0.77`이 마스터로 추가됨
- **단, 추가된 마스터는 할당된 해시슬롯이 없어 데이터를 보유할 수 없다.** 데이터를 저장하려면 **리샤딩으로 직접 해시슬롯을 할당**해야 한다.

#### 복제본으로 추가하기
마스터 추가와 동일하되 **`--cluster-slave`** 옵션 추가.
```bash
redis-cli --cluster add-node <추가할 노드 IP:PORT> <기존 노드 IP:PORT> --cluster-slave \
  [--cluster-master-id <기존 마스터 ID>]
```
- `--cluster-master-id` 지정 시 → 그 마스터의 복제본으로 연결.
- 미지정 시 → **임의의 마스터의 복제본**으로 연결되며, 클러스터가 비대칭이면 **복제본이 적게 연결된 마스터**를 파악해 그 마스터의 복제본이 되도록 지정해 균형을 맞춘다.

```bash
$ redis-cli --cluster add-node 192.168.0.77:6379 192.168.0.11:6379 --cluster-slave
>>> Adding node 192.168.0.77:6379 to cluster 192.168.0.11:6379
...
>>> Configure node as replica of 192.168.0.11:6379.
[OK] New node added correctly.
```

### 5-3. 노드 제거하기

클러스터에서 노드를 제거할 때 **`del-node`** 커맨드를 사용한다.
```bash
redis-cli --cluster del-node <기존 노드 IP:PORT> <삭제할 노드 ID>
```

- **복제본**은 바로 제거 가능.
- **마스터를 제거**하려면 먼저 **빈 노드(슬롯이 없는 상태)** 로 만들어야 한다 → 리샤딩으로 해당 마스터의 해시슬롯을 전부 다른 마스터로 옮긴 뒤 제거.

```bash
$ redis-cli --cluster del-node 192.168.0.11:6379 73173c7c742a5659a25e41e0cf288fe24429e2fd
>>> Removing node 73173c7c742a5659a25e41e0cf288fe24429e2fd from cluster
```

## 6. 레디스 클러스터의 데이터 마이그레이션

운영 중인 클러스터의 데이터를 **다른 (소스) 레디스에서 클러스터로 전달**받을 수 있다.

```bash
redis-cli --cluster import 192.168.0.11:6379 --cluster-from 192.168.0.88:6379 \
  --cluster-copy
```
- `import` 받을 클러스터 노드 + `--cluster-from`에 소스 노드를 지정.
- 소스 데이터를 클러스터 내 마스터로 해시슬롯에 맞춰 분산 저장.
- 마이그레이션 중에도 원본 데이터는 정상 접근 가능.
- `--cluster-copy` : 원본 데이터를 **복사**(소스에서 삭제하지 않음).
- 단, 마이그레이션 도중 소스/클러스터 노드에 변경이 반영되지 않으므로, **양쪽 모두 중단 상태에서 진행하거나 추가된 데이터는 미반영**될 수 있음에 주의.

마이그레이션 로그 예시:
```
Importing 65 keys from DB 0
Migrating list:342 to 192.168.0.22:6379: OK
Migrating myhash:323 to 192.168.0.22:6379: OK
Migrating tag:IT:posts to 192.168.0.11:6379: OK
...
```

## 7. 복제본을 이용한 읽기 성능 향상

기본적으로 클러스터에서 **읽기/쓰기 모두 마스터**가 처리한다. 복제본 노드로 연결을 시도해도 마스터로 리디렉션된다.

```bash
$ redis-cli -h 192.168.0.55
192.168.0.55:6379> get hello
(error) MOVED 866 192.168.0.11:6379       # 일반 모드: 에러만

$ redis-cli -h 192.168.0.55 -c
192.168.0.55:6379> get hello
-> Redirected to slot [866] located at 192.168.0.11:6379   # 클러스터 모드: 마스터로 리디렉션
"world"
192.168.0.11:6379>
```

#### READONLY 모드 — 복제본 직접 읽기
마스터에 읽기 부하가 집중될 때, **읽기는 복제본에서** 수행하도록 분배하면 읽기 성능을 향상시킬 수 있다. 복제본 연결을 `READONLY` 모드로 변경하면 클라이언트가 복제본 데이터를 직접 읽을 수 있다.

```bash
$ redis-cli -h 192.168.0.55 -c
192.168.0.55:6379> readonly
OK
192.168.0.55:6379> get hello
"world"                                    # 복제본에서 직접 읽음(리디렉션 X)
```

> 쓰기는 마스터, 읽기는 복제본으로 분배 -> 읽기가 확장됨(Read scaling).

## 8. 레디스 클러스터 동작 방법 (내부 메커니즘)

클러스터가 내부적으로 어떻게 동작하는지 자세히 알아두면 운영 중 클러스터 구성에서 문제가 발생했을 때 원인 파악이 쉬워진다.

### 8-1. 하트비트 패킷 (Heartbeat Packet)

클러스터 노드들은 지속적으로 서로의 상태를 확인하기 위해 **PING, PONG 패킷**을 주고받는다. 이 두 패킷을 묶어서 **하트비트 패킷**이라 한다. 일반적으로 클러스터가 주고받는 유형의 패킷에 가십 섹션이 추가된 형태를 띤다.

#### 하트비트 패킷의 구성
**공통 헤더**
- 노드 ID
- 현재 에포크 / 구성 에포크
- 플래그 (마스터/복제본 여부, 노드 플래그 등)
- 비트맵 (마스터가 제공하는 해시슬롯의 비트맵 정보, 복제본이면 마스터의 비트맵)
- TCP 포트 (발신 노드의 TCP 포트)
- 클러스터 포트 (발신 노드의 클러스터 간 커뮤니케이션 포트)
- 클러스터 상태 (발신 노드 관점에서 본 클러스터 상태 = down/ok)
- 마스터 노드 ID (복제본인 경우 마스터의 노드 ID)

**가십 섹션** (발신 노드가 알고 있는 다른 노드 정보를 포함)
- 노드 ID
- IP / 포트 쌍
- 노드 플래그

가십 섹션을 통해 노드는 **다른 노드를 직접 확인하지 않고도** 다른 노드들의 상태(자신이 알고 있는 일부 노드의 뷰)를 알 수 있다. 이를 이용해 장애도 감지하게 된다.

### 8-2. 해시슬롯 구성이 전파되는 방법

레디스 클러스터에서 사용한 키의 해시슬롯 정보(어떤 해시슬롯이 어느 마스터에 있는지)는 다음 두 가지 방법으로 전파된다.

1. **하트비트 패킷**: 마스터 노드가 PING, PONG 패킷을 보낼 때 **항상 자기가 갖고 있는 해시슬롯을 패킷 데이터에 추가**.
2. **업데이트 메시지**: 하트비트 패킷에는 발신 노드의 **구성 에포크 값**이 포함돼 있다. 패킷을 보낸 노드의 에포크 값이 **오래됐다면**, 패킷을 받은 노드가 **신규 에포크의 구성 정보를 포함한 업데이트 메시지**를 보내, 하트비트를 보낸 노드의 해시슬롯 구성을 업데이트한다.

#### 전파 과정 예시
클러스터 시작 시 모든 노드의 해시슬롯은 `NULL`로 초기화된다.
```
0     -> NULL
1     -> NULL
2     -> NULL
...
16383 -> NULL
```

노드 A가 "해시슬롯 1, 2를 갖고 있다"고 주장(구성 에포크 3)하는 패킷 수신 시:
```
0     -> NULL
1     -> A [3]
2     -> A [3]
...
16383 -> NULL
```

페일오버 발생 후, 현재 에포크 3보다 큰 **에포크 4**를 가진 패킷에서 노드 B가 해시슬롯 1, 2를 갖고 있다고 주장하면:
```
0     -> NULL
1     -> B [4]
2     -> B [4]
```
- 해시슬롯 구성의 변경은 **페일오버와 리샤딩 중에만** 발생한다.
- 둘 다 **에포크가 증가하는 작업**이므로, 작업 이후 변경 사항을 클러스터 전체에 전파.
- 클러스터의 모든 노드는 **가장 큰 구성 에포크 값을 가진 노드에 동의**하므로, 모든 노드 값이 일관되게 업데이트된다.

### 8-3. 노드 핸드셰이크 (Node Handshake)

한 노드가 클러스터에 합류하려면 **`CLUSTER MEET`** 커맨드를 다른 노드에 보낸다.

- 커맨드를 수신한 노드는 자신이 아는 다른 노드들에게 전파
- 신규 노드를 모르는 노드는 `CLUSTER MEET`를 통해 신규 연결을 맺는다.
- 이미 아는 노드면 커맨드를 무시
- 이런 방식으로 클러스터 내부 모든 노드가 **풀 메쉬(full-mesh) 연결**을 이룬다.
- `CLUSTER MEET`는 **방향성이 없다** → A가 B에 보내 연결됐다면 B가 A에 다시 보낼 필요 없다.

> **이점**: 이미 아는 노드끼리만 통신하므로, **클러스터 외부의 다른 클러스터와 우연히 연결되는 것을 방지**할 수 있다.

### 8-4. 클러스터 라이브 재구성 (Live Reconfiguration)

클러스터가 정상 운영되는 동안 새 마스터 추가 / 기존 마스터 제거를 할 수 있다. 클러스터에서 노드 추가와 삭제는 **본질적으로 동일한 작업**(해시슬롯 이동)으로 볼 수 있다.

- **노드 추가** : 빈 노드를 클러스터에 추가한 뒤, 일부 해시슬롯을 기존 마스터에서 신규 마스터로 옮긴다.
- **노드 삭제** : 삭제할 노드의 해시슬롯을 다른 노드로 옮긴 뒤 빈 노드로 만들어 제거

#### 해시슬롯 이동의 내부 커맨드
해시슬롯 8을 노드 A -> B로 옮기는 경우:
```
A에게:  CLUSTER SETSLOT 8 MIGRATING B
B에게:  CLUSTER SETSLOT 8 IMPORTING A
```
이 상태에서 키 조회 요청이 오면:
- 키가 **A에 아직 존재**하면 A에서 처리
- 키가 **없으면** (이미 마이그레이션됐거나 새 키 생성이면) -> B로 리디렉션. 단, 이 키에 대한 새 키 생성은 B에서 수행되어, 마이그레이션 중에도 정합성이 유지됨.

이동 완료 후:
```
모든 마스터에게:  CLUSTER SETSLOT 8 NODE B
```

#### 관련 커맨드
- **`CLUSTER GETKEYSINSLOT <slot> <count>`** : 지정한 해시슬롯이 가진 키를 반환. 이 키들을 `MIGRATE` 커맨드로 노드 A -> B로 옮긴다.
- **`MIGRATE target-host target-port key target-database id timeout`** : 대상 인스턴스에 연결해 **키 하나**를 전송. A -> B로 전송하고 OK를 받으면 A의 기존 데이터를 삭제한다. **개별 키 단위로는 원자적**이라, 외부 클라이언트가 봤을 때 그 키가 A·B 양쪽에 동시에 존재하는 순간은 없다(전송 -> 삭제가 한 단위).

### 8-5. 리디렉션 (Redirection)

레디스 클러스터가 반환하는 리디렉션에는 **MOVED**와 **ASK** 두 종류가 있다.

| 구분 | MOVED 리디렉션 | ASK 리디렉션 |
|------|---------------|-------------|
| 의미 | "이 키는 (영구적으로) 저 노드에 있다" | "이번 요청만 저 노드에서 수행하라" (임시) |
| 발생 시점 | 해시슬롯 자체가 다른 노드로 영구 이동 | 해시슬롯 마이그레이션이 **진행 중**일 때 |
| 클라이언트 동작 | cluster map 갱신 (캐싱된 정보 변경) | **맵을 갱신하지 않음**, 이번 요청만 해당 노드로 |
| 속도 영향 | 큰 영향 없음 | 자주 발생하면 속도가 매우 느려질 수 있음 |

> 실시간으로 트래픽이 많은 서비스에서는 속도가 매우 중요하므로, 두 리디렉션을 명확히 구분해 구현됐다.

#### MOVED 리디렉션
레디스 노드는 클라이언트가 보낸 키가 단일 키 커맨드인지/다중 키 커맨드인지, 연관된 키가 모두 동일한 해시슬롯에 있는지 파악한 뒤, 키가 속한 해시슬롯을 갖고 있다면 응답한다. 아니라면 어떤 노드에 해당 해시슬롯이 있는지 찾아 **MOVED 에러로 클라이언트에 응답**한다.

```
GET x
-MOVED 2345 192.168.0.22:6379
```
- "키 `x`의 해시슬롯(2345)을 가진 마스터는 `192.168.0.22:6379`다"
- 클라이언트는 받은 노드의 IP·포트로 키를 재요청.
- 클라이언트는 "해시슬롯 2345 → 192.168.0.22:6379" 정보를 **캐싱**해 다음부터 바로 올바른 노드로 보낸다.
- 클라이언트가 해시슬롯 매핑을 안정적으로 캐싱하면, **모든 클라이언트가 모든 노드와 연결을 맺어 SPOF(단일 장애점)가 없는** 효율적 서비스로 사용 가능

#### ASK 리디렉션
ASK는 **해시슬롯이 이동되는 과정(마이그레이션 중)** 에서만 발생한다. 이 리디렉션을 받은 클라이언트는 **해당 키에 대해 단 한 번만** 전환한다. 클라이언트의 맵은 갱신하지 않으므로 user:1을 조회하면 다음에도 다시 원래 노드로 간다.

ASK 리디렉션 받은 클라이언트 동작:
- 리디렉션 오류가 반환한 노드 정보로 쿼리를 재전송. 단, 이 리디렉션을 이용한 쿼리는 **한 번만 적용**
- 리디렉션을 받았으면 기준이 되는 노드(B)에 `ASKING` 커맨드를 먼저 보내고 쿼리를 다시 보낸다.
- 리디렉션을 받은 값으로 클라이언트의 **해시슬롯 맵을 업데이트하지 않음**

```
시나리오: 해시슬롯 8을 A -> B로 옮기는 중, user:1은 아직 A에 있는 경우
1. 클라이언트가 A에 get user:1 요청
2. A: 키가 아직 있으므로 Value 반환 (해시슬롯 8 이전 안 끝났고 user:1은 A에 남음)

시나리오: 해시슬롯 8을 A -> B로 옮기는 중, user:1은 이미 B로 옮겨진 경우
1. 클라이언트가 A에 get user:1 요청
2. A: (error) ASK -> "B로 가서 ASKING 후 다시 요청하라"
3. 클라이언트는 ASK 요청을 보내고(=ASKING), 그 후 쿼리를 B로 전송
   → MOVED가 아닌 ASK이므로 마이그레이션 완료되면 8번 해시슬롯은 B로 이동될 예정,
     그러나 클라이언트가 맵을 갱신하면 안 됨(아직 이전 진행 중이므로)
```

### 8-6. 장애 감지와 페일오버

레디스 클러스터에서는 **대부분의 노드가 특정 노드에 접근할 수 없다는 것을 인지하면 해당 노드의 상태를 변경**한다. 장애 감지에 사용되는 플래그가 **PFAIL**과 **FAIL**이다.

#### PFAIL 플래그 (Possible failure)
- 특정 노드에 **`NODE_TIMEOUT` 시간 이상 도달할 수 없는 경우** 해당 노드에 PFAIL 표시.
- 마스터, 복제본 관계없이 클러스터 내 모든 노드는 다른 노드를 PFAIL로 플래그할 수 있다.
- "도달할 수 없다"는 것은 노드에 **PING 패킷을 보냈지만 `NODE_TIMEOUT` 시간 동안 PONG을 받지 못한 상태**를 의미한다.
- `NODE_TIMEOUT`은 항상 RTT(Round Trip Time)보다 오랫동안 PONG을 받지 못했을 때 해당 노드에 다시 PING을 보내는 방식으로 안정성을 향상시킨다.

#### FAIL 플래그
- 실제로 마스터 노드에 장애가 발생했다고 인지해 **페일오버를 트리거하려면** 노드가 PFAIL이 아닌 **FAIL 상태**여야 한다.
- 만약 A 노드가 B를 PFAIL 상태로 플래그했다면, 이후 클러스터 내 다른 노드의 하트비트 패킷에서 B의 상태에 대한 정보를 듣는다.
- 이때 일정 시간 내에 다른 노드들에게서 B에 대한 PFAIL 또는 FAIL 응답을 받으면, A는 B를 FAIL로 플래그한다.
- FAIL로 플래그한 노드는 다른 모든 노드에게 B가 FAIL이라는 정보를 직접 전달(전파)해 **B에 접속한 모든 노드가 B에 장애가 발생했음에 동의**하게 한다.
- FAIL은 대다수의 노드에서 해당 노드에 장애가 발생했음을 동의한 상태임을 의미한다.

#### 복제본 선출 (Replica Election)
다음 조건에서 복제본은 페일오버를 직접 시도한다.
- 마스터가 **FAIL 상태**다.
- 마스터는 **1개 이상의 해시슬롯**을 갖고 있다.
- 마스터와의 복제가 끊어진 지 오래되지 않았다.

위 조건을 모두 만족하면, 복제본은 마스터로 선출되기 위해 **자신의 `currentEpoch`(현재 에포크) 값을 1 증가**시키고, 그 값을 자신의 새 선출 에포크로 삼아 다른 마스터 인스턴스에 투표를 요청한다. (선출에 성공하면 이 에포크가 새 `configEpoch`가 된다.)

```
(1) FAILOVER_AUTH_REQUEST  ← 복제본이 마스터들에게 페일오버 투표 요청
(2) FAILOVER_AUTH_ACK       ← 마스터들이 긍정 응답(투표)
```

**복제본 선출 동작 과정**
1. 우선 복제본은 클러스터의 모든 마스터 노드에 `FAILOVER_AUTH_REQUEST` 패킷을 보내 투표를 요청.
2. 요청을 받은 마스터는 `FAILOVER_AUTH_ACK` 패킷으로 긍정적인 응답을 보낼 수 있다.
3. 이때 마스터는 **동시에 다른 복제본을 승격시키는 것을 방지**하기 위해 `NODE_TIMEOUT * 2` 시간 동안은 같은 마스터에 대한 다른 복제본에게는 투표할 수 없다.
4. 복제본은 자신이 받은 ACK 중 **현재 에포크 값보다 작은 에포크**는 무시한다 (이전 버전의 투표 방지).
5. 복제본은 다수의 마스터로부터 ACK를 받은 뒤(과반수), `NODE_TIMEOUT * 4`만큼의 지연 후에 다시 새로운 투표를 시도할 수 있다.
6. 마스터가 FAIL 상태가 된 이후 복제본은 다음과 같이 계산된 **지연 딜레이를 가진 뒤 투표를 시작**한다.

```
DELAY = 500ms + 랜덤 지연 시간(0~500ms) + SLAVE_RANK * 1000ms
```
- 랜덤한 지연 시간을 이용해 여러 복제본에서 동시에 투표를 시작하는 충돌을 방지.
- **`SLAVE_RANK`** : **이미 동기화한 복제 오프셋**을 기준으로 매겨지는 랭크. 마스터가 FAIL 상태가 되면, 복제 데이터가 가장 최신(=복제 오프셋이 가장 앞선, 가장 많이 복제한)인 복제본이 **랭크 0**, 그다음이 **랭크 1**... 식으로 매겨진다. **랭크가 낮을수록(=데이터가 최신일수록) 딜레이가 짧아 더 빨리 투표를 시작**한다.

> 결국 **가장 최신 데이터를 가진 복제본이 가장 먼저 투표를 시작 → 마스터로 승격될 가능성이 높아** 데이터 유실을 최소화한다.
