# 9장. 센티널 (Sentinel)

## 1. 고가용성 기능의 필요성

### 1.1 복제만으로는 부족하다
레디스는 **인메모리 데이터베이스**다. 모든 데이터를 메모리에서 관리하므로, 백업 설정이 없으면 인스턴스 재시작 시 모든 데이터가 유실된다.

복제본을 구성하면 마스터 장애 시에도 데이터는 복제본 노드에 남아 있다. **하지만 클라이언트가 마스터에 직접 연결된 상태였다면**, 사람이 직접 다음 과정을 거쳐야 장애를 복구할 수 있다.

### 1.2 수동 복구 3단계
```
① 복제본 노드에 REPLICAOF NO ONE 실행
   → 복제 연결을 끊고 독립 마스터로 승격

② 애플리케이션 코드에서 레디스 엔드포인트를
   복제본의 IP로 변경

③ 변경된 애플리케이션을 재배포
```

### 1.3 수동 복구의 문제점
- **다운타임이 길다**: 사람이 인지 → 명령 실행 → 코드 수정 → 배포까지 시간 소요.
- **2차 장애 위험**: look-aside 캐시 구성에서는 레디스가 죽은 동안 모든 읽기가 원본 소스(예: MySQL)로 몰려 **원본 DB 부하가 급증**, 운영 중인 서비스 전체로 장애가 번질 수 있다.

이 과정을 **자동으로** 처리해 줄 장치가 센티널이다.

## 2. 센티널이란?

### 2.1 정의
레디스의 자체 고가용성 기능. 데이터를 저장하는 기존 레디스 인스턴스와는 **다른 역할을 하는 별도의 프로그램**이며, 보통 같은 서버에서 다른 포트(기본 26379)로 실행된다. 자동 페일오버 기능으로 마스터 장애 시에도 레디스를 계속 사용할 수 있게 해 **다운타임을 최소화**한다.

### 2.2 센티널의 3가지 기능 (상세)

**① 모니터링 (Monitoring)**
마스터·복제본 인스턴스의 상태를 실시간으로 확인한다. 주기적으로 PING을 보내 응답 여부로 상태를 파악한다.

**② 자동 페일오버 (Automatic Failover)**
마스터의 비정상 상태를 감지해 정상 상태의 복제본 중 하나를 마스터로 승격시킨다. 기존 마스터에 연결돼 있던 복제본들은 새롭게 승격된 마스터에 연결된다.

**③ 인스턴스 구성 정보 안내 (Configuration Provider)**
센티널은 클라이언트에게 현재 구성에서의 마스터 정보를 알려준다. 페일오버가 발생하면 변경된 마스터 정보를 재전달하기 때문에, **페일오버가 발생하더라도 클라이언트가 레디스 엔드포인트 정보를 변경할 필요가 없다.**

### 2.3 센티널 동작 방식
```
       ①  Master 질의
client ───────────────────▶  Sentinel  Sentinel  Sentinel
       ◀───────────────────    │         │         │
       ②  Master 주소 반환       └────┬────┴─────────┘
                                    │ 감시(PING)
       ③  Master로 직접 연결          ▼
client ──────────────────────▶ Redis Master ── Redis Replica
```
- 클라이언트는 레디스에 직접 붙기 전, **센티널에게 "현재 마스터가 누구냐"를 먼저 질의**한다(①).
- 센티널이 마스터 주소를 반환하면(②), 클라이언트는 그 주소로 **직접 연결**한다(③).
- 즉 데이터 read/write는 클라이언트 ↔ 레디스 간 직접 이뤄지고, 센티널은 "주소 안내소" 역할만 한다.

## 3. 분산 시스템으로 동작하는 센티널

### 3.1 SPOF 방지가 목적
**SPOF(Single Point Of Failure)** 는 하나의 서비스에 문제가 생겼을 때 전체 시스템이 영향받는 지점을 뜻한다.

복제와 자동 페일오버로 고가용성을 확보하는 이유는 **레디스가 SPOF가 되는 것을 방지**하기 위함이다. 그런데 고가용성을 위해 도입한 센티널이 1대뿐이라 그 센티널이 SPOF가 돼 버린다면 애초에 도입하는 의미가 없다.

### 3.2 최소 3대 이상
센티널은 그 자체로 SPOF가 되는 것을 방지하기 위해 **최소 3대 이상일 때 정상 동작**하도록 설계됐다. 하나의 센티널에 이상이 생겨도 다른 센티널이 계속 역할을 수행한다.

> 클라이언트는 센티널에 먼저 연결해 마스터 정보를 받아온다. 만약 마스터·복제본은 정상인데 센티널 1대에 문제가 생겨도, **다른 센티널에서 마스터 정보를 정상적으로 받아올 수 있다.**

### 3.3 쿼럼(quorum)
센티널은 **오탐을 줄이기 위해** 쿼럼이라는 개념을 사용한다.
- **쿼럼** = 마스터가 비정상 동작을 한다는 것에 **동의해야 하는 센티널의 수**.
- 쿼럼을 만족하는 경우 페일오버를 시작한다.
- 일반적으로 센티널 3개일 때 쿼럼은 **2**로 설정. 즉 최소 2대 이상이 마스터 비정상 상태에 동의하면 페일오버 프로세스를 시작한다.

### 3.4 홀수로 구성하는 이유
센티널은 쿼럼을 이용한 **과반수 선출** 개념을 사용하기 때문에 **3대 이상의 홀수로 구성**하는 것이 좋다. 보통 3대로 구성하며, 좀 더 견고하게 장애에 대처하려면 5대로 구성할 수 있다.

> **왜 홀수인가?** 짝수(예: 4대)로 구성하면 2:2로 표가 갈렸을 때 과반수를 못 만들어 리더 선출이 실패할 수 있다. 홀수는 이런 교착을 방지하고, 같은 대수 대비 동일한 내결함성을 더 적은 노드로 확보한다.

## 4. 센티널 인스턴스 배치 방법

### 4.1 배치 원칙
센티널 인스턴스는 물리적으로 **서로 영향받지 않는 서버**에서 실행되는 것이 좋다. 마스터의 장애를 제대로 감지하려면 마스터와 운명을 같이하면 안 되므로, **서로 다른 가용 영역(AZ)에 배치**하는 것이 일반적이다.

### 4.2 복제본 2개 구성 (가장 안정적)
```
        Server A
     ┌────────────┐
     │Redis Master│
     │  Sentinel  │
     └────────────┘
  Server B          Server C
┌─────────────┐   ┌─────────────┐
│Redis Replica│   │Redis Replica│
│  Sentinel   │   │  Sentinel   │
└─────────────┘   └─────────────┘
        quorum: 2
```
- 보통 하나의 서버에 **레디스 프로세스와 센티널 프로세스를 동시에** 실행한다.
- 서버 리소스에 여유가 있다면 이처럼 복제본 2개 구성이 **가장 안정적**이다.

**페일오버 시나리오**
1. 서버 A 장애로 마스터·센티널 접근 불가 -> 서버 B, C의 센티널이 "마스터 접근 불가"에 동의 → 페일오버 진행.
2. 서버 B의 복제본이 새 마스터로 선출되면, 서버 C의 복제본은 **서버 B를 바라보도록 복제 연결을 변경**.
3. 기존 마스터를 보던 클라이언트는 모두 서버 B의 새 마스터로 연결되고, 새 커넥션도 서버 B 주소를 전달받는다.
4. 서버 A가 복구되면, 센티널이 **서버 A의 레디스를 서버 B(새 마스터)의 복제본이 되도록 자동 연결**한다. → 이 과정은 센티널 판단으로 자동 구성되며 **운영자 개입이 불필요**하다.

### 4.3 복제본 1개 구성 (리소스 절약)
```
   Server A         Server B
┌────────────┐   ┌─────────────┐
│Redis Master│   │Redis Replica│
│  Sentinel  │   │  Sentinel   │
└────────────┘   └─────────────┘
        Server C
     ┌──────────┐
     │ Sentinel │   <- 센티널만 (최저 사양)
     └──────────┘
        quorum: 2
```
- 경우에 따라 복제본 1개로도 충분한 서비스가 있다. 이때 2대 서버(A, B)에는 레디스+센티널을, **나머지 1대(C)에는 센티널 프로세스만** 실행.
- 서버 C는 데이터를 저장하지도, 클라이언트 요청을 받지도 않으므로 **최저 사양 스펙으로 구성**해도 된다.
- 리소스 비용을 아끼면서도 센티널 3대를 유지해 마스터 장애를 감지하고 자동 페일오버를 수행할 수 있는 안정적인 구조.

## 5. 센티널 인스턴스 실행하기

### 5.1 실행 예제 구성
```
   192.168.0.11      192.168.0.22
┌────────────┐    ┌─────────────┐
│Redis Master│    │Redis Replica│
│  Sentinel  │    │  Sentinel   │
└────────────┘    └─────────────┘
      192.168.0.33
     ┌──────────┐
     │ Sentinel │
     └──────────┘
   quorum: 2
   Redis Port: 6379
   Sentinel Port: 26379
```
- 2대 서버(192.168.0.11, 192.168.0.22)에는 레디스+센티널을, 192.168.0.33에는 센티널만.
- 쿼럼 2, 모든 레디스는 6379, 모든 센티널은 26379 포트 사용.

### 5.2 ① 복제 연결 먼저 (복제본 노드에서)
센티널 프로세스를 실행하기 전, 마스터-복제본 노드 간 복제 연결을 먼저 만들어 둔다.
```
REPLICAOF 192.168.0.11 6379
```

### 5.3 ② sentinel.conf 작성
센티널 프로세스를 띄우려면 `sentinel.conf`라는 별도의 구성 파일이 필요하다.
```
port 26379
sentinel monitor master-test 192.168.0.11 6379 2
```
- **`port`**: 센티널 프로세스가 실행될 포트 (위 예제는 26379).
- **`sentinel monitor <마스터이름> <ip> <port> <quorum>`**:
  - 모니터링할 마스터에 **이름**을 부여하고 주소·쿼럼을 지정.
  - 마스터 이름에는 특수문자·공백 불가. **알파벳, 숫자, `.`, `-`, `_` 만** 사용 가능.
  - **복제본 정보는 직접 입력하지 않아도 된다.** 센티널 프로세스가 시작되면 마스터에 연결된 복제본을 **자동으로 찾아내는 과정**을 거친다.

### 5.4 ③ 센티널 실행 (두 가지 방법, 동일 동작)
```bash
# 방법 1: redis-sentinel 이용
redis-sentinel /path/to/sentinel.conf

# 방법 2: redis-server에 --sentinel 옵션
redis-server /path/to/sentinel.conf --sentinel
```
> 두 방법 모두 명시된 `sentinel.conf` 파일로 센티널을 시작한다. **지정된 위치에 파일이 없거나, 해당 경로에 데이터를 쓸 수 없으면 인스턴스는 시작되지 않는다.** (센티널은 실행 중 알아낸 구성을 이 파일에 다시 기록하기 때문)

## 6. 센티널 정보 조회

### 6.1 센티널 접속
레디스 프로세스에 접근할 때와 동일하게 `redis-cli`로 센티널에 직접 접근한다.
```bash
redis-cli -p 26379
```
- 접속하면 센티널이 모니터링 중인 **마스터·복제본 노드 정보**와 **함께 모니터링 중인 다른 센티널 정보**를 확인할 수 있다.
- 단, **레디스 인스턴스가 가진 데이터(키-값)는 확인할 수 없다.**

### 6.2 `SENTINEL master <master-name>`
마스터의 IP, 포트, 연결된 복제본 개수 등 다양한 정보를 확인한다.
```
sentinel> SENTINEL master master-test
 1) "name"          2) "master-test"
 3) "ip"            4) "192.168.0.11"
 5) "port"          6) "6379"
 7) "runid"         8) "953ae6a589449c13ddefaee3538d356d287f509b"
 9) "flags"        10) "master"
 ...
21) "down-after-milliseconds"  22) "5000"
 ...
29) "config-epoch"  30) "1"
31) "num-slaves"    32) "1"
33) "num-other-sentinels"  34) "2"
35) "quorum"        36) "2"
37) "failover-timeout"     38) "60000"
39) "parallel-syncs"       40) "1"
```

**확인하면 좋은 플래그 3가지**
| 플래그 | 의미 |
|--------|------|
| `num-other-sentinels` | 같은 마스터를 모니터링하는 **다른** 센티널 수. 예제는 2 → 자신 외 2대가 더 존재 |
| `flags` | 마스터 상태. 정상은 `master`, 비정상이면 `s_down` 또는 `o_down` 등으로 변경 |
| `num-slaves` | 현재 마스터에 연결된 복제본 개수. 예제는 1 |

### 6.3 `SENTINEL replicas <master-name>`
마스터에 연결된 복제본의 자세한 정보(주소, runid, flags=`slave` 등)를 확인한다.

### 6.4 `SENTINEL sentinels <master-name>`
같은 마스터를 함께 모니터링하는 **다른 센티널**의 정보(flags=`sentinel`)를 확인한다. (자기 자신 제외)

### 6.5 `SENTINEL ckquorum <master-name>`
마스터를 바라보는 센티널 수가 설정한 쿼럼 값(및 과반수)을 만족하는지 확인한다.
```
# 센티널 3대 모두 정상 (쿼럼 2)
sentinel> SENTINEL ckquorum master-test
OK 3 usable Sentinels. Quorum and failover authorization can be reached

# 1대 장애, 2대 정상 (쿼럼 2 충족)
OK 2 usable Sentinels. Quorum and failover authorization can be reached

# 2대 장애, 1대만 정상 (쿼럼 미달)
(error) NOQUORUM 1 usable Sentinels. Not enough available Sentinels to reach
the specified quorum for this master. Not enough available Sentinels to reach
the majority and authorize a failover
```
> 정상 센티널 대수가 쿼럼보다 작으면, 마스터 장애가 발생해도 투표를 진행할 수 없어 **페일오버를 자동 실행할 수 없다.**

## 7. 페일오버 테스트

운영 서비스 투입 전, 마스터 프로세스에 이슈가 생겼을 때 정상적으로 페일오버가 동작하는지 확인하는 것이 좋다. 이에 대한 두 가지 방법이 있다.

### 7.1 수동 페일오버 — `SENTINEL FAILOVER <master name>`
```
SENTINEL FAILOVER master-test
```
- 센티널에 직접 접속해 이 커맨드를 사용하면 **다른 센티널의 동의를 구하지 않고도 페일오버를 즉시 발생**시킬 수 있다.
- **실제 마스터 상태가 정상이었을 때도** 이 커맨드를 쓰면 마스터-복제본 간 **롤 체인지(role change)** 가 발생한다.
- 센티널-복제본 간 네트워크 단절 등으로 페일오버가 실패하지 않는지, 애플리케이션 커넥션이 롤 체인지된 마스터에 정상 연결되는지 등을 확인할 수 있다.

### 7.2 자동 페일오버 — 마스터 강제 종료
```bash
redis-cli -h <master-host> -p <master-port> shutdown
```
- 직접 마스터 노드에 장애를 발생시켜 센티널이 이를 인지하는지 확인.
- 센티널은 주기적으로 마스터에 PING을 보내 상태를 파악하는데, `sentinel.conf`의 **`down-after-milliseconds`** 시간 동안 응답이 없으면 비정상으로 판단해 페일오버를 트리거한다.
- 이 옵션의 **기본값은 30,000밀리초(= 30초)**. 따라서 마스터 프로세스를 다운시킨 뒤 **30초가 지나면** 페일오버가 발생하는지 확인할 수 있다.

## 8. 센티널 운영하기

### 8.1 패스워드 인증
마스터·복제본 노드에 `requirepass`/`masterauth`로 패스워드를 설정한 경우, 센티널 설정 파일에서도 패스워드를 지정해야 한다.
```
sentinel auth-pass <master-name> <password>
```
> 센티널 구성에서는 장애 시 **모든 레디스 노드가 마스터가 될 가능성**이 있으므로, 하나의 복제 그룹 내 `requirepass`와 `masterauth` 값은 **모든 노드에서 동일하게** 설정해야 한다.

### 8.2 복제본 우선순위 (`replica-priority`)
모든 레디스 인스턴스는 `replica-priority` 파라미터를 가진다. 센티널은 페일오버 시 각 복제본의 이 값을 확인하고 **가장 작은 노드를 마스터로 선출**한다.
- 기본값 100이며, 이 값이 **`0`이면 해당 복제본은 절대 마스터로 선출되지 않는다.** (예: 백업 전용 복제본, DR 사이트 복제본)

### 8.3 운영 중 구성 정보 변경
센티널은 실행 도중 모니터링할 마스터를 추가·제거·변경할 수 있다.
> **주의**: 마스터를 모니터링하는 센티널이 여러 대라면 **각각의 센티널에 모두 설정을 적용**해야 한다. 한 센티널의 설정 변경은 다른 센티널로 **전파되지 않는다.**

```
SENTINEL MONITOR <master name> <ip> <port> <quorum>   # 새 마스터 모니터링 시작
SENTINEL REMOVE <master name>                          # 모니터링 중단 (내부 상태에서 완전 제거)
SENTINEL SET <name> [<option> <value> ...]             # 특정 마스터 파라미터 변경
```

**`SENTINEL SET` 예시**
```
sentinel> SENTINEL SET master-test down-after-milliseconds 1000
OK
sentinel> SENTINEL SET master-test quorum 1
OK
```

**`SENTINEL CONFIG GET/SET` (레디스 6.2버전 이상)**
각 마스터에 종속되지 않는 센티널 고유 설정값을 런타임에 변경할 수 있다. (레디스 노드의 `CONFIG GET/SET`과 유사)
```
sentinel> SENTINEL CONFIG GET announce*
1) "announce-hostnames"  2) "no"
3) "announce-ip"         4) ""
5) "announce-port"       6) "0"
```

### 8.4 센티널 초기화 — `SENTINEL RESET <master name>`
센티널은 비정상이라 판단한 노드도 **임의로 모니터링을 멈추지 않는다.** 페일오버된 뒤 접근 불가한 옛 마스터나, 장애로 접근 불가한 복제본도 계속 모니터링을 시도한다.

따라서 더 이상 사용하지 않는 노드를 목록에서 지우려면 `SENTINEL RESET`으로 상태 정보를 초기화해야 한다.
```
SENTINEL RESET <master name>   # 특정 마스터
SENTINEL RESET *               # 모니터링 중인 전체 마스터
```
- RESET을 실행하면 센티널의 상태 정보를 초기화하고, 모니터링 중인 마스터·복제본·다른 센티널 정보를 새로 고친다.

### 8.5 센티널 노드의 추가/제거

**노드 추가**
- 마스터를 모니터링하도록 설정한 새 센티널을 실행하면, **자동 검색 메커니즘**에 의해 기존 센티널의 `known-list`에 자동 추가된다.
- 한 번에 여러 대를 추가할 때는 **10초 이상 간격**을 두면서 1개씩 천천히 추가하는 것이 오류 발생 가능성을 줄인다.
- 추가 완료 후 `SENTINEL MASTER <name>`의 `num-other-sentinels` 값으로 정상 추가를 확인한다.

**노드 제거**
- 센티널끼리는 오랜 시간 응답이 없어도 `known-list`에서 지우지 않는다.
- 따라서 제거할 센티널 **프로세스를 먼저 종료**한 뒤, `SENTINEL RESET *`로 모니터링 정보를 리셋한다.
- 센티널 인스턴스 사이에는 **최소 30초의 대기 시간**을 두고 차례대로 실행하는 것이 좋다.
- 제거 완료 후에도 `num-other-sentinels` 값으로 정상 제거를 확인.

## 9. 센티널의 자동 페일오버 과정

여러 센티널 노드가 레디스를 함께 감시하므로 상태에 대한 오탐을 줄일 수 있다. 페일오버는 다음 6단계로 진행되며, 각 단계는 `sentinel.log`에 기록을 남긴다.

```
① 장애 감지 → sdown    ② odown 전환    ③ epoch 증가
        │                   │                │
        ▼                   ▼                ▼
④ 리더 선출 ──────▶ ⑤ 복제본 승격 ──────▶ ⑥ 복제 연결 변경 → 완료
```

### 9.1 ① 마스터의 장애 상황 감지 → sdown
- 센티널은 `down-after-milliseconds` 지정 값 이상 동안 마스터에 보낸 PING에 **유효한 응답**을 받지 못하면 마스터가 다운됐다고 판단한다.
- **유효한 응답**: `+PONG`, `-LOADING`, `-MASTERDOWN`. 그 외 다른 응답이나 무응답은 모두 유효하지 않다고 판단.
- 예: `down-after-milliseconds`가 30,000(= 30초)인데 마스터가 29초마다 응답하면, 센티널은 마스터를 **정상**으로 인지한다.
- 하나의 센티널이 응답을 늦게 받으면 마스터 상태를 우선 **`sdown`(subjectively down, 주관적 다운)** 으로 플래깅한다.

```
sentinel.log
+sdown
```

### 9.2 ② odown 전환
- sdown으로 플래깅한 센티널은 다른 센티널들에게 다음 커맨드를 보내 장애 사실을 전파한다.
  ```
  SENTINEL is-master-down-by-addr <master-ip> <master-port> <current-epoch> <*>
  ```
- 커맨드를 받은 센티널들은 해당 마스터의 장애 인지 여부를 응답한다. **자기 자신을 포함해 쿼럼 값 이상**의 센티널이 마스터 장애를 인지하면, 마스터 상태를 **`odown`(objectively down, 객관적 다운)** 으로 변경한다.

```
sentinel.log
+sdown
+odown
```

**odown은 마스터 전용**
> 센티널은 모든 레디스 노드를 모니터링하므로, 복제본 노드 장애도 인지해 복제본을 `sdown`으로 플래깅한다. **하지만 복제본을 `odown`으로 변경하거나 장애를 전파하지는 않는다.** 장애 전파는 오직 **마스터 노드에 대해서만** 이뤄진다.
> 단, 페일오버 시 **`sdown` 상태인 복제본은 마스터로 승격되도록 선택되지 않는다.**

### 9.3 ③ epoch 증가
- 처음으로 마스터를 odown으로 인지한 센티널이 페일오버 과정을 시작한다. 시작 전 우선 **epoch(에포크)** 값을 1 증가시킨다.
- epoch는 각 마스터에서 발생한 **페일오버의 버전**을 관리하는 증가하는 숫자값이며, 처음 페일오버 시 epoch는 1이 된다.
- 새 페일오버마다 1씩 증가하며, **동일한 epoch 값을 이용해 페일오버 진행 중 모든 센티널이 같은 작업을 시도하고 있음을 보장**한다.

```
sentinel.log
+new-epoch
+try-failover
```

### 9.4 ④ 센티널 리더 선출
- epoch를 증가시킨 센티널은 다른 센티널에게 **리더로 선출되기 위해 투표하라는 메시지**(`vote-for-leader`)를 보낸다. 이때 증가시킨 epoch를 함께 전달한다.
- 메시지를 받은 센티널이 **전달받은 epoch가 자신의 epoch보다 크면**, 자신의 epoch를 증가시킨 뒤 리더에게 투표하겠다는 응답을 보낸다.
- 투표 요청 시 전달받은 epoch가 **자신의 epoch와 같으면**, 이미 리더로 선출한 센티널의 id를 응답한다.
- **하나의 epoch에서 센티널은 하나의 센티널에만 투표할 수 있고, 투표 결과는 변경할 수 없다.**

```
sentinel.log
+vote-for-leader
+elected-leader
```

> **NOTE — 과반수와 쿼럼**
> - 센티널이 마스터를 `sdown → odown`으로 변경하려면 **쿼럼 값 이상**의 센티널 동의가 필요하다.
> - 하지만 페일오버를 실제로 시도하기 위해 센티널 리더를 선출할 때는, 쿼럼 값이 아니라 **실제 센티널 개수 중 과반수 이상**의 동의를 얻어야만 리더가 선출된다.
> - 쿼럼 값보다 큰 센티널이 동의해도 그 수가 과반수보다 작으면 **페일오버는 발생하지 않는다.**
> - 예: 센티널 5개·쿼럼 2일 때, 마스터 상태는 sdown→odown으로 바뀌어 페일오버가 트리거될 수는 있으나, **2개의 센티널만 동의하면 리더를 선출할 수 없어 페일오버가 발생하지 않는다.** (과반수 = 3 필요)

### 9.5 ⑤ 복제본 선정 후 마스터로 승격
과반수 이상의 센티널이 페일오버에 동의했다면, 리더 센티널이 마스터가 될 적당한 복제본을 선정한다. 마스터로부터 **오랜 기간 연결이 끊겼던 복제본은 승격 자격이 없으며**, 자격 있는 복제본은 다음 순서로 선출된다.

```
1순위: redis.conf의 replica-priority가 낮은 복제본 (0이면 후보 제외)
2순위: 마스터로부터 더 많은 데이터를 수신한 복제본 (master_repl_offset이 큰 쪽)
3순위: 2순위까지 동일하면, runID가 사전 순으로 작은 복제본
```
- 작은 runID 선택에 특별한 의미는 없으며, 임의로 하나의 노드를 선택하는 방식이다.
- 선정된 복제본에는 **`slaveof no one`**(= `REPLICAOF NO ONE`) 커맨드를 수행해 기존 마스터로부터의 복제를 끊는다.

```
sentinel.log
+failover-state-select-slave
+selected-slave
+failover-state-send-slaveof-noone
+failover-state-wait-promotion
+promoted-slave
```

### 9.6 ⑥ 복제 연결 변경 → 장애 조치 완료
- 기존 마스터에 연결돼 있던 다른 복제본들이 새로 승격된 마스터의 복제본이 되도록, 복제본마다 **`replicaof <new-ip> <new-port>`** 커맨드를 수행해 복제 연결을 변경한다.
- 복제 그룹의 모든 센티널 노드에서도 레디스 구성 정보를 변경한다.
- 모든 과정이 완료된 뒤 센티널은 **새로운 마스터를 모니터링**한다.

```
sentinel.log
+failover-state-reconf-slaves
+slave-reconf-sent
+slave-reconf-inprog
+slave-reconf-done
+config-update-from sentinel
+failover-end
+switch-master      <- 페일오버 완료
```

**페일오버 로그 전체 흐름 요약**
```
+sdown → +odown → +new-epoch → +try-failover → +vote-for-leader
→ +elected-leader → +failover-state-select-slave → +selected-slave
→ +failover-state-send-slaveof-noone → +promoted-slave
→ +failover-state-reconf-slaves → +slave-reconf-{sent,inprog,done}
→ +failover-end → +switch-master
```

## 10. 스플릿 브레인 현상 (Split Brain)

### 10.1 정의
**스플릿 브레인**이란 네트워크 파티션 이슈로 분산 환경의 데이터 저장소가 끊어지고, **끊긴 두 부분이 각각을 정상적인 서비스라고 인식**하는 현상이다.

### 10.2 발생 과정
```
              ✂ 네트워크 단절
Sentinel A  │  Sentinel B   Sentinel C
    │       │      │            │
    └───────┼──────┴────────────┘
  Redis     │   (과반수 B,C가 이쪽)
  Master    │   Redis Replica → 새 Master로 승격
            ✂
```
1. 마스터+Sentinel A가 한쪽, 복제본+Sentinel B·C가 다른 쪽으로 네트워크 단절
2. 단절이 길어지면 센티널 B, C는 마스터 노드 접근이 정상적이지 않다고 감지 -> 복제본을 마스터로 승격
3. **과반수 이상의 센티널(B, C)이 같은 네트워크 파티션에 존재**하므로 복제본 승격이 가능하다.
4. 그런데 마스터 인스턴스에는 장애가 없고 **단지 네트워크 단절**만 일어난 경우라면, 하나의 복제본에 **마스터가 2개**가 생기는 스플릿 브레인이 발생한다.

### 10.3 결과
- 기존 client는 옛 마스터에, 새 client는 센티널이 알려준 새 마스터에 각각 쓰기를 하게 되어 **데이터가 갈라진다.**
- 네트워크 복구 시 한쪽 마스터는 복제본으로 강등되며, 그동안의 쓰기가 **유실**될 수 있다.

### 10.4 완화책
- **`min-replicas-to-write`** / **`min-replicas-max-lag`**: "복제본 N대 이상에 닿지 않으면 쓰기 거부"를 설정해, 고립된 옛 마스터가 쓰기를 받지 못하도록 막는다.
- 센티널을 서로 다른 가용 영역에 분산 배치해 네트워크 파티션 영향을 줄인다.
