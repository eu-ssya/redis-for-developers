# 개발자를 위한 레디스 - 6장 정리 · 메시지 브로커로서의 레디스


## 도입 — 왜 메시지 브로커가 필요한가

> 최근의 서비스 아키텍처는 여러 모듈이 서로 느슨하고 적절하게 연결시킨 구조를 채택하고 있다.
> 이런 형태의 구조는 다양한 장점을 갖고 있지만, 모듈 간 서로 탄탄한 상호 작용이 필요하기 때문에 효율적인 메시징 솔루션, 메시지 브로커를 필요로 한다.
> 서비스 간 커넥션이 실패하는 상황은 언제나 발생할 수 있다. 서비스 배포처럼 예정된 작업 도중 발생할 수도 있고, 예기치 못한 장애로 인해 통신이 안 될 상황도 고려해야 한다.
> 따라서 모듈 간의 통신에서는 되도록 비동기 통신을 사용하는 것을 권장하며, 동기 통신의 횟수를 최대한 줄이는 것이 바람직하다.
>
> — 김가림, 『개발자를 위한 레디스』 6장 도입

마이크로서비스 환경에서는 서비스 A가 서비스 B를 동기로 호출하면 다음과 같은 문제가 발생합니다.

- B가 잠시 다운되거나 배포 중이면 A의 요청도 같이 실패
- B의 응답이 느려지면 A도 느려짐 (latency 전염)
- 트래픽이 몰리면 B의 부하가 A의 응답시간을 결정

서비스 간 통신이 불가능한 상황이 바로 장애로 이어지지 않게, **당장 메시지를 처리하지 못하더라도 보낸 메시지를 어딘가에 쌓아둔 뒤 나중에 처리할 수 있는 채널**을 만들어 주는 것 — 이것이 메시지 브로커의 핵심 역할입니다.

```
[동기 통신]
A → B 호출 → B 응답 대기 → A 다음 작업
              ⚠ B가 죽으면 A도 실패

[비동기 통신 + 메시지 브로커]
A → Broker (메시지 저장)
              ↓ (나중에)
              B가 살아나면 가져가 처리
A는 즉시 다음 작업 진행
```

메시지 브로커는 크게 **메시징 큐(Messaging Queue)** 와 **이벤트 스트림(Event Stream)** 두 가지 형태로 나뉩니다.

## 메시징 큐와 이벤트 스트림의 차이

### 용어 — 생산자/소비자 vs 발행자/구독자

| 형태 | 메시지 만드는 쪽 | 메시지 읽는 쪽 |
|------|------------------|-----------------|
| **메시징 큐** | 생산자 (Producer) | 소비자 (Consumer) |
| **이벤트 스트림** | 발행자 (Publisher) | 구독자 (Subscriber) |

용어가 다르지만 본질은 같습니다 — "넣는 쪽 vs 빼는 쪽". 다만 두 형태의 동작 방식이 다르므로 용어도 구분해 씁니다.

---

### 방향성 — Push vs Pull

**메시징 큐 (Push 모델)**

생산자가 소비자의 큐로 데이터를 **직접 푸시**합니다.

```
Producer ──push── Queue-A ──→ Consumer-A
Producer ──push── Queue-B ──→ Consumer-B
```

2개의 서비스에 같은 메시지를 보내야 할 때, 메시징 큐를 이용한다면 **생산자는 2개의 각각 다른 메시징 큐에 각각 데이터를 push**해야 합니다. 즉 생산자가 소비자 수만큼 책임을 지는 구조.

**이벤트 스트림 (Pull 모델)**

생산자는 스트림의 특정 저장소에 **하나의 메시지**를 보낼 수 있고, 메시지를 읽어가고자 하는 소비자들은 스트림에서 같은 메시지를 **pull**할 수 있습니다.

```
                  ┌──→ Subscriber-A (pull)
Publisher ──push── Stream ──┤
                  └──→ Subscriber-B (pull)
                  └──→ Subscriber-C (pull)
```

**메시지를 복제해서 저장하지 않아도 된다**는 점이 핵심 이점.

> 💡 **방향성이 미치는 영향**
> 메시징 큐는 "이 일을 누가 처리할지" 명확한 작업 분배에 강하고, 이벤트 스트림은 "이 사실을 알아야 할 모든 사람이 알게" 하는 광역 전파에 강합니다.

---

### 데이터 영속성 — 일회성 vs 보관

**메시징 큐**

소비자가 데이터를 읽어갈 때 **큐에서 데이터를 삭제**합니다. 한 번 소비된 메시지는 사라집니다.

**이벤트 스트림**

구독자가 읽어간 데이터는 **바로 삭제되지 않고**, 저장소의 설정에 따라 특정 기간 동안 저장될 수 있습니다.

#### 새로운 소비자/구독자가 추가될 때의 차이

이 차이가 가장 극명하게 드러나는 시나리오입니다.

**메시징 큐의 경우**

```
T=1  Producer → 메시지 1 → Consumer-A 수신 → 큐에서 삭제
T=2  Consumer-B 추가
T=3  Producer → 메시지 2 → ?
```

여기서 Consumer-B는 **오직 메시지 2만** 받을 수 있습니다. 메시지 1은 이미 사라졌습니다.

**이벤트 스트림의 경우**

```
T=1  Publisher → 메시지 1 → Stream에 저장
T=2  Subscriber-A 수신 (메시지 1)
T=3  Subscriber-B 추가 → Stream에 남아있는 메시지 1, 2 모두 조회 가능
```

새로 추가된 서비스도 **스트림에 남아 있는 이전 데이터의 히스토리**를 볼 수 있습니다.

| 비교 축 | 메시징 큐 | 이벤트 스트림 |
|---------|-----------|----------------|
| 메시지 보관 | 소비 즉시 삭제 | 일정 기간 보관 |
| 신규 소비자의 과거 메시지 접근 | ❌ 불가능 | ✅ 가능 |
| 메시지 재처리 | ❌ 어려움 | ✅ 가능 (재구독) |
| 저장 공간 | 적음 (소비 즉시 비움) | 보관 기간만큼 누적 |

---

### 선택 기준

| 상황 | 적합한 형태 |
|------|-------------|
| 일대일, 한 서비스가 다른 서비스에 동작을 지시 | **메시징 큐** |
| 일대다/다대다, 한 이벤트를 여러 서비스가 알아야 함 | **이벤트 스트림** |
| 메시지 재처리·이력 추적이 중요 | **이벤트 스트림** |
| 작업 분배 (job queue) | **메시징 큐** |

> 💡 **실무 예시**
> - 메시징 큐 적합: 결제 처리 작업, 이메일 발송 작업, 이미지 변환 작업 (워커 한 명이 처리하면 끝)
> - 이벤트 스트림 적합: 주문 생성 이벤트 (재고팀·배송팀·분석팀이 각자 소비), 사용자 행동 로그 (분석·마케팅·추천 시스템이 각자 활용)

## 메시지 브로커로서의 레디스 — 3가지 도구

레디스는 메시지 브로커 역할을 위한 세 가지 도구를 제공합니다.

| 도구 | 형태 | 핵심 특징 |
|------|------|-----------|
| **Pub/Sub** | 이벤트 스트림 (단, 영속성 없음) | 가장 가벼운 fire-and-forget |
| **List** | 메시징 큐 | LPUSH/RPOP + 블로킹 지원 |
| **Stream** | 이벤트 스트림 (완전형) | 아파치 카프카에서 영감, append-only, consumer group |

이전 시리즈에서 다룬 자료구조들을 다시 만나게 됩니다. **2-3장에서 자료구조로 배웠던 String/Hash/Set이 4-5장에서 캐싱·세션의 도구가 됐던 것처럼**, 6장에서는 List와 Stream이 메시징 도구로 재등장합니다.

---

## Pub/Sub — 가벼운 fire-and-forget

레디스에서 제공하는 pub/sub을 사용하면 **빠르고 간단한 방식으로 메시지를 전달할 수 있는 메시지 브로커**를 구현할 수 있습니다.

### 기본 동작

발행자가 특정한 채널에 데이터를 전송하면 **이 채널을 듣고 있는 모든 소비자는 데이터를 바로 소비**할 수 있습니다.

```
Publisher → PUBLISH news.tech "Redis 7.4 released"
                          ↓
            ┌──→ Subscriber-A (SUBSCRIBE news.tech)
            ├──→ Subscriber-B (SUBSCRIBE news.tech)
            └──→ Subscriber-C (PSUBSCRIBE news.*)
```

레디스의 pub/sub에서 모든 데이터는 **한 번 채널 전체에 전파된 뒤 삭제되는 일회성**의 특징을 가지며, 메시지가 잘 전달됐는지 등의 정보는 **보장하지 않습니다**.

따라서 완벽하게 메시지가 전달돼야 하는 상황에는 적합하지 않을 수 있지만, **fire-and-forget 패턴**이 필요한 간단한 알림 서비스에서는 유용하게 사용될 수 있습니다.

> 📌 **fire-and-forget 패턴**
> 비동기 프로그래밍에서 사용되는 디자인 패턴으로, 어떤 작업을 실행하고 그 결과에 대한 응답을 기다리지 않고 바로 다음 코드를 실행하는 것. 주로 성능 향상이나 비동기 작업을 수행할 때 사용되며, 작업의 완료나 결과에 대한 처리가 필요하지 않을 때 유용하게 사용됩니다. 예를 들어 **로깅 이벤트 발행, 통계 데이터 수집**과 같이 작업의 성공 또는 실패에 대한 관심이 없는 경우에 활용될 수 있습니다.
>
> 단, 이 패턴을 사용할 때는 결과 확인이나 오류 처리를 고려하지 않고 작업을 진행하므로 신뢰성이 그렇게 필요하지 않은 곳에서만 사용하는 게 좋습니다.

### Pub/Sub의 메타데이터 부재

레디스의 pub/sub은 매우 가볍기 때문에 **최소한의 메시지 전달 기능만 제공**합니다.

- **발행자**는 메시지를 채널로 보낼 수 있을 뿐, 어떤 구독자가 메시지를 읽어가는지, 정상적으로 모든 구독자에게 메시지가 전달됐는지 확인할 수 없음
- **구독자** 또한 메시지를 받을 수 있지만 해당 메시지가 언제 어떤 발행자에 의해 생성됐는지 등의 메타데이터는 알 수 없음
- 한 번 전파된 데이터는 **레디스에 저장되지 않으며**, 단순히 메시지의 통로 역할만 함

만약 특정 구독자에 장애가 생겨 메시지를 받지 못했다 하더라도 그 사실을 알 수 없기 때문에, **정합성이 중요한 데이터를 전달하기에는 적합하지 않을 수 있습니다**. 이럴 경우 애플리케이션 레벨에서 메시지의 송수신과 관련한 로직을 추가해야 할 수 있습니다.

---

### PUBLISH — 메시지 발행

```
PUBLISH channel message
```

레디스에서는 `PUBLISH` 커맨드를 이용해 데이터를 전파할 수 있습니다.

```bash
> PUBLISH news.tech "Redis 7.4 released"
(integer) 3
```

위 커맨드를 수행하면 `news.tech`라는 채널을 수신하고 있는 모든 서버들에 `"Redis 7.4 released"`라는 메시지가 전파됩니다.

메시지가 전파된 후에는 **메시지를 수신한 구독자의 수**가 반환됩니다. 위 예에서는 3명의 구독자가 받았다는 의미.

> 💡 **반환값이 0이면?**
> 아무도 듣고 있지 않다는 뜻. pub/sub은 일회성이므로 그 메시지는 그대로 사라집니다. 이게 보장 없는 fire-and-forget의 본질.

---

### SUBSCRIBE — 메시지 구독

```
SUBSCRIBE channel [channel ...]
```

`SUBSCRIBE` 커맨드를 이용하면 특정 채널을 구독할 수 있습니다.

```bash
> SUBSCRIBE news.tech
Reading messages... (press Ctrl-C to quit)
1) "subscribe"
2) "news.tech"
3) (integer) 1
```

클라이언트가 위의 커맨드를 수행하면 `news.tech` 채널을 동시에 구독하기 시작합니다. 여러 채널을 동시에 구독할 수도 있습니다.

```bash
> SUBSCRIBE news.tech news.health news.sports
```

이후 누군가 해당 채널에 PUBLISH하면 다음과 같은 메시지를 받습니다.

```
1) "message"           # 메시지 타입
2) "news.tech"          # 채널 이름
3) "Redis 7.4 released"  # 본문
```

---

### PSUBSCRIBE — 패턴 매칭 구독

`PSUBSCRIBE` 커맨드를 사용하면 **일치하는 패턴에 해당하는 채널을 한 번에 구독**할 수 있으며, 이때 레디스는 glob-style 패턴을 지원합니다.

```bash
> PSUBSCRIBE news.*
```

위 커맨드를 입력하면 `news.social`, `news.tech`, `news.health` 등 **앞부분이 `news.`로 시작하는 모든 채널**에 전파된 메시지를 모두 수신할 수 있습니다.

SUBSCRIBE와 마찬가지로 동시에 여러 문자열을 구독하는 것도 가능합니다.

#### 메시지 타입의 차이 — message vs pmessage

PSUBSCRIBE로 구독했을 때 받은 메시지는 `message`가 아닌 **`pmessage` 타입**으로 전달되며, SUBSCRIBE 커맨드를 이용해 메시지를 구독하는 방식과 구분됩니다.

```
1) "pmessage"        # 패턴 매칭 메시지
2) "news.*"           # 매칭된 패턴
3) "news.tech"        # 실제 채널
4) "Redis 7.4 released"  # 본문
```

> ⚠️ **중복 수신 주의**
> 만약 구독자가 `SUBSCRIBE news.1`과 `PSUBSCRIBE news.*`로 동시에 구독하고 있을 때 `news.1` 채널에 메시지가 발행되면, **구독자는 2개의 메시지를 받게 됩니다** (message 1개 + pmessage 1개). 애플리케이션에서 메시지 타입으로 구분해 중복 처리를 피해야 합니다.

> 📌 **glob-style 패턴이란?**
> 와일드카드 비스무리한 것으로 보면 됩니다. 2-3장에서 다룬 `KEYS *` 패턴과 동일한 문법입니다.
> - `*` — 0개 이상 임의 문자
> - `?` — 정확히 1개 임의 문자
> - `[abc]` — a, b, c 중 하나
>
> 예: `news.*`, `user.?.login`, `event.[abc].*`

### 구독자가 사용할 수 있는 명령어

클라이언트가 **구독자로 동작할 때는 새로운 채널을 구독할 수 있지만, pub/sub과 관련되지 않은 다른 커맨드를 수행할 수는 없습니다**.

구독자가 수행할 수 있는 커맨드는 다음 9가지뿐입니다.

- `SUBSCRIBE` / `UNSUBSCRIBE` — 채널 구독/해지
- `PSUBSCRIBE` / `PUNSUBSCRIBE` — 패턴 구독/해지
- `SSUBSCRIBE` / `SUNSUBSCRIBE` — 샤드 채널 구독/해지 (클러스터 환경용)
- `PING` — 살아있는지 확인
- `RESET` — 클라이언트 상태 리셋
- `QUIT` — 연결 종료

> ⚠️ **이건 흔한 실수 포인트**
> Spring Data Redis에서 `RedisMessageListenerContainer`를 등록한 커넥션에서 `redisTemplate.opsForValue().get()`을 호출하면 에러가 납니다. 구독 상태의 커넥션은 위 9개 외 어떤 명령도 받지 않기 때문입니다. **구독용 커넥션과 일반 작업용 커넥션을 분리**해야 합니다.

### 한계와 적합한 사용처

| 항목 | Pub/Sub의 동작 |
|------|----------------|
| 메시지 영속성 | ❌ 없음 (일회성) |
| 메시지 전달 보장 | ❌ At-most-once (한 번 시도하고 끝) |
| 구독자 부재 시 | 메시지 사라짐 |
| 발행자/구독자 식별 | ❌ 불가능 |
| 메시지 메타데이터 (timestamp, ID 등) | ❌ 없음 |
| 재처리 | ❌ 불가능 |
| 속도 | ✅ 매우 빠름 |
| 구현 난이도 | ✅ 매우 간단 |

**적합한 사용처**

- 실시간 채팅의 메시지 brokering (메시지는 어차피 클라이언트에 전달되면 충분)
- 캐시 무효화 신호 전파 ("user:42 캐시 지워라" 같은 알림)
- 실시간 알림 (지나간 알림은 의미 없음)
- 분산 노드 간 상태 변경 알림 (예: 설정 변경 broadcast)

**부적합한 사용처**

- 결제·주문 같은 정합성 중요한 메시지
- 작업 큐 (워커가 못 받으면 작업 사라짐)
- 이벤트 sourcing (메시지 이력이 필요)

## List를 메시징 큐로 사용하기

레디스의 자료구조 중 하나인 **list는 큐로 사용하기 적절한 자료구조**입니다.

레디스에는 tail과 head에서 데이터를 넣고 뺄 수 있는 `LPUSH`, `LPOP`, `RPUSH`, `RPOP` 커맨드가 존재하기 때문에, **애플리케이션 특성에 맞는 메시징 큐를 직접 구현**할 수 있다는 장점이 있습니다.

> 📌 **이전 시리즈와의 연결**
> 2-3장에서 자료구조로 학습한 List가 여기서는 **메시징 큐의 자료구조**로 재등장합니다. 같은 List도 어떤 명령어 조합으로 쓰느냐에 따라 전혀 다른 도구가 됩니다 — 이게 레디스가 다재다능한 이유입니다.

### LPUSH / RPOP 기본 큐 패턴

가장 단순한 FIFO 큐는 `LPUSH`(왼쪽으로 넣고) + `RPOP`(오른쪽에서 빼기) 조합입니다.

```bash
# Producer 측
> LPUSH tasks "send-email-1"
(integer) 1
> LPUSH tasks "send-email-2"
(integer) 2
> LPUSH tasks "send-email-3"
(integer) 3

# 현재 상태 (head ← → tail)
# [send-email-3, send-email-2, send-email-1]

# Consumer 측 (FIFO 순서대로 빼기)
> RPOP tasks
"send-email-1"
> RPOP tasks
"send-email-2"
> RPOP tasks
"send-email-3"
```

`LPUSH`는 list의 왼쪽(head)에 추가하고, `RPOP`은 오른쪽(tail)에서 빼므로 가장 먼저 들어간 것이 가장 먼저 나옵니다 — 표준 FIFO 큐.

> 💡 **반대 방향도 가능**
> `RPUSH` + `LPOP`도 동일한 FIFO 동작. 어느 쪽으로 통일할지는 팀 컨벤션 문제. 다만 같은 큐에서 push 방향과 pop 방향이 헷갈리면 안 되므로 한 방향으로 일관되게 사용해야 합니다.

### 블로킹 기능 — BLPOP / BRPOP

레디스를 이벤트 큐로 사용할 경우 **블로킹 기능** 또한 유용하게 사용할 수 있습니다.

#### 폴링의 문제

이벤트 기반 구조에서 시스템은 이벤트 루프를 돌며 신규로 처리할 이벤트가 있는지 체크합니다.

> 📌 **폴링(polling)이란?**
> 이벤트 루프는 이벤트 큐에 새 이벤트가 있는지 체크하며, 새로운 이벤트가 없을 경우 정해진 시간 동안 대기한 뒤 다시 이벤트 큐에 데이터가 있는지 확인하는 과정을 반복합니다.

폴링 프로세스가 진행되는 동안 **애플리케이션과 큐의 리소스가 불필요하게 소모**될 수 있습니다.

또한 이벤트 큐에 이벤트가 들어왔을 수 있지만, **폴링 인터벌 시간 동안은 대기한 뒤 다시 확인하는 과정을 거치기 때문에 이벤트를 즉시 처리할 수 없다**는 단점이 있습니다.

```
Worker → 큐 확인 → 비어있음 → 1초 대기
        ↓ (이 사이에 메시지 도착 — 1초 손실)
Worker → 큐 확인 → 메시지 발견 → 처리
```

#### 블로킹의 작동

이때 list의 **블로킹 기능**을 사용하면 이와 같은 불필요함을 줄일 수 있습니다.

`BRPOP`과 `BLPOP`은 각각 `RPOP`과 `LPOP`에 블로킹을 추가한 커맨드입니다. 클라이언트가 `BLPOP`을 사용해 데이터를 요청했을 때:

- list에 **데이터가 있으면 즉시 반환**
- **데이터가 없을 경우**:
  - list에 데이터가 들어올 때까지 기다린 후에 들어온 값을 반환
  - 또는 클라이언트가 설정한 **타임아웃 시간만큼 대기**한 후에 nil값을 반환

```bash
> BRPOP tasks 3
# tasks에 데이터가 입력될 때까지 최대 3초 동안 대기
# 3초가 경과하면 nil을 반환
```

타임아웃 값을 **0으로 설정하면 데이터가 리스트에 들어올 때까지 제한 없이 기다리라**는 의미로 쓰입니다.

```bash
> BRPOP tasks 0
# 데이터가 들어올 때까지 무한 대기
```

#### 여러 클라이언트가 동시 대기 시

하나의 리스트에 대해 **여러 클라이언트가 동시에 블로킹될 수 있으며, 리스트에 데이터가 입력되면 가장 먼저 요청을 보낸 클라이언트가 데이터를 가져갑니다**.

```
Worker-1: BRPOP tasks 0  (대기 중, 가장 먼저 들어옴)
Worker-2: BRPOP tasks 0  (대기 중)
Worker-3: BRPOP tasks 0  (대기 중)

Producer: LPUSH tasks "job-1"
        ↓
Worker-1만 "job-1" 수신, Worker-2/3는 계속 대기
```

이런 방식으로 **워커 풀(worker pool)** 을 자연스럽게 구현할 수 있습니다. 여러 워커가 같은 큐를 듣고 있다가, 작업이 들어오면 한 워커만 받아 처리.

> 💡 **폴링 vs 블로킹 비교**
> | 항목 | 폴링 | 블로킹 (BLPOP/BRPOP) |
> |------|------|---------------------|
> | 메시지 도착 시 처리 | 인터벌만큼 지연 | **즉시** |
> | 유휴 시 리소스 | 인터벌마다 쿼리 (낭비) | 대기 (효율적) |
> | 구현 복잡도 | 약간 복잡 (스케줄링) | 간단 (한 줄) |
> | 적합한 상황 | 메시지 빈도 매우 높을 때 | **일반적인 경우** |

---

### BRPOP이 2개 값을 반환하는 이유

`BRPOP`은 `RPOP`과 다르게 **2개의 데이터를 반환**합니다.

```bash
> BRPOP tasks 0
1) "tasks"          # 첫 번째: 팝된 리스트의 키 값
2) "send-email-1"   # 두 번째: 반환된 데이터의 값
```

첫 번째는 **팝된 리스트의 키 값**을 반환하고, 두 번째에 반환된 **데이터의 값**을 반환합니다.

> 📌 **bi의 약자라는 의미로 2개라는 뜻**일 수 있습니다 (책의 추정). 다만 이런 설계의 진짜 이유는 따로 있습니다.

이렇게 설계된 이유는 **동시에 여러 개의 리스트에서 대기할 수 있게 하기 위해서**입니다.

```bash
> BRPOP high-priority normal-priority low-priority 0
1) "high-priority"        # 어느 큐에서 왔는지
2) "urgent-task"

# high-priority가 가장 먼저 체크되고, 비어있으면 normal, 그 다음 low 순으로 체크
# 어디서든 첫 번째로 들어온 데이터를 받음
```

여러 큐를 동시에 모니터링할 때 **어느 큐에서 메시지가 왔는지 알아야 처리 로직을 분기할 수 있으므로**, 키 이름을 함께 반환하는 것입니다.

> 💡 **실무 활용**
> 우선순위 큐를 이렇게 구현할 수 있습니다 — high, normal, low 세 큐를 만들고 `BRPOP high normal low 0`로 대기. high가 있으면 그것부터, 없으면 normal, 그것도 없으면 low. 추가 코드 없이도 우선순위 처리 가능.

---

### 원형 큐 — LPUSH + LRANGE

만약 특정 아이템을 **계속해서 반복 접근해야 하는 클라이언트**, 혹은 여러 개의 클라이언트가 **병렬적으로 같은 아이템에 접근**해야 하는 클라이언트에서는 원형 큐(circular queue)를 이용해 아이템을 처리하고 싶을 수 있습니다.

#### 일괄 입력

`LPUSH`를 통해 한꺼번에 여러 개를 집어넣을 수 있습니다.

```bash
> LPUSH circular "item-1" "item-2" "item-3" "item-4" "item-5" "item-6" "item-7"
(integer) 7
```

#### 인덱스 순서로 조회

`LRANGE`로 인덱스 순으로 리스트를 뽑을 수 있습니다.

```bash
> LRANGE circular 0 -1
1) "item-7"
2) "item-6"
3) "item-5"
4) "item-4"
5) "item-3"
6) "item-2"
7) "item-1"
```

`LPUSH`로 head에 넣었기 때문에 가장 나중에 push된 `item-7`이 인덱스 0에 위치합니다. `LRANGE key 0 -1`은 처음부터 끝까지 모두 반환.

> 💡 **원형 큐 활용 예**
> 라운드로빈 방식의 작업 분배, 채팅 봇의 응답 후보 회전, 일정 개수의 최근 이력 관리 등에서 유용합니다.

## Stream — 본격 스트림 플랫폼

레디스의 stream을 사용하면 **레디스를 완벽한 스트림 플랫폼으로 사용**할 수 있습니다.

### List의 한계와 Stream의 등장

List를 메시징 큐로 쓸 때 한계가 있습니다:

- **소비 즉시 삭제** — 다른 컨슈머는 그 메시지를 못 봄
- **메타데이터 부재** — 누가/언제 보낸 메시지인지 알 수 없음
- **재처리 불가** — 한 번 RPOP하면 끝
- **컨슈머 그룹 개념 없음** — 같은 메시지를 여러 그룹이 각자 처리하는 시나리오 어려움

이런 한계를 해결하기 위해 등장한 것이 Stream입니다. 레디스 stream은 **아파치 카프카 시스템에서 영감을 받아 만들어진 자료구조**로, 데이터는 계속해서 추가되는 방식으로 저장됩니다 — **append-only**.

---

### 핵심 개념 — append-only, consumer group, ID

#### Append-Only

새 메시지는 stream의 끝에 추가만 됩니다. 기존 메시지는 변경되지 않으며, 명시적으로 삭제하지 않는 한 보관됩니다.

#### Consumer / Consumer Group

소비자와 소비자 그룹이라는 개념을 이용하면 **카프카에서와 비슷하게 데이터의 분산 처리를 구현**할 수 있습니다.

```
                                    ┌── Consumer-1 (worker-A)
Stream "orders" ──→ Consumer Group ─┼── Consumer-2 (worker-B)
                                    └── Consumer-3 (worker-C)
                  (그룹 내 분산 처리)

같은 Stream을 또 다른 그룹이 별도로 소비 가능:
Stream "orders" ──→ Group "billing"   (결제팀 워커들)
                  → Group "analytics" (분석팀 워커들)
                  → Group "shipping"  (배송팀 워커들)
```

- **한 그룹 안에서**는 메시지가 워커들에게 분산 처리 (작업 분배)
- **그룹 간에는** 같은 메시지를 모두 받음 (광역 전파)

이 구조는 메시징 큐의 장점(분산 처리)과 이벤트 스트림의 장점(다대다 전파) 둘 다 가집니다.

#### ID

각 메시지는 자동 생성되는 ID를 가집니다.

```
1700000000000-0
   ↑              ↑
   타임스탬프(ms)  같은 ms 내 순번
```

이 ID 덕분에:
- 시간대별로 메시지를 검색할 수 있음
- 마지막으로 처리한 위치를 ID로 기억해 재개 가능
- 정확한 순서 보장

---

### 주요 명령어

| 명령어 | 용도 |
|--------|------|
| `XADD stream * field value ...` | 메시지 추가 (ID 자동 생성) |
| `XLEN stream` | 메시지 개수 조회 |
| `XRANGE stream start end` | ID 범위로 조회 |
| `XREAD COUNT n STREAMS stream id` | 특정 ID 이후 메시지 읽기 |
| `XGROUP CREATE stream group $ MKSTREAM` | consumer group 생성 |
| `XREADGROUP GROUP g c COUNT n STREAMS s >` | 그룹으로 메시지 읽기 (분산 처리) |
| `XACK stream group id` | 메시지 처리 완료 확인 |
| `XPENDING stream group` | 미처리 메시지 조회 |
| `XCLAIM stream group consumer min-idle id` | 다른 컨슈머의 메시지 인수 |

#### 기본 사용 예

```bash
# 메시지 추가
> XADD orders * userId 100 amount 50000
"1700000000000-0"

> XADD orders * userId 200 amount 30000
"1700000000001-0"

# 전체 조회
> XRANGE orders - +
1) 1) "1700000000000-0"
   2) 1) "userId"  2) "100"  3) "amount"  4) "50000"
2) 1) "1700000000001-0"
   2) 1) "userId"  2) "200"  3) "amount"  4) "30000"

# Consumer group 생성
> XGROUP CREATE orders order-workers $ MKSTREAM
OK

# 그룹으로 메시지 받기
> XREADGROUP GROUP order-workers worker-1 COUNT 5 STREAMS orders >
# > 표시: 그룹이 아직 안 받은 새 메시지만

# 처리 후 확인
> XACK orders order-workers 1700000000000-0
(integer) 1
```

#### 핵심 활용

stream에 저장되는 메시지를 **실시간으로 리스닝하며 소비**할 수도 있으며, **저장돼 있는 데이터를 시간대별로 검색**하는 것도 가능합니다.

> 💡 **Kafka와의 비교**
> Stream은 Kafka의 핵심 개념(append-only, consumer group, offset)을 가져왔지만 Kafka 전체를 대체하진 않습니다.
> - 메시지 처리량이 초당 수만~수십만이면 Redis Stream으로 충분
> - 초당 수백만 이상의 throughput이나 영구 보관(months)이 필요하면 Kafka가 더 적합
> - 그러나 이미 운영 중인 레디스가 있고 메시징을 추가하고 싶다면 별도 인프라 없이 stream으로 시작 가능

## 3가지 메시지 브로커 도구 비교

| 비교 축 | Pub/Sub | List | Stream |
|---------|---------|------|--------|
| **메시지 패턴** | 이벤트 스트림 (일회성) | 메시징 큐 | 이벤트 스트림 (완전형) |
| **메시지 보관** | ❌ 없음 (즉시 사라짐) | 소비 시 삭제 | ✅ 보관 (명시적 trim 전까지) |
| **전달 보장** | At-most-once | At-least-once (애플리케이션 로직 필요) | At-least-once (XACK) |
| **새 구독자의 과거 메시지** | ❌ 불가능 | ❌ 불가능 | ✅ XRANGE로 조회 |
| **재처리** | ❌ | ❌ | ✅ ID로 재시작 |
| **메시지 ID/메타데이터** | ❌ | ❌ (직접 구현) | ✅ 자동 |
| **블로킹 대기** | 본질적으로 블로킹 | ✅ BLPOP/BRPOP | ✅ XREAD BLOCK |
| **컨슈머 그룹** | ❌ | ❌ | ✅ XGROUP |
| **속도** | ✅ 매우 빠름 | ✅ 매우 빠름 | 약간 무거움 |
| **구현 난이도** | ✅ 매우 간단 | 간단 | 보통 |
| **적합한 상황** | 실시간 알림, fire-and-forget | 작업 큐, 워커 풀 | 이벤트 sourcing, 다대다 전파 |

### 선택 의사결정 흐름

```
메시지를 잃어도 되는가?
├── YES → Pub/Sub
└── NO
    ├── 한 메시지를 한 워커만 처리하면 되는가? (작업 분배)
    │   ├── YES → List + BRPOP
    │   └── NO (여러 그룹이 각자 처리)
    │       └── Stream + Consumer Group
    └── 메시지 이력 추적/재처리 필요한가?
        ├── YES → Stream (필수)
        └── NO → List로 충분
```

---

## 실무 활용 패턴

### Pub/Sub — 캐시 무효화 신호

여러 애플리케이션 서버가 각자 로컬 캐시(Caffeine 등)를 가지고 있고, DB가 갱신될 때 모든 서버의 로컬 캐시를 무효화해야 하는 시나리오.

```
DB 갱신 → PUBLISH cache.invalidate "user:42"
                  ↓ 모든 서버 동시 수신
        SUBSCRIBE cache.invalidate → 로컬 캐시 evict
```

5장에서 다룬 Cache Invalidation 전략의 **분산 환경 버전**입니다.

### List + BRPOP — 이메일 발송 워커

```
[API 서버]
HTTP 요청 → LPUSH email-queue {to, subject, body}
            (즉시 200 응답)

[Email Worker (3대)]
while True:
  result = BRPOP email-queue 0
  send_email(result[1])
  # 비동기 처리, API 응답 속도와 분리
```

워커가 3대 떠 있어도 각 메시지는 한 워커만 받아 처리합니다. 워커 수를 늘려 처리량 scale-out 가능.

### Stream + Consumer Group — 주문 이벤트 fan-out

```
[Order Service]
XADD orders * orderId 12345 userId 42 amount 50000

[Billing Group]
XREADGROUP GROUP billing worker-1 COUNT 10 STREAMS orders >
→ 결제 처리

[Shipping Group]
XREADGROUP GROUP shipping worker-1 COUNT 10 STREAMS orders >
→ 배송 준비

[Analytics Group]
XREADGROUP GROUP analytics worker-1 COUNT 10 STREAMS orders >
→ 매출 집계
```

**한 번 발행된 주문 이벤트를 결제·배송·분석 세 그룹이 각자 자기 처리 속도로 소비**합니다. 분석 시스템이 잠시 다운돼도 주문 처리는 영향 없음.

## 핵심 정리 (Takeaways)

### 1. 동기 통신 줄이고 비동기로
마이크로서비스 환경에서 서비스 간 통신은 **되도록 비동기로**. 동기 통신은 latency 전염과 cascading failure의 원인. 메시지 브로커가 그 비동기 채널을 제공.

### 2. 메시징 큐와 이벤트 스트림은 본질적으로 다른 도구
- **메시징 큐**: Push 모델, 일대일 작업 분배, 소비 즉시 삭제
- **이벤트 스트림**: Pull 모델, 다대다 전파, 보관 후 재처리 가능

용도를 헷갈리면 시스템이 깨집니다. 작업 분배에 stream을 쓰거나 이력 추적에 큐를 쓰면 곤란합니다.

### 3. 레디스는 3가지 도구를 모두 제공
- **Pub/Sub** — 가장 가벼움. fire-and-forget만.
- **List** — 메시징 큐. BLPOP/BRPOP으로 폴링 없는 워커 풀.
- **Stream** — 본격 스트림 플랫폼. Kafka의 가벼운 대안.

같은 인스턴스에서 세 도구를 동시에 사용할 수 있다는 점이 레디스의 강점.

### 4. Pub/Sub은 메시지를 잃을 수 있다
구독자가 일시적으로 다운돼 있으면 그 사이 발행된 메시지는 사라집니다. **신뢰성이 필요한 곳에는 절대 사용 금지**. 적합한 곳: 실시간 채팅 알림, 캐시 무효화 신호, 분산 노드 broadcast.

### 5. List + BRPOP은 폴링을 없애준다
이벤트 큐에서 폴링하지 마세요. `BLPOP` / `BRPOP`으로 데이터 도착 즉시 처리. 여러 워커가 같은 큐에서 대기하면 자연스러운 워커 풀이 됩니다. 여러 큐에서 동시 대기도 가능 — 우선순위 큐 패턴.

### 6. Stream은 Kafka의 가벼운 대안
이미 운영 중인 레디스가 있고 메시징을 추가하고 싶다면 stream으로 시작하세요. 별도 인프라 도입 없이 Kafka 핵심 개념(append-only, consumer group, offset)을 얻을 수 있습니다. 단, 초대형 throughput이 필요하다면 Kafka로 가야 합니다.

### 7. 구독 상태 커넥션은 일반 명령을 받지 않는다
`SUBSCRIBE` 상태의 커넥션은 pub/sub 관련 명령 9개만 받습니다. Spring Data Redis 등에서 **구독용과 일반 작업용 커넥션을 반드시 분리**하세요. 흔한 실수 포인트.
