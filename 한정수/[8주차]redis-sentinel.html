개발자를 위한 레디스 · Chapter 09

센티널로 만드는 고가용성 레디스

복제만으로는 마스터 장애 시 수동 복구가 필요하다. 센티널(Sentinel)은 마스터를 감시하다 장애를 감지하면 자동 페일오버로 복제본을 새 마스터로 승격시킨다. 왼쪽에서 개념을 읽고, 오른쪽 터미널에서 실제 SENTINEL 커맨드를 실행해 마스터·복제본·센티널 상태와 페일오버 단계가 어떻게 변하는지 직접 확인해 보자.

고가용성 · HA quorum · 쿼럼 sdown · odown epoch · 리더선출 split brain
Section 01

왜 센티널이 필요한가

복제만으로는 부족하다 — 자동 장애 조치라는 마지막 퍼즐

복제만 쓸 때의 수동 복구 문제 09.1 · 배경

레디스는 인메모리 데이터베이스라 백업이 없으면 재시작 시 데이터가 모두 사라진다. 복제본을 구성하면 마스터 장애 시에도 데이터는 복제본에 남는다. 하지만 클라이언트가 마스터에 직접 연결돼 있었다면, 사람이 직접 다음 3단계를 거쳐야 복구된다.

  1. 복제본 노드에 REPLICAOF NO ONE을 실행해 복제 연결을 끊고 독립 마스터로 승격
  2. 애플리케이션 코드에서 레디스 엔드포인트를 복제본 IP로 변경
  3. 변경된 애플리케이션을 재배포
그림 9-1 복제 구성에서 마스터 장애 시 수동 복구 과정

이 수동 과정은 다운타임이 길고, look-aside 캐시 구성에서는 그 사이 원본 소스(DB)로 부하가 급증해 2차 장애로 번질 수 있다. 그래서 자동으로 이 과정을 처리해 줄 무언가가 필요하다.

센티널의 3가지 기능 09.2 · 개념

센티널은 데이터를 저장하는 레디스 인스턴스와는 별개의 프로그램이다. 같은 서버에서 다른 포트로 실행되며, 데이터를 갖지 않고 오직 감시·조정 역할만 한다. 세 가지 핵심 기능을 제공한다.

① 모니터링 (Monitoring)

마스터·복제본 인스턴스의 상태를 주기적인 PING으로 실시간 확인한다.

② 자동 페일오버 (Automatic Failover)

마스터의 비정상 상태를 감지하면 정상 복제본 중 하나를 새 마스터로 승격시키고, 나머지 복제본을 새 마스터에 재연결한다.

③ 인스턴스 구성 정보 안내 (Configuration Provider)

클라이언트는 레디스에 직접 붙기 전 센티널에게 "지금 마스터가 누구냐"고 먼저 질의한다. 페일오버로 마스터가 바뀌어도 센티널이 새 마스터 주소를 알려주므로 클라이언트는 엔드포인트를 바꿀 필요가 없다.

그림 9-2 · 센티널 동작 방식 — client는 센티널에 마스터를 질의(①②)한 뒤 마스터에 직접 연결(③)
흐름 요약 client → (Master 질의) → Sentinel → (Master 주소 반환) → client → (직접 연결) → Redis Master. 오른쪽 터미널의 SENTINEL get-master-addr-by-name이 바로 이 ③번 동작이다.

Section 02

분산 시스템과 쿼럼

센티널 스스로가 SPOF가 되지 않으려면 — 홀수, 그리고 quorum

SPOF와 최소 3대 구성 09.3 · 분산

SPOF(Single Point Of Failure)는 한 곳이 고장 나면 전체가 멈추는 지점이다. 고가용성을 위해 센티널을 도입했는데 정작 센티널이 1대뿐이라면 그 센티널이 새로운 SPOF가 된다. 그래서 센티널은 최소 3대 이상일 때 정상 동작하도록 설계됐다. 하나가 죽어도 나머지가 역할을 이어간다.

과반수 선출 개념을 쓰기 때문에 3대, 5대처럼 홀수로 구성하는 것이 좋다. 보통 3대, 더 견고하게는 5대를 쓴다.

쿼럼(quorum)이란 09.4 · 핵심

쿼럼은 "마스터가 비정상이다"라고 동의해야 하는 최소 센티널 수다. 쿼럼을 만족하면 페일오버 과정을 시작한다. 센티널 3개일 때 보통 쿼럼은 2로 설정한다 — 즉 최소 2대가 마스터 장애에 동의하면 페일오버가 트리거된다. 

# sentinel.conf — 마스터 이름·주소·쿼럼 지정
sentinel monitor master-test 192.168.0.11 6379 2
주의 · 쿼럼 ≠ 과반수 마스터를 sdown→odown으로 바꾸는 데는 쿼럼이면 충분하다. 하지만 실제 페일오버를 위한 리더 선출에는 전체 센티널의 과반수 동의가 필요하다. 이 둘의 차이는 Section 05에서 직접 본다.
Try it · 쿼럼 충족 여부 확인
센티널 배치 방법 (복제본 2개 / 1개) 09.5 · 배치

센티널은 물리적으로 서로 영향받지 않는 서버, 가급적 다른 가용 영역에 배치한다. 마스터 장애를 제대로 감지하려면 마스터와 운명을 같이하면 안 되기 때문이다.

복제본 2개 구성 (가장 안정적)

Server A(Master+Sentinel), B(Replica+Sentinel), C(Replica+Sentinel). 각 서버에 레디스와 센티널을 함께 띄운다.

복제본 1개 구성 (리소스 절약)

A(Master+Sentinel), B(Replica+Sentinel), C(Sentinel만). C는 데이터를 저장하지도, 클라이언트 요청을 받지도 않으므로 최저 사양으로 구성해도 된다. 그래도 센티널 3대를 유지해 과반수 판단이 가능하다.

그림 9-3·9-6 · 배치 방법 — 왼쪽이 가장 안정적, 오른쪽은 C를 센티널 전용 최저 사양으로 절약
이 Playground 구성 192.168.0.11(Master+Sentinel), 192.168.0.22(Replica+Sentinel), 192.168.0.33(Sentinel만). 바로 위에서 설명한 "복제본 1개 구성"이다. 오른쪽 인스펙터에서 확인하라.

sentinel.conf 와 실행 두 방법 09.7 · 설정

센티널 프로세스를 띄우려면 sentinel.conf라는 별도 구성 파일이 필요하다.

port 26379
sentinel monitor master-test 192.168.0.11 6379 2

port는 센티널 프로세스 포트(기본 26379). sentinel monitor는 감시할 마스터에 이름을 붙이고(알파벳·숫자·.·-·_만 가능) 주소와 쿼럼을 지정한다. 복제본 정보는 적지 않아도 된다 — 센티널이 마스터에 물어 복제본을 자동으로 찾아낸다.

실행 두 방법 (동일 동작)

# 방법 1
redis-sentinel /path/to/sentinel.conf

# 방법 2
redis-server /path/to/sentinel.conf --sentinel
주의 지정된 위치에 sentinel.conf가 없거나 그 경로에 쓰기 권한이 없으면 센티널은 시작되지 않는다. 센티널은 실행 중 알아낸 구성을 이 파일에 다시 기록하기 때문이다.

SENTINEL master — 마스터 상태 09.8 · 조회

센티널 인스턴스에 접속하면 마스터·복제본·다른 센티널 정보를 볼 수 있다(레디스 데이터 자체는 못 본다). SENTINEL master <name>으로 마스터의 IP·포트·연결된 복제본 수 등을 확인한다.

특히 발표/면접에서 짚으면 좋은 플래그 3가지:

  • num-other-sentinels — 같은 마스터를 감시하는 다른 센티널 수 (예제는 2)
  • flags — 마스터 상태. 정상은 master, 비정상이면 s_down/o_down
  • num-slaves — 마스터에 연결된 복제본 수

SENTINEL ckquorum — 페일오버 가능 여부 09.9 · 조회

SENTINEL ckquorum <name>은 현재 정상 센티널 수가 쿼럼(그리고 과반수)을 만족하는지 점검한다. 센티널을 죽여 가며 결과가 어떻게 바뀌는지 보자.

Section 05

자동 페일오버 6단계

sdown → odown → epoch↑ → 리더선출 → 승격 → 재연결 → 완료

자동 페일오버 6단계 — 감지(sdown)부터 switch-master까지. 각 단계는 sentinel.log에 기록을 남긴다
① 장애 감지 & sdown(주관적 다운) 09.10 · 페일오버

센티널은 주기적으로 마스터에 PING을 보낸다. down-after-milliseconds(기본 30,000ms = 30초) 동안 유효한 응답(+PONG, -LOADING, -MASTERDOWN)을 못 받으면 마스터가 다운됐다고 판단한다.

한 센티널이 먼저 응답을 못 받으면 마스터를 우선 sdown(subjectively down, 주관적 다운)으로 플래깅한다. 아직 "나 혼자 생각"이라는 뜻이다.

예시 down-after-milliseconds가 30,000(=30초)인데 마스터가 29초마다 응답한다면, 센티널은 이 마스터를 여전히 정상으로 본다.
Try it · 마스터를 셧다운시켜 페일오버 시작
② odown(객관적 다운)으로 전환 09.11 · 페일오버

sdown으로 플래깅한 센티널은 다른 센티널들에게 SENTINEL is-master-down-by-addr <ip> <port> <epoch> *를 보내 장애 사실을 전파한다. 자기 자신을 포함해 쿼럼 이상의 센티널이 장애에 동의하면 마스터 상태를 odown(objectively down, 객관적 다운)으로 변경한다.

그림 9-11·9-12 · sdown(1대의 주관적 판단) → 전파 → odown(쿼럼 이상의 객관적 합의)
중요 · odown은 마스터 전용 센티널은 복제본 장애도 감지해 복제본을 sdown으로 플래깅하지만, 복제본을 odown으로 만들거나 장애를 전파하지는 않는다. 장애 전파는 오직 마스터에 대해서만 일어난다. 단, sdown 상태인 복제본은 페일오버 시 새 마스터 후보에서 제외된다. 
# sentinel.log 에 남는 기록
+sdown
+odown
Try it · 페일오버 진행 상태 확인
③ epoch 증가 & ④ 리더 선출 09.12 · 페일오버

처음 마스터를 odown으로 인지한 센티널이 페일오버를 시작한다. 먼저 epoch(에포크) 값을 1 증가시킨다. 에포크는 페일오버의 버전 번호로, 새 페일오버마다 1씩 늘어 모든 센티널이 같은 작업을 한다는 것을 보장한다.

에포크를 올린 센티널은 다른 센티널에게 vote-for-leader 메시지를 보내 리더 선출 투표를 요청한다. 한 에포크에서 센티널은 딱 한 번만 투표할 수 있고 결과를 바꿀 수 없다.

그림 9-13 센티널 리더 선출 — 투표 요청에 받은 epoch가 더 크면 자신도 올린 뒤 투표
NOTE · 과반수와 쿼럼 odown 전환에는 쿼럼이면 되지만, 리더 선출에는 전체 센티널의 과반수 동의가 필요하다. 예: 센티널 5개·쿼럼 2일 때 마스터를 odown으로 만들 수는 있으나, 2대만 동의하면 리더를 못 뽑아 페일오버는 일어나지 않는다. 
+new-epoch
+try-failover
+vote-for-leader
+elected-leader

⑤ 복제본 승격 & ⑥ 재연결 → 완료 09.13 · 페일오버

리더 센티널은 마스터가 될 적당한 복제본을 다음 순서로 고른다.

  1. replica-priority가 낮은 복제본 (0이면 후보 제외)
  2. 마스터로부터 더 많은 데이터를 받은 복제본 (master_repl_offset이 큰 쪽)
  3. 그래도 같으면 runID가 사전 순으로 작은 복제본 (사실상 임의 선택)

선정된 복제본에 slaveof no one(=REPLICAOF NO ONE)을 실행해 독립 마스터로 만들고, 기존 마스터에 붙어 있던 다른 복제본들에는 replicaof new-ip new-port를 실행해 새 마스터를 바라보게 한다. 모든 과정이 끝나면 센티널은 새 마스터를 모니터링하기 시작한다. 

+failover-state-select-slave
+selected-slave
+failover-state-send-slaveof-noone
+promoted-slave
+slave-reconf-sent / -inprog / -done
+failover-end
+switch-master   ← 페일오버 완료

Section 06

센티널 운영 & 스플릿 브레인

런타임 구성 변경, 노드 추가/제거, 그리고 분산 시스템의 함정

런타임 구성 변경 (SET / MONITOR / REMOVE) 09.14 · 운영

패스워드 인증

마스터/복제본에 requirepass·masterauth를 걸었다면 sentinel.conf에도 sentinel auth-pass <name> <pw>를 지정한다. 모든 노드가 마스터가 될 수 있으니 복제 그룹 내 패스워드는 동일하게 맞춘다.

복제본 우선순위

모든 레디스는 replica-priority(기본 100)를 갖는다. 페일오버 시 값이 가장 작은 노드가 마스터로 선출되며, 0이면 절대 마스터가 되지 않는다.

런타임 변경 커맨드

SENTINEL SET으로 특정 마스터의 파라미터를, SENTINEL MONITOR/REMOVE로 감시 대상을 추가/제거한다. 6.2+에서는 SENTINEL CONFIG GET/SET으로 센티널 자체 설정도 바꾼다. 단, 이 변경은 각 센티널에 개별 적용해야 하며 다른 센티널로 자동 전파되지 않는다. 

SENTINEL SET master-test down-after-milliseconds 1000
SENTINEL SET master-test quorum 1
SENTINEL MONITOR mymaster 192.168.0.55 6379 2
SENTINEL REMOVE mymaster

센티널 초기화 & 노드 추가/제거 09.15 · 운영

센티널은 비정상이라 판단한 노드도 임의로 감시를 멈추지 않는다. 페일오버 뒤 접근 불가한 옛 마스터나 죽은 복제본도 계속 PING한다. 더 이상 안 쓰는 노드를 목록에서 지우려면 SENTINEL RESET <name>(또는 *)으로 상태 정보를 초기화한다.

노드 추가

새 센티널을 마스터 감시 설정으로 실행하면 자동 검색 메커니즘에 의해 기존 센티널들의 known-list에 추가된다. 한 번에 여러 대를 넣을 땐 10초 이상 간격으로 1개씩 천천히 추가해 오류 가능성을 줄인다.

노드 제거

센티널끼리는 오래 응답이 없어도 known-list에서 안 지운다. 따라서 제거할 센티널 프로세스를 종료한 뒤 SENTINEL RESET *로 정보를 리셋한다. 작업 사이엔 최소 30초 간격을 둔다. 추가·제거 후엔 num-other-sentinels 값으로 정상 반영을 확인한다.

스플릿 브레인 현상 09.16 · 함정

스플릿 브레인(split brain)은 네트워크 파티션으로 분산 시스템이 둘로 끊기고, 끊긴 두 부분이 각각 자신을 정상이라 인식하는 현상이다.

그림 9-14처럼 마스터+Sentinel A가 한쪽, 복제본+Sentinel B·C가 다른 쪽으로 단절됐다고 하자. 단절이 길어지면 과반수 센티널(B·C)이 같은 파티션에 있으므로 이들은 복제본을 새 마스터로 승격시킨다. 그런데 원래 마스터에는 장애가 없었다면 — 이제 마스터가 2개가 되어 버린다.

기존 client는 옛 마스터에, 새 client는 센티널이 알려준 새 마스터에 각각 쓰기를 하게 되어 데이터가 갈라진다. 네트워크가 복구되면 한쪽 마스터는 복제본으로 강등되며 그동안의 쓰기가 유실될 수 있다.

그림 9-14·9-15 · 스플릿 브레인 — 과반수 센티널이 있는 쪽이 복제본을 승격시켜 마스터가 둘이 된다
완화책 min-replicas-to-write / min-replicas-max-lag로 "복제본 N대 이상에 닿지 않으면 쓰기 거부"를 설정해 고립된 옛 마스터의 쓰기를 막는다. 센티널을 가용 영역에 분산 배치하는 것도 파티션 영향을 줄인다.
