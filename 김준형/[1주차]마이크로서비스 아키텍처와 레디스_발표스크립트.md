## 슬라이드 1. [title] 개발자를 위한 레디스

안녕하세요. 오늘은 **"개발자를 위한 레디스"**라는 주제로 발표를 진행하겠습니다.

레디스라고 하면 많은 분들이 "캐시 서버" 정도로 떠올리실 텐데요, 사실 레디스는 캐시를 훨씬 넘어서는 다재다능한 인메모리 데이터베이스입니다.

오늘 발표에서는 **왜 레디스가 필요해졌는지**, 그 배경부터 시작해서 레디스의 핵심 특징과 메시지 브로커로서의 활용까지 살펴보겠습니다.

## 슬라이드 2. [comparison] 모놀리틱 vs 마이크로서비스

레디스 이야기에 앞서, 먼저 소프트웨어 아키텍처의 흐름을 짚어보겠습니다.

**왼쪽의 모놀리틱 아키텍처**는 전통적인 개발 모델입니다. 전체 애플리케이션을 하나의 통합된 패키지로 개발하고 배포하죠. 작은 규모에서는 운영이 단순하다는 장점이 있지만, 서비스가 커지면 문제가 생깁니다. 한 모듈을 수정하면 전체를 다시 배포해야 하고, 한 곳의 장애가 전체 시스템 장애로 이어집니다. 특정 파트만 확장하고 싶어도 서비스 전체를 확장해야 해서 리소스 낭비가 발생하죠.

**오른쪽의 마이크로서비스 아키텍처**는 기능별로 독립된 서비스가 각각 동작하는 구조입니다. 배포, 테스트, 확장이 서비스별로 독립적으로 수행될 수 있고, 각 서비스에 맞는 기술 스택을 자유롭게 선택할 수 있습니다. 한 서비스의 장애가 다른 서비스에 큰 영향을 주지 않아 운영 안정성도 향상됩니다.

물론 모든 상황에 마이크로서비스가 답은 아닙니다. 소규모 팀에서는 오히려 관리 복잡도가 증가할 수 있죠. 하지만 빠르게 성장하는 서비스라면 마이크로서비스 채택이 큰 도움이 됩니다.

## 슬라이드 3. [content] 데이터 저장소 요구사항의 변화

아키텍처가 바뀌면, 당연히 데이터 저장소에 대한 요구사항도 달라집니다.

모놀리틱 시대에는 **Oracle, MySQL 같은 관계형 데이터베이스**가 표준이었습니다. 모든 데이터를 하나의 DB에서 관리하니까, 중앙 집약적인 RDBMS가 잘 맞았던 거죠. ACID 특성 — 원자성, 일관성, 독립성, 지속성 — 을 보장하면서 복잡한 관계를 다루는 데 강점이 있었습니다.

하지만 최근에는 상황이 많이 달라졌습니다. **비정형 데이터**가 폭발적으로 증가하고 있거든요. 그래프 데이터, 실시간 시계열 데이터, JSON 데이터 같은 것들을 관계형 DB의 정형화된 테이블에 억지로 넣으려면 변환 과정이 필요하고, 성능 이슈도 발생합니다.

그리고 마이크로서비스에서 가장 중요한 건, **각 서비스가 독립된 데이터 저장소를 가져야 한다**는 점입니다. 하나의 거대한 DB를 모든 서비스가 공유하면 서비스 간 결합도가 높아져서 마이크로서비스의 장점이 사라지니까요. 이런 배경에서 NoSQL이 등장하게 됩니다.

---

## 슬라이드 4. [card-grid] NoSQL의 핵심 특징 4가지

NoSQL은 "Not Only SQL", 관계형 데이터베이스의 한계를 넘어서기 위해 고안된 데이터베이스입니다. 마이크로서비스의 데이터 저장소로 쓰일 수 있도록 여러 요구사항을 만족시키는데요, 핵심 특징 4가지를 살펴보겠습니다.

첫 번째, **실시간 응답**입니다. 사람은 100밀리초만 넘어가도 지연을 인지한다고 합니다. 그중 3분의 1은 눈 깜빡이는 시간이에요. 마이크로서비스에서는 개별 서비스가 병목이 되면 안 되기 때문에, 데이터 저장소의 응답 속도가 0~1ms 이내여야 합니다.

두 번째, **확장성**입니다. 계획된 확장뿐만 아니라, 새해 첫날 트래픽이나 세일 이벤트 같은 예상치 못한 트래픽 급증에도 유연하게 대응할 수 있어야 합니다.

세 번째, **고가용성**입니다. 데이터 저장소를 사용할 수 없는 상황은 곧 서비스 장애입니다. 장애 상황에서 신속하게 복구돼 항상 사용 가능한 상태를 유지해야 합니다.

네 번째, **단순성과 유연성**입니다. 서비스가 세분화될수록 관리 포인트는 늘어나니까 데이터 저장소는 간단하게 쓸 수 있어야 하고, 동시에 비정형 데이터를 적합한 형식으로 저장할 수 있는 유연성도 필요합니다.

---

## 슬라이드 5. [card-grid] NoSQL 데이터 저장소 유형 4가지

NoSQL은 데이터를 저장하는 방식에 따라 크게 4가지 유형으로 나뉩니다.

첫 번째, **그래프 유형**입니다. 노드, 에지, 속성으로 데이터 간의 관계를 표현합니다. SNS에서 친구 추천이나, 쇼핑몰의 상품 추천 서비스에 유용하죠. 대표적으로 Neo4j가 있습니다.

두 번째, **칼럼 유형**입니다. 데이터를 행이 아닌 열 기준으로 저장합니다. 대량 데이터에 대한 집계 쿼리를 빠르게 처리할 수 있어서 BI 분석이나 데이터 웨어하우스에 적합합니다. Apache Cassandra, HBase가 대표적이죠.

세 번째, **문서 유형**입니다. JSON 형태로 데이터를 저장하기 때문에 개발자의 코드와 동일한 구조로 직관적으로 사용할 수 있습니다. 스키마가 고정되지 않아 유연성이 크고요. MongoDB가 대표적입니다.

네 번째, **키-값 유형**입니다. 가장 단순하고, 가장 빠릅니다. 모든 값은 키에 연결되어 있고, 키로 값을 검색합니다. 구조의 단순성 덕분에 수평적 확장이 쉽고, 실시간 서비스에서 빠른 응답 속도를 보장합니다. 그리고 바로 이 키-값 유형의 대표주자가 — **레디스**입니다.

## 슬라이드 6. [title] Redis — Remote Dictionary Server

자, 이제 본격적으로 레디스 이야기를 해보겠습니다.

Redis. **Remote Dictionary Server**의 약자입니다. 이름 그대로 원격 딕셔너리 서버, 즉 키-값 구조의 데이터를 원격으로 관리할 수 있는 서버라는 뜻이죠.

레디스는 고성능 키-값 유형의 **인메모리 NoSQL 데이터베이스**이자, **오픈 소스** 기반의 데이터 저장소입니다. 2017년부터 2021년까지 5년 연속으로 Stack Overflow에서 개발자들에게 가장 사랑받는 데이터베이스로 선정되기도 했죠.

그런데 레디스가 단순히 인기만 있는 게 아닙니다. 실제로 어떤 수치들이 레디스의 강력함을 뒷받침하는지 보시겠습니다.

## 슬라이드 7. [stat-grid] 레디스 핵심 수치

레디스를 숫자로 말씀드리겠습니다.

**응답 속도: sub-millisecond**. 대부분의 커맨드가 1밀리초 이내에 처리됩니다. 모든 데이터가 메모리에 있기 때문에 디스크 접근이 필요 없거든요. RAM의 접근 속도는 HDD나 SSD보다 현저히 빠릅니다.

**스레드: 싱글 스레드** 기반 이벤트 루프. 정확히는 메인 스레드 1개와 별도 스레드 3개, 총 4개의 스레드로 동작하지만, 클라이언트 커맨드 처리는 싱글 스레드입니다. 최소 하나의 코어만 있어도 동작하죠.

**지원 언어: 100개 이상의 오픈 소스 클라이언트**. Java, Python, Node.js, Go 등 다수의 언어를 지원합니다.

**클라우드: 주요 벤더 모두 지원**. AWS ElastiCache, Google Cloud Memorystore, Azure Cache, 국내에서는 NHN CLOUD EasyCache, NAVER CLOUD DB for Redis까지. 멀티 클라우드 환경에서도 일관된 성능을 제공합니다.

## 슬라이드 8. [card-grid] 레디스의 5대 특징

레디스의 핵심 특징 5가지를 정리해보겠습니다.

**첫째, 실시간 응답**. 인메모리 데이터베이스이기 때문에 디스크 접근 없이 데이터를 처리합니다. 온디스크 DB에서는 페이지 단위로 디스크에서 메모리로 로딩을 반복해야 하지만, 레디스는 그 과정 자체가 없습니다.

**둘째, 다양한 자료 구조 — 단순성**. 키-값 저장소이지만, 값에 문자열뿐 아니라 Hash, Set, Sorted Set, HyperLogLog, Stream 등 다양한 자료 구조를 저장할 수 있습니다. 프로그래밍의 기본 자료 구조와 밀접해서, 관계형 DB에서 발생하는 **임피던스 불일치** 문제를 해소해줍니다.

**셋째, 고가용성**. 복제를 통해 데이터를 여러 서버에 분산시키고, 센티널이 장애를 자동 탐지해서 페일오버를 수행합니다. 엔드포인트를 변경할 필요 없이 자동 복구되죠.

**넷째, 확장성**. 클러스터 모드를 사용하면 데이터가 자동으로 샤딩되어 저장됩니다. 애플리케이션은 어떤 샤드에 데이터가 있는지 신경 쓸 필요가 없고, 클러스터 버스 프로토콜로 노드들이 서로를 감시하면서 자동 페일오버까지 지원합니다.

**다섯째, 클라우드 네이티브**. 컨테이너, 오케스트레이션, 데브옵스 환경에 최적화되어 있고, 멀티 클라우드 전략에서도 일관된 성능과 기능을 제공합니다.

## 슬라이드 9. [diagram] 싱글 스레드 이벤트 루프

레디스를 사용할 때 **반드시** 알아두셔야 할 특징이 있습니다. 바로 싱글 스레드 이벤트 루프입니다.

레디스는 클라이언트의 커맨드를 **이벤트 루프 기반의 싱글 스레드**로 처리합니다. 이게 장점이자 주의점인데요.

**장점**부터 말씀드리면, 멀티스레드에서 필요한 동기화나 잠금 메커니즘이 필요 없습니다. 덕분에 구조가 단순하고, 적은 리소스로도 안정적이고 빠르게 요청을 처리할 수 있죠. 코어 하나만으로도 좋은 성능을 냅니다.

하지만 **주의해야 할 점**도 있습니다. 싱글 스레드라는 건, 한 사용자가 오래 걸리는 커맨드를 실행하면 다른 모든 사용자가 그 커맨드가 끝날 때까지 대기해야 한다는 뜻입니다. 책에서도 이 부분을 강조하는데요, 이 특징 때문에 **인적 장애가 발생할 가능성이 매우 높다**고 합니다.

레디스는 메모리에서 동작하기 때문에 대부분의 커맨드는 빠르지만, 반환이 느린 특정 커맨드들이 존재합니다. 예를 들어 KEYS 같은 커맨드죠. 이런 위험한 커맨드만 주의해서 사용해도 장애 가능성을 크게 줄일 수 있습니다.

## 슬라이드 10. [card-grid] 메시지 브로커로서의 레디스

레디스를 데이터 저장소로만 생각하기 쉽지만, 사실 레디스는 **메시지 브로커**로서도 매우 유용합니다.

마이크로서비스에서는 각 서비스가 완전히 분리되어 있기 때문에, 서비스 간에 비동기적으로 데이터를 전달할 수 있는 **통신 채널**이 필요합니다. 레디스는 이를 위해 3가지 방식을 제공합니다.

**pub/sub** — 가장 간단한 메시징 기능입니다. 채널에 데이터를 던지면 듣고 있는 모든 소비자가 즉시 받아갑니다. fire-and-forget 패턴에 적합하죠.

**list** — 메시징 큐로 사용하기 딱 좋습니다. push/pop 연산이 빠르고, 데이터가 들어올 때까지 대기하는 블로킹 기능도 지원합니다.

**stream** — 아파치 카프카에서 영감을 받은 자료 구조입니다. append-only 방식으로 저장되고, 소비자 그룹을 통한 분산 처리, 시간대별 검색까지 가능합니다.

각각의 차이점을 좀 더 자세히 비교해보겠습니다.

---

## 슬라이드 11. [comparison] pub/sub vs list vs stream

세 가지 메시징 방식을 비교해보겠습니다.

**pub/sub**은 데이터가 전달된 뒤 **즉시 삭제**됩니다. 일회성이에요. 연결이 끊긴 소비자는 그 사이의 메시지를 받을 수 없습니다. 대신 가장 빠르고 가장 간단합니다. 실시간 알림 서비스처럼 "보내고 잊어버리는" 패턴에 적합합니다.

**list**는 데이터가 소비될 때까지 **저장이 유지**됩니다. 하나의 소비자가 pop하면 해당 데이터는 list에서 제거되죠. 1:1 메시지 전달, 작업 큐 같은 패턴에 적합합니다. 블로킹 기능 덕분에 폴링 없이도 새 데이터를 기다릴 수 있고요.

**stream**은 가장 강력합니다. 데이터가 **영구적으로 보관**되면서 append-only로 계속 쌓입니다. 카프카처럼 소비자 그룹이 있어서 데이터의 분산 처리가 가능하고, 시간대별로 검색도 할 수 있습니다. 이벤트 소싱이나 로그 처리 같은 복잡한 메시징 시나리오에 적합하죠.

정리하면, 간단한 알림은 pub/sub, 작업 큐는 list, 본격적인 스트림 처리는 stream. 요구사항에 맞게 선택하시면 됩니다.

## 슬라이드 12. [closing] 마무리 — 핵심 요약

마지막으로 오늘 내용을 정리하겠습니다.

소프트웨어 아키텍처가 **모놀리틱에서 마이크로서비스로** 진화하면서, 데이터 저장소에 대한 요구사항도 함께 변했습니다. 더 이상 관계형 데이터베이스만으로는 모든 것을 커버할 수 없게 되었고, 그 흐름에서 **NoSQL**이 등장했습니다.

레디스는 키-값 유형의 인메모리 NoSQL 데이터베이스로서, **실시간 응답, 다양한 자료 구조, 고가용성, 손쉬운 확장성, 클라우드 네이티브** — 이 5가지 특징으로 마이크로서비스의 요구사항을 충실히 만족시킵니다.

그리고 레디스는 단순한 데이터 저장소를 넘어서, **pub/sub, list, stream**을 통해 메시지 브로커로서도 강력한 역할을 합니다.

다만 한 가지 꼭 기억하실 점 — 레디스는 **싱글 스레드**입니다. 오래 걸리는 커맨드 하나가 전체 시스템을 멈출 수 있으니, 위험한 커맨드는 반드시 주의해서 사용하시기 바랍니다.

이상으로 발표를 마치겠습니다. 감사합니다.

## 별첨 A. 센티널 & 클러스터

### 센티널 — 고가용성(HA) 전담

센티널은 마스터-레플리카 구조를 감시하는 별도 프로세스입니다. 핵심 역할은 4가지 — **감시(Monitoring)**, **알림(Notification)**, **자동 페일오버(Automatic Failover)**, **설정 제공(Configuration Provider)**. 클라이언트는 센티널에게 "지금 마스터가 누구야?"를 질의하면 되므로, 페일오버 후에도 엔드포인트를 바꿀 필요가 없습니다.

**과반수 합의**로 동작하기 때문에 최소 **3대**를 권장합니다. 2대면 1대 다운 시 과반수 미달로 페일오버가 불가능해지거든요.

페일오버 흐름: **SDOWN**(개별 센티널이 "다운 같다" 판단) → **ODOWN**(quorum 이상 동의하면 "확실히 다운") → 센티널 리더 선출 → 레플리카 중 최신 데이터 보유자를 새 마스터로 승격

### 클러스터 — 수평 확장 + HA

클러스터는 데이터를 **`CRC16(key) mod 16384`** 공식으로 16,384개 슬롯에 분산합니다. 여기서 **해시 슬롯**은 키와 노드 사이의 **중간 추상화 레이어**입니다. 키 → 슬롯 매핑은 불변이고, 슬롯 → 노드 매핑만 바꾸기 때문에 노드를 추가하거나 제거할 때 슬롯 단위로 유연하게 재분배할 수 있습니다.

여러 마스터가 슬롯을 나눠 담당하므로 쓰기 확장이 가능하고, 페일오버도 자체 내장되어 있어 센티널이 불필요합니다.

그리고 키 안에 `{...}` 중괄호가 있으면 중괄호 안의 문자열만 해시 계산에 사용됩니다. 이것이 **해시 태그**입니다. `user:{1000}:profile`과 `user:{1000}:cart`는 둘 다 `CRC16("1000")`으로 계산되어 **같은 슬롯에 확정 배치**됩니다. `MGET`, `MULTI/EXEC`, Lua 스크립트 같은 멀티키 연산이 같은 슬롯에서만 동작하기 때문에, 원자적으로 묶어야 할 키들을 해시 태그로 같은 슬롯에 강제 배치하는 거죠. 다만 남발하면 특정 슬롯에 키가 몰려 병렬성이 깨지니 주의가 필요합니다.

### 한눈에 비교

| 항목 | 센티널 | 클러스터 |
|------|--------|----------|
| 목적 | HA만 | HA + 수평 확장 |
| 마스터 수 | 단일 | 다수 |
| 쓰기 확장 | 불가 | 가능 |
| 멀티키 연산 | 제약 없음 | 같은 슬롯에서만 |
| 운영 복잡도 | 낮음 | 높음 |

### 둘을 함께 쓰나요?

**아닙니다.** 공식 문서는 "Sentinel은 Cluster를 사용하지 않을 때를 위한 것"이라고 명시합니다. 클러스터가 Gossip 프로토콜로 페일오버를 자체 처리하므로, 함께 쓰면 두 시스템이 마스터/레플리카 관계를 서로 다르게 관리해 충돌합니다. 데이터셋이 단일 서버 메모리 내라면 센티널, 수평 확장이 필요하면 클러스터 — 둘 중 하나만 선택합니다.

### 핫 파티션 문제

클러스터에서 자주 빠지는 함정입니다. 99대 노드 클러스터라도 **핫 키 하나는 항상 같은 슬롯, 같은 노드에 집중**됩니다. 인기 콘텐츠 키나 실시간 랭킹 키에 트래픽이 몰리면, 해당 노드만 과부하이고 나머지 98대는 idle 상태가 됩니다.

완화 방법은 세 가지입니다.
- **키 분산(Key Splitting)**: 하나의 핫 키를 `hot_key:{shard_0}`, `hot_key:{shard_1}`, ... 등 여러 이름으로 분리합니다. 해시태그가 다르므로 각각 다른 슬롯/노드에 배치되고, 애플리케이션이 랜덤이나 해시로 분기해서 접근하면 트래픽이 분산됩니다. 쓰기는 모든 분산 키에, 읽기는 그중 하나에서 — 가장 직접적인 해결책입니다.
- **레플리카 읽기**: `READONLY` 명령으로 읽기 트래픽을 레플리카에 분산. 쓰기가 아닌 읽기 핫 키에 효과적입니다.
- **클라이언트 사이드 캐시**: Redis 6+ `CLIENT TRACKING`으로 애플리케이션 로컬 캐시 활용 — 노드 요청 자체를 줄임

### 주의할 점

- 센티널: `quorum=1`은 오판 위험. Split-brain 대비로 `min-replicas-to-write` 설정 권장.
- 클러스터: 다른 슬롯 키끼리 멀티키 명령 시 **CROSSSLOT 에러**. 해시태그 `{}`로 해결 가능하나 남용하면 데이터 쏠림 발생.
- 공통: 복제는 **비동기** — 페일오버 시 마지막 미복제 데이터는 유실 가능.

## 별첨 B. 싱글 스레드와 백그라운드 스레드

### 메인 스레드가 하는 일

메인 스레드 하나가 **모든 클라이언트 커맨드를 순차 실행**합니다. I/O 멀티플렉싱(Linux `epoll`, macOS `kqueue`)으로 수천 개 소켓을 동시에 감시하면서, 준비된 요청만 처리하는 이벤트 루프 구조입니다.

### 나머지 3개 스레드가 하는 일

메인 스레드를 블로킹할 수 있는 **느린 시스템 콜**을 분리한 것이 BIO(Background I/O) 스레드입니다.

| 스레드 | 하는 일 | 왜 분리했나 |
|--------|---------|------------|
| `CLOSE_FILE` | 파일 디스크립터 닫기 | 대형 파일 unlink가 수 ms 걸림 |
| `AOF_FSYNC` | AOF 디스크 동기화 | fsync가 수십 ms 걸림 |
| `LAZY_FREE` | 대형 키 메모리 해제 | 수백 MB 해제에 수백 ms 걸림 |

이 3개는 커맨드 실행에는 관여하지 않습니다. 커맨드 처리만 놓고 보면 여전히 **순수 싱글 스레드**입니다.

### 장점과 주의사항

**장점**: 잠금(lock) 불필요 → 데드락/레이스 컨디션 없음. 모든 커맨드가 원자적으로 실행. 적은 CPU로 높은 처리량.

**그러면 클러스터에서는?** 클러스터의 각 노드는 독립적인 Redis 프로세스입니다. 서로 다른 노드에 있는 키에 대한 명령은 **실제로 병렬 처리**됩니다. 다만 같은 노드 안에서는 여전히 싱글 스레드를 통과하므로 순차 처리입니다. 즉 클러스터에서 병렬성은 곧 **샤딩 설계의 문제** — 키가 얼마나 잘 분산되었는지가 동시 처리량을 결정합니다.

**주의**: 싱글 스레드이므로 **느린 커맨드 하나가 전체를 멈춥니다.**

| 위험한 커맨드 | 대안 |
|--------------|------|
| `KEYS *` | `SCAN` |
| `SMEMBERS` (대형) | `SSCAN` |
| `HGETALL` (대형) | `HSCAN` |
| `DEL` (대형 키) | `UNLINK` (비동기 삭제) |

특히 `DEL` vs `UNLINK` — 대형 키를 `DEL`하면 메인 스레드가 수백 ms 블로킹. `UNLINK`는 keyspace에서 O(1)로 분리하고 실제 해제는 `LAZY_FREE` 스레드에 위임합니다. 프로덕션에서는 **항상 UNLINK**.

## 별첨 C. 인메모리 저장소 & AOF/RDB 영속성

### 인메모리의 장단점

**장점**: **RAM 100ns** vs SSD 100μs vs HDD 10ms — 각 단계 1,000배 차이. 단일 인스턴스 **100K+ ops/sec**.

**단점**: RAM은 디스크 대비 수십 배 비쌈. 물리 메모리 한계 존재. 프로세스 종료 시 데이터 소멸. `maxmemory-policy`로 eviction 전략 설정 필수.

### SSoT(Single Source of Truth)로는 선호되지 않는 이유

레디스는 구조적으로 **데이터 유실 가능성이 존재**합니다. RDB는 스냅샷 간격(수 분), AOF everysec은 최대 1초, 비동기 복제는 페일오버 시 미전달 쓰기 — 어느 조합이든 "유실 제로"를 보장하지 않습니다. 공식 문서도 `WAIT` 명령을 써도 "강한 일관성이 보장되지 않는다"고 명시하고 있죠.

따라서 일반적인 아키텍처에서는 **RDBMS가 SSoT**, 레디스는 캐시/세션/실시간 보조 저장소로 역할을 분리하는 패턴이 선호됩니다. 다만 절대 못 쓴다는 건 아닙니다 — 선착순 쿠폰 발급, 게임 리더보드, 실시간 입찰처럼 **속도가 내구성보다 중요하거나, 약간의 유실을 허용할 수 있는 도메인**에서는 primary DB로 활용하기도 합니다.

### RDB — 스냅샷 방식

`fork()` → 자식 프로세스가 **COW(Copy-on-Write)**로 메모리를 바이너리 파일(`dump.rdb`)로 직렬화. 부모는 계속 서비스.

- 컴팩트하고 복구가 빠름. DR 백업에 적합.
- 단점: 스냅샷 **간격 사이 데이터 유실**, 대용량에서 fork 지연, 쓰기 부하 시 COW로 메모리 최대 2배.

### AOF — 명령어 로그 방식

모든 쓰기 커맨드를 텍스트(RESP 프로토콜)로 파일에 append. 재시작 시 replay로 복원.

| appendfsync | 내구성 | 최대 유실 | 성능 |
|-------------|--------|----------|------|
| `always` | 최고 | 거의 없음 | 느림 |
| `everysec` (권장) | 높음 | 1초 | 빠름 |
| `no` | 낮음 | 수십 초 | 가장 빠름 |

파일이 커지면 `BGREWRITEAOF`로 압축. (예: INCR 1,000번 → SET 최종값 하나)

### 공식 권장: RDB + AOF 병행

| 상황 | 전략 |
|------|------|
| 캐시 전용 | RDB만 |
| 데이터 손실 최소화 | **RDB + AOF 병행** |
| AOF만 | 공식 비권장 |

재시작 시 AOF 우선 로드(더 완전한 데이터). `aof-use-rdb-preamble yes`로 AOF 앞부분을 RDB 형식으로 저장하면 복구 속도도 확보할 수 있습니다.
