# 2장. 레디스 시작하기 — 온전히 이해하기

## 이 장의 본질

> "레디스를 일단 올리는 방법"이 아니라,
> **"레디스를 운영 환경에 올릴 때 왜 OS를 손봐야 하는가"를 이해하는 장**.

레디스는 **싱글 스레드 + fork() 기반 영속화라는** 특이한 설계를 갖는다. 이 때문에 일반 OS 기본값(일반 웹서버용) 위에 그냥 얹으면 반드시 문제가 난다. 2장의 모든 튜닝은 이 특이성에서 거꾸로 나온 요구사항이다.

## 설치 (요점만)

| 환경 | 방법 | 특징 |
|---|---|---|
| 리눅스 (소스) | `wget` → `make` → `make PREFIX=... install` | 운영 권장. 다중 인스턴스·오프라인 설치 가능 |
| 리눅스 (패키지) | `yum install redis` + remi repo | 빠르지만 파일 산재, 다중 인스턴스 불리 |
| macOS | `brew install redis` + `brew services start redis` | 개발용 |
| Windows | WSL 위에서 리눅스 방식 | 개발용 |

설치 후 `bin/redis-server redis.conf`로 포그라운드 기동하면, 로그에 **시스템 변수 경고가** 뜬다. 이게 바로 다음 단계(OS 튜닝)의 입력이다.

## ⭐ 운영 환경 튜닝 — "왜" 중심

네 가지 모두, 앞서 말한 **"싱글 스레드 + fork()"** 특성에서 파생된다.

### 1) 파일 디스크립터 (`ulimit -n`)

**왜?** 레디스 기본 `maxclients = 10000`. 프로세스 내부 예약 FD 32개를 합쳐 **최소 10032개의** open files가 필요. 부족하면 레디스가 **알아서 maxclients를 깎는다**(연결 끊김 유발).

**어떻게?** `/etc/security/limits.conf`에
```
* hard nofile 100000
* soft nofile 100000
```

### 2) THP (Transparent HugePage) 비활성화

**왜?** THP는 일반 앱에선 TLB 오버헤드를 줄여주지만, **레디스처럼 메모리를 촘촘히 쓰는 DB는** 오히려 레이턴시가 튄다. 특히 fork() 시 CoW 비용이 커진다.

**어떻게?**
```bash
echo never > /sys/kernel/mm/transparent_hugepage/enabled
```
영구 적용은 `/etc/rc.local`에 동일 구문 + `chmod +x /etc/rc.d/rc.local`.

### 3) `vm.overcommit_memory = 1` ⭐ 가장 중요

**왜?** 레디스는 RDB/AOF 저장·복제 시 **`fork()`로 자식 프로세스를** 만든다. 부모·자식이 같은 페이지를 공유하다가 **쓰기 발생 시점에 복사(CoW)한다**. 쓰기가 많으면 **일시적으로 메모리 할당이 폭증한다**. 기본값 0에서는 "필요 메모리 > 남은 메모리"면 커널이 할당 거부 → **백그라운드 저장 실패**.

**어떻게?**
```bash
# 영구
echo 'vm.overcommit_memory = 1' >> /etc/sysctl.conf
# 즉시
sysctl vm.overcommit_memory=1
```

### 4) `somaxconn`, `tcp_max_syn_backlog`

**왜?** `redis.conf`의 `tcp-backlog`(기본 511)는 OS의 두 값을 **초과할 수 없다**. 리눅스 기본값은 128이라 조용히 잘린다.

**어떻게?**
```
net.ipv4.tcp_max_syn_backlog = 1024
net.core.somaxconn = 1024
```

## `redis.conf` 핵심 키

| 키 | 기본값 | 이해 포인트 |
|---|---|---|
| `port` | 6379 | — |
| `bind` | 127.0.0.1 ::1 | 외부 접속 허용하려면 외부 인터페이스 IP 추가. `0.0.0.0`은 보안상 지양 |
| `protected-mode` | yes | **비밀번호 없으면 로컬만 허용하는** 안전망 |
| `requirepass` / `masterauth` | 없음 | 복제 사용하면 **둘을 같은 값으로** |
| `daemonize` | no | yes면 백그라운드 + pid 파일 |
| `dir` | `./` | 로그·덤프 저장 경로 — **반드시 명시** |

## 실행·접속·첫 CRUD

```bash
bin/redis-server redis.conf       # 시작 (daemonize=yes면 백그라운드)
bin/redis-cli shutdown             # 종료
redis-cli -h <ip> -p <port> -a <pw>   # 접속

# 대화형
127.0.0.1:6379> SET hello world
OK
127.0.0.1:6379> GET hello
"world"

# 커맨드라인 1회 실행
redis-cli PING   # → PONG
```

파이썬 클라이언트:
```python
import redis
r = redis.Redis(host='localhost', port=6379, db=0)
r.get('hello')   # b'world'
```

# 3장. 레디스 기본 개념 — 온전히 이해하기

## 이 장의 본질

> "모든 데이터는 **키로** 식별된다.
> 값은 **자료구조로** 저장된다.
> 자료구조마다 **복잡도와 연산 특성이** 다르며, 이것이 설계 선택을 결정한다."

그리고 2장의 사슬이 이어진다: **싱글 스레드이므로, 오래 걸리는 커맨드 하나가 전 클라이언트를 막는다.** 3장의 모든 "주의"는 여기서 나온다.

## 자료 구조 지도 (7 + 확장 2 = 9)

```
기본 5종       확장 2종              독립 구조
─────────     ──────────            ──────────
string   ──→  bitmap (bit 연산)
list
hash
set      ──→  (교·합·차집합)
sorted set ─→ geospatial (위경도)
                                    hyperloglog (카디널리티 추정)
                                    stream (메시지 브로커, 7장)
```

### string — 값이 한 덩어리인 경우

- **특성**: 키-값 1:1. 최대 512MB. **binary-safe** → JPEG·직렬화 바이트도 그대로 가능.
- **대표 패턴**: 세션 값, 캐시, **원자적 카운터**.
- **핵심 커맨드**
  - `SET k v [NX|XX]`: NX = 없을 때만, XX = 있을 때만
  - `GET`, `MSET`/`MGET` (왕복 시간 절감)
  - `INCR`/`INCRBY`/`DECR`/`DECRBY` — **원자적**. 여러 클라이언트 동시 INCR이 절대 유실/중복되지 않는다.

> "원자적"의 의미: `읽기 → 증가 → 쓰기` 3단계가 **중간에 끊기지 않음**. 싱글 스레드이기에 당연하지만, 이 보장이 분산 카운터의 기반이 된다.

### list — 순서 있는 큐/스택

- **특성**: 최대 42억. 양 끝 연산은 **O(1)**, 중간 접근은 **O(n)**.
- **대표 패턴**: 작업 큐, 로그 버퍼, 최근 N개 유지.
- **핵심 커맨드**
  - `LPUSH`/`RPUSH` (양 끝 삽입), `LPOP`/`RPOP` (양 끝 제거) — O(1)
  - `LRANGE k 0 -1` (전체 조회, 음수 인덱스)
  - `LTRIM k start stop` (범위 밖 삭제, 반환 없음)
  - `LINSERT BEFORE|AFTER pivot v`, `LSET i v`, `LINDEX i` — **중간 조작은 O(n)**

> **고정 길이 큐 패턴** (실무 빈출):
> ```
> LPUSH logdata <data>
> LTRIM logdata 0 999
> ```
> tail 삭제는 O(1). "쌓고 나중에 배치 삭제"보다 훨씬 싸다.

### hash — 객체 표현

- **특성**: 필드-값 쌍 묶음. **필드를 동적으로 추가 가능**(RDBMS 컬럼과 다른 점).
- **대표 패턴**: 객체 1건을 통째 캐싱 (예: `Product:123`).
- **핵심 커맨드**
  - `HSET k f1 v1 [f2 v2 ...]`
  - `HGET k f`, `HMGET k f1 f2`, `HGETALL k`

> 같은 객체를 저장해도, 필드 집합이 들쭉날쭉하면 hash가, 고정이면 RDB가 낫다. **"데이터 모양"으로 저장소를 고르는 감각이** 이 장의 교훈.

### set — 중복 없는 집합

- **특성**: 순서 없음, 중복 없음. **집합 연산 지원**.
- **대표 패턴**: 태그, 유니크 방문자, 친구 교집합.
- **핵심 커맨드**
  - `SADD k v...`, `SMEMBERS k`, `SREM k v`, `SPOP k` (랜덤 제거)
  - `SINTER`, `SUNION`, `SDIFF`

### sorted set (zset) — 스코어로 정렬된 집합

- **특성**: **(스코어, 값)** 쌍. 스코어 순 정렬, 같은 스코어는 사전순. 인덱스 접근 **O(log n)**.
- **대표 패턴**: 랭킹 보드, 시간순 피드, 우선순위 큐.
- **핵심 커맨드**
  - `ZADD k [NX|XX|LT|GT] score val [score val ...]`
    - `LT`/`GT`: 더 작을/클 때만 업데이트 (최고/최저 스코어 유지에 유용)
  - `ZRANGE k start stop [BYSCORE|BYLEX] [REV] [LIMIT o c] [WITHSCORES]`

> **인덱스 접근이 잦다면 list가 아니라 sorted set**. 직관과 다른 이 사실이 3장의 핵심 함정 중 하나.

**ZRANGE의 3가지 모드**

| 옵션 | start/stop 의미 | 특수값 |
|---|---|---|
| (기본) | 인덱스 | `0 -1` = 전체 |
| `BYSCORE` | 스코어 (포함). `(` 접두 = 미포함 | `-inf`, `+inf` |
| `BYLEX` | 사전식 문자열. `[` 포함 / `(` 미포함 | `-`, `+` (전체) |

`REV` 사용 시 **start/stop을 뒤집어 전달해야** 한다.

### bitmap — string의 비트 뷰

- **특성**: string을 **2³² 비트 배열로** 사용. 독립 자료구조가 아니다.
- **대표 패턴**: 유저별 y/n 플래그 (출석, 기능 노출 등). **40억 유저 y/n이 512MB에** 들어간다.
- **핵심 커맨드**: `SETBIT k offset 0|1`, `GETBIT`, `BITFIELD` (여러 비트 세트), `BITCOUNT` (1의 개수).

### hyperloglog — 카디널리티 추정

- **특성**: 입력 데이터 자체를 저장하지 않고 해시로 추정. **크기 고정 12KB**, 오차 약 0.81%, 최대 2⁶⁴개 아이템.
- **대표 패턴**: "고유 방문자 수", "고유 검색어 수" 같은 **정확도 < 공간효율인** 지표.
- **핵심 커맨드**: `PFADD`, `PFCOUNT`.

### geospatial — 위경도 저장

- **특성**: 내부적으로 **sorted set**. `GEOADD k lon lat member` 순서에 주의.
- **대표 패턴**: "내 주변 N km 가게".
- **핵심 커맨드**: `GEOADD`, `GEOPOS`, `GEODIST k a b KM`, `GEOSEARCH ... BYRADIUS|BYBOX`.

### stream — 메시지 브로커 (개요)

- append-only, 카프카 영향, 소비자 그룹 개념 지원. **7장에서 본격 학습**.

## 키 자동 생성·삭제 3원칙

여러 아이템을 담는 자료구조(list/hash/set/sorted set/stream)는 **명시 생성·삭제가 없다**.

1. **없던 키에 아이템 넣으면 키가 자동 생성**.
2. **마지막 아이템 삭제 시 키도 자동 삭제** (단, `stream` 예외).
3. **없는 키에 읽기 전용 커맨드는 에러 대신 "빈 것처럼" 반환** (`LLEN`=0, `LPOP`=nil, `EXISTS`=0).

또한 **타입이 맞지 않으면 `WRONGTYPE` 에러** — string에 LPUSH하면 실패.

## 키 공통 커맨드

| 커맨드 | 설명 | 비고 |
|---|---|---|
| `EXISTS k [k ...]` | 존재 확인 | 1/0 반환 |
| `KEYS pattern` | 패턴 매칭 전체 조회 | ⚠ **프로덕션 금지** |
| `SCAN cursor [MATCH][COUNT][TYPE]` | 커서 기반 스캔 | cursor 0→0 복귀까지 반복 |
| `SORT k [BY][GET][LIMIT][ASC/DESC][ALPHA][STORE]` | list/set/zset 정렬 반환 | 문자열은 `ALPHA` |
| `RENAME` / `RENAMENX` | 이름 변경 | NX = 대상 없을 때만 |
| `COPY src dst [DB n] [REPLACE]` | 복사 | |
| `TYPE k` | 자료 구조 확인 | |
| `OBJECT ENCODING/IDLETIME` | 내부 저장 형태·유휴 시간 | |

## ⭐ 싱글 스레드와 "쓰지 마 3형제"

2장의 사슬이 여기서 실전 규칙이 된다. **실행 시간이 긴 커맨드 하나가 전체를 멈춘다**.

| 위험 커맨드 | 무엇이 문제인가 | 안전한 대체 |
|---|---|---|
| `KEYS *pattern*` | 전 키 스캔, 수 초간 블로킹, 모니터링 health check 실패 → 페일오버 유발 가능 | **`SCAN`** (커서, 10개 내외씩) |
| 큰 키에 `DEL` | 아이템 100만 개 삭제 = 동기 FLUSHALL과 동급 | **`UNLINK`** (백그라운드 스레드가 삭제) |
| 동기 `FLUSHALL` | 모든 키 삭제 완료까지 응답 불가 | **`FLUSHALL ASYNC`** |
| 큰 키에 `SMEMBERS` / `HGETALL` / `ZRANGE 0 -1` | 컬렉션 전체 반환 | **`SSCAN` / `HSCAN` / `ZSCAN`** |

**SCAN의 함정**: `MATCH`/`TYPE` 필터는 **스캔 후 반환 전 필터링이다**. 빈 배열이 여러 번 나올 수 있고, **cursor 0이** 돌아올 때까지 반복해야 완전 조회가 된다.

**Lazy free 옵션** (기본 `no`, 레디스 7 기준)
- `lazyfree-lazy-user-del = yes` → 모든 `DEL`이 자동으로 `UNLINK`처럼 동작
- `lazyfree-lazy-user-flush = yes` → `FLUSHALL`도 비동기

## 키의 만료 시간 (TTL)

```
EXPIRE     k seconds       [NX|XX|GT|LT]
EXPIREAT   k unix-ts-sec   [NX|XX|GT|LT]
EXPIRETIME k               → 만료 시각. 없음 -1 / 키 없음 -2
TTL        k               → 남은 초. 없음 -1 / 키 없음 -2
```

**옵션의 의미**
- `NX`: 만료시간 없을 때만 설정 (기존 TTL 덮지 않음)
- `XX`: 이미 있을 때만 갱신
- `GT`: 새 값이 기존보다 **클 때만** (TTL 연장)
- `LT`: 새 값이 기존보다 **작을 때만** (TTL 단축)

밀리초 버전: `PEXPIRE`, `PEXPIREAT`, `PEXPIRETIME`, `PTTL`.

# 두 장을 관통하는 한 문장

> **"레디스는 싱글 스레드로 메모리에서 돌고, 영속화는 fork()로 한다."**
>
> 이 한 줄에서
> - 2장의 모든 **OS 튜닝이** 나오고 (FD, THP, overcommit, backlog),
> - 3장의 모든 **자료구조 복잡도 경고와** **"쓰지 마 3형제"가** 나온다.

2장·3장을 따로 외우지 말고, **이 문장→세부 규칙으로** 재구성하는 훈련이 "온전히 이해"의 핵심이다.
