Architecture

아키텍처의 진화

소프트웨어 아키텍처는 중앙 집약적인 모놀리틱에서 독립적 모듈의 마이크로서비스로 변화하고 있다.

Monolithic

모놀리틱

  • 전체 애플리케이션을 하나의 패키지로 배포
  • 한 모듈 장애 → 전체 시스템 장애
  • 수직 확장만 가능 (서비스 전체 스케일업)
  • 하나의 중앙 집약적 관계형 DB

Microservice

마이크로서비스

  • 기능별 독립 모듈로 개별 배포
  • 장애 격리 — 다른 서비스에 영향 없음
  • 서비스별 독립 수평 확장
  • 서비스별 최적 DB 자유 선택

Data Paradigm

RDBMS만으로는
부족하다

비정형 데이터의 폭발적 증가, 실시간 처리 요구, 스키마 유연성의 필요 — 관계형 데이터베이스만으로 모든 상황을 커버할 수 없게 되었다.

관계형 데이터베이스의 핵심 — ACID
원자성(Atomicity) · 일관성(Consistency) · 독립성(Isolation) · 지속성(Durability)

관계형 DB가 나쁜 것이 아니라, 모든 상황에 최적은 아니다.

⚠ 핵심: 비즈니스 특성과 데이터 형태를 고려하지 않고 RDBMS만 고집하면 복잡하고 비효율적인 시스템이 될 수 있다.

Key Features

레디스의 5대 특징

인메모리 성능부터 클라우드 네이티브까지, 레디스가 제공하는 5가지 핵심 기능.

🚀

실시간 응답

모든 데이터를 RAM에서 처리. 디스크 I/O 없이 압도적 속도.

복제 + 센티널 자동 페일오버. 엔드포인트 변경 없이 장애 복구.

📦

확장성

클러스터 모드 자동 샤딩. 클러스터 버스로 노드 간 상호 감시.

☁️

클라우드 네이티브

컨테이너·데브옵스 최적화. 멀티 클라우드에서 일관된 성능.

Threading Model

싱글 스레드, 그러나 빠르다

클라이언트 커맨드 처리는 이벤트 루프 기반 싱글 스레드. 메인 1개 + 백그라운드 3개, 총 4개 스레드로 동작.

⚠ 주의: 오래 걸리는 커맨드 하나가 전체를 블로킹할 수 있다. KEYS, FLUSHALL 등 O(N) 커맨드 사용에 주의해야 한다.

Appendix B

싱글 스레드의 진실

커맨드 처리는 순수 싱글 스레드. 느린 시스템 콜만 BIO 스레드로 분리.

📁

CLOSE_FILE

파일 디스크립터 닫기. 대형 파일 unlink가 수 ms 소요.

💿

AOF_FSYNC

AOF 디스크 동기화. fsync가 수십 ms 소요.

🗑️

LAZY_FREE

대형 키 메모리 해제. 수백 MB 해제에 수백 ms 소요.

클러스터에서 병렬성은? 각 노드는 독립 프로세스 — 서로 다른 노드의 키는 실제 병렬 처리. 같은 노드 안에서는 여전히 싱글 스레드. 즉, 병렬성 = 샤딩 설계의 문제.

⚠ 위험한 커맨드 → 대안:
KEYS * → SCAN · SMEMBERS(대형) → SSCAN · HGETALL(대형) → HSCAN · DEL(대형 키) → UNLINK (비동기 삭제)

Appendix C

왜 인메모리인가

각 단계별 약 1,000배의 속도 차이. 단, RAM은 디스크 대비 수십 배 비싸고 용량 한계가 있다.

100ns

RAM 접근 속도

100μs

SSD 접근 속도
(×1,000 느림)

10ms

HDD 접근 속도
(×100,000 느림)

100K+

ops/sec
(단일 인스턴스)

레디스는 구조적으로 데이터 유실 가능성이 존재한다. 일반적 아키텍처에서는 RDBMS가 SSoT, 레디스는 캐시/세션/실시간 보조 저장소로 역할을 분리하는 패턴이 선호된다.

단, 속도가 내구성보다 중요한 도메인(선착순 쿠폰, 게임 리더보드, 실시간 입찰)에서는 primary DB로 활용 가능.

⚠ 필수 설정: maxmemory-policy로 eviction 전략 설정 필수. 물리 메모리 한계 도달 시 데이터 자동 제거 정책을 미리 정해야 한다.
