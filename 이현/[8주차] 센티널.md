# 9장. 센티널 
## 고가용성 기능의 필요성

레디스는 복제(Replication)를 통해 데이터를 여러 노드에 복사할 수 있다.

```
Master
  │
  ├── Replica A
  └── Replica B
```

하지만 단순 복제만 구성했다고 해서 고가용성(HA)이 완성되는 것은 아니다.

마스터 노드가 장애가 발생하면 복제본에는 데이터가 존재하더라도 서비스는 계속 장애 상태가 된다.

왜냐하면 애플리케이션은 여전히 죽어있는 마스터를 바라보고 있기 때문이다.

기존에는 아래 과정을 운영자가 직접 수행해야 했다.

```
1. Replica 접속
2. REPLICAOF NO ONE 실행
3. 새로운 Master 승격
4. 애플리케이션 Redis 주소 변경
5. 배포
```

이 과정이 늦어지면 서비스 장애 시간이 길어진다.

특히 캐시 서버가 갑자기 사라지면 Cache Stampede가 발생해 DB까지 장애가 전파될 수 있다.

### 🤔 복제만으로는 왜 부족할까?

복제는 데이터를 복사하는 기능이다.

하지만 장애가 났을 때

- 누가 새 마스터가 될지
- 클라이언트는 어디로 연결할지
- 기존 노드는 어떻게 재구성할지

는 처리하지 못한다.

즉

```
Replication = 데이터 복사

Sentinel = 장애 감지 + 복구 자동화
```

이다.

# 센티널이란?

Sentinel은 Redis가 자체적으로 제공하는 고가용성(HA) 솔루션이다.

주요 목적은

```
Master 장애 감지
        ↓
자동 Failover
        ↓
서비스 지속
```

이다.

센티널을 사용하면 운영자가 직접 개입하지 않아도 장애 복구가 가능하다.

## 센티널의 주요 기능

### 1. 모니터링(Monitoring)

마스터와 복제본 상태를 지속적으로 감시한다.

```
PING
 ↓
응답 확인
 ↓
정상 여부 판단
```

### 2. 자동 페일오버(Failover)

마스터 장애 발생 시

정상적인 Replica 중 하나를 선택해 Master로 승격시킨다.

```
Master 장애
       ↓
Replica 선출
       ↓
Master 승격
       ↓
나머지 Replica 재연결
```

### 3. 구성 정보 제공(Configuration Provider)

클라이언트가 현재 Master 정보를 얻을 수 있도록 제공한다.

애플리케이션은 Sentinel에게 물어본 뒤 현재 Master로 연결할 수 있다.

```
Application
      ↓
 Sentinel
      ↓
현재 Master 주소 반환
```

# 센티널은 분산 시스템이다

센티널도 장애가 날 수 있다.

그래서 센티널은 여러 대가 함께 동작하도록 설계됐다.

```
Sentinel A
Sentinel B
Sentinel C
```

센티널끼리 서로 통신하면서 장애 여부를 판단한다.

## Quorum

Quorum은

"마스터가 죽었다고 동의한 Sentinel 수"

를 의미한다.

예시

```
Sentinel 3대
Quorum 2
```

이면

```
A : 죽음
B : 죽음
C : 정상
```

2표 이상이 모여야 장애로 판단한다.

### 🤔 왜 3대 이상이 권장될까?

과반수 투표 때문이다.

```
1대 → 의미 없음
2대 → 동점 발생
3대 → 과반수 가능
5대 → 더 안정적
```

그래서 보통

```
3대
5대
7대
```

같은 홀수 구성을 사용한다.

# 센티널 배치 전략

## 2개의 Replica가 있는 경우

```
서버 A
 ├ Redis(Master)
 └ Sentinel

서버 B
 ├ Redis(Replica)
 └ Sentinel

서버 C
 ├ Redis(Replica)
 └ Sentinel
```

가장 이상적인 구조다.

### 장애 발생

```
A 장애
```

↓

```
B 또는 C 선출
```

↓

```
새 Master 승격
```

↓

```
나머지 Replica 재구성
```

↓

```
서비스 계속 운영
```

### A 서버 복구

복구되면 Sentinel이 자동으로 구성한다.

```
A
 ↓
Replica 재편입
```

운영자가 별도 작업할 필요가 없다.

## 최소 구성

비용을 아끼기 위한 구조

```
서버 A
 Redis Master
 Sentinel

서버 B
 Redis Replica
 Sentinel

서버 C
 Sentinel만
```

서버 C는

- 데이터 저장 안함
- Redis 없음
- Sentinel만 존재

한다.

### 🤔 Sentinel만 있는 서버가 왜 필요할까?

투표권 때문이다.

```
A Sentinel
B Sentinel
C Sentinel
```

3표가 있어야 과반수 선출이 가능하다.

실제 데이터는 저장하지 않는다.

# 센티널 실행하기


예시

```bash

```

## Sentinel 설정 파일

```bash

```

### 설정 의미

### port

```bash
26379
```

센티널 포트

### sentinel monitor

```bash
sentinel monitor
```

형식

```bash
sentinel monitor <name> <ip> <port> <quorum>
```

예시

```bash
sentinel monitor master-test 192.168.0.11 6379 2
```

의미

```
master-test
 └ 마스터 별칭

192.168.0.11

6379
 └ Redis 포트

2

```

### 🤔 Replica 정보는 왜 안 적을까?

센티널은 시작 시

```
Master 접속
       ↓
Replica 자동 발견
```

을 수행한다.

그래서 Replica 정보를 직접 적지 않는다.

# 자동 페일오버 과정

## 1단계 - 주관적 장애(SDOWN)

한 Sentinel이

```
PING 실패
```

를 일정 시간 이상 확인하면

```
SDOWN
```

으로 판단한다.

Subjectively Down

즉

```
"내가 보기엔 죽은 것 같음"
```

이다.

## 2단계 - 객관적 장애(ODOWN)

다른 Sentinel에게 확인한다.

```
A : 죽은 것 같은데?
B : 나도
C : 나도
```

↓

```
Quorum 충족
```

↓

```
ODOWN
```

Objectively Down

즉

```
"우리 모두 죽었다고 판단"
```

이다.

### 🤔 SDOWN과 ODOWN 차이

SDOWN

```
개별 Sentinel 판단
```

ODOWN

```
다수 Sentinel 합의
```

## 3단계 - 리더 Sentinel 선출

Failover는 한 Sentinel만 수행한다.

여러 Sentinel이 동시에 실행하면 충돌하기 때문이다.

```
Leader Sentinel 선출
```

## 4단계 - 새로운 Master 선정

Replica 중 하나를 선택한다.

선정 기준

### 1순위

Replica 우선순위

```bash
replica-priority
```

값이 낮을수록 우선

### 2순위

복제 데이터 최신 여부

Replication Offset

### 3순위

Run ID

가장 작은 값

## 5단계 - Master 승격

```bash
REPLICAOF NO ONE
```

실행

```
Replica
   ↓
Master
```

변경

## 6단계 - 나머지 Replica 재구성

기존 Replica들은

```
새 Master
```

를 바라보도록 자동 변경된다.

## 7단계 - 기존 Master 복구

장애 서버가 복구되면

```
기존 Master
      ↓
Replica
```

로 자동 편입된다.

# 클라이언트는 어떻게 연결할까?

실무에서는 Redis 주소를 직접 넣기보다 Sentinel을 바라보게 한다.

```
Application
      ↓
 Sentinel
      ↓
 현재 Master 조회
```

장애 발생 시에도

애플리케이션 설정 변경 없이 새 Master를 사용할 수 있다.

## 기억해야 할 핵심

### 🤔 Sentinel은 데이터를 저장하는가?

아니다.

Sentinel은

```
감시
투표
선출
Failover
```

만 담당한다.

### 🤔 Sentinel이 해결하는 문제는?

복제 환경에서

```
장애 감지
↓
새 Master 선출
↓
Replica 재구성
↓
클라이언트 연결 전환
```

을 자동화해 Redis의 고가용성을 제공한다.

즉

```
Replication = 데이터 복사

Sentinel = 장애 복구 자동화

Cluster = 장애 복구 + 샤딩
```

로 이해하면 된다.


## 스플릿 브레인(Split Brain)

스플릿 브레인은 네트워크 단절(Network Partition)로 인해

서로 다른 노드가 자신이 정상적인 Master라고 판단하는 현상을 의미한다.

```text
        네트워크 단절

   Master A  XXXXXXXXX  Sentinel

        │
        │
    Replica B
````

Sentinel은 Master와 통신할 수 없게 되면

Master가 장애가 발생했다고 판단할 수 있다.

```text
Master A
    ↓
ODOWN 판단
    ↓
Replica B 승격
    ↓
새로운 Master 생성
```

하지만 실제로는 기존 Master A가 죽은 것이 아니라

단순히 네트워크가 끊어진 상태일 수 있다.

그러면 다음과 같은 상황이 발생한다.

```text
Master A (기존 Master)

Master B (새로운 Master)
```

두 개의 Master가 동시에 존재하게 된다.

이 상태를 Split Brain이라고 한다.

````

#### 🤔 왜 위험할까?

```text
사용자1 → Master A 쓰기

사용자2 → Master B 쓰기
````

데이터가 서로 다른 노드에 기록된다.

```text
A 데이터 ≠ B 데이터
```

결국 데이터 정합성이 깨진다.


### 예시

```text
10:00
Master A 정상

10:01
네트워크 단절

10:02
Sentinel이 장애 판단

10:03
Replica B 승격

10:04
A도 계속 서비스 중
```

결과

```text
Master A
Master B
```

동시 존재


### Redis는 어떻게 대응할까?

Redis는 다음 설정을 제공한다.

#### min-replicas-to-write

최소 몇 개의 Replica가 연결되어 있어야 쓰기를 허용할 것인가

```bash
min-replicas-to-write 1
```

#### min-replicas-max-lag

Replica와의 복제 지연 허용 시간

```bash
min-replicas-max-lag 10
```

예시

```bash
min-replicas-to-write 1
min-replicas-max-lag 10
```

의 의미

```text
정상 Replica가 1개 이상 존재하고

복제 지연이 10초 이하일 때만

Master 쓰기 허용
```

이다.

네트워크 단절 시 Replica와 연결이 끊어지면

기존 Master는 쓰기 요청을 거부한다.

```text
Replica 연결 끊김
        ↓
쓰기 차단
        ↓
데이터 분기 방지
```


#### 🤔 Sentinel이 Split Brain을 완벽하게 막아줄까?

아니다.

Sentinel은 장애를 감지하고 Failover를 수행할 뿐이다.

Split Brain 자체를 완벽하게 방지하는 기능은 아니다.

그래서 실무에서는

* Sentinel 과반수 구성
* min-replicas-to-write 설정
* min-replicas-max-lag 설정
* 안정적인 네트워크 구성

을 함께 사용한다.
