복제는 백업이 아니라 고가용성의 시작이다

01 · Replication Structure

레디스 복제는 Single Master 구조다

Redis는 하나의 복제 그룹 안에서 멀티 마스터를 지원하지 않는다. 쓰기 기준점은 마스터 하나이며, 복제본은 마스터에서 변경된 데이터를 그대로 받아와 같은 상태를 유지한다.

Master 모든 데이터 입력 처리
Replica 읽기 전용 복제본
멀티 마스터 미지원

한 복제 그룹에서 동시에 쓰기를 받는 마스터를 여러 개 두는 구조가 아니다.

마스터는 복제본이 아님

쓰기 기준점이 되는 마스터는 다른 노드의 변경을 받아오는 복제본 역할을 하지 않는다.

변경분은 Master에서 Replica로

모든 데이터 입력은 마스터에서 처리되고, 복제본은 그 변경 데이터를 전달받는다.

핵심 정리복제본은 읽기 확장과 장애 대응을 돕는 노드다. 데이터의 원본 기준은 마스터이며, 복제본에서 직접 데이터를 쓰는 방식으로 멀티 마스터처럼 운영하지 않는다.
03 · REPLICAOF

복제 구조 구성하기

복제본이 될 노드에서 REPLICAOF 커맨드를 실행하면 마스터와 복제 연결이 시작된다. 

127.0.0.1:6379> REPLICAOF <master-ip> <master-port>
OK
1
Replica 노드에서 REPLICAOF 실행

복제본이 될 노드 B에 마스터 노드 A의 IP와 포트를 입력한다.

2
복제 연결 시작

노드 B는 노드 A의 데이터를 받아오기 시작한다.

3
쓰기 요청은 Master로

서비스 애플리케이션은 기본적으로 마스터 노드 A를 바라봐야 한다. 모든 업데이트 커맨드는 A에서 실행된다.

4
Master 장애 시 연결 전환

기본 복제만 사용하는 경우 Master 장애 시 자동 전환은 일어나지 않는다. 자동 페일오버가 없다면 Replica를 새 Master로 승격하고 애플리케이션 연결 설정도 변경해야 한다.

확장 구조마스터에 여러 복제본을 연결할 수 있고, 복제본 아래에 또 다른 복제본을 둘 수도 있다. 다만 한 복제 그룹의 최상위 write 노드는 항상 하나의 마스터다.
04 · Failover

Master 장애 시 페일오버는 별도 구성이 필요하다

Redis 기본 복제는 데이터를 Master에서 Replica로 복사하는 기능이다. Master 장애를 감지하고 Replica를 자동 승격하는 기능까지 포함하지는 않는다. 자동 페일오버가 필요하면 Sentinel이나 Cluster를 함께 구성한다.

수동 페일오버

운영자가 Replica에서 REPLICAOF NO ONE을 실행해 새 Master로 승격한다. 이후 애플리케이션 접속 주소도 새 Master로 변경해야 한다.

Redis Sentinel

Sentinel이 Master 상태를 감시하다가 장애를 감지하면 Replica 하나를 새 Master로 승격하고, 다른 Replica들이 새 Master를 바라보도록 조정한다.

Redis Cluster

데이터를 여러 Master에 나눠 저장하는 샤딩 구조다. 각 Master에 Replica를 둘 수 있고, 장애 시 해당 Replica가 승격된다. 

# 수동으로 Replica를 Master로 승격
127.0.0.1:6379> REPLICAOF NO ONE
OK
05 · Password

패스워드가 있는 Master에 복제본 연결하기

마스터 노드에 requirepass가 설정되어 있다면, 복제본 노드에는 masterauth를 설정해야 한다.

Master

127.0.0.1:6379> CONFIG SET requirepass mypasswd
OK
127.0.0.1:6379> CONFIG REWRITE
OK

Replica

127.0.0.1:6379> CONFIG SET masterauth mypasswd
OK
127.0.0.1:6379> CONFIG REWRITE
OK
주의CONFIG SET은 실행 중인 인스턴스에 즉시 반영되지만 설정 파일까지 바꾸지는 않는다. 재시작 후에도 유지하려면 CONFIG REWRITE가 필요하다.
05 · Full Resynchronization

복제 메커니즘: 처음 연결될 때는 RDB 기반 전체 동기화

복제 연결은 자동으로 이뤄지지만 내부에서는 꽤 많은 일이 발생한다. 초기 연결에서는 마스터가 RDB 스냅숏을 만들고 복제본이 이를 로드한다.

1
REPLICAOF로 복제 연결 시도

복제본이 마스터에게 동기화를 요청한다.

2
Master가 fork 후 RDB 생성

마스터는 자식 프로세스를 생성하고 RDB 스냅숏을 만든다.

3
복제 버퍼에 변경 작업 저장

RDB 생성 중 마스터에서 수행된 변경 명령은 RESP 형태로 복제 버퍼에 쌓인다.

4
RDB를 Replica로 전달

RDB 파일 생성이 끝나면 복제본으로 전송한다.

5
Replica 기존 데이터 삭제 후 RDB 로드

복제본은 기존 내용을 비우고 전달받은 RDB를 메모리에 로드한다.

6
복제 버퍼 replay

RDB 생성 동안 쌓인 변경 명령을 복제본에 전달해 최종 상태를 맞춘다.

실무 포인트복제 연결은 단순 네트워크 연결이 아니다. fork, RDB 생성, 디스크 또는 소켓 전송, 복제본 로딩, 버퍼 replay가 필요하다. 그래서 대용량 Redis에 Replica를 붙이는 작업은 성능 영향이 생길 수 있는 작업으로 보고 트래픽이 적은 시간에 진행하거나, 모니터링과 장애 대응 담당자를 정해 두고 진행하는 것이 좋다.
06 · Disk / Diskless

디스크를 사용하는 복제와 Diskless 복제

Redis 7.0 이전에는 repl-diskless-sync 기본값이 no였지만, Redis 7.0 이후에는 기본값이 yes이다.

디스크를 사용하는 방식

RDB 생성Master Disk
→
Replica 전송Network
→
Disk 저장 후 로드Replica
  • 복제 속도가 디스크 I/O 처리량에 영향을 받는다.
  • NAS 같은 원격 디스크를 사용하면 더 느려질 수 있다.
  • 하나의 RDB 파일을 여러 복제본에 전달하기 쉽다.

Diskless 방식

RDB 생성pipe
→
Socket StreamMaster → Replica
→
Replica Disk 저장then load
  • 마스터가 RDB를 디스크에 저장하지 않고 소켓으로 바로 전송한다.
  • 디스크 I/O가 느리고 네트워크가 빠른 환경에서 유리하다.
  • 복제본은 안정성을 위해 받은 RDB를 디스크에 저장한 뒤 로드한다.
항목 Disk 기반 Diskless
마스터 RDB 저장 디스크에 저장 디스크 저장 없이 소켓으로 전송
병목 디스크 I/O 네트워크 / 복제본 처리
동시 Replica 연결 기존 RDB를 여러 복제본에 전달 가능 하나의 복제 연결 중 다른 복제본은 대기할 수 있음
관련 옵션 repl-diskless-sync no repl-diskless-sync yes, repl-diskless-sync-delay
repl-diskless-sync-delayDiskless 복제에서 새로운 복제 연결이 들어오면 기본 5초 대기한다. 이 시간 안에 다른 복제본도 연결되면 여러 복제본에 한 번에 RDB stream을 보낼 수 있다. 
# 기본값 예시
repl-diskless-sync-delay 5
07 · Asynchronous Replication

정상 복제 상태에서는 비동기 방식으로 동작한다

Redis는 마스터에서 커맨드를 처리한 뒤 복제본 반영 완료를 기다리지 않고 클라이언트에게 응답한다. 그래서 복제 구조에서도 짧은 지연 시간과 높은 성능을 유지한다.

ClientSET user:1
→
Master 저장command 처리
→
Client OK먼저 응답
⇢
Replica 전파비동기
유실 가능성마스터가 클라이언트에게 OK를 보낸 직후, 복제본으로 명령이 전달되기 전에 마스터가 비정상 종료되면 그 변경 데이터는 복제본에 없을 수 있다.

장점

복제본 응답을 매번 기다리지 않기 때문에 write latency가 낮고 처리량이 높다.

단점

마스터는 쓰기를 처리하자마자 OK를 반환하고, 그 변경 내용은 잠시 뒤 복제본으로 전달된다. 이 짧은 시간 동안 복제본을 읽으면 방금 쓴 값이 아직 안 보일 수 있다.

08 · Replication ID & Offset

복제 ID와 오프셋은 “같은 데이터 계보에서 어디까지 왔는가”를 나타낸다

모든 Redis 인스턴스는 복제 기능을 사용하지 않아도 replication ID를 가진다. 복제 ID는 오프셋과 쌍으로 존재하며, 데이터가 수정되는 커맨드가 수행될 때마다 오프셋이 증가한다.

Master
replid = aaa · offset 915
Replica A
replid = aaa · offset 901
Replica B
replid = aaa · offset 915
해석복제 ID가 같고 오프셋도 같으면 두 노드는 같은 데이터 상태라고 볼 수 있다. 위 예시에서 Replica B는 마스터와 동일하지만, Replica A는 offset 901까지만 받아 아직 일부 변경을 받지 못했다.
INFO REPLICATION으로 확인하기role · offset · lag

복제본 연결 전 Master

> INFO Replication
role:master
connected_slaves:0
master_replid:e3b06d3eba522894a240a8a9ce3e808dd5ccfd7a
master_repl_offset:709
repl_backlog_size:67108864

복제본 연결 후 Master

role:master
connected_slaves:1
slave0:ip=127.0.0.1,port=6002,state=online,offset=709,lag=0
master_repl_offset:807

Replica

role:slave
master_host:127.0.0.1
master_port:6001
master_link_status:up
slave_repl_offset:723
slave_read_only:1
master_replid:e3b06d3eba522894a240a8a9ce3e808dd5ccfd7a
09 · Partial Resynchronization

부분 재동기화: 매번 RDB 전체를 다시 받지 않기 위한 장치

네트워크가 잠깐 끊길 때마다 전체 재동기화를 수행하면 리소스 낭비가 크다. 그래서 Redis는 복제 백로그 버퍼와 PSYNC를 이용해 부분 재동기화를 시도한다.

1
복제 연결 끊김

Replica는 마지막으로 받은 replication ID와 offset을 기억한다.

2
Master는 backlog에 명령 저장

연결이 끊긴 동안에도 최근 변경 명령을 복제 백로그 버퍼에 보관한다.

3
Replica가 PSYNC 요청

자신의 replication ID와 offset을 마스터에 전달한다.

4
Backlog에 필요한 데이터가 있으면 Partial Resync

RDB 전체 전송 없이 누락된 명령만 전달한다.

Replica offset 900
→
Backlog 901~915 보유
→
901~915만 전송
상황 동작
replication ID가 같고 backlog에 필요한 offset이 있음 부분 재동기화 수행
backlog에 필요한 데이터가 없음 전체 재동기화 수행
복제본이 보낸 replication ID가 현재 마스터와 불일치 전체 재동기화 수행 
# 복제 백로그 크기
repl-backlog-size 1mb

# 복제본이 모두 끊긴 후 백로그를 유지할 시간
repl-backlog-ttl 3600
실무 포인트백로그가 클수록 부분 재동기화가 가능한 시간이 길어진다. 네트워크가 불안정하거나 복제본이 자주 재연결되는 환경에서는 backlog 크기가 중요하다.
10 · Secondary Replication ID

Secondary 복제 ID는 장애 전환 후 부분 재동기화를 돕는다

Replica가 Master로 승격되면 새로운 replication ID를 갖는다. 이때 승격 전 사용하던 이전 Master의 replication ID를 secondary replication ID로 잠시 보관해, 기존 Replica들이 전체 재동기화 대신 부분 재동기화를 시도할 수 있게 한다.

기존 Masterreplid = aaa
×
Replica 승격new replid = bbb
→
secondary replidold replid = aaa
왜 필요한가?

장애 전환 직후 다른 Replica들은 아직 이전 Master의 replid와 offset을 기억하고 있다. 새 Master가 이전 replid 정보를 보관하고 있으면 “같은 데이터 계보”로 판단해 누락분만 받을 수 있다.

없다면?

replid가 완전히 달라진 것으로 판단되어 부분 재동기화를 하지 못하고, RDB를 다시 받는 전체 재동기화가 발생할 가능성이 커진다. 

# INFO Replication에서 볼 수 있는 값
master_replid:bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb
master_replid2:aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa
second_repl_offset:915
발표 포인트Primary replid는 현재 Master의 새 복제 ID이고, Secondary replid는 승격 전의 이전 복제 ID다. 장애 전환 직후 Replica들이 불필요하게 전체 동기화를 다시 하지 않도록 돕는 안전장치로 보면 된다.
11 · Read Only Replica

복제본은 기본적으로 읽기 전용으로 동작한다

Redis 2.6 이후 복제본은 기본적으로 읽기 전용이다. 클라이언트가 복제본에 연결해도 데이터를 읽는 커맨드만 수행할 수 있고, SET 같은 변경 명령은 수행할 수 없다. 

# 관련 옵션
replica-read-only yes
읽기 전용이 필요한 이유

복제본이 임의로 데이터를 변경하면 마스터와 데이터셋이 달라진다. Redis 복제 구조에서는 마스터 데이터가 기준이다.

쓰기 요청의 기준

모든 데이터 입력은 마스터 노드에서 처리한다. 복제본은 마스터에서 변경된 데이터를 그대로 받아온다.

옵션을 끄면?replica-read-only no로 복제본에 직접 write할 수는 있지만, 전체 재동기화가 발생하면 복제본에서 직접 쓴 데이터는 사라질 수 있다.
중요복제본에 직접 쓴 데이터는 로컬에서만 유지된다. 해당 복제본 아래에 또 다른 복제본이 있더라도 그 로컬 변경은 하위 복제본으로 전파되지 않는다. 하위 복제본은 최상위 마스터의 복제 프로토콜을 전달받는다.
12 · Stale Data

유효하지 않은 복제 데이터: 마스터와 복제본이 일치하지 않는 순간

복제본이 마스터와 연결이 끊겼거나, 복제 연결이 시작됐지만 아직 완료되지 않았다면 복제본의 데이터는 유효하지 않을 수 있다.

Master Write최신 데이터
→
Network Lag전파 지연
→
Replica Read이전 데이터

replica-serve-stale-data

yes 기본값 데이터가 유효하지 않다고 판단되어도 읽기 요청에 응답한다. 가용성을 우선한다.

13 · Replication without Backup

백업 없이 복제만 사용할 때 발생할 수 있는 위험

복제를 사용하더라도 마스터와 복제본에서 백업 기능을 사용하는 것이 권장된다. 백업을 사용하지 않는다면 마스터의 자동 재시작 설정이 오히려 데이터를 지우는 방향으로 동작할 수 있다.

1
백업 없는 Master와 Replica 운영

두 노드는 메모리에만 데이터를 가지고 있다.

2
Master 장애 후 자동 재시작

백업 파일이 없으면 Redis 프로세스가 재시작되면서 메모리 내용은 초기화된다.

3
Replica가 Master에 다시 연결

Replica에는 데이터가 남아 있지만 마스터와 복제 연결을 시도한다.

4
빈 데이터셋 전파

재시작된 Master가 빈 데이터셋을 Replica로 전달하면, Replica에 남아 있던 데이터도 사라질 수 있다.

결론데이터 안정성을 위해 복제 기능을 사용할 경우 백업 기능도 함께 사용하는 것이 좋다. 백업을 사용하지 않는다면 마스터 인스턴스의 자동 재시작은 활성화하지 않는 것을 권장한다.

백업 기능 사용

마스터가 재시작될 때 RDB/AOF 파일을 읽어 데이터를 복원하고, 복원된 데이터가 복제본에 전달된다.

백업 없이 자동 재시작 미사용

장애 후 바로 빈 Master를 띄우지 않고, 애플리케이션 연결을 데이터가 남아 있는 Replica로 변경하거나 Replica의 데이터를 백업받아 Master에 복원한 뒤 재시작한다.
