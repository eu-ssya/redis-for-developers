# 1장. 마이크로서비스 아키텍처와 레디스

## NoSQL의 등장 배경
### 모놀리틱 아키텍처
전통적인 모델로, 전체 애플리케이션을 하나의 패키지로 개발·배포하는 방식이다. 모든 시스템이 연결되어 관리된다. 규모가 작을 땐 운영이 쉽지만, 커지면 유지보수 복잡도가 증가한다. 모듈 하나만 수정해도 전체를 다시 배포해야 하며 빌드와 배포 시간이 길어진다. 트래픽 급증이나 복잡한 트랜잭션 요구에 유연하게 대처하기 힘들다.

### 마이크로서비스 아키텍처
독립된 모듈을 조립해 하나의 서비스를 만드는 구조다. 프로젝트가 빠르게 성장한다면 민첩한 대응을 위해 이 구조를 채택하는 것이 유리하다. 하지만 소규모 팀에서는 서비스 분리로 인한 관리 포인트와 운영 부담이 늘어날 수 있다.

### 데이터 저장소 요구사항의 변화
모놀리틱에서는 주로 관계형 데이터베이스(RDBMS)를 사용했다. 행과 열로 된 테이블 간의 관계를 규정하는데, 서비스가 커질수록 이 관계와 쿼리가 매우 복잡해진다.
최근에는 정해진 형태가 없는 비정형 데이터가 늘어나고 있다. RDBMS의 정형화된 테이블로는 관리가 어렵다. 또한 RDBMS는 구조 변경 시 DBA의 승인이 필요한 등 절차가 까다로워 개발 속도가 저하될 수 있지만, NoSQL은 개발팀이 바로 구조를 바꿀 수 있어 더 빠른 개발이 가능하다.


## NoSQL이란
말 그대로 SQL을 사용하지 않는 데이터 저장소라는 뜻을 내포한다. 일반적인 특징은 다음과 같다.

* **실시간 응답:** 사람은 100ms만 넘어도 지연을 인지한다. 따라서 지연은 0~1ms 이내여야 한다.
* **확장성:** 세일 기간이나 이벤트 등 예상치 못한 트래픽 증가에 유연하게 확장 가능하다.
* **고가용성:** 장애 상황에서 신속하게 복구되어 항상 사용할 수 있어야 한다.
* **유연성:** 비정형 데이터를 저장하는 다양한 방법을 제공한다.
* **단순성:** 서비스별로 적절한 데이터 모델(멀티 모델)을 선택해 관리 포인트를 줄이고 싶어 한다.


## NoSQL 데이터 저장소 유형
* **그래프 유형:** 엔티티 간의 관계를 노드와 에지로 저장한다. SNS 친구 추천이나 쇼핑몰 관심 상품 추천 서비스에 유용하다.
* **칼럼 유형:** 행이 아닌 열을 기준으로 데이터를 저장한다. 유연한 스키마를 가질 수 있으며 BI 분석, 빅데이터 처리에 적합하다. (예: 카산드라, HBase)
* **문서 유형:** JSON 형태로 데이터를 저장한다. 코드와 데이터 구조가 동일해 직관적이고 유연하다. (예: MongoDB)
* **키-값 유형:** 가장 단순하고 빠르다. 저장이 간단해서 수평적 확장이 쉽다. 레디스가 대표적이다.


## 레디스(Redis)란?
Remote Dictionary Server의 약자로, 고성능 키-값 유형의 인메모리 NoSQL 데이터베이스다.

### 레디스의 특징
**1. 실시간 응답 (빠른 성능)**
온디스크 데이터베이스의 데이터는 영구적으로 디스크에 저장되며, 캐싱되어 있지 않은 데이터를 찾을 때는 직접 디스크에 가서 데이터를 검색해야 한다. 
이때 디스크에 저장된 데이터는 바로 찾을 수 없으며, 페이지 단위로 메모리에 올린 뒤 메모리에서 데이터를 찾고, 없는 경우 다른 페이지를 디스크에서 가져와 메모리에 올린 뒤 찾는 과정을 반복한다. 디스크 접근은 메모리보다 훨씬 느리므로 접근 빈도가 늘면 성능이 저하된다. 레디스는 인메모리 방식이라 매우 빠르다.

**2. 단순성 및 자료 구조**
문자열뿐만 아니라 Hash, Set 등 프로그래밍 기본 자료 구조와 밀접한 다양한 타입을 지원한다. 덕분에 추가 가공 없이 앱에서 바로 사용 가능하며, RDBMS와 프로그래밍 언어 간의 구조 차이에서 오는 '임피던스 불일치'를 해소해 준다.

**3. 주의 할 점**
레디스는 싱글 스레드로 동작한다. 
이 말은 한 사용자가 오래 걸리는 커맨드를 수행한다면, 다른 사용자는 그 쿼리가 완료될 때까지 대기해야 한다는 것이다. 
반환이 느린 특정 커맨드가 존재하니, 이런 커맨드만 주의해서 사용하면 된다.

**4. 고가용성**
자체적으로 고가용성 기능을 제공한다. 복제를 통해 데이터를 여러 서버에 분산시킬 수 있으며, 센티널은 장애 상황을 탐지해 자동으로 페일오버를 시켜준다. 애플리케이션이 센티널을 이용해 레디스에 연결하는 구조에서는 마스터에 장애가 발생하더라도 레디스로의 엔드포인트를 변경할 필요 없이 페일오버가 완료돼 정상화된 마스터 노드를 사용할 수 있다.

**5. 확장성**
레디스에서 클러스터 모드를 사용한다면 쉽게 수평적 확장이 가능하다. 데이터는 클러스터 내에서 자동으로 샤딩된 후 저장되며, 여러 개의 복제본이 생성될 수 있다. 인스턴스끼리 '클러스터 버스' 프로토콜로 서로 감시하며 장애 발생 시 자동으로 페일오버되어 고가용성을 유지한다.


## 마이크로서비스 아키텍처와 레디스
### 데이터 저장소로서의 레디스
MSA에서 각 서비스별 개별 저장소로 쓰기 알맞다. 설치가 쉽고 리소스를 적게 쓰면서 처리량이 많다. 로드밸런서 같은 추가 서비스 없이도 가용성 확보가 충분하다. 데이터는 AOF나 RDB 형식으로 디스크에 주기적으로 저장 가능하다.

### 메시지 브로커로서의 레디스
서비스 간 비동기 통신 채널로 활용하기 좋다.
* **Pub/Sub:** 매우 빠르고 간단하며 1회성 전달(Fire-and-forget) 패턴의 알림 서비스에 유용하다.
* **List 자료 구조:** 메시징 큐로 사용하기 알맞다. 데이터 push/pop이 빠르고, 데이터가 들어올 때까지 대기하는 블로킹 기능을 쓸 수 있다.
* **Stream 자료 구조:** 카프카에서 영감을 받은 구조로, 소비자 그룹을 통한 분산 처리가 가능하다. stream 자료 구조를 이용하면 완벽한 스트림 플랫폼으로 사용할 수 있다.

## 궁금한 점 🤔
### 그럼 왜 카프카나 SQS 같은 메시지 큐를 따로 쓰는거지?
1. 데이터 영속성과 복구 능력 
   - 레디스: 인메모리 기반이라 속도는 압도적이지만, 메모리 용량 한계가 있다. 큐에 데이터가 과하게 쌓이면 시스템이 불안정해질 수 있고, 장애 발생 시 데이터 유실 위험이 상대적(카프카 대비)으로 높다. 
   - 카프카: 데이터를 디스크에 영구 저장한다. 메시지를 처리한 후에도 설정 기간(예: 7일) 동안 보관하므로, 나중에 문제가 생겨도 과거 데이터를 다시 읽어와(Replay) 복구하는 것이 가능하다.

2. 분산 처리 및 확장성 (Scale-out)
   - 레디스: 보통 단일 소비자 혹은 소수의 소비자가 데이터를 가져가는 구조에 최적화되어 있다.
   - 카프카: 파티션(Partition) 개념을 사용하여 데이터를 여러 서버에 쪼개 저장한다. 수십 대의 서버가 동시에 데이터를 읽어가는 병렬 처리가 가능해 초당 수백만 건의 대용량 트래픽 처리에 압도적이다.

3. 운영 편의성 (SQS)
   - 레디스: 직접 서버를 띄우고 메모리 관리, 센티널/클러스터 설정 등을 직접 운영해야 한다.
   - SQS: AWS의 완전 관리형 서비스다. 운영 리소스가 거의 들지 않으며, 트래픽이 갑자기 튀어도 AWS가 알아서 확장해주므로 관리가 매우 편리하다.

#### 상황별 선택 기준
    - 레디스 선택: 무엇보다 속도가 제일 중요하고, 데이터가 소실되어도 서비스 치명도가 낮은 경우. (ex. 게임 내 실시간 알림, 짧은 작업 대기열)
    - 카프카 선택: 데이터 양이 엄청나게 많고, 절대 누락되면 안 되며 나중에 다시 조회(Replay)할 필요가 있는 경우. (ex: 대규모 이벤트 로그, 결제 데이터 흐름)
    - SQS 선택: 관리 부담을 최소화하고 싶고, 서비스 간의 느슨한 결합을 통해 비동기로 안전하게 메시지를 전달하고 싶을 때. (ex: 회원가입 후 이메일 발송 등)
