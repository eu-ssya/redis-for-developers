# 6장. 레디스를 메시지 브로커로 사용하기
## 메시징 큐와 이벤트 스트림

최근 서비스 아키텍처는 하나의 거대한 애플리케이션이 모든 책임을 가지는 구조보다 여러 모듈이나 서비스가 느슨하게 연결되는 구조를 많이 사용한다.

예를 들면 아래와 같은 상황이다.

```
주문 서비스
↓
결제 서비스
↓
알림 서비스
↓
배송 서비스
```

이때 모든 서비스가 서로 직접 호출만 한다면 문제가 생길 수 있다.

```
주문 서비스 → 결제 서비스 직접 호출
주문 서비스 → 알림 서비스 직접 호출
주문 서비스 → 배송 서비스 직접 호출
```

이 구조에서는 한 서비스가 잠시 느려지거나 장애가 나면 호출하는 쪽 서비스까지 영향을 받을 수 있다.

예를 들어 주문은 정상적으로 생성됐는데 알림 서비스가 잠시 장애라면, 주문 자체가 실패해야 하는가?

대부분의 경우 그렇지 않다.

주문은 성공시키고, 알림은 나중에 다시 처리할 수 있으면 된다.

이처럼 서비스 간 통신에서 즉시 처리하지 않아도 되는 작업은 비동기적으로 분리하는 것이 좋다.

```
주문 생성
↓
메시지 브로커에 이벤트 저장
↓
알림 서비스가 나중에 읽어서 처리
```

메시지 브로커는 서비스 사이에서 메시지를 전달하거나 저장하는 중간 매개체 역할을 한다.

즉 메시지를 보내는 쪽과 받는 쪽이 서로 직접 강하게 연결되지 않도록 도와준다.

메시지 브로커는 크게 두 가지 형태로 생각할 수 있다.

- 메시징 큐
- 이벤트 스트림

메시징 큐는 보통 하나의 메시지를 하나의 소비자가 처리하는 구조에 가깝다.

```
Producer
↓
Queue
↓
Consumer
```

작업을 큐에 넣어두고, 소비자가 하나씩 꺼내 처리한다.

- 이메일 발송 작업
- 푸시 알림 작업
- 이미지 리사이징 작업
- 결제 후 정산 작업
- 로그 후처리 작업

이벤트 스트림은 메시지를 단순히 꺼내서 제거하는 구조보다, 이벤트 로그를 일정 기간 저장하고 여러 소비자가 각자 필요한 위치부터 읽는 구조에 가깝다.

```
Producer
↓
Event Stream
↓        ↓        ↓
Consumer A Consumer B Consumer C
```

- 주문 생성 이벤트
- 결제 완료 이벤트
- 사용자 행동 로그
- 실시간 분석 데이터
- IoT 센서 데이터

메시징 큐와 이벤트 스트림은 비슷해 보이지만 관점이 다르다.

| 구분 | 메시징 큐 | 이벤트 스트림 |
| --- | --- | --- |
| 목적 | 작업을 전달하고 처리 | 이벤트 흐름을 저장하고 소비 |
| 메시지 소비 | 보통 한 소비자가 처리 | 여러 소비자가 독립적으로 읽을 수 있음 |
| 데이터 저장 | 소비 후 제거되는 경우가 많음 | 일정 기간 저장 가능 |
| 재처리 | 구조에 따라 어려울 수 있음 | 특정 ID 또는 시간 범위부터 다시 읽기 쉬움 |
| 대표 사용 | 작업 큐, 비동기 처리 | 이벤트 로그, 실시간 분석, 팬아웃 |

메시징 큐는 “해야 할 일을 쌓아두는 곳”에 가깝고, 이벤트 스트림은 “발생한 일을 기록해두고 여러 곳에서 읽는 곳”에 가깝다.

### 🤔 **메시지와 이벤트는 뭐가 다를까?**

메시지는 넓은 개념이다.

서비스 간에 전달되는 데이터라면 메시지라고 부를 수 있다.

이벤트는 그중에서도 “이미 발생한 사실”을 표현하는 메시지에 가깝다.

```
명령형 메시지
- 이메일을 보내라
- 결제를 취소하라
- 재고를 차감하라

이벤트
- 주문이 생성됐다
- 결제가 완료됐다
- 회원이 탈퇴했다
```

이벤트는 보통 과거형으로 이름 짓는 것이 자연스럽다.

```
OrderCreated
PaymentCompleted
UserSignedUp
```

이벤트 기반 구조에서는 한 서비스가 어떤 일이 발생했다는 사실만 발행하고, 그 이후 어떤 서비스가 어떤 후속 작업을 할지는 느슨하게 분리할 수 있다.

### 🤔 **동기 통신을 아예 쓰면 안 될까?**

그건 아니다.

즉시 응답이 필요한 작업은 동기 통신이 더 적합하다.

예를 들어 사용자가 상품 상세 페이지를 보고 있는데 상품 정보를 즉시 보여줘야 한다면 동기 조회가 자연스럽다.

반면 아래 작업은 비동기 처리가 더 적합하다.

- 주문 완료 후 이메일 발송
- 결제 완료 후 포인트 적립
- 리뷰 작성 후 알림 발송
- 로그 적재
- 통계 집계

핵심은 “사용자 응답을 막아야 하는 작업인가?”를 기준으로 판단하는 것이다.

사용자 응답에 반드시 필요한 작업이면 동기 처리하고, 나중에 처리해도 되는 부가 작업이면 메시지 브로커로 분리하는 것이 좋다.

### 레디스를 메시지 브로커로 사용하기

레디스는 단순한 캐시로만 사용하는 것이 아니라 메시지 브로커로도 사용할 수 있다.

레디스에서 메시지 브로커를 구현하는 방법은 크게 세 가지로 볼 수 있다.

```
1. Pub/Sub
2. List
3. Stream
```

각 방식은 성격이 다르다.

| 방식 | 특징 | 적합한 상황 |
| --- | --- | --- |
| Pub/Sub | 메시지를 즉시 전달하고 저장하지 않음 | 실시간 알림, 채팅, 단순 이벤트 전파 |
| List | push/pop 기반 큐 구현 가능 | 간단한 작업 큐 |
| Stream | 메시지를 로그처럼 저장하고 ID 기준으로 읽을 수 있음 | 이벤트 스트림, 로그성 데이터 처리 |

Pub/Sub은 가장 가볍다.

발행자가 채널에 메시지를 보내면 구독 중인 클라이언트가 즉시 메시지를 받는다.

하지만 메시지가 저장되지 않기 때문에, 구독자가 잠시 끊겨 있으면 그 사이 발행된 메시지는 받을 수 없다.

List는 큐처럼 사용할 수 있다.

생산자가 리스트에 작업을 넣고, 소비자가 리스트에서 작업을 꺼내 처리한다.

블로킹 명령어를 사용하면 소비자가 계속 폴링하지 않고 데이터가 들어올 때까지 대기할 수 있다.

Stream은 카프카와 비슷한 개념으로 설계된 레디스 자료구조이다.

데이터를 append-only 방식으로 저장하고, ID를 기준으로 필요한 범위의 메시지를 읽을 수 있다.

또한 메시지마다 ID가 붙기 때문에 단순 Pub/Sub처럼 지나가면 사라지는 구조가 아니라, 저장된 데이터를 기준으로 다시 조회할 수 있다.

### 🤔 **세 방식 중 뭘 써야 할까?**

가볍게 생각하면 아래처럼 고를 수 있다.

```
메시지를 놓쳐도 괜찮고, 지금 연결된 구독자에게만 보내면 된다
→ Pub/Sub

간단한 작업 큐가 필요하고, 한 작업을 한 소비자가 처리하면 된다
→ List

메시지를 저장해두고 ID 기준으로 다시 읽어야 한다
→ Stream
```

예를 들어 실시간 접속자에게 “새 공지 알림”을 뿌리는 정도라면 Pub/Sub으로 충분할 수 있다.

하지만 결제 완료 후 정산 처리처럼 반드시 처리되어야 하는 작업이라면 Pub/Sub은 위험하다.

이런 경우에는 Stream처럼 메시지 저장과 처리 확인이 가능한 구조를 고려해야 한다.

## 레디스의 pub/sub

Pub/Sub은 Publish/Subscribe의 줄임말이다.

- Publish
    - 메시지를 발행하는 것
- Subscribe
    - 특정 채널을 구독하는 것

레디스의 Pub/Sub 구조는 매우 단순하다.

```
Publisher
↓
Channel
↓       ↓       ↓
Subscriber Subscriber Subscriber
```

발행자는 특정 채널에 메시지를 보낸다.

해당 채널을 구독 중인 구독자들은 메시지를 즉시 받는다.

레디스 서버는 메시지를 저장하지 않고 단순히 통로 역할만 한다.

```
메시지 발행
↓
현재 구독 중인 클라이언트에게 전달
↓
저장하지 않고 사라짐
```

이런 구조는 빠르고 단순하지만, 메시지 유실 가능성을 전제로 한다.

Pub/Sub의 특징은 아래와 같다.

- 메시지를 저장하지 않는다.
- 메시지 처리 완료 여부를 별도로 추적하지 않는다.
- 구독자가 연결되어 있지 않으면 메시지를 받을 수 없다.
- 같은 채널을 구독 중인 모든 구독자에게 메시지가 전달된다.
- fire-and-forget 패턴에 적합하다.

### 🤔 **fire-and-forget이란?**

fire-and-forget은 메시지를 보내고 결과를 기다리지 않는 방식이다.

```
작업 요청 발행
↓
응답 기다리지 않음
↓
다음 로직 진행
```

예를 들어 아래와 같은 작업에 적합하다.

- 간단한 알림 전파
- 채팅 메시지 브로드캐스트
- 실시간 상태 변경 알림
- 내부 이벤트 트리거
- 모니터링 이벤트 전파

반대로 아래처럼 반드시 처리되어야 하는 작업에는 적합하지 않다.

- 결제 승인
- 재고 차감
- 주문 확정
- 정산 처리
- 쿠폰 발급

이런 작업은 메시지 유실이 치명적일 수 있으므로 Stream, Kafka, RabbitMQ 같은 더 안정적인 메시징 구조를 고려하는 것이 좋다.

### 메시지 publish하기

메시지를 발행할 때는 `PUBLISH` 명령어를 사용한다.

```bash
> PUBLISH channel message
```

예를 들어 `notice` 채널에 메시지를 발행한다.

```bash
> PUBLISH notice "hello redis"
(integer) 1
```

반환값은 메시지를 받은 구독자 수이다.

```
(integer) 1
```

위 결과는 현재 `notice` 채널을 구독 중인 클라이언트가 1명이라는 뜻이다.

구독자가 없다면 메시지는 저장되지 않고 사라진다.

```bash
> PUBLISH notice "hello redis"
(integer) 0
```

이 경우 메시지를 받은 구독자가 없다는 뜻이다.

하지만 메시지는 나중을 위해 보관되지 않는다.

여러 채널을 구분해서 사용할 수도 있다.

```bash
> PUBLISH order.created "orderId=1"
> PUBLISH order.paid "orderId=1"
> PUBLISH user.signup "userId=10"
```

채널 이름은 애플리케이션에서 이벤트 타입처럼 사용할 수 있다.

```
order.created
order.paid
order.cancelled
user.signup
user.deleted
```

### 🤔 **채널 이름은 어떻게 짓는 게 좋을까?**

보통 도메인과 이벤트를 조합해서 짓는 것이 좋다.

```
도메인.이벤트
```

```
order.created
order.completed
payment.failed
notification.requested
chat.room.1
```

일관된 네이밍을 사용하면 구독자가 어떤 이벤트를 받는지 파악하기 쉽다.

단, Pub/Sub 채널은 Redis Key와 같은 데이터 저장 키가 아니다.

채널은 메시지가 지나가는 통로에 가깝다.

### 메시지 구독하기

메시지를 구독할 때는 `SUBSCRIBE` 명령어를 사용한다.

```bash
> SUBSCRIBE notice
```

구독을 시작하면 해당 클라이언트는 Pub/Sub 모드로 들어간다.

이 상태에서는 일반적인 Redis 명령어를 자유롭게 실행하기보다 구독 관련 명령을 중심으로 동작한다.

구독 중인 상태에서 다른 클라이언트가 메시지를 발행하면 아래와 같이 메시지를 받는다.

```bash
1) "message"
2) "notice"
3) "hello redis"
```

각 값의 의미는 아래와 같다.

```
1) 메시지 타입
2) 채널 이름
3) 실제 메시지
```

여러 채널을 동시에 구독할 수도 있다.

```bash
> SUBSCRIBE notice event chat
```

패턴 기반 구독도 가능하다.

```bash
> PSUBSCRIBE order.*
```

이 경우 아래 채널의 메시지를 받을 수 있다.

```
order.created
order.paid
order.cancelled
```

### 🤔 **SUBSCRIBE와 PSUBSCRIBE는 뭐가 다를까?**

`SUBSCRIBE`는 정확한 채널 이름을 구독한다.

```bash
> SUBSCRIBE order.created
```

`PSUBSCRIBE`는 패턴에 맞는 채널을 구독한다.

```bash
> PSUBSCRIBE order.*
```

패턴 구독은 편리하지만 너무 넓은 범위를 구독하면 원하지 않는 메시지까지 받을 수 있다.

따라서 운영 환경에서는 채널 네이밍 규칙을 명확하게 정하는 것이 중요하다.

### 🤔 **Pub/Sub은 메시지를 다시 읽을 수 있을까?**

기본 Pub/Sub은 다시 읽을 수 없다.

메시지는 구독 중인 클라이언트에게 전달된 뒤 저장되지 않는다.

```
구독자 연결 중
→ 메시지 수신 가능

구독자 연결 끊김
→ 그 사이 메시지 유실
```

따라서 “나중에 다시 읽기”가 필요하면 Pub/Sub이 아니라 Stream을 사용해야 한다.

### 클러스터 구조에서의 pub/sub

Redis Cluster에서는 데이터를 여러 노드에 나누어 저장한다.

```
Node A → hash slot 일부 담당
Node B → hash slot 일부 담당
Node C → hash slot 일부 담당
```

일반적인 key-value 명령어는 키의 해시 슬롯에 따라 특정 노드에서 처리된다.

하지만 기존 Pub/Sub은 키 기반 데이터가 아니기 때문에 채널 메시지를 특정 해시 슬롯에만 묶어 처리하지 않는다.

기본 Pub/Sub에서는 클러스터의 어느 노드에 발행하더라도, 다른 노드에 연결된 구독자도 메시지를 받을 수 있어야 한다.

이를 위해 클러스터 내부에서는 메시지가 전체 노드로 전파된다.

```
Client A
↓ PUBLISH channel
Node 1
↓
클러스터 버스 통해 전체 노드로 전파
↓        ↓        ↓
Node 1   Node 2   Node 3
↓        ↓        ↓
Subscriber Subscriber Subscriber
```

이 방식은 사용하기는 편하지만, 클러스터 규모가 커질수록 비효율이 생길 수 있다.

모든 Pub/Sub 메시지가 전체 노드로 퍼지면 내부 네트워크 트래픽이 증가한다.

### 🤔 **클러스터 Pub/Sub은 왜 부담이 될까?**

일반 key 명령어는 특정 슬롯을 담당하는 노드로만 가면 된다.

하지만 일반 Pub/Sub 메시지는 클러스터 전체에 전파될 수 있다.

```
일반 key 명령
→ 관련 슬롯의 노드로 이동

일반 Pub/Sub
→ 모든 노드에 전파
```

채널 수나 메시지 수가 적을 때는 큰 문제가 아닐 수 있다.

하지만 실시간 채팅, 알림, 상태 이벤트처럼 메시지가 매우 많아지면 클러스터 내부 버스 트래픽이 부담이 될 수 있다.

그래서 Redis 7부터는 클러스터 환경에서 Pub/Sub을 더 효율적으로 사용하기 위한 sharded Pub/Sub이 제공된다.

### sharded pub/sub

Sharded Pub/Sub은 Redis 7에서 도입된 기능이다.

기존 Pub/Sub은 클러스터 전체로 메시지를 전파하는 구조라면, Sharded Pub/Sub은 메시지를 특정 shard 범위 안에서만 전파한다.

```
일반 Pub/Sub
→ 클러스터 전체 노드로 전파

Sharded Pub/Sub
→ 해당 shard 내부로만 전파
```

Sharded Pub/Sub에서 사용하는 명령어는 일반 Pub/Sub과 다르다.

```bash
> SPUBLISH channel message
> SSUBSCRIBE channel
> SUNSUBSCRIBE channel
```

```bash
> SSUBSCRIBE order.events
```

다른 클라이언트에서 메시지를 발행한다.

```bash
> SPUBLISH order.events "orderId=1"
```

구독자는 메시지를 받는다.

```bash
1) "smessage"
2) "order.events"
3) "orderId=1"
```

Sharded Pub/Sub에서는 채널도 해시 슬롯 알고리즘에 따라 특정 슬롯에 매핑된다.

따라서 해당 shard를 담당하는 노드로 메시지 전파 범위가 제한된다.

Sharded Pub/Sub의 장점은 클러스터 내부 전파 비용을 줄일 수 있다는 점이다.

```
전체 노드 broadcast 감소
↓
클러스터 버스 트래픽 감소
↓
Pub/Sub 확장성 향상
```

### 🤔 **일반 Pub/Sub과 Sharded Pub/Sub 차이**

| 구분 | 일반 Pub/Sub | Sharded Pub/Sub |
| --- | --- | --- |
| 발행 명령어 | PUBLISH | SPUBLISH |
| 구독 명령어 | SUBSCRIBE | SSUBSCRIBE |
| 전파 범위 | 클러스터 전체 | 해당 shard 중심 |
| 장점 | 사용이 단순함 | 클러스터 확장성에 유리 |
| 적합한 상황 | 소규모 Pub/Sub | 클러스터에서 Pub/Sub 트래픽이 많은 경우 |

### 🤔 **Sharded Pub/Sub을 쓰면 메시지 저장이 될까?**

아니다.

Sharded Pub/Sub도 Pub/Sub이다.

즉 메시지를 저장하거나 처리 완료 상태를 관리하는 기능이 생기는 것은 아니다.

달라지는 핵심은 “클러스터 내부 전파 범위”이다.

```
저장 여부
일반 Pub/Sub: 저장 안 함
Sharded Pub/Sub: 저장 안 함

차이
일반 Pub/Sub: 전체 전파
Sharded Pub/Sub: shard 단위 전파
```

따라서 메시지 유실이 문제라면 Sharded Pub/Sub이 아니라 Stream을 고려해야 한다.

## 레디스의 list를 메시징 큐로 사용하기

Redis List는 양쪽 끝에서 데이터를 넣고 뺄 수 있는 자료구조이다.

```
LEFT  [ item3, item2, item1 ]  RIGHT
```

대표적인 명령어는 아래와 같다.

```bash
LPUSH
RPUSH
LPOP
RPOP
```

List를 사용하면 간단한 큐를 만들 수 있다.

예를 들어 오른쪽에 넣고 왼쪽에서 꺼내면 FIFO 큐처럼 동작한다.

```bash
> RPUSH queue job1
> RPUSH queue job2
> RPUSH queue job3

> LPOP queue
"job1"

> LPOP queue
"job2"

> LPOP queue
"job3"
```

반대로 왼쪽에 넣고 오른쪽에서 꺼내도 FIFO처럼 사용할 수 있다.

```bash
> LPUSH queue job1
> LPUSH queue job2
> LPUSH queue job3

> RPOP queue
"job1"

> RPOP queue
"job2"

> RPOP queue
"job3"
```

중요한 점은 생산자와 소비자가 일관된 방향을 사용해야 한다는 것이다.

```
RPUSH + LPOP
또는
LPUSH + RPOP
```

List 기반 큐 구조는 단순하다.

```
Producer
↓ RPUSH
Redis List
↓ LPOP
Consumer
```

- 이메일 발송 작업 큐
- 썸네일 생성 작업 큐
- 로그 후처리 작업 큐
- 배치성 비동기 작업

### 🤔 **List 큐는 Pub/Sub보다 뭐가 좋을까?**

Pub/Sub은 구독자가 없으면 메시지가 사라진다.

List는 데이터가 리스트에 저장된다.

```
Pub/Sub
→ 현재 구독자가 없으면 메시지 유실

List
→ 소비자가 나중에 와도 리스트에 남아 있으면 처리 가능
```

그래서 간단한 작업 큐라면 Pub/Sub보다 List가 더 안전하다.

하지만 List도 완전한 메시지 브로커는 아니다.

소비자가 메시지를 꺼낸 직후 장애가 나면 그 메시지는 이미 리스트에서 제거된 상태일 수 있다.

```
Consumer가 RPOP
↓
메시지 리스트에서 제거됨
↓
처리 도중 Consumer 장애
↓
메시지 유실 가능
```

이 문제를 줄이려면 처리 중 리스트를 따로 두거나, 처리 상태를 별도로 관리하는 구조가 필요하다.

### list의 EX 기능

Redis 6.2부터 `LPUSHX`, `RPUSHX`처럼 기존에 있던 X 계열 명령 외에도 `LPOP`, `RPOP`에서 count 옵션을 사용할 수 있게 되었다.

하지만 책 목차에서 말하는 `list의 EX 기능`은 보통 “존재할 때만 push하는 X 계열 명령어”를 의미하는 맥락으로 이해하면 된다.

대표적으로 아래 명령어가 있다.

```bash
LPUSHX key element [element ...]
RPUSHX key element [element ...]
```

`LPUSHX`는 리스트가 이미 존재할 때만 왼쪽에 데이터를 추가한다.

```bash
> LPUSHX queue job1
(integer) 0
```

`queue`라는 리스트가 존재하지 않으면 아무 일도 하지 않는다.

먼저 리스트를 만든 뒤 사용하면 값이 추가된다.

```bash
> RPUSH queue job0
(integer) 1

> LPUSHX queue job1
(integer) 2
```

`RPUSHX`도 마찬가지로 리스트가 존재할 때만 오른쪽에 값을 추가한다.

```bash
> RPUSHX queue job2
(integer) 3
```

X 계열 명령어는 “없으면 만들지 않는다”는 점이 특징이다.

일반 `LPUSH`, `RPUSH`는 키가 없으면 새 리스트를 만든다.

```bash
> RPUSH queue job1
(integer) 1
```

하지만 `LPUSHX`, `RPUSHX`는 키가 없으면 생성하지 않는다.

```bash
> RPUSHX not-exists-queue job1
(integer) 0
```

### 🤔 **왜 굳이 존재할 때만 넣는 기능이 필요할까?**

큐가 특정 상태에서만 활성화되어야 할 때 유용하다.

예를 들어 어떤 작업 큐를 운영자가 명시적으로 열어둔 경우에만 작업을 넣고 싶을 수 있다.

```
큐가 존재한다
→ 작업 추가 가능

큐가 없다
→ 작업 추가하지 않음
```

또는 잘못된 큐 이름으로 인해 의도치 않게 새로운 큐가 생성되는 것을 막고 싶을 때도 사용할 수 있다.

```
오타 난 큐 이름
→ 일반 RPUSH면 새 큐 생성
→ RPUSHX면 아무 것도 안 함
```

실무에서는 큐 이름이 동적으로 만들어질 때 이런 실수를 막는 데 도움이 된다.

### 🤔 **EX가 만료 시간 EX 옵션이랑 같은 뜻일까?**

아니다.

Redis에서 `SET key value EX seconds`의 `EX`는 expire seconds를 의미한다.

```bash
> SET token abc EX 60
```

하지만 `LPUSHX`, `RPUSHX`의 `X`는 key가 존재할 때만 실행한다는 의미로 이해하면 된다.

즉 TTL의 EX와 List X 계열 명령어는 다른 개념이다.

### list의 블로킹 기능

List를 큐로 사용할 때 가장 불편한 점은 소비자가 계속 데이터를 확인해야 한다는 것이다.

```
while true:
    data = LPOP queue
    if data is not null:
        process(data)
    sleep(1)
```

이런 방식은 폴링이다.

데이터가 없는데도 계속 Redis에 요청을 보내야 한다.

```
큐 비어 있음
↓
조회
↓
없음
↓
잠시 대기
↓
다시 조회
```

트래픽도 낭비되고, sleep 간격 때문에 처리 지연도 생긴다.

이를 해결하기 위해 Redis는 블로킹 pop 명령어를 제공한다.

대표적인 명령어는 아래와 같다.

```bash
BLPOP
BRPOP
BLMOVE
```

`BLPOP`은 리스트가 비어 있으면 데이터가 들어올 때까지 대기한다.

```bash
> BLPOP queue 0
```

마지막 인자는 timeout이다.

```
0
→ 무한 대기

10
→ 최대 10초 대기
```

다른 클라이언트가 데이터를 넣으면 대기 중인 클라이언트가 즉시 값을 가져간다.

```bash
> RPUSH queue job1
```

대기 중인 클라이언트는 아래와 같은 응답을 받는다.

```bash
1) "queue"
2) "job1"
```

`BRPOP`도 비슷하지만 오른쪽에서 값을 꺼낸다.

```bash
> BRPOP queue 0
```

### 🤔 **BLPOP과 BRPOP 차이**

| 명령어 | 꺼내는 방향 | 일반 명령어 기준 |
| --- | --- | --- |
| BLPOP | 왼쪽 | LPOP의 블로킹 버전 |
| BRPOP | 오른쪽 | RPOP의 블로킹 버전 |

큐를 FIFO로 사용하려면 push 방향과 pop 방향을 맞춰야 한다.

```bash
RPUSH + BLPOP
```

또는

```bash
LPUSH + BRPOP
```

블로킹 기능을 사용하면 소비자는 데이터가 들어올 때까지 기다릴 수 있다.

```
Consumer
↓
BLPOP queue 0
↓
큐가 비어 있으면 대기
↓
Producer가 데이터 추가
↓
즉시 Consumer가 가져감
```

이 방식은 폴링보다 효율적이다.

```
폴링
→ 계속 Redis에 물어봄

블로킹
→ Redis가 깨워줄 때까지 기다림
```

### 🤔 **블로킹 큐도 메시지 유실이 없을까?**

그렇지는 않다.

`BLPOP`, `BRPOP`은 데이터를 꺼내는 순간 리스트에서 제거한다.

소비자가 데이터를 받은 뒤 처리 중에 죽으면 메시지가 유실될 수 있다.

```
BRPOP으로 job1 꺼냄
↓
queue에서 job1 제거됨
↓
처리 중 서버 장애
↓
job1 재처리 불가
```

이 문제를 줄이기 위해 과거에는 `RPOPLPUSH` 또는 `BRPOPLPUSH`를 사용해 작업 큐와 처리 중 큐를 분리하기도 했다.

최근에는 `LMOVE`, `BLMOVE`를 사용할 수 있다.

```bash
> BLMOVE ready processing RIGHT LEFT 0
```

의미는 아래와 같다.

```
ready 리스트의 오른쪽에서 꺼냄
↓
processing 리스트의 왼쪽으로 이동
↓
이동한 값을 반환
```

이렇게 하면 작업을 꺼내는 동시에 처리 중 리스트로 옮길 수 있다.

```
ready queue
↓
processing queue
↓
처리 성공 후 LREM으로 제거
```

하지만 이런 구조를 직접 만들면 재시도, 타임아웃, 중복 처리, 모니터링을 애플리케이션에서 직접 관리해야 한다.

그래서 처리 보장과 재처리가 중요하면 단순 List 큐만으로는 부족할 수 있다.

### list를 이용한 원형 큐

원형 큐는 데이터를 꺼낸 뒤 다시 큐에 넣어 반복적으로 순회하는 구조이다.

```
A → B → C
↑       ↓
← ← ← ←
```

Redis List에서는 `RPOPLPUSH` 또는 `LMOVE`를 사용해 비슷한 구조를 만들 수 있다.

예를 들어 리스트에 작업 대상이 들어 있다.

```bash
> RPUSH workers worker1 worker2 worker3
```

`RPOPLPUSH`를 사용하면 오른쪽에서 값을 꺼내 왼쪽으로 다시 넣을 수 있다.

```bash
> RPOPLPUSH workers workers
"worker3"
```

리스트는 아래처럼 회전한다.

```
초기
[worker1, worker2, worker3]

RPOPLPUSH 후
[worker3, worker1, worker2]
```

다시 실행한다.

```bash
> RPOPLPUSH workers workers
"worker2"
```

결과

```
[worker2, worker3, worker1]
```

즉 같은 리스트에서 pop과 push를 동시에 수행하면 리스트를 회전시키는 효과가 있다.

Redis 6.2 이후에는 `LMOVE`를 사용할 수 있다.

```bash
> LMOVE workers workers RIGHT LEFT
```

의미

```
workers 오른쪽에서 꺼냄
↓
workers 왼쪽에 넣음
```

원형 큐는 특정 아이템들을 반복적으로 접근해야 할 때 사용할 수 있다.

- 여러 API 키를 번갈아 사용
- 여러 서버 또는 워커를 라운드로빈 방식으로 선택
- 반복 처리 대상 순회
- 일정한 후보군을 계속 순환 조회

### 🤔 **원형 큐는 일반 작업 큐와 뭐가 다를까?**

일반 작업 큐는 한 번 처리한 데이터를 큐에서 제거한다.

```
job1 처리
↓
큐에서 제거
```

원형 큐는 꺼낸 데이터를 다시 넣는다.

```
item1 꺼냄
↓
다시 큐에 넣음
↓
나중에 또 사용
```

따라서 원형 큐는 “소모”가 아니라 “순환”이 목적이다.

### 🤔 **원형 큐 사용 시 주의할 점**

원형 큐는 편리하지만 모든 상황에 적합하지 않다.

반복 순회 대상이 바뀌는 경우에는 동시성 문제를 조심해야 한다.

예를 들어 여러 소비자가 동시에 같은 리스트를 회전시키면 순서가 예상과 달라질 수 있다.

또한 특정 아이템이 실패했을 때 다시 뒤로 넣을지, 제외할지, 별도 실패 큐로 옮길지 정책을 정해야 한다.

```
성공
→ 다시 순환

일시 실패
→ 다시 순환

영구 실패
→ 제거 또는 실패 큐 이동
```

## Stream

Redis Stream은 Redis 5.0부터 제공되는 자료구조이다.

Stream은 로그처럼 데이터가 계속 뒤에 추가되는 append-only 구조이다.

```
stream
↓
entry1
entry2
entry3
entry4
```

기본적으로 한 번 추가된 메시지는 명시적으로 삭제하거나 trim하지 않는 한 stream에 남아 있을 수 있다.

Pub/Sub과 가장 큰 차이는 메시지를 저장한다는 점이다.

```
Pub/Sub
→ 메시지 전달 후 사라짐

Stream
→ 메시지를 stream에 저장
```

List와의 차이는 메시지마다 ID가 붙고, 저장된 이벤트를 범위 기준으로 조회할 수 있다는 점이다.

```
List
→ push/pop 기반 큐

Stream
→ 메시지 ID 기반 로그 + 범위 조회 + 저장된 이벤트 재조회
```

Stream은 아래 기능을 제공한다.

- 메시지 저장
- 시간 기반 ID
- 범위 조회
- 블로킹 읽기
- 메시지 ID
- 범위 조회
- 저장된 이벤트 재조회
- 메시지 재처리
- 스트림 길이 제한

### 레디스의 Stream과 아파치 카프카

Redis Stream은 Apache Kafka에서 영향을 받은 자료구조로 볼 수 있다.

둘 다 이벤트 데이터를 append-only 방식으로 저장하고, 소비자가 데이터를 읽는 구조를 제공한다.

```
Producer
↓
Stream / Topic
↓
Consumer
```

하지만 완전히 같은 시스템은 아니다.

Kafka는 대규모 분산 로그 플랫폼이고, Redis Stream은 Redis 내부 자료구조로 제공되는 스트림 기능이다.

공통점

- 생산자가 메시지를 추가한다.
- 소비자가 메시지를 읽는다.
- 메시지에 ID 또는 offset 개념이 있다.
- 저장된 메시지를 ID 기준으로 읽을 수 있다.
- 이벤트 로그처럼 사용할 수 있다.

차이점

| 구분 | Redis Stream | Apache Kafka |
| --- | --- | --- |
| 저장 단위 | Redis key 하나가 stream | topic / partition |
| 메시지 식별 | stream entry ID | partition 내 offset |
| 저장 위치 | Redis 메모리 중심 | 디스크 기반 로그 중심 |
| 확장 방식 | Redis Cluster 기반 | 브로커/파티션 기반 |
| 운영 복잡도 | 상대적으로 단순 | 상대적으로 복잡 |
| 적합한 규모 | Redis 기반의 중소규모 이벤트 처리 | 대규모 이벤트 스트리밍 플랫폼 |

Redis Stream은 이미 Redis를 사용하고 있고, 별도 Kafka 클러스터를 운영하기 부담스러운 상황에서 유용하다.

반면 매우 큰 규모의 이벤트 스트리밍, 장기 보관, 복잡한 파티셔닝, 대용량 데이터 처리 파이프라인이 필요하다면 Kafka가 더 적합할 수 있다.

### 🤔 **Redis Stream이 Kafka를 대체할 수 있을까?**

상황에 따라 일부 역할은 대체할 수 있다.

하지만 항상 대체 가능한 것은 아니다.

Redis Stream이 적합한 경우

- 이미 Redis를 운영하고 있다.
- 메시지 처리량이 Redis로 감당 가능한 수준이다.
- 이벤트를 너무 오래 보관하지 않는다.
- 운영 복잡도를 낮추고 싶다.
- 저장된 이벤트를 기준으로 다시 조회해야 한다.

Kafka가 더 적합한 경우

- 매우 큰 규모의 이벤트 로그를 장기간 보관해야 한다.
- 여러 시스템이 대규모로 이벤트를 구독한다.
- 파티션 기반 수평 확장이 중요하다.
- 데이터 파이프라인의 중심 플랫폼이 필요하다.
- 디스크 기반 내구성과 장기 보관이 중요하다.

즉 Redis Stream은 “Redis 안에서 제공되는 강력한 스트림 자료구조”이고, Kafka는 “전문적인 분산 이벤트 스트리밍 플랫폼”에 가깝다.

### 스트림이란?

스트림은 연속적인 데이터의 흐름을 의미한다.

```
event1 → event2 → event3 → event4
```

각 데이터는 시간 순서에 따라 계속 추가된다.

예를 들어 사용자의 행동 로그를 생각해볼 수 있다.

```
10:00 로그인
10:01 상품 조회
10:03 장바구니 추가
10:05 결제 시도
10:06 결제 완료
```

이런 이벤트들은 하나의 흐름을 이룬다.

Redis Stream에서는 이런 데이터를 entry 단위로 저장한다.

```
stream key
↓
entry id: 1700000000000-0
fields: userId=1, action=login

entry id: 1700000001000-0
fields: userId=1, action=view_product
```

각 entry는 field-value 쌍으로 구성된다.

Hash와 비슷하게 여러 필드와 값을 저장할 수 있다.

```bash
> XADD user-events * userId 1 action login
```

여기서 `user-events`는 stream key이다.

- 는 Redis가 자동으로 ID를 생성하라는 뜻이다.

```bash
"1700000000000-0"
```

Stream ID는 보통 아래 구조를 가진다.

```
milliseconds-sequence
```

```
1700000000000-0
1700000000000-1
1700000000001-0
```

앞부분은 밀리초 시간이고, 뒷부분은 같은 밀리초 안에서의 시퀀스 번호이다.

### 🤔 **Stream ID는 왜 필요할까?**

Stream은 메시지를 저장하기 때문에 각 메시지를 식별할 수 있어야 한다.

ID가 있으면 아래 작업이 가능하다.

- 특정 ID 이후의 메시지 읽기
- 특정 범위의 메시지 조회
- 마지막으로 읽은 위치 기억
- 재처리 대상 추적
- 마지막으로 읽은 위치 관리

Pub/Sub에는 이런 ID가 없다.

그래서 Pub/Sub은 지나간 메시지를 다시 읽을 수 없다.

Stream은 ID를 기준으로 메시지를 다시 조회할 수 있다.

### 데이터의 저장

Redis Stream에 데이터를 저장할 때는 `XADD` 명령어를 사용한다.

```bash
> XADD stream-key ID field value [field value ...]
```

보통 ID 자리에는 `*`를 사용한다.

```bash
> XADD Email * subject "hello" body "first email"
"1700000000000-0"
```

- 를 사용하면 Redis가 자동으로 ID를 생성한다.

하나의 entry는 여러 field-value 쌍을 가질 수 있다.

```bash
> XADD order-events * orderId 1 event CREATED amount 30000
"1700000001000-0"
```

저장된 데이터는 대략 아래와 같은 형태로 볼 수 있다.

```
order-events
└─ 1700000001000-0
   ├─ orderId: 1
   ├─ event: CREATED
   └─ amount: 30000
```

Stream은 key 하나가 하나의 stream이다.

```
Redis Key: order-events
Data Type: Stream
```

Kafka에서 topic을 만드는 것과 달리 Redis Stream은 별도의 생성 명령 없이 `XADD`를 실행하면 stream이 생성된다.

```bash
> XADD Email * message "I am first email"
```

이 명령을 실행하면 `Email`이라는 stream key가 없더라도 자동으로 생성된다.

저장된 stream 데이터는 `XRANGE`로 범위 조회할 수 있다.

```bash
> XRANGE Email - +
```

- 는 가장 작은 ID를 의미하고, `+`는 가장 큰 ID를 의미한다.

```bash
> XRANGE Email - + COUNT 2
```

특정 범위만 조회할 수도 있다.

```bash
> XRANGE Email 1700000000000-0 1700000009999-0
```

역순 조회는 `XREVRANGE`를 사용한다.

```bash
> XREVRANGE Email + - COUNT 10
```

### 🤔 **Stream 데이터는 계속 쌓이면 어떻게 될까?**

Stream은 append-only 구조이기 때문에 계속 쌓일 수 있다.

그대로 두면 메모리를 계속 사용한다.

따라서 운영 환경에서는 길이 제한이나 삭제 전략이 필요하다.

대표적으로 `XADD`에서 `MAXLEN` 옵션을 사용할 수 있다.

```bash
> XADD order-events MAXLEN 1000 * orderId 1 event CREATED
```

대략적인 trimming을 사용하려면 `~`를 붙일 수 있다.

```bash
> XADD order-events MAXLEN ~ 1000 * orderId 1 event CREATED
```

정확히 1000개로 맞추는 것보다 근사치로 자르는 방식이 성능에 유리할 수 있다.

또는 `XTRIM` 명령어로 별도 정리할 수 있다.

```bash
> XTRIM order-events MAXLEN 1000
```

Stream은 단순 저장뿐 아니라 읽기 방식도 중요하다.

기본 읽기는 `XREAD`를 사용한다.

```bash
> XREAD STREAMS Email 0
```

의미는 `Email` stream에서 ID 0 이후의 메시지를 읽겠다는 뜻이다.

새로운 메시지를 기다리며 읽고 싶다면 `BLOCK` 옵션을 사용할 수 있다.

```bash
> XREAD BLOCK 0 STREAMS Email $
```

`$`는 현재 시점 이후에 들어오는 새 메시지부터 읽겠다는 뜻이다.

```
0
→ 처음부터 읽기

$
→ 지금 이후 새 메시지부터 읽기
```

### 🤔 **XREAD에서 0과 $는 뭐가 다를까?**

`0`은 처음부터 읽겠다는 의미에 가깝다.

```bash
> XREAD STREAMS Email 0
```

이미 저장된 과거 메시지도 읽을 수 있다.

`$`는 현재 마지막 ID를 기준으로 이후에 들어오는 새 메시지를 기다린다.

```bash
> XREAD BLOCK 0 STREAMS Email $
```

따라서 실시간 구독처럼 사용하고 싶다면 `$`를 사용할 수 있다.

하지만 애플리케이션이 재시작됐을 때 놓친 메시지까지 읽으려면 마지막으로 처리한 ID를 저장해두고 그 ID 이후부터 읽어야 한다.

Stream 데이터 저장 구조를 정리하면 아래와 같다.

```
Redis Stream Key
├─ Entry ID 1
│  ├─ field1: value1
│  └─ field2: value2
├─ Entry ID 2
│  ├─ field1: value1
│  └─ field2: value2
└─ Entry ID 3
   ├─ field1: value1
   └─ field2: value2
```
