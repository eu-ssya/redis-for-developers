# 7장. 레디스 데이터 백업 방법

## 레디스에서 데이터를 영구 저장하기

Redis의 모든 데이터는 메모리에서 관리된다.

따라서 서버 장애나 Redis 인스턴스 재시작 시 메모리에 있던 데이터가 모두 사라질 수 있다.

```text
Redis 실행
↓
메모리에 데이터 저장
↓
프로세스 종료
↓
데이터 유실 가능
```

복제(replication)를 구성하면 데이터가 실시간으로 복제본에 전달되지만, 이것은 가용성을 위한 기능이다.

```text
복제
→ 장애 대응 / 가용성

백업
→ 데이터 복구
```

예를 들어 운영자의 실수로 데이터를 삭제하면 해당 작업도 복제본에 그대로 반영된다.

```text
FLUSHALL 실행
↓
Replica에도 동일 반영
```

그래서 Redis를 영구 저장소 용도로 사용한다면 디스크 백업이 필요하다.

Redis에서 제공하는 대표적인 백업 방식은 아래 두 가지이다.

```text
1. RDB
2. AOF
```

| 구분       | RDB            | AOF                  |
| -------- | -------------- | -------------------- |
| 저장 방식    | snapshot 저장    | write command append |
| 저장 형태    | binary         | RESP protocol        |
| 복구 속도    | 빠름             | 상대적으로 느림             |
| 데이터 안정성  | snapshot 시점 기준 | append 정책 기준         |
| 특정 시점 복구 | 어려움            | 가능                   |
| 파일 크기    | 상대적으로 작음       | 커질 수 있음              |

AOF와 RDB를 동시에 사용할 수도 있다.

Redis는 복원 시 AOF 파일이 존재하면 AOF를 우선 사용한다.

```text
RDB + AOF 존재
↓
AOF 우선 로드
```

Redis가 AOF를 더 내구성이 높은 방식이라고 판단하기 때문이다.

#### 🤔 Redis는 언제 데이터를 복원할 수 있을까?

Redis는 서버가 재시작될 때만 데이터를 복원할 수 있다.

```text
Redis 시작
↓
RDB / AOF 파일 확인
↓
메모리 복원
```

실행 중인 인스턴스가 현재 메모리를 날리고 파일을 다시 읽어오는 기능은 지원하지 않는다.


# RDB 방식의 데이터 백업

RDB는 특정 시점의 메모리 상태를 snapshot 형태로 저장한다.

```text
현재 메모리 상태
↓
dump.rdb 생성
```

즉 “메모리를 사진 찍듯 저장하는 방식”이다.

RDB 특징

* snapshot 기반 저장
* binary 형태 저장
* 빠른 복구 가능
* 시점 단위 백업 가능
* 마지막 snapshot 이후 데이터 유실 가능

예를 들어 아래 작업이 있었다고 가정한다.

```bash
SET key1 a
SET key1 apple
SET key2 b
DEL key2
```

RDB에는 최종 메모리 상태만 저장된다.

```text
key1 = apple
```

#### 🤔 RDB는 왜 복구 속도가 빠를까?

이미 snapshot 형태로 저장된 데이터를 그대로 메모리에 적재하기 때문이다.

```text
RDB
→ snapshot 로드

AOF
→ 명령 replay
```

#### 🤔 RDB 단점은?

snapshot 이후 데이터는 유실될 수 있다.

```text
12:00 snapshot 생성
↓
12:05 데이터 변경
↓
12:06 장애 발생
```

이 경우 12:00 이후 데이터는 복구할 수 없다.


## 특정 조건에 자동으로 RDB 파일 생성

Redis는 save 옵션으로 자동 snapshot 조건을 설정할 수 있다.

```bash
save <기간(초)> <변경된 키 개수>
```

예시

```bash
save 900 1
save 300 10
save 60 10000
```

의미

```text
900초 동안 1개 이상 변경
→ snapshot 생성

300초 동안 10개 이상 변경
→ snapshot 생성

60초 동안 10000개 이상 변경
→ snapshot 생성
```

RDB 파일 이름과 저장 경로도 지정 가능하다.

```bash
dbfilename dump.rdb
dir /data/redis
```

#### 🤔 save 조건은 AND일까 OR일까?

OR 조건이다.

```text
조건 중 하나라도 만족
↓
RDB 생성
```


## RDB 저장 비활성화

RDB 저장을 사용하지 않으려면 아래처럼 설정한다.

```bash
save ""
```

실행 중인 인스턴스에서는 아래처럼 변경 가능하다.

```bash
CONFIG SET save ""
```

현재 설정 확인

```bash
CONFIG GET save
```

설정 파일까지 반영하려면 아래 명령 사용

```bash
CONFIG REWRITE
```

#### 🤔 CONFIG SET만 하면 redis.conf도 수정될까?

아니다.

```text
CONFIG SET
→ 실행 중 인스턴스에만 반영
```

재시작 후에도 유지하려면 `CONFIG REWRITE`가 필요하다.


## 수동으로 RDB 파일 생성

대표 명령어

```bash
SAVE
BGSAVE
```

둘 다 snapshot 생성 커맨드지만 동작 방식이 다르다.


### SAVE 커맨드

```bash
SAVE
```

동기 방식으로 파일 저장

```text
SAVE 실행
↓
Redis 블로킹
↓
RDB 생성 완료
```

파일 저장 완료 전까지 모든 클라이언트 요청이 차단된다.

운영 환경에서는 거의 사용하지 않는다.

#### 🤔 왜 SAVE는 위험할까?

메모리 전체를 스캔하고 파일 저장이 끝날 때까지 Redis가 멈추기 때문이다.

```text
SAVE 실행
↓
1분 저장 소요
↓
1분 동안 요청 차단
```


### BGSAVE 커맨드

```bash
BGSAVE
```

fork로 자식 프로세스를 생성한다.

```text
부모 프로세스
→ 요청 처리 계속

자식 프로세스
→ RDB 생성
```

실무에서는 대부분 BGSAVE를 사용한다.

#### 🤔 BGSAVE 중 다시 BGSAVE를 호출하면?

에러가 발생한다.

하지만 `SCHEDULE` 옵션을 사용할 수 있다.

```bash
BGSAVE SCHEDULE
```

현재 작업이 끝난 뒤 다시 BGSAVE를 수행한다.


## LASTSAVE 커맨드

마지막 snapshot 생성 시점을 확인할 수 있다.

```bash
LASTSAVE
```

유닉스 타임스탬프를 반환한다.

```text
마지막 RDB 저장 시각 확인 가능
```


## 복제를 사용할 경우 자동으로 RDB 파일 생성

복제 연결 시에도 RDB가 사용된다.

```text
Replica 연결
↓
Master가 RDB 생성
↓
Replica 전송
↓
메모리 복원
```

복제가 일정 시간 끊겼다가 다시 연결되는 경우에도 RDB 파일 전송이 발생할 수 있다.

#### 🤔 왜 초기 복제에서 RDB를 사용할까?

전체 데이터를 snapshot 하나로 빠르게 전송할 수 있기 때문이다.

```text
메모리 전체 상태
↓
압축된 snapshot 전달
```


# AOF 방식의 데이터 백업

AOF는 Redis에서 수행된 write 명령어를 append-only 방식으로 저장한다.

```text
SET user:1 kim
LPUSH queue job1
DEL key1
```

Redis 재시작 시에는 이 명령들을 replay해서 데이터를 복원한다.

```text
AOF replay
↓
메모리 복원
```

AOF 활성화

```bash
appendonly yes
```

파일 설정

```bash
appendfilename "appendonly.aof"
appenddirname "appendonlydir"
```

Redis 7부터는 AOF가 여러 파일로 분리되어 저장된다.

```text
base.rdb
+
incr.aof
+
manifest
```

#### 🤔 AOF는 모든 명령을 그대로 저장할까?

아니다.

메모리에 영향을 주는 명령만 저장한다.

```bash
DEL non_existing_key
```

같이 실제 데이터 변경이 없는 작업은 저장되지 않는다.

또 일부 명령은 내부적으로 다른 형태로 저장된다.

예시

```bash
BRPOP
→ AOF에서는 RPOP로 저장
```

```bash
INCRBYFLOAT
→ 결과값 기준 SET 형태 저장
```

아키텍처마다 부동소수점 처리 방식이 다를 수 있기 때문이다.


# AOF 파일을 재구성하는 방법

AOF는 append-only 방식이기 때문에 파일 크기가 계속 증가한다.

예를 들어

```bash
INCR counter
```

를 100번 실행하면

```text
메모리
→ 최종값만 존재

AOF
→ 100번 실행 내역 존재
```

그래서 주기적인 AOF 재구성(rewrite)이 필요하다.

```text
현재 메모리 상태 기준
↓
새로운 compact AOF 생성
```

중요한 점은 기존 AOF 파일을 압축하는 것이 아니라 현재 메모리 상태를 다시 저장하는 방식이라는 점이다.

#### 🤔 AOF 재구성 시 기존 AOF 파일을 읽을까?

아니다.

```text
기존 AOF 사용 X
현재 메모리 상태 사용 O
```


# Redis 7 이전 AOF 구조

7 이전에는 하나의 AOF 파일로 관리했다.

```text
RDB preamble(binary)
+
RESP command log
```

재구성 과정

```text
1. fork
2. 새 임시파일 생성
3. 기존 AOF + 인메모리 버퍼 이중 기록
4. 새 파일로 교체
```

단점

* 동일 로그 이중 저장
* binary + text 혼합 저장
* 관리 복잡


# Redis 7 이후 AOF 구조

Redis 7부터는 구조가 변경되었다.

```text
base.rdb
+
incr.aof
+
manifest
```

manifest 파일은 Redis가 현재 어떤 AOF 파일을 사용 중인지 관리한다.

```text
appendonly.aof.manifest
```

재구성 과정도 단순화되었다.

```text
1. fork
2. 새 base 파일 생성
3. 신규 incr.aof 기록
4. manifest 교체
5. 이전 파일 삭제
```

기존 구조보다 비효율이 줄어들었다.


# 자동 AOF 재구성

대표 설정

```bash
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

의미

```text
현재 AOF 크기가
기존 base 크기 대비 100% 증가
+
64MB 이상
↓
자동 rewrite
```

관련 값 확인

```bash
INFO Persistence
```

```text
aof_current_size
aof_base_size
```

#### 🤔 왜 min-size 옵션이 필요할까?

AOF 크기가 너무 작은 상황에서도 rewrite가 반복되면 비효율적이기 때문이다.

```text
1KB → 2KB
↓
100% 증가
↓
rewrite 반복
```


# 수동 AOF 재구성

명령어

```bash
BGREWRITEAOF
```

자동 rewrite와 동일하게 동작한다.

```text
fork
↓
새로운 compact AOF 생성
```


# AOF 타임스탬프

Redis 7부터 지원된다.

```bash
aof-timestamp-enabled yes
```

AOF 저장 시 타임스탬프도 함께 기록된다.

```text
#TS:1669532240
SET a b
```

이를 이용하면 특정 시점 복구(Point-In-Time Recovery)가 가능하다.

예를 들어 실수로 `FLUSHALL` 실행 시

```text
SET a b
↓
FLUSHALL
```

타임스탬프 기준으로 FLUSHALL 이전 시점까지 복구 가능하다.

복구 도구

```bash
redis-check-aof --truncate-to-timestamp
```

#### 🤔 타임스탬프 기능은 언제부터 지원될까?

Redis 7부터 지원된다.


# AOF 파일 복원

AOF 파일 손상 시 아래 도구 사용 가능

```bash
redis-check-aof
```

문제 발견 시

```bash
redis-check-aof --fix
```

손상된 마지막 부분을 잘라낸다.

#### 🤔 왜 AOF가 손상될 수 있을까?

append 도중 Redis가 비정상 종료될 수 있기 때문이다.

```text
append 중 장애
↓
마지막 command 일부만 저장
```


# AOF 파일의 안정성

Redis는 실제 디스크에 바로 저장하지 않는다.

```text
WRITE system call
↓
OS buffer 저장
↓
이후 디스크 flush
```

FSYNC는 버퍼 내용을 디스크에 강제로 기록하는 시스템 콜이다.

Redis는 APPENDFSYNC 옵션으로 이를 제어한다.


## APPENDFSYNC no

```text
WRITE만 수행
```

가장 빠르다.

최대 약 30초 데이터 유실 가능성 존재


## APPENDFSYNC always

```text
WRITE + FSYNC 매번 수행
```

가장 안전하지만 가장 느리다.


## APPENDFSYNC everysec

```text
WRITE 수행
+
1초마다 FSYNC
```

기본 설정

```text
성능
+
안정성
```

실무에서는 보통 `everysec` 사용을 권장한다.

#### 🤔 no 옵션이 위험한 이유는?

OS가 실제 디스크 flush를 최대 30초까지 미룰 수 있기 때문이다.

```text
서버 장애
↓
최대 30초 데이터 유실 가능
```


# 백업을 사용할 때 주의할 점

RDB/AOF 사용 시 `maxmemory`를 너무 크게 설정하지 않는 것을 권장한다.

BGSAVE, BGREWRITEAOF는 fork 기반으로 동작한다.

Redis는 Copy-On-Write(COW)를 사용한다.

```text
fork
↓
메모리 공유
↓
쓰기 발생 시 page 복사
```

최악의 경우 기존 메모리의 2배 가까운 메모리를 사용할 수 있다.

```text
메모리 부족
↓
OOM 발생
↓
Redis 프로세스 종료 가능
```

또한 아래 제약도 있다.

```text
RDB 저장 중
→ AOF rewrite 불가

AOF rewrite 중
→ BGSAVE 불가
```
